01 目标与三角色

构建一个 agent loop system，三个角色各有独立上下文 + 独立记忆，形成闭环：

角色职责硬约束 LOOPS XXVIII
Planner把模糊的人话翻成 sprint spec绝不碰代码
Generator写一切禁止给自己打分
Evaluator读 diff / 跑测试 / 试 app第一条消息就被告知"代码是坏的，你的任务是证明它"（adversarial）
LOOPS.md 的核心警告："绝不混角色。模型一旦自评就变谄媚，loop 悄悄收敛到 slop。" 这就是为什么要三个独立上下文——不是为了并行，而是为了防止自我评分的污染。

02 来源 A：LOOPS.md 的 agent loop 规则

来自 gist Karpathy-Michaels CLAUDE.md + LOOPS.md 的 TIER 6（规则 XXVII–XXXV）。主旨："大多数 agent 系统不是死在弱模型，而是死在弱 harness。模型能写代码、审代码、对着 rubric 验证自己的输出，但它自己无法决定何时停、何时重启、结果写哪——那是 loop 的活。"

规则一句话对你的设计的意义
XXVII Write the Loop, Not the Promptprompt 打一次，loop 自主跑。在反复打磨一条消息而非定义可重复流程 = 还在 prompting 时代loop = gather → reason → act → verify → repeat
XXVIII Separate the Roles三角色、三上下文、三 system prompt，绝不混你的三角色直觉被 endorse，且补 adversarial evaluator + 禁自评
XXIX Negotiate the Contract Firstgenerator 写第一行前，先把"完成"定义成可测断言 checklist。小 app ~27 条合理，10 条太少会 rubber-stampevaluator 评的是 contract，不是 vibe。这是把 run 从坏 demo 变可用产品的唯一改动
XXX Write to Disk, Not to Context"Context window 会撒谎：压缩、腐烂、用你没写的摘要藏起一小时前说的话。磁盘上的文件不撒谎。"独立记忆 = 磁盘文件，不是对话历史
XXXI Let the Loop Restart跑偏了全删重来，别打成考古。第 9 次删、第 11 次交付可用 = loop 正常状态全在磁盘，所以能删能重启
XXXII Score the Subjective品味可评分：定轴（design/originality/craft/functionality）、加权重、用参考样例标定（好 vs slop）主观任务可用 LLM evaluator，但必须 rubric + 参考样例
XXXIII Read the Traces把 agent 输出 pipe 到文件，grep 出判断偏离你的那一刻，针对那一瞬改 prompt调 agent = 读 transcript，不是再跑实验
XXXIV Delete the Harnessharness 是为弥补模型写的；模型变强后一半成负担，每版本重读并删脚手架别单调增长
XXXV The Bottleneck Always Movescoding→planning→verification→taste。永远不完成，只是暴露下一个瓶颈loop 的意义是让下一个瓶颈可见

XXX 落到具体文件：最少 4 个 LOOPS XXX

feature_list.json   # 在建什么
progress.md          # 做完 vs 待办
contract.md          # 可测的成功标准（evaluator 评这个）
log.md               # append-only，每条 `## [YYYY-MM-DD] op | title`
关键标准："模型应该能崩溃、丢掉 session，然后读三个文件就接上。如果你的状态没法用三个文件描述，那它太复杂了。"

03 来源 B：autoresearch/program.md 的实战实践

来自 karpathy/autoresearch——让 LLM 自主做 ML 研究实验的 loop 规约。要点：

实践含义
LOOP FOREVER，自主不停不问人，跑到被手动打断
第一个 run 永远是 baseline先建基准再改
evaluator = 客观指标（evaluate_bpb）代码判定胜负，不是 LLM
git 当状态机改进→保留 commit 推进分支；持平/变差→git reset 回退
结构化追加日志 = 记忆results.tsv（commit/bpb/内存/状态/描述），append-only，不入 git
上下文卫生输出重定向到 run.log，grep 只取指标
严格 in/out-of-scopeprepare.py 只读、train.py 可改、不加依赖
简单性准则相同收益下越简单越好；删代码拿等价结果 = 赢
超时即杀>10 分钟 kill，记 crash，回退
卡住不是重试，是升级策略读论文、重读 in-scope 文件、组合 near-miss、激进改架构

05 Agent SDK 是什么 + 三种实现路径

Anthropic 官方代码库（npm @anthropic-ai/claude-agent-sdk / pip claude-agent-sdk），让你用代码控制一个能调工具、自主循环的 agent，并完全控制它的循环和记忆。

对比是什么loop 在哪记忆
Claude Code CLI给人用的终端工具CLI 内部，看不见会话级
Anthropic API裸 API，发消息收回复自己从零写自己管 messages[]
Agent SDKAPI 上加"现成 agent 能力"（工具、循环、文件访问）SDK 管，你能介入能拿到每步状态
# 三角色各持独立 history = 独立记忆（档 B）
planner   = Agent(sys=PLANNER_PROMPT,   history=[])  # 禁碰代码
generator = Agent(sys=GENERATOR_PROMPT, history=[])  # 禁自评
evaluator = Agent(sys=EVAL_PROMPT,       history=[])  # 第一句: "代码是坏的,证明它"

while True:
    plan     = await planner.run(goal + read_state_files())   # 读磁盘
    code     = await generator.run(plan)
    verdict  = await evaluator.run(code, read("contract.md"))
    append("log.md", plan, code, verdict)              # append-only, 不靠 history
    if verdict.pass: break
    planner.history.append({"role": "user", "content": verdict.feedback})
不一定要 SDK：evaluator 能写客观代码、loop 简单时，裸 API + while + 文件记忆 就够。SDK 在"需要 agent 自主调一堆工具（读写文件、跑命令）"时才值得引入。

07 决策清单与下一步

问题选项影响
你的任务有客观指标吗？有 → 档 A 没有 → 档 B决定 evaluator 是代码还是 rubric-LLM，决定单 agent 还是三角色
SDK 语言TypeScript Python决定骨架写法
是否需要自主调工具是 → Agent SDK 否 → 裸 API + while是否引入 SDK

无论哪档，都要落实的 5 条

  1. Contract 先行：generator 写代码前，把"完成"写成可测断言 checklist（XXIX）。
  2. 记忆 = 磁盘文件：4 文件集，append-only log，状态能用 3 文件恢复（XXX）。
  3. evaluator 独立 + adversarial：三独立上下文，evaluator 第一句"代码是坏的，证明它"（XXVIII）。
  4. 输出进文件，grep 提取：原始输出不进 loop 的 context（autoresearch 上下文卫生）。
  5. loop forever + 重启权 + 升级策略：跑偏整删重来；卡住读论文/组合 near-miss（XXXI + autoresearch）。
