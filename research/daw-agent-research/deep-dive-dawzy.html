DAWZY：LLM Agent 如何控制 REAPER DAW

首个经同行评审的 DAW Agent 系统——用 LLM 代码生成替代 API 调用， 以 Human-in-the-Loop 范式实现自然语言驱动的音乐制作

📄 NeurIPS 2025 AI4Music Workshop 👥 Aaron Elkins et al. (SDSU) 📅 2025 年 12 月 🔓 开源 (GitHub)

1. 项目定位与核心问题

现代 DAW（如 REAPER）拥有 1000+ 个 Action、数百个 API 函数和极其复杂的菜单层级。 新用户需要花费大量时间学习界面和记忆操作路径，而非专注于创作本身。 DAWZY 要解决的核心问题是：

"DAWZY uses LLM-based code generation as a novel way to significantly reduce the time users spend familiarizing themselves with large DAWs."

— DAWZY 论文, NeurIPS 2025 AI4Music Workshop

与传统的 "文本到音乐" 生成工具（如 Suno、Udio）不同，DAWZY 不替代 DAW， 而是作为 DAW 内部的 AI 协作伙伴——保持 DAW 作为创作中心，通过自然语言降低操作复杂性。

2. 三层系统架构

DAWZY 的架构分为三个清晰的层次，数据从用户意图流向 DAW 执行：

👤
User Interaction
文本 / 语音 / 哼唱输入 → 捕获用户创作意图
⬇
🌐
Electron Gateway
Electron.js 应用 — 最小化 GUI，语音优先，实时解释操作
⬇
🧠
Processing Layer
GPT-based LLM + MCP Tools → 查询 REAPER 状态 → 生成 ReaScript (Lua)
⬇
⚡
Execution Layer
可逆 ReaScript 执行 → 精确、可审计的 DAW 操作
⬇
🎛️
REAPER DAW
轨道、效果器、MIDI、路由、自动化 — 完整会话状态

2.1 User Interaction Layer（用户交互层）

用户交互层是一个基于 Electron.js 的轻量级应用，设计为 voice-first（语音优先）工作流：

  • 文本输入："创建一个带压缩器和混响的人声轨道"
  • 语音输入：直接说出操作指令，通过 STT 转化为文本
  • 哼唱输入：哼唱旋律，系统将其转化为 MIDI 数据
  • 实时解释：Agent 用自然语言解释每一步操作，支持学习

Electron 作为 Gateway 的优势在于：跨平台（Windows/macOS/Linux）、 可以轻松集成 Web API（STT/TTS）、保持最小化界面不干扰 DAW 工作区。

2.2 Processing Layer（处理层）— 核心创新

这是 DAWZY 最核心的创新层。它负责将用户意图转化为可执行的 DAW 操作代码：

🔑 关键机制：State-Grounded Code Generation（状态感知的代码生成）

传统方法让 LLM 直接调用 API 时，模型容易 "幻觉" 出不存在的函数名或错误的参数索引。 DAWZY 的解决方案是让 LLM 生成完整的 Lua ReaScript 代码而非 API 调用序列， 并在生成前先查询 REAPER 的实时会话状态作为上下文。

处理层的工作流程：

  1. 状态查询：通过 MCP Tools 获取当前 REAPER 会话的完整状态—— 有多少轨道、每个轨道的名称/音量/效果器、当前选中的轨道、项目 BPM 等
  2. 上下文构建：将用户意图 + 会话状态 + ReaScript API 文档组装为 LLM prompt
  3. 代码生成：GPT-based LLM 生成一段完整的 Lua ReaScript 代码
  4. 代码验证：检查生成的代码是否语法正确、是否引用了有效的 API 函数
  5. 可逆性保证：在执行前设置 Undo Point，确保操作可以回退

2.3 Execution Layer（执行层）

生成的 ReaScript 代码通过 REAPER 的脚本引擎执行。REAPER 原生支持 Lua、Python 和 EEL2 三种脚本语言，DAWZY 选择 Lua 因为：

  • LLM 对 Lua 的代码生成质量普遍优于 EEL2（训练数据更多）
  • REAPER 的 Lua API 文档最为完善
  • Lua 代码可读性好，便于用户审计和理解
  • 执行速度快，满足实时交互需求

3.1 ReaScript 代码生成示例

当用户说 "把所有名字包含 'drum' 的轨道静音" 时，DAWZY 生成的代码：

-- DAWZY 生成的 ReaScript (Lua) -- 用户意图: 静音所有名字包含 "drum" 的轨道 local num_tracks = reaper.CountTracks(0) for i = 0, num_tracks - 1 do local track = reaper.GetTrack(0, i) local _, name = reaper.GetTrackName(track, "") if name:lower():find("drum") then reaper.SetMediaTrackInfo_Value(track, "B_MUTE", 1) reaper.ShowConsoleMsg("已静音: " .. name .. "\n") end end

这段代码展示了代码生成方式的优势：

  • 循环遍历：可以处理任意数量的轨道
  • 条件判断：字符串匹配实现精确筛选
  • 反馈信息：通过 Console 输出操作日志
  • 可审计：用户可以阅读并理解每一步操作

3.2 MCP Tools 的角色

DAWZY 使用 Model Context Protocol (MCP) 工具作为 LLM 和 REAPER 之间的桥梁：

  • 状态查询工具：获取轨道列表、效果器参数、路由配置等
  • 上下文注入：将查询结果作为 LLM prompt 的一部分，确保代码生成基于真实状态
  • API 文档工具：向 LLM 提供 ReaScript API 的函数签名和用法

MCP 工具的关键作用是将 LLM 锚定在真实的 DAW 状态上， 而不是让模型"猜测"当前的轨道数量、名称或配置。这是减少幻觉的核心机制。

4. 技术栈与实现细节

🌐 Electron.js Gateway

跨平台桌面应用框架，承载语音输入(STT)、语音输出(TTS)和用户交互界面

🧠 GPT-based LLM

核心推理引擎，负责理解意图、查询状态、生成 ReaScript 代码

🔌 MCP Tools

Model Context Protocol 工具集，提供 REAPER 状态查询和 API 文档访问

📝 ReaScript (Lua)

REAPER 原生脚本语言，LLM 生成的代码在此执行

🎵 MusicGen (可选)

独立的音乐生成模块 (musicgen-venv)，用于 AI 音乐内容生成

🎛️ REAPER DAW

目标 DAW，提供 1000+ Action 和完整的脚本 API

4.1 部署架构

DAWZY 使用两个独立的 Python 虚拟环境，反映了系统的模块化设计：

# DAWZY 部署结构 dawzy-venv/ # 核心 Agent 环境 ├── Electron Gateway # 语音/文本交互 ├── LLM Client # GPT API 调用 ├── MCP Tools # REAPER 状态查询 └── ReaScript Bridge # 代码注入与执行 musicgen-venv/ # 音乐生成环境 (独立依赖) ├── MusicGen Model # Meta MusicGen ├── Audio Processing # 音频后处理 └── MIDI Generation # MIDI 内容生成 REAPER/ # DAW 主机 ├── Lua Scripts/ # DAWZY 注入的脚本 └── Project Files/ # 用户音乐项目

分离两个环境的原因是 MusicGen 需要 PyTorch/CUDA 等重型依赖， 与核心 Agent 的轻量级依赖（API 客户端、MCP 库）冲突。

6. 核心设计原则

原则一：Human-in-the-Loop（人在回路）

DAWZY 不会自主执行操作——它生成操作建议，由用户确认后执行。 这确保创作者始终保持对作品的控制权，AI 是协作者而非替代者。

原则二：State-Grounded（状态锚定）

每次操作前都查询 REAPER 的实时状态。LLM 不是基于 "假设" 生成代码， 而是基于 "实际" 会话状态。这是减少幻觉的关键。

原则三：Reversible Edits（可逆编辑）

所有操作都设置 Undo Point。用户可以安全地实验， 不满意就 Ctrl+Z 回退。这大大降低了使用 AI 的心理门槛。

原则四：Explain-as-you-go（边做边解释）

Agent 用自然语言解释每一步操作，帮助用户理解 DAW 的工作原理。 这使 DAWZY 不仅是工具，也是一个学习助手。

原则五：DAW as Creative Hub（DAW 为创作中心）

DAWZY 不试图替代 DAW 的界面——它保持 DAW 作为创作中心， 自己只是一个最小化的辅助窗口。所有的视觉编辑、精细调节仍在 DAW 中完成。

8. 局限性与未来方向

论文诚实地讨论了 DAWZY 的局限性：

⚠️ 复杂编排能力有限

涉及多步骤、跨轨道的复杂编排操作成功率较低。需要更精细的 prompt 工程和更好的上下文管理。

⚠️ 依赖 GPT API

目前依赖商业 GPT API，成本和网络延迟是问题。论文计划 fine-tune 开源 LLM (如 Qwen3) 替代。

⚠️ 仅支持 REAPER

架构设计是 REAPER 特定的。虽然原理可以迁移到其他 DAW，但需要针对不同脚本 API 重新实现。

⚠️ 音频理解缺失

DAWZY 无法 "听" 音频内容——它只理解 DAW 的结构状态（轨道、效果器），不理解音频信号本身。

论文提到的后续研究方向

  • Fine-tune 开源 LLM：在 ReaScript 语料上微调 Qwen3 等开源模型，减少对 GPT API 的依赖
  • 音频分析集成：将音频分析工具作为 MCP Tool，让 Agent 能够 "听" 并基于音频特征做决策
  • 多 DAW 支持：将架构适配到 Ableton Live、Logic Pro 等其他 DAW
  • 更大规模用户研究：扩大评估范围，收集更多用户反馈

9. 为什么 DAWZY 是最具代表性的项目？

🔑 五大代表性因素

  1. 唯一经过同行评审的 DAW Agent：NeurIPS 2025 Workshop 接收，学术可信度最高
  2. 提出了 "代码生成优于 API 调用" 的关键洞察：这一发现对整个 DAW Agent 领域具有指导意义
  3. 完整实现了三层架构：从语音输入到代码生成到 DAW 执行的全链路，而非概念原型
  4. 开源且有用户评估：代码公开可复现，且有正式的用户研究验证
  5. 集成了所有关键技术：LLM + MCP + Electron + ReaScript + 语音/哼唱输入，是技术栈最全面的项目

DAWZY 对 DAW Agent 化领域的启示

  • State-grounded code generation 是最佳实践——让 LLM 先 "看到" 真实状态，再生成代码
  • 可逆性是用户信任的基础——Undo 功能看似简单，实则是 Human-in-the-Loop 的关键
  • MCP 作为中间层的价值——标准化的工具接口使 LLM 能够可靠地获取 DAW 状态
  • 语音优先的交互设计——音乐制作场景中，手通常在乐器/控制器上，语音是最自然的交互方式
  • DAW 不应被替代——Agent 应该增强 DAW，而非取代它
