语音、哼唱与生成素材

DAWZY 最有意思的地方不只是“文本控制 DAW”，而是把语音、哼唱转 MIDI、AI beat 生成都接进同一个 REAPER 工作流。这一页拆这些多模态入口的代码路径。

首页DAWZY 深挖入口与动态脚本MCP 与 ReaPyHarness来源
本页重点：多模态不是额外按钮，而是 DAW agent 的核心输入设计。音乐人经常通过说、唱、哼、拖音频、弹 MIDI 来表达意图；DAWZY 把其中三类接到了同一条 agent 执行链里。

Renderer 分流

前端 UI 将三种输入合流到聊天界面：文本直接发给 agent；语音先录音、转录，再当文本消息发送；哼唱则绕过 LLM，对音频做 pitch tracking，直接导入 MIDI。

`src/renderer.js`：文本与哼唱的分支
async function sendMessage(message, whistle=false) {
  addMessageToChat('user', message, true, null, whistle);

  let response;
  if (whistle) {
    await window.electronAPI.humToMIDI();
    response = 'Added MIDI track to REAPER';
  } else {
    response = await window.electronAPI.takeScreenshot(message);
  }

  addMessageToChat('assistant', response, true, messageIndex, whistle);
}

这个设计有一个微妙点：哼唱路径并不让 LLM 解释旋律或决定编曲，而是直接生成 MIDI 并导入。它很务实，但如果要更 agent 化，可以在导入后让 LLM 读取新 track 状态，解释旋律、建议和声或自动配器。

语音转文本

语音路径由 JS 启动 Python 录音脚本，生成 `input.wav`，停止后用 OpenAI Whisper API 转录。转录结果会回到 `sendMessage()`，因此后续完全复用文本 agent 链路。

`src/speech_to_text.js`：启动/停止录音
export function startRecording(filename = 'input.wav') {
  const recordScript = path.join(process.cwd(), 'ReaPy_Utils', 'record.py');
  recordingProcess = spawn('python', [recordScript, filename]);
}

export function stopRecording() {
  if (!recordingProcess) return;
  recordingProcess.kill('SIGINT');
  recordingProcess = null;
}
`src/speech_to_text.js`：Whisper 转写
export async function transcribeFromMic(filename = 'input.wav') {
  const transcription = await openai.audio.transcriptions.create({
    file: fs.createReadStream(filename),
    model: 'whisper-1'
  });
  return transcription.text.trim();
}

语音路径的工程风险主要是文件名固定、录音进程异常、转录延迟和隐私。产品化应让用户看到录音状态、转录文本可编辑，并避免把所有录音写到同一个 `input.wav`。

哼唱转 MIDI

哼唱路径使用 Spotify 的 Basic Pitch CLI，把 `input.wav` 转成 `input_basic_pitch.mid`，然后调用 `add_media.py` 导入 REAPER。这里没有 LLM 参与，属于确定性音频工具链。

`src/hum_to_midi.js`：Basic Pitch 转 MIDI 并导入
export async function humToMIDI() {
  const inputMidiPath = path.join(projectRoot, 'input_basic_pitch.mid');
  const inputWavPath = path.join(projectRoot, 'input.wav');
  const addMediaScript = path.join(projectRoot, 'ReaPy_Utils', 'add_media.py');

  await exec(`rm -f "${inputMidiPath}"`);
  await exec(`basic-pitch "${projectRoot}" "${inputWavPath}"`);
  await exec(`python "${addMediaScript}" "${inputMidiPath}" true`);
}
`ReaPy_Utils/add_media.py`：MIDI 导入后的音色补丁
if len(sys.argv) > 2 and sys.argv[2] == 'true':
    project = reapy.Project()
    track = project.tracks[track_index]
    params = track.fxs[0].params
    params[5] = 1.41421356237

`true` 参数会给导入的 MIDI track 设置 ReaSynth 相关参数。这里的 magic number 需要注释和抽象：至少应标明它对应哪个插件参数、为什么设到这个值、是否跟 REAPER/插件版本有关。

Beat 生成

Beat 生成路径由 MCP 工具触发，而不是 UI 直接触发。模型可以在理解用户描述后调用 `generate_beat(description)`，生成 wav 并导入工程。

`ReaPy_Utils/beat_generation.py`：MusicGen-small 生成音频
import audiocraft
import sys

model = audiocraft.models.MusicGen.get_pretrained(
    'facebook/musicgen-small'
)
descriptions = sys.argv[1:]
wav_tensor = model.generate(descriptions, progress=True)

audiocraft.data.audio.audio_write(
    'beat', wav_tensor[0].cpu(), model.sample_rate
)

这个实现很短，但说明 DAW agent 的素材生成可以被工具化：LLM 不直接生成音频，而是选择一个生成器、传入描述、等待 artifact，再把 artifact 导入宿主。

可测试点

链路当前实现建议测试
语音转文本`input.wav` -> Whisper -> 文本消息。录音中断、空音频、噪声、转录错误可编辑、API 失败降级。
哼唱转 MIDI`basic-pitch` 输出固定 MIDI 文件，再导入 REAPER。无音高输入、多音输入、节奏偏移、输出 MIDI 是否包含 note、导入后 track 数变化。
Beat 生成MusicGen-small 生成固定 `beat.wav`。生成超时、采样率/长度检查、重复 job 覆盖、导入后媒体 item 校验。
Agent 合流文本/语音走 LLM，哼唱走确定性链路。每条链路都应写 action log，并在 REAPER 状态变化后刷新快照。
一个可迁移的产品点：哼唱导入后，可以自动追加一个 agent turn：“我已经把你的哼唱放到新 MIDI track。要不要我根据它生成和弦、低音线或鼓组？”这样多模态输入就不是孤立工具，而会自然进入创作对话。
