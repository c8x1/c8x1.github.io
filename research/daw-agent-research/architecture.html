架构模式

DAW agent 的稳定性取决于「模型决策」和「宿主 mutation」之间的契约。MCP 是入口，不是安全边界本身。

推荐分层

LLM / Agent Client
  └─ Planner: user intent → typed edit plan
      └─ MCP Protocol Layer: session + capability negotiation
          └─ MCP Server Features: tools/resources/prompts
              └─ Deployment Auth: OAuth/resource indicators for HTTP where applicable
                  └─ Domain Guard: permissions, dry-run, schema validation, rollback strategy
                      └─ DAW Bridge: ReaScript / Ableton SDK / OSC / MIDI / AppleScript / sockets
                          └─ Host DAW: project/session state + audio engine

桥接模式比较

模式代表优势限制适合任务
REAPER ReaScript/APILua、Python、EEL scripts；REAPER actions/API functions较适合完整工程自动化、状态快照、RPP diff 与 undo block。脚本本身不是数据库式 atomic transaction；错误恢复需要 harness 设计。project inspection、track/item/MIDI/FX automation、render。
Controller scriptingFL Studio Python MIDI scripting、Bitwig controller extensions适合控制器、即时互动、transport/mixer/clip 触发。官方定位偏 controller integration；完整 project/session editing 能力需按具体 API 与项目验证。live control、hardware mapping、低风险 macro。
Ableton Remote Script/LOMableton-mcp 类仓库、ableton-js社区已有大量实现；可控制 Live Object Model。非官方路径容易受版本影响；headless 测试困难。Live session 操作、Arrangement View、device params。
AbletonOSCideoforms/AbletonOSC、AbletonOSC MCP wrappersOSC 成熟，适合即时控制、跨语言。OSC message grammar 和状态同步需要自建；错误型别较弱。transport、track/mixer、live performance control。
Ableton Extensions SDKloophole、ableton-sdk-mcp、extension skills官方 public beta，JS/TS，能直接和 Live Set 结构互动。截至 2026-06 仍是 Live 12.4.5 Suite beta；安全与发布模式还在演化。workflow automation、project analysis、structured edit tools。
MIDI/Control SurfaceLogic/Cubase/Mackie Control、MIDI MCP宿主兼容面广，对即时 performance 友好。语义层弱，不容易知道「第 3 个旋钮其实是什么」。硬体/混音台式控制、低风险宏。
文件/交换格式DAWProject、.als/.flp parser、openDAW最利于 diff、versioning、offline harness。专有格式逆向风险；离线变更不代表 DAW 会接受。模板生成、跨 DAW exchange、静态分析。

MCP 授权与信任边界

截至 2026-06-15，MCP latest specification 指向 2025-11-25。对 DAW agent 而言，安全设计应按「MCP 只是协议入口，DAW mutation 另需领域安全层」处理。

风险要求DAW agent 含义
Token passthroughMCP server 不应接受未明确签发给自身的 token；HTTP 部署需验证 audience/resource。不能把 DAW MCP server 当作通用 API proxy；插件下载、云素材、Git、文件系统需分 scope。
SSRF / session hijack对远端资源、回调 URL、session id 和重连做校验。防止模型透过素材 URL、project path 或 plugin installer 间接读私有数据或劫持 session。
Local MCP compromise本地 server 最小权限、明确 allowlist、隔离工作目录。DAW project、音频 stems、插件扫描目录、shell/render 命令要分开授权。
Per-client consentclient/使用者需看到工具风险、资源范围与操作预览。删 track、覆盖 audio、安装插件、上传 stems 应进 human gate。

Project File / Versioning

项目文件可以成为 agent 状态源，但不能简化成「直接改档」。REAPER `.rpp` 较适合文本 diff；Ableton Live Set / export 工具与解析器可做静态分析；DAWProject 可做跨宿主交换，但要评估插件、automation、media path、preset 和 routing 的信息损失。

格式/来源适用注意事项
REAPER RPPfixture diff、版本控制、离线检查。仍需让 REAPER 加载验证；外部媒体和插件 state 需一起封存。
Ableton Live Set / Export静态分析、版本回归、clip/warp metadata 抽取。版本兼容与专有内容要保守处理；不要把 parser 成功等同于 DAW 可接受。
DAWProject跨宿主 fixture、交换格式、agent 中间表示。跨宿主转换可能丢失插件、automation 或宿主特有 routing。
openDAW / Web DAWheadless 或 browser-native harness 参考。可控性高，但不代表商用 DAW session 的全部约束。

工具设计原则

少即是多

不要把宿主 API 一比一暴露给 LLM。先暴露音乐语义工具，例如 create_drum_pattern、apply_reference_mix_profile、insert_arrangement_markers，再由 guard 层翻译成低阶 API。

读写分离

读取资源可宽，写入工具要窄。所有 mutation 工具都应支持 dry-run、preview diff、undo token、affected object list。

显式状态

每次写入前都 refresh state；工具输出必须包含 pre-state hash、post-state hash、warnings、render/eval hooks。

人类审批

大范围 destructive 操作、插件新增、音频覆盖、外部下载、运行任意脚本都应有人类 gate。
