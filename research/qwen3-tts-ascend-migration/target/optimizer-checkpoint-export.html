学习轨道：入门教程

优化器、Checkpoint 与导出

在 SFT 循环里把镜头拉近到优化器、反向步进与每轮 checkpoint 导出三段。AdamW 为什么只优化 qwen3tts.model、clip 1.0 的全局范数裁剪、每轮末 copytree+改 config+unwrap+删 speaker_encoder+注入 codec_embedding[3000]+写 safetensors 的五步流程——把 speaker 向量注入表索引这一“注册”机制讲透，并逐段标出昇腾 910B 的迁移风险（保存已 .to(cpu) 无风险、多卡 unwrap/DDP、speaker 注入是纯索引赋值与硬件无关）。

学习目标

  • 说清 AdamW 只优化 qwen3tts.model.parameters() 的范围含义，以及为何 speaker_encoder 的梯度被 .detach() 切断、保存时整段删除
  • 按 L115-121 顺序描述反向累积→clip_grad_norm_(1.0)→step→zero_grad 四步，并区分 sync_gradients 同步步与累积步
  • 逐行列出每轮保存五步（copytree / 改 config / unwrap+state_dict.to(cpu) / 删 speaker_encoder / 注入 codec_embedding[3000] / save_file），说出每步入口行号
  • 解释 speaker 向量注入 codec_embedding.weight[3000] 的“注册”机制：推理时 spk_id→3000 表索引即取该说话人，与硬件无关

前置知识

  • 读过 SFT 训练循环总览页（setup/collate/前向/反向/保存五段划分）
  • 理解 PyTorch optimizer.step/zero_grad 与梯度累积
  • 知道 accelerate 的 prepare/unwrap_model/accumulate/sync_gradients 约定

总览：从优化器到导出的三段

总览页把 SFT 循环切成五段，本页把其中三段——优化器构造、反向步进、每轮 checkpoint 导出——展开到行级。这三段都集中在 finetuning/sft_12hz.py 的 L60 与 L115-121、L126-158。优化器只在一处构造（L60），反向每 step 跑一次（L115-121），保存每轮末只在 main process 跑一次（L126-158）。它们之间靠 accelerate 的 ownership 约定串起来：prepare 返回的 model/optimizer 是后续唯一句柄，unwrap_model 在保存时才把 DDP 外壳剥掉取原始权重。下图是三段的输入输出与边界。

两个张量/对象贯穿这三段：target_speaker_embedding（L83-84 首步缓存、L156 注入到表索引 3000）是音色定制的载体；talker.model.codec_embedding.weight（L155 取出、L156 行赋值）是说话人注册的落点。target_speaker_embedding 是模块级全局（L30 声明、L32 global），跨 step/epoch 持续持有首步 speaker 向量——这是保存时能拿到它的前提。

优化器：AdamW 与只优化 .model 的范围

L60 是整条循环唯一的优化器构造点：AdamW(qwen3tts.model.parameters(), lr=args.lr, weight_decay=0.01)。args.lr 默认 2e-5（L39 parser.add_argument("--lr", default=2e-5)），weight_decay=0.01 是硬编码值。关键在第一个参数：qwen3tts.model.parameters() 只收集 qwen3tts.model 这棵子树下的参数，而不是整个 Qwen3TTSModel 实例的全部属性。

    model, optimizer, train_dataloader = accelerator.prepare(

qwen3tts 是 Qwen3TTSModel 的实例，它顶层有 model（含 talker/code_predictor 及其 codec_embedding/text_embedding 等子模块）和 speaker_encoder 等并列属性。把 .model.parameters() 传给 AdamW，意味着 talker 与 code_predictor 的全部权重都会被优化——包括第 0 层 codec 的 codec_embedding 与 1-15 层残差的 15 张独立嵌入表。而 speaker_encoder 的参数不在 .model 下、不在优化器里，本来就收不到梯度。

更彻底的是 L82 speaker_embedding = model.speaker_encoder(...).detach() 的 .detach()：即便 speaker_encoder 的权重在优化器里（实际不在），.detach() 也把从 ref_mels 到 speaker 向量的整条计算图切断，speaker 向量作为常量参与前向。所以 speaker_encoder 在训练里只读不练，保存时还会被整段删除（L150-153）。这与“优化 .model”是两道保险：参数范围不包含 + 计算图被 detach。

范围设计的后果： 被更新的只有 talker/code_predictor 的权重，不包含 speaker_encoder。音色定制不是靠训练 speaker_encoder 的新权重，而是靠把首步 speaker 向量（L83-84 只在 target_speaker_embedding is None 时赋值一次）固化到 codec_embedding.weight[3000]。理解这一点是理解保存段“注册”机制的前提。
NPU 迁移 · 优化器构造无 NPU 算子AdamW(...) 只是 CPU 上建优化器对象、注册参数引用，不含任何 NPU kernel。风险点不在这行本身，而在它随后被 accelerator.prepare 包裹（L62-64）：prepare 会把 optimizer 的参数组绑到 accelerate 的状态机（梯度累积、bf16、设备路由）。910B 上要确保 accelerate 已配置 torch_npu 后端，否则 step() 时 optimizer 会在错误的 device 上下文里更新权重。

逐行往下看。L115 accelerator.backward(loss)：accelerate 按当前是否同步步决定梯度 scale/unscale 与是否真反传；bf16 下不需要 GradScaler（原生支持）。L117 if accelerator.sync_gradients: 只在累积到第 4 步、需要真正步进时为真——这是累积步与同步步的分水岭。

L118 accelerator.clip_grad_norm_(model.parameters(), 1.0) 对 model.parameters() 全局范数裁剪到 1.0。注意这里是 model（prepare 返回值，可能是 DDP 包装），不是 qwen3tts.model——但 prepare 包裹的是同一个 qwen3tts.model，所以裁剪范围与优化器范围一致。L120 optimizer.step() 更新权重，L121 optimizer.zero_grad() 清零梯度，为下个 micro-step 准备。

NPU 迁移 · clip_grad_norm 全局 reduceclip_grad_norm_ 要先算所有参数梯度的平方和再开方——这是一次跨所有参数的 reduce。单卡上纯 elementwise+reduce，torch_npu 原生支持，风险低。多卡时 DDP 下这次 reduce 会和梯度 allreduce 一起在 HCCL 上走，后端必须是 hccl 而非 nccl。裁剪阈值 1.0 是固定配方，迁移时保持原值，先复现再调。
NPU 迁移 · backward 的 DDP/HCCL 同步多卡时 prepare 把 model 包成 DDP，accelerator.backward(loss) 触发梯度 allreduce，后端必须 HCCL。这是单机多卡段的硬前置：要么 accelerate config 选 torch_npu 后端 + hccl，要么迁到 MindSpeed 训练引擎不走 accelerate。单卡迁移先验证 accelerate+torch_npu 的 device 路由，多卡再碰 HCCL 拓扑与 launch 配置。此风险与总览页反向段一致，本页不重复展开。

后四步处理权重本身。L147-148 unwrapped_model = accelerator.unwrap_model(model) 把 DDP/accelerate 外壳剥掉拿原始 model，state_dict = {k: v.detach().to("cpu") for k, v in unwrapped_model.state_dict().items()} 全量逐键 .detach().to("cpu")——这一步把所有权重搬到 CPU，是保存段无 NPU 风险的关键。理解这点：state_dict 里每个张量已是独立 CPU 拷贝，与原模型权重脱钩。

L150-153 删除所有 speaker_encoder 前缀的键：drop_prefix = "speaker_encoder"、keys_to_drop = [k for k in state_dict if k.startswith(drop_prefix)]、循环 del state_dict[k]。推理时不再需要 speaker_encoder（音色已固化进 codec_embedding 表），所以从 checkpoint 里彻底剔除，减小体积、避免误用。这与 L82 的 .detach()、L60 的优化范围形成完整闭环：训练不练、计算图切断、导出删除。

L155-156 是音色定制的落点：weight = state_dict['talker.model.codec_embedding.weight'] 取出这张嵌入表（CPU 张量），state_dict['talker.model.codec_embedding.weight'][3000] = target_speaker_embedding[0].detach().to(weight.device).to(weight.dtype) 把首步缓存的 speaker 向量写到表的第 3000 行。注意 target_speaker_embedding[0] 取的是首步缓存的第 0 个样本向量，.to(weight.device) 在这里 weight 已是 CPU，所以是 CPU 上的行赋值。L157-158 save_file(state_dict, model.safetensors) 用 safetensors 落盘。

speaker 的“注册”机制： 训练得到的不是新 speaker_encoder 权重，而是把首步 speaker 向量直接写到 codec_embedding 表的第 3000 行。推理时 spk_id（config 里 spk_id[speaker_name]=3000）映射到 3000 这个表索引即取该说话人向量。这条逻辑是纯索引赋值，与硬件无关——NPU 上只需保证 .to(weight.device) 的 device 一致（保存段 weight 已在 CPU）。

深读：speaker 向量如何变成表里的一行

把贯穿三段的 speaker 数据流收束一下。ref 音频的 mel 过 speaker_encoder 得到一个 speaker 向量，首步（L83-84 if target_speaker_embedding is None）缓存进模块级全局；训练全程它作为常量参与前向（L91 写入 codec 第 6 位）；每轮末被注入到 codec_embedding.weight[3000]（L155-156）。推理时不再有 speaker_encoder，而是直接按 spk_id=3000 查这张表。下图是这条数据流的完整生命周期。

这个设计的精妙在于：音色定制不需要重训 speaker_encoder。speaker_encoder 在 Base 模型里已预训练好（能从任意 ref mel 提取稳定 speaker 向量），SFT 只是用它提取一次目标说话人向量，然后把向量固化进 codec_embedding 表的一个保留索引（3000）。新音色 = 表里新增一行，而不是新模型。这也是为什么保存段要删 speaker_encoder 权重——推理管线用不到它，表索引 3000 已足够。
