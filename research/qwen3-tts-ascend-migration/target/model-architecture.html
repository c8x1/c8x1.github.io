学习轨道：入门教程

复合模型架构与生成流

按 modeling_qwen3_tts.py 的真实类层次逐段拆解 Qwen3-TTS 的复合结构：顶层 Qwen3TTSForConditionalGeneration = talker + speaker_encoder（可选）+ speech_tokenizer；talker 又内嵌 code_predictor 与 codec/text 双 embedding 表。讲清 talker 主前向、16 层残差 codec 的 embedding 汇总、forward_sub_talker_finetune 的 MTP 损失路径，以及推理时 sub-talker 逐层补全 codec 的生成流——每段标出昇腾 910B 迁移的算子风险点。

学习目标

  • 在 modeling_qwen3_tts.py 里定位顶层复合类、talker、code_predictor、speaker_encoder 四个构造点并说出它们的组合关系
  • 说清 talker 的 codec_embedding 与 text_embedding 双表、text_projection 投影对齐维度后相加的 embedding 汇总机制
  • 区分主 talker 的 codec_head CE 与 sub-talker(code_predictor) 的多组 lm_head MTP 损失，说明 forward_sub_talker_finetune 如何把 talker 隐状态喂给 sub-talker
  • 解释推理生成流里 codec_hiddens 16 路求和(1,keepdim) 再拼 trailing_text_hidden 的逐 token 组装，与训练前向的差别

前置知识

  • 读过 Qwen3-TTS 的 SFT 训练循环与 16 层 codec 布局
  • 熟悉 HF transformers 的 PreTrainedModel/GenerationMixin 组合
  • 理解 GQA 注意力与 RoPE

总览：复合模型的四层组合

Qwen3-TTS 不是单一 transformer，而是一个嵌套复合模型。最外层 Qwen3TTSForConditionalGeneration（modeling_qwen3_tts.py:1813）持有 self.talker、可选的 self.speaker_encoder、以及推理时装载的 self.speech_tokenizer。talker 本身又是 Qwen3TTSTalkerForConditionalGeneration（:1564），它内部再嵌一个 self.model（真正的解码器骨架 Qwen3TTSTalkerModel）、一个 self.codec_head（主 token 头）、一个 self.code_predictor（sub-talker / MTP 模块）。四层嵌套是这个模型理解迁移的关键：改 attention 实现会同时穿透 talker 与 code_predictor 的所有层。

读这张图要注意两条轴：组合轴（谁持有谁）和双轴 embedding。talker 的解码器骨架 Qwen3TTSTalkerModel 同时持两张嵌入表——codec_embedding (vocab_size=3072, hidden_size=1024) 给 codec token，text_embedding (text_vocab_size, text_hidden_size=2048) 给文本 token；两者维度不同（1024 vs 2048），所以中间必须过一个 text_projection（Qwen3TTSTalkerResizeMLP，2048→2048→1024）把文本侧拉到 codec 侧的 1024 维再相加。code_predictor 自己另有一组 15 张嵌入表（num_code_groups-1=15，见 :1030-1032）与 15 个 lm_head（:1167-1169），对应第 1-15 残差层——这与 SFT 页讲的 16 层 codec 完全对齐：第 0 层走 talker 的 codec_embedding + codec_head，第 1-15 层走 code_predictor。

配置契约：talker_config 与 codec 字段

模型结构由 configuration_qwen3_tts.py 的三层 config 契约决定。最外层 Qwen3TTSConfig（:454）持有 talker_config 与 speaker_encoder_config 两个子配置（sub_configs 声明，:460-463）。Qwen3TTSTalkerConfig（:259）又把 code_predictor_config 声明为自己的子配置（sub_configs = {"code_predictor_config": Qwen3TTSTalkerCodePredictorConfig}，:368）。这套嵌套的 config 树是from_pretrained 能从一份 config.json 还原整棵模型树的依据。

Talker 主前向：双 embedding 汇总与 codec_head

Qwen3TTSTalkerModel.forward（:1457）是 talker 解码器骨架的标准 transformer 前向，但它有一个多模态 RoPE 位置的细节值得精读。L1500 position_ids = cache_position.view(1,1,-1).expand(3, batch, -1) 把位置 id 扩成 3 维（temporal/height/width），L1504-1508 再从 4 维（含 text 一维）里拆出 text_position_ids。这是为 vision/text 混合序列准备的 MRoPE——纯文本时三轴相同，但代码仍走这条路径。迁移到 NPU 时 MRoPE 的 cumsum/expand 算子无特殊风险，但要注意 get_rope_index（:1746）里的 attention_mask.cumsum(-1) 是 CPU 上算的位置规划，不能与 GPU/NPU 的 attention mask 混用 dtype。

NPU 迁移 · create_causal_mask 在 NPUcreate_causal_mask / create_sliding_window_causal_mask（来自 transformers.masking_utils，:32-33）会在 NPU 上生成大 mask 张量。长序列时这是个 (b, 1, t, t) 的布尔/浮点张量，显存压力与 FA2 的无显存注意力正相反。若迁到 sdpa，CANN 的 SDPA 路径仍接受 4D mask，但建议小样本先验证 mask 的 dtype（fp32/bf16）与 torch_npu 是否对齐；sliding_window 路径（code_predictor）更要注意，swa 的窗口 mask 生成逻辑在 NPU 上未必有融合算子。
NPU 迁移 · MRoPE 位置规划get_rope_index（:1746-1800）用 attention_mask.float().cumsum(-1) + masked_fill_ + expand(3,-1,-1) 在 CPU/NPU 上规划 3D 位置。cumsum 与 masked_fill_ 都是 elementwise，NPU 原生支持，无算子风险；但要确认 attention_mask 的 device 与模型 device 一致，否则 .to(attention_mask.device)（:1796）会触发跨设备拷贝。纯文本路径下三轴相同，等价 1D RoPE，可作数值对齐基线。

forward_sub_talker_finetune：MTP 残差损失路径

SFT 训练里损失 = 主 talker CE + 0.3×sub-talker MTP。sub-talker 的入口是 forward_sub_talker_finetune（:1612），它把 talker 最后一层隐状态（已被 codec_mask 拉平成一维，只剩真 codec 位置）当作 sub-talker 的输入序列首位的 embedding，再拼上 16 层 codec 各自的查表 embedding，喂给 code_predictor 算多组残差 logits。

    def forward_sub_talker_finetune(self, codec_ids, talker_hidden_states):
        assert len(codec_ids.shape) == 2
        assert len(talker_hidden_states.shape) == 2
        assert codec_ids.shape[0] == talker_hidden_states.shape[0]
        assert talker_hidden_states.shape[1] == self.config.hidden_size
        assert codec_ids.shape[1] == self.config.num_code_groups

        sub_talker_inputs_embeds = [talker_hidden_states.unsqueeze(1)]

        for i in range(self.config.num_code_groups - 1):
            if i == 0:
                sub_talker_inputs_embeds.append(self.get_input_embeddings()(codec_ids[:, :1]))
            else:
                sub_talker_inputs_embeds.append(self.code_predictor.get_input_embeddings()[i-1](codec_ids[:, i:i+1]))
        sub_talker_inputs_embeds = torch.cat(sub_talker_inputs_embeds, dim=1)
        
        sub_talker_outputs = self.code_predictor.forward_finetune(inputs_embeds=sub_talker_inputs_embeds,
                                                                 labels=codec_ids[:, 1:])
        
        sub_talker_logits = sub_talker_outputs.logits
        sub_talker_loss = sub_talker_outputs.loss
        return sub_talker_logits, sub_talker_loss

NPU 迁移 · sub-talker 自定义 forwardforward_sub_talker_finetune 是自定义 forward，不走 GenerationMixin，所以 attention 实现不会经 generate 路径派发，而是直接由 code_predictor 内部 Qwen3TTSAttention 的 ALL_ATTENTION_FUNCTIONS 派发（:942）。这意味着改 sdpa 时这一段也会命中——但它没有 KV cache（forward_finetune 不传 past_key_values 的增量用法），纯前向，NPU 上风险集中在 code_predictor 的 5 层注意力与 sliding_window mask。建议迁移后单独对 forward_sub_talker_finetune 做一次小样本数值 diff（主 loss + 0.3×sub loss vs FA2 基线）。
0.3 权重的物理意义： 主 talker 学第 0 层（粗粒度码本）的 next-token，sub-talker 在 talker 隐状态上补 1-15 残差层。把 talker 隐状态塞进 sub-talker 输入第 0 位，相当于让残差预测“继承”主模型已抽取的上下文表示——这与 DeepSeek 的 MTP（multi-token prediction）思路同源。迁移时这条隐状态传递链不能断：若 talker 与 code_predictor 装载到不同 device，sub_talker_inputs_embeds 会跨设备拼张量报错。
