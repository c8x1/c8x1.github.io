25Hz Tokenizer 静态依赖图

按固定 SHA 022e286b 的源码，逐段拆 25Hz speech tokenizer 的静态依赖：Whisper 风格语义编码器（conv1/conv2 → 32 层 ResidualAttentionBlock → avg_pool → proj）在中途插入 GRVQ 量化，与 12Hz 的 V2 编码器（自研 transformer + 16 层 RVQ、codebook 2048）形成帧率/版本分野。画出依赖图、跟踪 mel→aftercnn→token 的形状演化，并标出 STFT/attention/argmax 等算子在昇腾 910B 上的迁移方向。

学习目标

  • 沿 import 与继承链说出 25Hz tokenizer 的静态依赖：modeling → speech_vq(WhisperEncoderVQ) → whisper_encoder(WhisperEncoder) + core_vq(GRVQ)
  • 区分 25Hz(V1) 与 12Hz(V2) 两套 tokenizer：编码器主干、量化器类型、codebook 大小、num_quantizers 的差异
  • 跟踪 mel (n_mels,T_mel) → aftercnn (T_cnn) → token (T_tok) 的形状演化，说出 conv2 stride2 与 avg_pool 各砍半一次的作用
  • 指出 whisper STFT、flash_attn_varlen、EuclideanCodebook 的 argmax/EMA 更新这几处算子在 NPU 上的保守迁移方向

前置知识

  • 熟悉 PyTorch 的 Conv1d / nn.Embedding / F.stft / F.one_hot
  • 了解向量量化（VQ/RVQ）的基本概念：码本、最近邻、残差
  • 读过 Qwen3-TTS 包结构，知道 tokenizer 是 talker 的上游离散化器

总览：25Hz tokenizer 的静态依赖

25Hz tokenizer 的全部可运行代码集中在 qwen_tts/core/tokenizer_25hz/ 一个目录里，依赖关系是线性单向的：顶层 modeling_qwen3_tts_tokenizer_v1.py 聚合 encoder 与 decoder 两个子模型，encoder 的实际主干是 vq/speech_vq.py 里的 WhisperEncoderVQ，它继承自 vq/whisper_encoder.py 的 WhisperEncoder 并在中途挂接 vq/core_vq.py 的 GRVQ 量化器。下图把这条 import/继承链画清楚——这是本页要讲的全貌，后续每节都是对其中某个节点的展开。

三个要点先记住。第一，WhisperEncoderVQ 继承 WhisperEncoder（class WhisperEncoderVQ(WhisperEncoder)），所以 conv1/conv2/positional_embedding/blocks/avg_pooler/proj 这些算子是父类定义、子类复用，子类只额外加 VQ 相关字段。第二，modeling 文件同时聚合了 encoder（本页主题）和 decoder（DiT 扩散 + BigVGAN 声码器），但 25Hz tokenizer 的语义离散化只发生在 encoder 内部，decoder 不参与量化。第三，core_vq.py 是从 Meta EnCodec 移植的 VQ 实现（文件头有 Meta 版权与 lucidrains MIT 声明），25Hz 走其中的 GRVQ 分支而非朴素 RVQ。

配置层：帧率链与 25Hz/12Hz 分野

Qwen3TTSTokenizerV1Config（configuration_..._v1.py:280）把 encoder/decoder 两个子配置与四个采样率常量捏在一起。四个常量决定了整条帧率链：input_sample_rate=24000 是原始音频；encode_downsample_rate=1920 是“每个输出 token 对应多少音频采样”（24000/1920≈12.5Hz，这是 talker 端消费的音频 codec 帧率）；decode_downsample_rate 同理用于 BigVGAN 上采样。而模型名里的 25Hz 指的是语义编码器在 VQ 前后的 token 帧率，由 hop、conv stride、avg_pool、vq ds_rate 一级级降下来（见下一节形状跟踪）。

与 12Hz(V2) tokenizer 对照一下，差异是系统级的，不是调参。12Hz 版的 model_type="qwen3_tts_tokenizer_12hz"，encoder 是 Qwen 自研的 sliding-attention transformer（不是 Whisper 架构），量化器是朴素 VectorQuantization 串成 RVQ（num_quantizers=16、codebook_size=2048）；25Hz 版用 Whisper 风格主干 + GRVQ 分组残差量化（codebook_size=32768、单组单量化器）。下表收齐对比。

NPU 迁移 · GRVQ 单量化器 vs 16 层 RVQ25Hz 版默认 num_groups=1, num_quantizers=1（speech_vq.py:219-220），即 GRVQ 退化为“单组单层 VQ”——每个 token 只查一次 32768 项的码本，而非 12Hz 版的 16 层残差串行查表。迁移含义：码本表 (32768, 1280) 在 bf16 下约 80MB，单卡可放下；但 F.embedding 的 gather 在 NPU 上是 index 算子，codebook_size=32768 的 one-hot（F.one_hot(...,32768)）会产生大稀疏矩阵，训练时 EMA 更新路径（ema_inplace、scatter_add_）需核验 torch_npu 支撑。推理只走 encode/decode 两条路径（无 EMA），风险低很多。

形状跟踪：mel → aftercnn → token

把一条音频进编码器到出 token 的形状演化走一遍，这是后续算子风险分析的基础。入口在 modeling_..._v1.py:1309 的 speech2mel：对每条 speech 调 get_mel_audio 得 mel，搬到 conv1 的 device/dtype。get_mel_audio（whisper_encoder.py:118）内部调 log_mel_spectrogram，输出形状 (n_mels=128, T_mel)，其中 T_mel = ceil(audio_len / hop)，hop=160。

qwen_tts/core/tokenizer_25hz/vq/whisper_encoder.py · L110–L126

def get_T_after_cnn(L_in, dilation=1):
    for (padding, kernel_size, stride) in eval("[(1,3,1)] + [(1,3,2)] "):
        L_out = L_in + 2 * padding - dilation * (kernel_size - 1) - 1
        L_out = 1 + L_out // stride
        L_in = L_out
    return L_out


def get_mel_audio(audio, padding=False, audio_vq_ds_rate = 1, n_mels = 128):
    audio_len = len(audio)
    if padding:
        reduction = 160 * 2 * audio_vq_ds_rate
        audio_pad = math.ceil(audio_len / reduction) * reduction - audio_len
        mel = log_mel_spectrogram(audio, n_mels=n_mels, padding=audio_pad)
    else:
        mel = log_mel_spectrogram(audio, n_mels=n_mels)  # [F,T]
    return mel

形状演化至此清晰：(audio_len,) → STFT → (128, T_mel) → conv1+conv2 → (n_state=1280, T_cnn)，permute 成 (T_cnn, 1280) 喂 transformer blocks。32 层 block 后若过 avg_pooler(2) 再砍半 → (T_cnn//2, 1280)，最后 proj 到 output_dim=3584。在 mel2code 里 indice_lens = [T // audio_vq_ds_rate for T in audio_aftercnnlens]（L1332）——VQ 的 ds_rate 再对 token 数做一次除法，pad_sequence 把不定长的 indices 拼成 (b, T_tok) 的 batch。

NPU 迁移 · whisper STFT 钉 CPUlog_mel_spectrogram（whisper_encoder.py:62）用 torch.stft(audio, N_FFT=400, HOP_LENGTH=160, return_complex=True)，再 .abs()**2、mel filter 矩阵乘、log10+clamp。这整段在 get_mel_audio→speech2mel 里对每条音频单独跑。建议保持 CPU：与 cosyVoice2/12Hz 的 STFT 钉 CPU 同一判断——CANN 的 STFT 复数路径（return_complex=True + torch.view_as_real）在多版 torch_npu 上不稳，且这里是预处理、对吞吐不敏感。mel 算完再 .to(conv1.weight.device) 搬 NPU，不冲突。注意 mel_filters 用 lru_cache 加载 npz（L43-59），device 切换时缓存失效需重建。

GRVQ 量化：分组残差与 Euclidean 码本

量化的实际入口是 WhisperEncoderVQ._do_quantize（speech_vq.py:239），在编码器 forward 的第 audio_vq_layers 层被调用（L320-321）。它先把 x (T,D) unsqueeze 成 (1,T,D)，过 downsample conv 对齐帧率，再调 self.audio_quantizer.encode(x) 得 indices、decode(indices) 得量化向量。注意 L255-259：训练时 raise NotImplementedError，所以这条 VQ 路径只在推理/编码时走，不参与梯度。

量化器本体是 DistributedGroupResidualVectorQuantization（core_vq.py:477），“Group Then RVQ”结构。forward（L502）先把输入在通道维 torch.chunk 成 num_groups 份，每组丢给一个 DistributedResidualVectorQuantization，再把各组 indices/loss stack 回来。25Hz 配置 num_groups=1, num_quantizers=1，所以实际只有一组一量化器——GRVQ 的“分组”和“残差”能力在此配置下都退化，等价于单个 VQ。

class DistributedGroupResidualVectorQuantization(nn.Module):
    """Efficient distributed group residual vector quantization implementation.
    Follows Algorithm 1. in https://arxiv.org/abs/2305.02765
    Group Then rvq
    """
    def __init__(self, *,
                 num_groups,
                 num_quantizers,
                 quantize_dropout: bool = False,
                 rand_num_quant: tp.Optional[tp.List] = None,
                 **kwargs):
        super().__init__()
        self.rvqs = nn.ModuleList(
            [
                DistributedResidualVectorQuantization(
                    num_quantizers=num_quantizers,
                    quantize_dropout=quantize_dropout,
                    rand_num_quant=rand_num_quant,
                    **kwargs
                )
                for _ in range(num_groups)
            ]
        )
        self.num_groups = num_groups

    def forward(self, x, n_q: tp.Optional[int] = None):
        x_lst = torch.chunk(x, chunks=self.num_groups, dim=1)
        all_quantized_out = []
        all_indices = []
        all_losses = []
        for mod, item in zip(self.rvqs, x_lst):
            quantized_out, out_indices, out_losses = mod(item, n_q)
            all_quantized_out.append(quantized_out)
            all_indices.append(out_indices)
            all_losses.append(out_losses)

        out_losses = torch.stack(all_losses, dim=1).mean(dim=1)

        return torch.cat(all_quantized_out, dim=1), torch.stack(all_indices, dim=1), out_losses

残差量化的核心在 DistributedResidualVectorQuantization.forward（core_vq.py:388）：residual = x，逐量化器 quantized, indices, loss = layer(quant_in, buffers)，然后 residual = residual - quantized、quantized_out += quantized——经典 RVQ 算法（文件头引用 arxiv 2107.03312）。L417-427 还有个 q0_ds_ratio 分支：第 0 个量化器前后用 F.interpolate 做时域 2 倍 resample，让第 0 层在更低帧率量化、再插值回原帧率。

    def forward(self, x, n_q: tp.Optional[int] = None):
        quantized_out = torch.zeros_like(x)
        residual = x
        bb, cc, tt = x.shape
        device = x.device

        all_losses = []
        all_indices = []
        all_sub_quants = []
        n_q = n_q or len(self.layers)

        should_quantize_dropout = self.training and self.quantize_dropout and self.rand_num_quant is not None
        if should_quantize_dropout:
            rand_quantize_dropout_index = random.choice(self.rand_num_quant)

            null_indices_shape = (x.shape[0], x.shape[2])
            null_indices = torch.full(null_indices_shape, -1., device=device, dtype=torch.long)
            null_loss = torch.full((1,), 0., device=device, dtype=x.dtype)
            null_sub_quant = torch.full(x.shape, -1, device=device, dtype=x.dtype)

        for quantizer_index, layer in enumerate(self.layers[:n_q]):
            # dropout except the first quantizer
            if should_quantize_dropout and quantizer_index >= rand_quantize_dropout_index:
                all_indices.append(null_indices)
                all_losses.append(null_loss)
                all_sub_quants.append(null_sub_quant)
                continue

            quant_in = residual
            if self.q0_ds_ratio > 1 and quantizer_index == 0:
                quant_in = F.interpolate(quant_in, size=[tt//2])
            quantized, indices, loss = layer(quant_in, [
                self.inited[quantizer_index],
                self.cluster_size[quantizer_index],
                self.embed[quantizer_index],
                self.embed_avg[quantizer_index]
            ])
            if self.q0_ds_ratio > 1 and quantizer_index == 0:
                quantized = F.interpolate(quantized, size=[tt])

            quantized_out = quantized_out + quantized

            all_losses.append(loss)
            all_sub_quants.append(quantized)

        out_losses, out_indices, out_sub_quants = map(torch.stack, (all_losses, all_indices, all_sub_quants))

最近邻查表落在 EuclideanCodebook.quantize（core_vq.py:186）：dist = -(x.pow(2).sum(1) - 2*x@embed + embed.pow(2).sum(0))，embed_ind = dist.max(dim=-1).indices——这就是把每个向量映射到码本里欧氏距离最小的那一项的 argmax。解量化 dequantize 用 F.embedding(embed_ind, self.embed) 做 gather。forward（L218）里训练分支还会做 F.one_hot(embed_ind, codebook_size)、ema_inplace 更新 cluster_size 与 embed_avg、laplace_smoothing 归一——但这些在 25Hz 推理路径（encode）里不走，因为 _do_quantize 训练时直接 raise。

NPU 迁移 · VQ argmax / EMA 算子EuclideanCodebook.quantize 用 dist.max(dim=-1).indices 取最近邻（core_vq.py:193）——这是 argmax over codebook，codebook_size=32768 维度的 reduce，NPU 原生支持。dist 的计算含 x.pow(2).sum + x @ embed（L188-192），embed 是 (D=1280, 32768) 的矩阵，x @ embed 是 (T, 1280) @ (1280, 32768) 的大 matmul，bf16 下 NPU 可跑但需注意 x.pow(2).sum(1, keepdim=True) 的数值范围（bf16 精度有限，建议此处临时升 fp32）。推理路径无 EMA/one_hot，风险集中在 matmul 与 argmax；若未来要训练 GRVQ，scatter_add_、F.one_hot(...,32768)、ema_inplace 的原地 mul_/add_ 需逐个核验 torch_npu 支撑。

算子清单与 NPU 风险表

NPU 迁移 · varlen cu_seqlens 路由Whisper encoder 把多段音频拼成一个长序列送 attention，用 cu_seqlens（whisper_encoder.py:364-365，由 itertools.accumulate 算累积长度）区分各段边界——这是 varlen attention 的标准接口。退到 manual 路径时，qkv_attention_manual（L214）用 cu_seqlens 做 Python for 循环逐段填充 padding 再批量 matmul。若改走 torch_npu 的 npu_fusion_attention，需确认其 varlen 接口（actual_seq_lens 参数）与这里 cu_seqlens 的语义一致——二者都是“段边界累积长度”，但 FA 用 [0, L1, L1+L2, ...]，torch_npu 接口可能要 actual_seq_lens=[L1, L2, ...]，转换时易错位。这是手动 fallback 之外最值得核验的一处。
NPU 迁移 · mel filter 缓存与 devicemel_filters(device, n_mels)（whisper_encoder.py:43）用 @lru_cache 缓存，从 assets/mel_filters.npz 读 numpy 再 torch.from_numpy().to(device)。NPU 上 device=npu:0 时缓存会重建一次，无大风险；但要注意 lru_cache 的 key 是 device 对象，多卡时每张卡各缓存一份，np.load 在 worker 进程里跑。迁移无硬卡点，只是首次 forward 会有一次 npz 读盘+搬运开销。assets 目录与 .npz 文件需随模型一起部署，否则 assert n_mels in {80,128} 之后会 FileNotFoundError。
一句话结论： 25Hz tokenizer 上 NPU 的两个硬卡点是 STFT（钉 CPU 绕过）和 varlen FlashAttention（退 manual 可跑、换 npu_fusion_attention 提速）；GRVQ 的 argmax/embedding 在推理路径上是低风险原生算子，只有训练路径（当前被 NotImplementedError 屏蔽）的 EMA/one_hot 才需额外核验。
