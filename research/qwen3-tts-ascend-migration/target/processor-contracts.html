Processor 输入与提示词契约

Qwen3TTSProcessor 是一层极薄的契约边界：它不拼装音频、不画提示词模板，只把 text 交给 Qwen2TokenizerFast 并产出 input_ids/attention_mask。真正把 text、ref_audio、audio_codes 拼成 batch 的活儿在 collate 与推理端——控制符 id 来自 config，提示词模板在 dataset/inference 端构造，再经 Processor token 化。本页沿这条契约链逐段拆：Processor 的 __call__ 契约、tokenizer 调用与输出字段、控制符 id 表、提示词模板三态、以及 Processor 在训练与推理两端各扮演什么角色。

学习目标

  • 说出 Qwen3TTSProcessor.__call__ 的输入契约（text 必填、padding 左填、return_tensors）与输出字段（input_ids/attention_mask 等 BatchFeature）
  • 分清 Processor 的边界：它只负责文本 token 化，提示词模板由 dataset/inference 端 _build_assistant_text 构造，控制符 id 由 config 提供、在 collate 里拼接
  • 列出 codec 控制符五元组（nothink/think_bos/think_eos/pad/bos/eos）的 id 与各自在 collate 通道 1 中的位置约定，并指出 index 6 的 speaker 预留位
  • 解释 Processor 在训练（dataset.__getitem__→collate）与推理（inference model）两端如何被同一份契约调用，以及 CPU/设备无关部分对 NPU 迁移意味着什么

前置知识

  • 熟悉 transformers 的 ProcessorMixin / BatchFeature 概念
  • 读过 SFT 训练循环页对 collate 16 层 codec 布局与 speaker 注入位 index 6 的描述
  • 理解 next-token 训练里 text token 与 codec 控制符在双通道 input_ids 里分居通道 0/1

总览：Processor 是契约边界，不是拼装器

初读 processing_qwen3_tts.py 会发现一个反直觉的事实：文件只有 106 行，Qwen3TTSProcessor.__call__ 既不碰音频、也不碰 codec，它只接受 text。这一层的设计哲学是契约边界而非加工中心——Processor 把“文本到 token id”这件唯一的事规范化成可调用的接口，真正的拼装（text + ref_audio + audio_codes → batch 张量）发生在下游 finetuning/dataset.py 的 __getitem__/collate_fn，以及推理端 qwen_tts/inference/qwen3_tts_model.py。控制符 id（im_start/im_end/codec_*）由 configuration_qwen3_tts.py 提供，提示词模板由 dataset/inference 端的字符串拼接产生，二者汇入 Processor 走 token 化。下图把这条契约链画清楚。

三个角色各司其职，务必分清：Processor 是纯文本接口，输入 text、输出 BatchFeature（含 input_ids 与 attention_mask）；config 是 token id 的权威来源，im_start_token_id=151644、im_end_token_id=151645、codec_nothink_id=4203 等都在这里声明；collate/inference 是拼装者，它把 config 的 id 写进 input_ids[:,:,1] 的控制符位、把 Processor 产出的文本 token 写进 input_ids[:,:,0]。Processor 自己从不知道 codec 控制符长什么样——它只是下游用来 token 化提示词的工具。

__call__ 的输入契约与 tokenizer 转发

__call__ 的契约很硬：第一参数 text 不能为 None，否则直接 raise ValueError。其余参数走 **kwargs，由 _merge_kwargs 与默认值合并：默认 padding=False、padding_side="left"。随后把 text 包成 list、调 self.tokenizer(text, **text_kwargs)，把 tokenizer 的返回原样塞进 BatchFeature。注意 tensor_type=kwargs.get("return_tensors")——这一行决定了输出是 python list 还是 torch 张量，下游 dataset 调用时传 return_tensors="pt" 才能拿到 LongTensor。

    def __call__(self, text=None, **kwargs) -> BatchFeature:
        """
        Main method to prepare for the model one or several sequences(s) and audio(s). This method forwards the `text`
        and `kwargs` arguments to Qwen2TokenizerFast's [`~Qwen2TokenizerFast.__call__`] if `text` is not `None` to encode
        the text. 

        Args:
            text (`str`, `List[str]`, `List[List[str]]`):
                The sequence or batch of sequences to be encoded. Each sequence can be a string or a list of strings
                (pretokenized string). If the sequences are provided as list of strings (pretokenized), you must set
                `is_split_into_words=True` (to lift the ambiguity with a batch of sequences).
        """

        if text is None:
            raise ValueError("You need to specify either a `text` input to process.")

        output_kwargs = self._merge_kwargs(
            Qwen3TTSProcessorKwargs,
            tokenizer_init_kwargs=self.tokenizer.init_kwargs,
            **kwargs,
        )
        if not isinstance(text, list):
            text = [text]

        texts_inputs = self.tokenizer(text, **output_kwargs["text_kwargs"])

        return BatchFeature(
            data={**texts_inputs},
            tensor_type=kwargs.get("return_tensors"),
        )

逐行看契约。L59-60 if text is None: raise ValueError——这是输入契约的守门，Processor 拒绝空文本调用，因为下游 tokenizer 没有意义可编码的对象。L62-66 output_kwargs = self._merge_kwargs(...) 把 Qwen3TTSProcessorKwargs._defaults（L20-25 声明的 padding/padding_side）与运行时 kwargs 及 tokenizer 自身的 init_kwargs 三方合并，确保 padding 默认值在不显式传时仍然生效。

L67-68 if not isinstance(text, list): text = [text] 把单条字符串规整成单元素 list，让后续调用形态统一。L70 texts_inputs = self.tokenizer(text, **output_kwargs["text_kwargs"]) 是整个 Processor 唯一的实质动作——把 text 交给 Qwen2TokenizerFast，返回 input_ids 与 attention_mask 等 tokenizer 标准输出。

L72-75 return BatchFeature(data={**texts_inputs}, tensor_type=kwargs.get("return_tensors")) 把 tokenizer 的字典包成 BatchFeature，并按 return_tensors 决定张量化。这一步是与设备无关的：tokenizer 在 CPU 上跑、BatchFeature 只是 dict 包装，tensor_type="pt" 产出的也是 CPU LongTensor。设备搬运发生在下游（dataset 里 ref_mels.to(model.device)、inference 里 input_id.to(self.device)），Processor 不参与。

NPU 迁移 · Processor 全程 CPU/设备无关Processor 的 __call__ 只调 CPU 上的 tokenizer、包一个 dict，不碰任何 NPU 算子。迁移到 910B 时这一层无需任何改动——既不需要改 attention 实现，也不涉及 HCCL。唯一要保证的是 return_tensors="pt" 产出的 CPU LongTensor 在下游被正确 .to(device) 搬到 NPU。注意 Qwen2TokenizerFast 底层是 Rust tokenizers，与硬件无关，跨平台行为一致，待真机验证的只是下游搬运路径，不是 Processor 本身。

控制符 id：config 是唯一权威

提示词模板里的 <|im_start|>/<|im_end|> 是字符串，最终要被 tokenizer 编成 id；而 codec 控制符是纯 id，不经过 tokenizer，直接由 collate 写进通道 1。两类 id 的唯一权威来源都是 config。Qwen3TTSConfig 顶层声明文本侧的 im_start/im_end/tts_pad/bos/eos（L472-476），Qwen3TTSTalkerConfig 声明 codec 侧的六元组（L393-399）。任何下游（collate、inference）要拿到这些 id，都得读 self.config.talker_config.codec_*_id，不能硬编码。

NPU 迁移 · id 表是静态整数、设备无关这些 token id 是config 里的静态整数，config 本身是 JSON 反序列化产物、纯 CPU/无算子。collate 用它们做 LongTensor 的位置赋值，全程 CPU。迁移时这一层无需改动，只要保证训练脚本装载的 config 与推理端是同一份 id 表——id 表错位会导致 codec 控制符查到错误的 embedding 行，是语义级 bug而非算子级风险。

提示词模板三态：训练/推理/ref

提示词模板不是 Processor 自己的属性，而是调用方在传 text 进 Processor 之前拼好的字符串。Qwen3-TTS 用 ChatML 风格的 <|im_start|> 分隔。模板有三个形态：训练端的 assistant 模板、推理端的 assistant 模板、推理端的 ref/instruct 模板。它们都在 dataset/inference 端用 f-string 拼接，再交给 Processor。

    def _tokenize_texts(self, texts: List[str]) -> List[torch.Tensor]:
        input_ids = []
        for text in texts:
            input = self.processor(text=text, return_tensors="pt", padding=True)
            input_id = input["input_ids"].to(self.device)
            input_id = input_id.unsqueeze(0) if input_id.dim() == 1 else input_id
            input_ids.append(input_id)
        return input_ids

推理端 _tokenize_texts（L278-285）与 dataset 端（L97-101）几乎同构，差别只在 L282 input_id = input["input_ids"].to(self.device)——推理端在拿到 token 后立刻搬到设备（self.device），而 dataset 端保持 CPU 张量交给 collate。这就是 Processor 契约在不同调用方的唯一设备差异：训练侧 token 留 CPU 等 collate 搬，推理侧 token 进 model.device。

NPU 迁移 · 模板拼接是字符串、搬到 NPU 才碰设备模板用 Python f-string 拼字符串、Processor 用 Rust tokenizers 编码，二者都纯 CPU、与设备无关。迁移到 910B 时，唯一要验的是推理端 L282 .to(self.device) 在 device 为 NPU 时能把 LongTensor 搬到 npu:0——这是 LongTensor 的 host→device 拷贝，torch_npu 原生支持，无算子风险。训练端则因 token 留 CPU、由 collate 产出 batch 后再在前向阶段统一搬 NPU，风险落在 .to(model.device) 而非 Processor。待真机验证的是 NPU 上的 host→device 路径时延，不是正确性。

与 collate 的衔接：token id 从哪来

理解了 Processor 只产文本 token、config 只产控制符 id，再看 collate 如何把它们拼成 input_ids (b,t,2) 就一目了然。通道 0 的文本段来自 Processor 产出的 text_ids（经 dataset L141 切尾），通道 0 的 pad/bos/eos 来自 config 的 tts_* id；通道 1 的控制符段全部来自 config 的 codec_* id，而真正的 codec 数据 audio_codec_0 来自数据集的 audio_codes[:,0]。codec_ids (b,t,16) 则把整 16 层 audio_codecs 塞进对应位置。

所以“collate 用的 token id 从哪来”有两条来源：Processor→tokenizer 提供文本段的 input id（im_start/im_end/普通词都经 tokenizer 编码）；config 提供 pad/bos/eos 与全部 codec 控制符的 id。audio_codes 本身不经过 Processor——它是已离散化好的整数（来自上游 codec 模型），直接当 codec_ids[:,:,0] 与 codec_ids[:,:,1:16] 写进 batch。这就是为什么 Processor 文件那么薄：它只占“文本→id”这一环，其余拼装都在 collate 里手工索引。

NPU 迁移 · collate 全 CPU 索引、不占 NPUcollate_fn 全程在 CPU LongTensor 上做位置赋值（input_ids[i, a:b, 0] = ...），跑在 DataLoader worker 进程里，不占 NPU 算子。迁移时这一段保持原样即可。唯一要留意的是 collate 产出的 LongTensor 进入前向后要 .to(device)——indexing 赋值不会改 dtype/device，所以 collate 的输出是 CPU LongTensor，前向侧搬运。待真机验证的是大批量下 CPU collate 的吞吐是否成为数据馈送瓶颈，而非算子兼容性。

推理与训练两端：同一份契约的两种调用

Processor 在两端被同一份契约调用（都走 self.processor(text=text, return_tensors="pt", padding=True)），但调用形态不同。训练侧由 TTSDataset.__getitem__（L97-101 的 _tokenize_texts）调用，每条样本调一次、产 CPU LongTensor、交给 collate 拼 batch；推理侧由 inference model 的 _tokenize_texts（L278-285）调用，产张量后立刻 .to(self.device)。两端共享 Qwen3TTSProcessorKwargs 的默认 padding 设定，所以 tokenizer 行为一致。

NPU 迁移 · 左填 padding 与左对齐生成Processor 默认 padding_side="left"（L20-25 的 Qwen3TTSProcessorKwargs._defaults）。左填是因果生成模型的标准设定：pad 在序列左侧，让真实 token 靠右、与生成方向对齐。这与设备无关，是 tokenizer 层面的约定。迁移到 910B 时无需改动——torch_npu 的 attention 实现同样假设因果 mask，左填语义在 NPU 上保持一致。要 待真机验证的只是长序列下左填 + 因果 mask 的显存布局，不是正确性。
