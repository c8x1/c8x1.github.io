学习轨道：入门教程

SFT 数据预处理与 Collate

逐段拆解 Qwen3-TTS 微调的数据管线：prepare_data.py 把音频压成 16 层 codec 写进 JSONL，TTSDataset.__getitem__ 在 CPU 上抽 mel（24kHz/n_fft=1024/hop=256），collate_fn 把一个 batch 拼成 7 个对齐张量——重点展开 (b,t,2) 双通道的位置布局、[3:8] 控制符与 index 6 的 speaker 注入位、codec_0_labels 的 -100 屏蔽、padding 与 mask 的协同，并标出 mel/STFT 留 CPU 的昇腾迁移判断。

学习目标

  • 说清 prepare_data.py 产出 JSONL 的字段（audio/text/ref_audio/audio_codes），并理解 audio_codes 是 (t,16) 的 16 层残差 codec
  • 逐行定位 __getitem__：文本套 assistant 模板、codec 转 long 张量、ref 音频过 extract_mels 抽 24kHz mel
  • 画出 collate_fn 的 (b,t,2) 双通道位置布局：[3:7]pad/[7]bos/[8..]text 与 [3:8]控制符/index6 speaker/[8..]codec_0，说出 index 6 为何是 speaker 注入位
  • 说清 7 个 batch 张量各自的形状、dtype、用途，以及 codec_0_labels 用 -100 屏蔽非 codec 位置的机制
  • 判断 mel/STFT 留 CPU 的迁移方向，并理解 collate 全程在 CPU 不占 NPU 的原因

前置知识

  • 读过《SFT 训练循环》的 collate 概览段（本页是其深入展开，不重复前向/反向/保存）
  • 理解 PyTorch Dataset/DataLoader 与 collate_fn 的职责边界
  • 了解 next-token 训练里 label 与 input 各 shift 一位、-100 在 CE 里被忽略

总览：从 JSONL 到 batch 张量

Qwen3-TTS 的微调数据管线分两步：离线编码用 finetuning/prepare_data.py 把每条音频过 12Hz tokenizer 压成离散 codec、与文本一起写进 JSONL；在线取数用 finetuning/dataset.py 的 TTSDataset，__getitem__ 负责单条样本的 mel 提取与文本分词，collate_fn 负责把一个 batch 拼成 7 个对齐到同一长度 t 的张量。整条管线全在 CPU：mel 抽取走 torch STFT 但跑在 worker 进程，collate 是纯索引赋值——不触碰 NPU。本页深入展开 collate 的位置布局细节，与《SFT 训练循环》的 collate 概览段互补，但把每个张量的填充逐位讲透。

长度 t 的来源：collate 取 batch 内最长样本的 text_len + codec_len，再加 8 个固定前缀位（见 dataset.py:149-151）。所以同一个 batch 里短样本会被 padding 到与最长样本对齐，对齐由 attention_mask/codec_mask 等掩码标记哪些是真位置。

prepare_data.py：离线把音频压成 16 层 codec

训练前要先跑一遍 prepare_data.py。它读一个 input JSONL（每行有 audio wav 路径、text、ref_audio），对每 32 条音频一批调用 tokenizer_12hz.encode，把输出 audio_codes 写回每行，输出新的 JSONL。注意 --device 默认 cuda:0，说明这步编码本是为 GPU 设计——但它是一次性离线产物，迁移时用 NPU 或 CPU 跑完即可，产物（JSONL）与硬件无关。

L32-35 Qwen3TTSTokenizer.from_pretrained(tokenizer_model_path, device_map=args.device) 装载 12Hz 编码器，权重放 cuda:0。L37-38 把整份 input JSONL 一次性 readlines 进内存再逐行 json.loads——全量加载，数据量大时要注意内存，但只是一次性脚本。

L43-54 主循环：把音频路径攒到 batch_audios，攒满 32 条（BATCH_INFER_NUM）就 tokenizer_12hz.encode(batch_audios) 批量编码。enc_res.audio_codes 是一个 list，每个元素是一条音频的 codec；L51 line['audio_codes'] = code.cpu().tolist() 把 GPU 张量拉回 CPU 再转 list 写 JSON。.cpu() 这一步说明编码确实发生在加速器上。L56-62 处理不满 32 条的尾部余数。

产物结构：输出 JSONL 每行是 {audio, text, ref_audio, audio_codes}（外加原行里已有的字段）。audio_codes 是 (t_codec, 16) 的二维整数列表——16 列就是 16 层残差 codec，第 0 层是粗粒度主码本，第 1-15 层是逐层残差修正。这个 (t,16) 形状是后面 collate 一切布局的源头。

audio_codes 的 (t,16) 是全篇的基石。prepare_data 产出的是 16 层；__getitem__ 把它原样转 torch.long 张量；collate 把 batch 维加上得到 codec_ids (b,t,16)；前向里第 0 层走 codec_embedding、第 1-15 层走 code_predictor 的 15 张独立嵌入表。理解 16 列的残差语义，才能理解后面两个损失为何分工。
NPU 迁移 · 离线编码 device 可换prepare_data.py 的 --device cuda:0 只是装载 12Hz tokenizer 的位置。这步是一次性离线产物，不在训练热路径上。迁移时可选：(1) 改 --device npu:0 让 12Hz tokenizer 在 NPU 跑（需核验其 conv/attention 算子在 CANN 支持）；(2) 直接用 CPU 跑完（编码量不大、吞吐不敏感）。无论哪条，产出的 JSONL 与硬件无关，训练时只用产物。建议先用 CPU 跑完离线编码，避免训练前再调 NPU 编码器。

__getitem__：单条样本的 mel 提取与分词

    def __getitem__(self, idx):
        item = self.data_list[idx]

        audio_path  = item["audio"]
        text        = item["text"]
        audio_codes = item["audio_codes"]
        language        = item.get('language','Auto')
        ref_audio_path  = item['ref_audio']

        text = self._build_assistant_text(text)
        text_ids = self._tokenize_texts(text)

        audio_codes = torch.tensor(audio_codes, dtype=torch.long)

        ref_audio_list = self._ensure_list(ref_audio_path)
        normalized = self._normalize_audio_inputs(ref_audio_list)
        wav,sr = normalized[0]

        ref_mel = self.extract_mels(audio=wav, sr=sr)

        return {
            "text_ids": text_ids[:,:-5],    # 1 , t
            "audio_codes":audio_codes,      # t, 16
            "ref_mel":ref_mel
        }

NPU 迁移 · librosa.load 纯 CPU_load_audio_to_np（dataset.py:43-50）用 librosa.load(sr=None, mono=True) 读音频。librosa 底层走 soundfile/audioread，是纯 CPU 库，无 NPU 路径。这是 DataLoader worker 进程里的调用，本就不该碰加速器。迁移时无需改动，保持 CPU。注意 sr=None 保留原始采样率，再由 extract_mels 的 assert sr==24000 强制 24kHz——所以数据源音频必须已是 24kHz。

extract_mels：24kHz mel 与 STFT 参数

extract_mels 被 @torch.inference_mode() 装饰（不建图，纯前向），断言 sr==24000，再调 qwen_tts.core.models.modeling_qwen3_tts.mel_spectrogram。参数是固定的：n_fft=1024, num_mels=128, sampling_rate=24000, hop_size=256, win_size=1024, fmin=0, fmax=12000，最后 .transpose(1,2) 把 mel 从 (b, n_mels, t_mel) 转成 (b, t_mel, 128)。

    @torch.inference_mode()
    def extract_mels(self, audio, sr):
        assert sr == 24000, "Only support 24kHz audio"
        mels = mel_spectrogram(
            torch.from_numpy(audio).unsqueeze(0), 
            n_fft=1024, 
            num_mels=128, 
            sampling_rate=24000,
            hop_size=256, 
            win_size=1024, 
            fmin=0, 
            fmax=12000
        ).transpose(1, 2)
        return mels

形状跟踪：torch.from_numpy(audio).unsqueeze(0) 把 (n_samples,) 波形升到 (1, n_samples)；mel_spectrogram 内部 torch.stft 产出 (1, n_mels=128, t_mel)；.transpose(1,2) 得 (1, t_mel, 128)。t_mel ≈ n_samples / hop_size = n_samples / 256——24kHz 下每秒约 93.75 mel 帧。这个 mel 是 speaker_encoder 的输入，形状与文本/codec 长度无关。

mel_spectrogram 内部（modeling_qwen3_tts.py:399-453）细节：先 librosa_mel_fn(sr, n_fft, n_mels, fmin, fmax) 算 mel 滤波器组（numpy，CPU），转 mel_basis 并 .to(device)；torch.hann_window(win_size).to(device) 建 Hann 窗；padding=(n_fft-hop_size)//2=384 做 reflect pad；最后 torch.stft(y, n_fft=1024, hop_length=256, win_length=1024, window=hann_window, center=False) 出复数谱，取能量再过 mel_basis 与 dynamic_range_compression（log）。

NPU 迁移 · mel/STFT 留 CPUtorch.stft + 复数谱取能量是 mel 抽取的核心。CANN 的 STFT 复数路径在历史版本里不稳（window/hann/reflect pad 组合易报算子不支持或精度偏差）。这条调用在 __getitem__ 里、属 DataLoader worker 的 CPU 数据管线，建议保持 CPU（与 cosyVoice2 STFT 钉 CPU 同一判断）。具体做法：保证 extract_mels 的输入 audio 是 CPU 张量（torch.from_numpy 已是 CPU），mel_basis/hann_window 的 .to(device) 中 device 即 CPU 张量的 device。不需上 NPU，且 mel 抽取对吞吐不敏感（每条样本只抽一次 ref mel）。
extract_mels 的 @torch.inference_mode() 装饰器很关键：它让 mel 抽取不进 autograd 图，mel 不会回传梯度。这与 speaker_encoder 后接 .detach() 一致——音色参考信号是条件输入而非可训练参数。迁移时这个装饰器在 NPU 上无影响（inference_mode 是 Python 层上下文管理，非算子）。

collate_fn：7 张量与双通道位置布局

进入 collate。L147 assert self.lag_num == -1——当前实现只支持无 lag 模式（lag_num 是残差层预测延迟参数，-1 表示不启用残差预测的特殊路径，走标准 next-token）。L149 item_length = [text_ids.shape[1] + audio_codes.shape[0] for b in batch] 算每条的 text_len + codec_len；L150 max_length = max(item_length) + 8 加 8 个固定前缀位；L151 b,t = len(batch), max_length。

    def collate_fn(self, batch):
        assert self.lag_num == -1

        item_length = [b['text_ids'].shape[1] + b['audio_codes'].shape[0] for b in batch]
        max_length = max(item_length) + 8
        b,t = len(batch),max_length

        input_ids   = torch.zeros((b,t,2),dtype=torch.long)
        codec_ids   = torch.zeros((b,t,16),dtype=torch.long)
        text_embedding_mask     = torch.zeros((b,t),dtype=torch.bool)
        codec_embedding_mask    = torch.zeros((b,t),dtype=torch.bool)
        codec_mask      = torch.zeros((b,t),dtype=torch.bool)
        attention_mask  = torch.zeros((b,t),dtype=torch.long)
        codec_0_labels  = torch.full((b, t), -100, dtype=torch.long)

L153-159 预分配 7 个张量，全 CPU：input_ids (b,t,2) long 全 0；codec_ids (b,t,16) long 全 0；text_embedding_mask (b,t) bool 全 False；codec_embedding_mask (b,t) bool 全 False；codec_mask (b,t) bool 全 False；attention_mask (b,t) long 全 0；codec_0_labels (b,t) long 预填 -100（torch.full(..., -100)）。-100 是 PyTorch CE 的默认 ignore_index，预填意味着所有非 codec 位置默认被损失忽略，后面只把真 codec 段覆盖成实际 token。

这 7 个张量就是 batch 的全部内容。注意 text_embedding_mask 与 codec_embedding_mask 在 return 时会 .unsqueeze(-1) 变成 (b,t,1)，以便与前向 embedding 的 (b,t,d) 广播相乘。input_ids 的最后一维是 2，正是双通道：通道 0 放文本 token，通道 1 放 codec 控制符——前向里 input_ids[:,:,0] 与 input_ids[:,:,1] 分别取出来查两套嵌入表。

上图是双通道的位置布局。通道 0（文本）：位 [:3] 放 text_ids 的前 3 个 token（assistant 模板头部）；位 [3:7] 填 tts_pad_token_id（4 个 pad）；位 [7] 放 tts_bos_token_id（文本段 BOS）；位 [8 : 8+text_len-3] 放 text_ids 第 3 个之后的 token；位 [8+text_len-3] 放 tts_eos_token_id；位 [8+text_len-2 : ...] 起填 pad 直到 codec 段。

通道 1（codec）：位 [:3] 留 0（与文本通道对齐的前缀位）；位 [3:8] 是 5 个控制符，依次 codec_nothink_id、codec_think_bos_id、codec_think_eos_id、位 6 是 0（留给 speaker 注入）、codec_pad_id；位 [8 : 8+text_len-3] 填 codec_pad_id（与文本段对齐，这段无 codec）；位 [8+text_len-2] 放 codec_bos_id；位 [8+text_len-1 : 8+text_len-1+codec_len] 放真正的第 0 层 codec audio_codec_0；末位接 codec_eos_token_id。

speaker 注入位 = index 6 是 collate 与前向的硬约定：collate 在通道 1 的位 6 填 0 占位、并把 codec_embedding_mask[6] = False（L201）排除它参与 codec 嵌入查表；前向 input_codec_embedding[:,6,:] = speaker_embedding 在此写入 speaker 向量。所以位 6 的 0 只是占位符，实际内容由前向注入。迁移时这个位置约定不能改，否则 speaker 向量写错位、音色定制失效。
NPU 迁移 · collate 全程 CPUcollate_fn 的全部操作是 torch.zeros/torch.full 预分配 + 切片索引赋值（input_ids[i, a:b, c] = ...），无 GPU/NPU 算子。它在 DataLoader worker 进程里跑（num_workers>0 时是子进程）。迁移到 NPU 时无需改动——张量产出后在训练循环里才 .to(model.device) 搬到 NPU。唯一要保证的是 worker 进程不被 NPU 初始化污染（DataLoader 默认 CPU worker 无此问题）。这是数据管线，不占 NPU 算力。

深读：位置约定与 mask 的协同不变量

把 collate 的位置约定收束成几条不变量，迁移时这些不变量必须保持。第一：8 个固定前缀位（max_length = item_length + 8），其中 [:3] 是文本前缀、[3:8] 是 codec 控制符区——这两个区在双通道里对齐。第二：位 6 永远是 speaker 注入位，collate 填 0 占位、codec_embedding_mask[6]=False，前向在此写 speaker 向量。

第三：文本段与 codec 段交错 1 位。通道 0 的文本 eos 在 8+tl-3，通道 1 的 codec bos 在 8+tl-2，codec_0 从 8+tl-1 起。这个错位设计让 codec bos 紧贴文本 eos 之后一位——前向 shift 时，文本末位的隐状态恰好预测 codec bos，codec bos 的隐状态预测第一个 codec token，形成连贯的 next-token 链。

第四：codec_0_labels 用 -100 做默认屏蔽。预填 torch.full((b,t), -100) 后只在 codec 段覆盖实际 token。这是 PyTorch CrossEntropyLoss(ignore_index=-100) 的标准用法——非 codec 位置不回传损失。迁移到 NPU 时，ignore_index 是 loss 算子的标准参数，CANN 的 CE 实现支持，无风险。但要确认前向里 label 的 shift 与此处一致：talker(inputs=[:,:-1], labels=codec_0_labels[:,1:])，label 砍第一位、输入砍最后一位。

第五：codec_mask 与 codec_0_labels 范围不同。codec_mask 只标真 codec token 段（不含 codec bos/eos、不含控制符），前向用它从隐状态拉平挑位置过 sub-talker；而 codec_0_labels 覆盖 codec 段 + 末尾 codec_eos。前者用于挑位置算 MTP，后者用于主 CE——两者职责分离。
