学习轨道：入门教程

SFT 训练循环

按固定 sft_12hz.py 的真实顺序逐段拆解整条 SFT 循环：accelerate setup、from_pretrained 的 FA2 硬绑、collate 的 16 层 codec 布局、speaker 注入与 embedding 汇总、talker 前向与 MTP sub-talker 损失、反向累积与优化、speaker 向量写回与 safetensors 导出——每段标出昇腾 910B 迁移的风险点与验证条件。

学习目标

  • 按源码顺序定位 setup、collate、前向、反向、保存五段，并说出每段入口的行号
  • 说清 collate 产出的 (b,t,2)/(b,t,16) 张量布局、16 层 codec 的通道分配与 speaker 注入位（index 6）
  • 区分已核验的静态 call site 与 CUDA 硬绑、accelerate 设备路由、HCCL 后端等待真机验证的迁移点
  • 解释主 Talker CE 与 sub-talker MTP 两个损失如何组合、为何输入与标签各 shift 一位

前置知识

  • 熟悉 PyTorch 单卡张量与训练循环
  • 读过包结构与推理 API
  • 理解 next-token 语言模型的 label shift

总览：训练循环的五段

整个 SFT 脚本 finetuning/sft_12hz.py 只有一个 train() 函数，162 行。它用 accelerate 包住一个（默认单进程）训练循环，目标是 12Hz Base 单说话人音色定制。循环按源码顺序分五段：setup 建加速器、装载模型、构造数据/优化器并交给 accelerator.prepare；collate（在 finetuning/dataset.py）把每条样本拼成固定布局的 batch 张量；前向 汇总 text/codec/speaker 三路 embedding 喂给 talker，并算 MTP sub-talker 辅助损失；反向 累积梯度、裁剪、步进；保存 在每轮末把首步缓存的 speaker 向量注入 codec_embedding 表并写 safetensors。下图是五段的数据流，accelerator.prepare 是 setup 到前向的分水岭。

三个张量贯穿全流程，务必先记住：input_ids (b,t,2) 的通道 0 是文本 token、通道 1 是 codec 控制符；codec_ids (b,t,16) 是 16 层残差 codec（第 0 层参与主 CE，第 1-15 层参与 sub-talker MTP）；codec_mask (b,t) 标记哪些时间位是真 codec（用来从 talker 隐状态里挑出真 codec 位置算 MTP）。t 是 batch 内最长样本的 text_len + codec_len + 8。

Setup 与 Accelerate ownership

setup 段先把模型、配置、数据、优化器逐个构造好，再一次性交给 accelerator.prepare。顺序很关键：from_pretrained 发生在 prepare 之前，所以装载时设定的 dtype 与 attention 实现是原始模型语义，不是 accelerate 包裹后的行为。Accelerator 的三个参数决定了整条循环：累积步、精度、日志后端。

NPU 迁移 · FA2 硬绑attn_implementation="flash_attention_2" 走 FlashAttention-2，CUDA 专属。昇腾 910B 上需改 "sdpa"（torch SDPA，torch_npu 会路由到 aclnnFlashAttention / npu_fusion_attention）或 "eager"。MindSpeed-MM 用 NPU 专用注意力实现可作对照。这一行不改，模型装载阶段就会在 NPU 上报算子不支持，是整条迁移的第一个硬卡点。
NPU 迁移 · accelerate 设备路由accelerate 默认按 CUDA 语义取 device/sync。在 NPU 上要么 accelerate config 选 torch_npu 后端并把 device 设为 npu:0，要么直接迁到 MindSpeed 训练引擎（不走 accelerate）。accelerator.prepare 在多卡时会注入 DDP wrapper，其后端必须是 hccl 而非 nccl——这是单机多卡段的硬前置。单卡迁移可先验证 accelerate + torch_npu 的 device 路由，多卡再换 HCCL。
mixed_precision="bf16" 在 910B 上无风险：bf16 原生支持，不需要 loss scaler。这一点与 FP16 路径不同，迁移时不必引入 GradScaler。

Collate：16 层 codec 的 batch 布局

collate_fn 在 finetuning/dataset.py，把一个 batch 的样本拼成 7 个对齐张量。关键事实：Qwen3-TTS 的 codec 是 16 层残差——__getitem__ 返回的 audio_codes 形状是 (t_codec, 16)，所以 batch 里 codec_ids 是 (b, t, 16)；而 input_ids 是 (b, t, 2)，通道 0 放文本 token，通道 1 放 codec 控制符。t 取 batch 内最长样本的 text_len + codec_len 再加 8 个固定前缀位。

finetuning/dataset.py · L149–L159
固定源码 ↗
        item_length = [b['text_ids'].shape[1] + b['audio_codes'].shape[0] for b in batch]
        max_length = max(item_length) + 8
        b,t = len(batch),max_length

        input_ids   = torch.zeros((b,t,2),dtype=torch.long)
        codec_ids   = torch.zeros((b,t,16),dtype=torch.long)
        text_embedding_mask     = torch.zeros((b,t),dtype=torch.bool)
        codec_embedding_mask    = torch.zeros((b,t),dtype=torch.bool)
        codec_mask      = torch.zeros((b,t),dtype=torch.bool)
        attention_mask  = torch.zeros((b,t),dtype=torch.long)
        codec_0_labels  = torch.full((b, t), -100, dtype=torch.long)

这 7 个张量：input_ids (b,t,2) long、codec_ids (b,t,16) long、text_embedding_mask (b,t) bool、codec_embedding_mask (b,t) bool、codec_mask (b,t) bool（标记哪些位置是真 codec）、attention_mask (b,t) long、codec_0_labels (b,t) long 填 -100（只对第 0 层 codec 算 CE，其余位置被忽略）。collate 全程在 CPU 张量上做索引填充——这是数据管线，本来就在 DataLoader 的 worker 进程里跑，不占 NPU。

通道 1 的 [3:8] 是 5 个控制符：codec_nothink_id、codec_think_bos_id、codec_think_eos_id、位置 6 留给 speaker（codec_embedding_mask[6] = False，前向会在那里把 speaker 向量写进去）、codec_pad_id。之后从 8+text_len-1 起放真正的第 0 层 codec 序列 audio_codec_0，末尾接 codec_eos_token_id。codec_0_labels 与通道 1 的 codec 段同位，只是用 -100 屏蔽非 codec 位置。

speaker 注入位 = index 6。这个位置约定来自 collate 与前向的协同：collate 在通道 1 预留位 6（codec_embedding_mask[6]=False），前向 input_codec_embedding[:,6,:] = speaker_embedding 在此写入。迁移时这个位置约定不能改，否则 speaker 注入错位、音色定制失效。
NPU 迁移 · mel/STFT 留 CPUmel_spectrogram（dataset.py:106-115）用 n_fft=1024, hop=256, num_mels=128, fmax=12000，24kHz，底层是 torch STFT 类算子。它在 __getitem__ 里跑、属 CPU 数据管线。建议保持 CPU（与 cosyVoice2 STFT 钉 CPU 同一判断：CANN 的 STFT 复数路径不稳，且这里是预处理、对吞吐不敏感）。不需上 NPU。

前向①：embedding 汇总与 speaker 注入

进入循环后，每个 step 在 accelerator.accumulate(model) 上下文里跑（grad 累积 4 步才 step 一次）。前向第一件事：把 ref 音频的 mel 过 speaker_encoder 得到 speaker 向量（.detach() 不参与训练），再把它写入 codec embedding 的第 6 位。然后把 text/codec/speaker 三路 embedding 加总成 input_embeddings (b,t,d)，再逐层累加第 1-15 层残差 codec 的 embedding。

逐行看前向组装。L82 speaker_embedding = model.speaker_encoder(ref_mels.to(model.device).to(model.dtype)).detach()：ref_mels 搬到模型 device/dtype，过 speaker_encoder，detach() 使其不回传梯度。

L83-84 首步把结果存进全局 target_speaker_embedding——这个值在保存阶段会被注入 codec_embedding.weight[3000]，是音色定制的载体。

L86-87 input_text_ids = input_ids[:,:,0]、input_codec_ids = input_ids[:,:,1] 拆双通道。L89-90 text_embedding 与 codec_embedding 各自查表再乘各自掩码（把 pad 位清零）。

L91 input_codec_embedding[:,6,:] = speaker_embedding 把 speaker 写进 collate 预留的位 6。L93 input_embeddings = text + codec 两路相加——同维投影后相加，等价于双路融合。

L95-98 for i in 1..15：用 code_predictor.get_input_embeddings()[i-1] 对 codec_ids[:,:,i] 查表（16 层残差的第 1-15 层，每层独立嵌入表），逐层乘 codec_mask 后累加进去。所以第 0 层 codec 走 input_ids 通道 1 的 codec_embedding，第 1-15 层走 code_predictor 的独立嵌入表——16 层残差结构在 embedding 阶段就体现出来了。

NPU 迁移 · .to(model.device)ref_mels.to(model.device)：device 在 accelerate 包裹后是 NPU 时，这里把 mel 搬到 NPU，speaker_encoder 是 conv/linear，NPU 可跑。注意 ref_mel 是 __getitem__ 已算好的 mel 张量，搬 NPU 不涉及 STFT（mel 已算完），所以与上条“STFT 留 CPU”不冲突。

前向②：Talker 前向与 MTP sub-talker 损失

embedding 汇总后喂给 talker，标签是第 0 层 codec 的下一 token（输入与标签各 shift 1，标准 next-token 设定）。除了主 talker 的 CE 损失，还从 hidden states 里用 codec_mask 挑出真 codec 位置过 sub-talker，算 MTP（多 token 预测）辅助损失，最终 loss = outputs.loss + 0.3 * sub_talker_loss。

finetuning/sft_12hz.py · L100–L113
固定源码 ↗
                outputs = model.talker(
                    inputs_embeds=input_embeddings[:, :-1, :],
                    attention_mask=attention_mask[:, :-1],
                    labels=codec_0_labels[:, 1:],
                    output_hidden_states=True
                )

                hidden_states = outputs.hidden_states[0][-1]
                talker_hidden_states = hidden_states[codec_mask[:, :-1]]
                talker_codec_ids = codec_ids[codec_mask]

                sub_talker_logits, sub_talker_loss = model.talker.forward_sub_talker_finetune(talker_codec_ids, talker_hidden_states)

                loss = outputs.loss + 0.3 * sub_talker_loss

L100-105 model.talker(inputs_embeds=…[:,:-1,:], attention_mask=…[:,:-1], labels=codec_0_labels[:,1:], output_hidden_states=True)：输入砍最后一位、标签砍第一位，实现“用第 t 位预测第 t+1 位”的 next-token 设定。

outputs.loss 是 talker 对第 0 层 codec 的 CE。L107 hidden_states = outputs.hidden_states[0][-1] 取最后一层隐状态。

L108-109 talker_hidden_states = hidden_states[codec_mask[:,:-1]]、talker_codec_ids = codec_ids[codec_mask]：用 mask 把真 codec 位置从二维序列拉平成一维，只保留有效 token。

L111 这两个一维序列交给 model.talker.forward_sub_talker_finetune(talker_codec_ids, talker_hidden_states)，返回 sub_talker_logits 与 sub_talker_loss。L113 最终 loss = outputs.loss + 0.3 * sub_talker_loss——主 CE 与 MTP 辅助损失按 0.3 权重组合。

NPU 迁移 · talker 注意力talker 内部 attention 走的还是装载时设的 FA2 实现。改 attn_implementation 会一次性影响 talker 与 code_predictor 的所有 attention。换 sdpa 后需验证：1) 数值与 FA2 对齐（bf16 下允许合理误差，建议用小样本做 max-abs diff）；2) output_hidden_states 仍能取到隐状态；3) forward_sub_talker_finetune 内部若有自定义 attn 也一并改。
0.3 这个 MTP 权重是 Qwen3-TTS 官方 SFT 配方里的固定值，不是可调超参意义上的“经验值”——迁移时保持原值，先复现再调。

深读：16 层残差 codec 与两个损失的分工

把贯穿全篇的 16 层 codec 收束一下。Qwen3-TTS 的语音 codec 是残差结构：把语音压成 16 层离散整数，第 0 层是粗粒度主码本，第 1-15 层是逐层残差修正。这个结构在脚本里体现在三处：collate 产出 codec_ids (b,t,16)；前向 embedding 阶段第 0 层走 codec_embedding、第 1-15 层走 code_predictor.get_input_embeddings()[i-1]；损失阶段主 talker 只对第 0 层算 CE（codec_0_labels），残差层由 sub-talker 的 MTP 损失负责。

这样设计的好处：主 talker 只学预测第 0 层的下一步（与语言模型 next-token 一致），负担轻；残差层的修正由 sub-talker 在真 codec 位置（codec_mask 挑出）单独监督，不污染主序列的位置语义。迁移到 NPU 时，forward_sub_talker_finetune 的内部 forward（在 qwen_tts/core/models/modeling_qwen3_tts.py）要整体核验算子，但它仍是 conv/linear/attention 组合，没有特殊 CUDA 算子。
