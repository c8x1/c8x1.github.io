学习轨道：入门教程

包结构与推理 API

从 qwen_tts 包目录逐层拆到推理入口：__init__ 导出什么、from_pretrained 如何用 AutoConfig/AutoModel/AutoProcessor 注册并装载、Qwen3TTSModel 的三个 generate_* 公开 API 表面与输入归一化管线、12Hz Base 单说话人音色克隆的最小入口。每段标出 FA2 硬绑、device 路由、speech_tokenizer/processor 装载在昇腾 910B 的注意点。

学习目标

  • 画出 qwen_tts 包目录与各模块职责，说出 __init__ 导出的三个公开符号
  • 按源码顺序讲清 from_pretrained 的三步：AutoConfig.register → AutoModel.register/AutoProcessor.register → AutoModel.from_pretrained 与 AutoProcessor.from_pretrained，并说明三处 register 为何不可省
  • 列出 Qwen3TTSModel 的三个 generate_* 公开 API 及各自绑定的 tts_model_type 守卫、返回 (List[np.ndarray], int) 的契约
  • 把 12Hz Base 单说话人音色克隆的最小调用链从 examples/test_model_12hz_base.py 走一遍，标出 device/FA2/processor 装载的迁移点

前置知识

  • 熟悉 HuggingFace AutoConfig/AutoModel/AutoProcessor 三件套
  • 理解 dataclass 与 @torch.no_grad 装饰器
  • 知道 TTS 的 codec 声码器概念

总览：包目录与装载调用链

Qwen3-TTS 的 Python 包叫 qwen_tts，顶层只有两个文件 + 两个子包。对迁移者而言，包结构决定你要读哪些模块，而 from_pretrained 的装载顺序决定迁移时第一个卡点在哪。这一页只讲推理侧的装载与 API 表面，不展开 talker 前向——那是 SFT 循环页的主题。先把目录与调用链一次摆清，后面再逐段拆。

inference/ 子包是迁移的真正入口：qwen3_tts_model.py（877 行）定义 Qwen3TTSModel 这个 HF 风格 wrapper，qwen3_tts_tokenizer.py（410 行）定义独立的 Qwen3TTSTokenizer。注意这俩 wrapper 都不是 nn.Module——Qwen3TTSModel 持有 self.model（真正的 Qwen3TTSForConditionalGeneration）与 self.processor，自己只做输入归一化、token 化、调用转发、输出切片。真正的算子全在 core/models 下，wrapper 层不引入新算子，所以迁移时 wrapper 几乎零改动，重心在装载参数与 core 模型。

from_pretrained：三处 register 与装载顺序

Qwen3TTSModel.from_pretrained 是整条推理装载的核心。它做三件事：先向 HF 的 AutoConfig/AutoModel/AutoProcessor 三张注册表注册 Qwen3 自家的 config/model/processor 类，再用 AutoModel.from_pretrained 装载模型、AutoProcessor.from_pretrained 装载 processor，最后把 model.generate_config 当默认生成参数塞进 wrapper。顺序不可调：register 必须在 AutoModel.from_pretrained 之前，否则 HF 不认识 model_type: qwen3_tts，会按默认类装载、类型断言失败。

逐行往下看。L108-110 三行 register：把字符串 "qwen3_tts" 绑到 Qwen3TTSConfig，再把这个 config 类绑到模型类 Qwen3TTSForConditionalGeneration 与 processor 类 Qwen3TTSProcessor。这三处 register 的语义：HF 的 Auto 体系靠 config.json 里的 model_type 字段反查注册表。没注册，AutoModel.from_pretrained 读到 model_type: qwen3_tts 会回退到默认实现，权重大概率对不上。

L112 model = AutoModel.from_pretrained(pretrained_model_name_or_path, **kwargs)：用户传的 device_map、dtype、attn_implementation 全在 **kwargs 里原样转发。这意味着FA2 与 device 的设置发生在此行，是迁移第一硬卡点。L113-116 isinstance 断言装载出来确实是 Qwen3TTSForConditionalGeneration，否则 TypeError——这是 register 失败时的兜底报错（注册表没命中就会落到别的类，这里立刻发现）。

L118 AutoProcessor.from_pretrained(..., fix_mistral_regex=True)：processor 装载带一个非标参数 fix_mistral_regex，用来修分词器的正则。这个参数走的是 HF AutoProcessor 的 **kwargs，不是标准签名里的命名参数——迁移时若 torch_npu/MindSpeed 侧 processor 实现不同，要确认这个 fix 是否仍被消费。fix_mistral_regex 是纯 CPU 的正则修正，不涉及算子，NPU 上无风险，只是要保证 processor 类被正确注册（L110 已做）。

L120 generate_defaults = model.generate_config：这个 generate_config 来自 core 模型初始化时读 generate_config.json（在 modeling_qwen3_tts.py 的 load_generate_config），把 max_new_tokens、temperature 等默认值挂在 model 上。wrapper 把它存成 self.generate_defaults，后面 _merge_generate_kwargs 用它做用户值/文件默认/硬默认的三级合并。L121 cls(model=..., processor=..., generate_defaults=...) 进 __init__。

__init__（L70-80）里有一处device 推断值得迁移者盯紧：self.device = getattr(model, "device", None)，若为 None 再 next(model.parameters()).device，再不行落 cpu。这意味着 wrapper 的 self.device 直接继承自装载时的 device_map——在 NPU 上若装载时 device_map="npu:0"，这里就是 npu:0，后续 _tokenize_texts 的 .to(self.device) 会把 token 张量搬去 NPU。

    def __init__(self, model: Qwen3TTSForConditionalGeneration, processor, generate_defaults: Optional[Dict[str, Any]] = None):
        self.model = model
        self.processor = processor
        self.generate_defaults = generate_defaults or {}

        self.device = getattr(model, "device", None)
        if self.device is None:
            try:
                self.device = next(model.parameters()).device
            except StopIteration:
                self.device = torch.device("cpu")
NPU 迁移 · FA2 硬绑在装载行AutoModel.from_pretrained(..., attn_implementation="flash_attention_2")（L112 转发）把整条 talker 与 code_predictor 的注意力实现钉死成 FA2，CUDA 专属。昇腾 910B 上需改 "sdpa"（torch SDPA，torch_npu 路由到 aclnnFlashAttention/npu_fusion_attention）或 "eager"。这是推理侧第一个硬卡点——与 SFT 训练页同一处风险同源（都是装载时设的 attn_implementation）。不改这一行，模型装载阶段就会在 NPU 报算子不支持，根本到不了 generate。MindSpeed-MM 的 NPU 专用注意力实现可作对照。
NPU 迁移 · device_map 路由示例传 device_map="cuda:0"（example L50），NPU 上要换 "npu:0"。两个后果要一起验：1) AutoModel.from_pretrained 内部按 device_map 搬权重——torch_npu 要能识别 npu:0 这个 device 字符串（需 import torch_npu 在前）；2) wrapper __init__ 的 self.device 推断会拿到 npu:0，后续 token 与 mel 的 .to(self.device) 全跟这个走。若 device_map 不被 torch_npu 接受，退化成 cpu，整条生成会极慢或在 CPU 上跑（不算错但失去迁移意义）。
NPU 迁移 · processor 装载AutoProcessor.from_pretrained（L118）装的是分词器 + 特征提取器，纯 CPU 组件，本身无 NPU 算子。但要注意：processor 装载依赖 L110 的 AutoProcessor.register——若在迁移环境里 import 顺序变了、或 core 的 Qwen3TTSProcessor 没被提前 import，注册表里没有这一项，AutoProcessor 会静默回退到 HF 默认 processor，分词结果可能对不上。fix_mistral_regex 是正则修正，CPU 安全。建议迁移时显式确认 tts.processor.__class__.__name__ 是预期类。

推理 API 表面：三个 generate_*

Qwen3TTSModel 暴露三个生成入口，各自绑一个 tts_model_type 守卫：Base 模型走 generate_voice_clone + create_voice_clone_prompt，VoiceDesign 走 generate_voice_design，CustomVoice 走 generate_custom_voice。三者返回类型统一：Tuple[List[np.ndarray], int]（波形列表 + 采样率）。守卫是硬拒绝——模型类型对不上直接 ValueError，不是软回退。

看 generate_voice_clone 的守卫。L548-554 if self.model.tts_model_type != "base": raise ValueError：模型 tts_model_type 来自 config（core 模型初始化时 self.tts_model_type = self.config.tts_model_type，在 modeling_qwen3_tts.py:1839）。所以同一个 wrapper 类，根据装载的 checkpoint 是 Base 还是 CustomVoice，决定能调哪个 API。迁移时这个守卫纯逻辑判断，与硬件无关，无需改。

    def _tokenize_texts(self, texts: List[str]) -> List[torch.Tensor]:
        input_ids = []
        for text in texts:
            input = self.processor(text=text, return_tensors="pt", padding=True)
            input_id = input["input_ids"].to(self.device)
            input_id = input_id.unsqueeze(0) if input_id.dim() == 1 else input_id
            input_ids.append(input_id)
        return input_ids

L601 gen_kwargs = self._merge_generate_kwargs(**kwargs) 是三级默认合并：用户显式传的值优先，否则取 generate_config.json 里的值，再否则用硬默认（do_sample=True, top_k=50, temperature=0.9, max_new_tokens=2048 等）。L603-610 才真正把组装好的 input_ids、voice_clone_prompt、languages 喂给 self.model.generate(...)——从这一行起进入 core 模型算子，不在本页范围。

generate 尾段 L612-633 是ref code 拼接与裁切：ICL 模式下生成的 codec 前面拼上 ref 的 codec 一起 decode，再按 ref 长度占比把 ref 段从最终 wav 切掉。这段在 CPU 上做 torch.cat 与切片索引，speech_tokenizer.decode 才回到 NPU。shape 上 codes (T_gen, Q) cat ref_code (T_ref, Q) 得 (T_ref+T_gen, Q)，decode 出 wav (W,) 后按 cut = T_ref/(T_ref+T_gen) * W 切前段。

NPU 迁移 · token .to(self.device)input_ids.to(self.device)（L282）把分词后的 long 张量搬去模型 device。NPU 上若 self.device=npu:0 这是一次小张量 H2D 搬运，无风险。但前提是 self.device 推断正确——若 from_pretrained 时 device_map 不被 torch_npu 识别，self.device 退化成 cpu，token 留 CPU，后面 model.generate 在 NPU 上会因输入在 CPU 报 device mismatch。所以迁移第一件事是确认 print(tts.device) 是 npu:0 而非 cpu。
NPU 迁移 · speech_tokenizer encode/decodeself.model.speech_tokenizer.encode（在 create_voice_clone_prompt L427）与 .decode（L620）是声码器/codec 模型，独立于 talker 的 attn_implementation，它装载时用的是 Qwen3TTSTokenizer 的参数（docstring 建议默认 attn_implementation="eager"，见 qwen3_tts_tokenizer.py:73）。迁移时两条注意力路径要分别验证：talker 改 sdpa，speech_tokenizer 可能本来就是 eager（安全）但也可能内部用了 FA——需读 core tokenizer 模型确认。decode 末尾 .cpu().numpy()（tokenizer L364）已落 CPU，声码器输出回 CPU 无 NPU 残留。

create_voice_clone_prompt：ref 音频到 prompt item

create_voice_clone_prompt 是 Base 模型独有的预处理：吃 ref 音频，产出 VoiceClonePromptItem 列表，里面装 ref_code（codec）、ref_spk_embedding（说话人向量）、两个模式开关、ref_text。它把“建 prompt”与“生成”拆成两步，让一个 prompt 可以复用到多条合成文本（example 里 case1b/2b/3b 就是这种用法）。
