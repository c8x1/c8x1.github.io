学习轨道：入门教程

12Hz Tokenizer 契约

逐段拆解 Qwen3-TTS 的 12Hz speech tokenizer：wav→token 的输入输出契约（24kHz 波形 → 12.5Hz×16 层残差码本）、encoder 复用 transformers Mimi、decoder 自研 pre_transformer+RVQ+ConvNeXt+SnakeBeta 上采样、它在 SFT 与推理流水线里作为 speech_tokenizer 的位置，以及帧率换算、STFT/mel/注意力/RVQ argmin 在昇腾 910B 上的迁移风险。源码固定在 QwenLM/Qwen3-TTS@022e286b。

学习目标

  • 说清 12Hz tokenizer 的 wav→token 契约：input_values (b,t) → audio_codes (T,16)，帧率由 input_sample_rate / encode_downsample_rate = 24000/1920 决定
  • 区分 encoder（复用 transformers MimiModel，conv encoder + RVQ）与 decoder（自研 pre_transformer 滑窗因果 Transformer + SplitRVQ + 上采样 ConvNeXt/SnakeBeta）的职责边界
  • 解释 12Hz tokenizer 与 25Hz tokenizer 的关系：同命名族、同 downsample_rate=1920、不同 decoder 结构与条件输入（25Hz 需 xvector+ref_mel，12Hz 仅 audio_codes）
  • 定位它在 SFT（sft_12hz.py init Qwen3-TTS-12Hz-Base）与推理（speech_tokenizer.encode/decode）流水线里的位置，标出 910B 迁移的算子级风险点

前置知识

  • 理解残差向量量化（RVQ）的多码本概念
  • 读过 Qwen3-TTS 包结构与 SFT 训练循环
  • 熟悉 Conv1d/ConvTranspose1d 的 stride 与上/下采样关系

总览：12Hz tokenizer 的契约与位置

12Hz speech tokenizer 是 Qwen3-TTS 把连续语音波形压成离散 token 的那一层，对应类 Qwen3TTSTokenizerV2Model（在 qwen_tts/core/tokenizer_12hz/）。它的契约很窄：encode 吃 (b, t) 的 24kHz 波形，吐 List[(T_i, 16)] 的残差码本索引；decode 吃 (b, T, 16) 的码本索引，吐 24kHz 波形 List[(samples_i,)]。帧率不是字面 12，而是 input_sample_rate / encode_downsample_rate = 24000 / 1920 = 12.5 Hz——12Hz 是品牌命名。这个 tokenizer 在 SFT 与推理里都通过 sft_12hz.py 的 --init_model_path Qwen/Qwen3-TTS-12Hz-1.7B-Base 与推理模型的 self.model.speech_tokenizer 接入，是 talker 之外唯一与音频物理量打交道的大模块。

把整条 tokenizer 当作两个可独立替换的子模块。encoder 是复用 transformers 的 MimiModel（Qwen3TTSTokenizerV2Encoder(MimiModel)），只把不需要的 upsample/decoder_transformer/decoder 置空，保留 Mimi 的卷积特征提取器与残差向量量化器——这意味着 encoder 侧的算子风险几乎全部落在 Mimi 实现里。decoder 是自研的 Qwen3TTSTokenizerV2Decoder：码本先过 SplitResidualVectorQuantizer 解码回连续表示，再过 pre_transformer（8 层滑窗因果 Transformer）、逐级 ConvTranspose1d 上采样 + ConvNeXt block、最后 SnakeBeta 激活 + Conv1d 出波形。两端用 encode_downsample_rate / decode_upsample_rate = 1920 这一对常数对齐帧率。

输入输出契约：帧率与张量形状

契约的核心是帧率换算。Qwen3TTSTokenizerV2Config 在 configuration_…_v2.py 里写死四个常数：input_sample_rate=24000、output_sample_rate=24000、encode_downsample_rate=1920、decode_upsample_rate=1920。帧率 = 输入采样率 / 下采样率 = 24000 / 1920 = 12.5 Hz，即每秒语音约 12.5 帧 token，每帧由 16 个码本索引组成（encoder_valid_num_quantizers=16）。反方向：1 帧 token 对应 1920 个波形采样点，decode 用 audio_lengths = (codes[...,0] > -1).sum(1) * decode_upsample_rate 把有效帧数还原成采样点数。

encode 的返回形状在 docstring 与代码里都明确：audio_codes 是 List[torch.LongTensor]，每条样本形状 (codes_length_i, num_quantizers) = (T_i, 16)——时间维在前、16 层码本在后。注意它是 list 不是单张量，因为各样本时长不同、未做 batch padding。encode 内部先把 (b,t) 波形 unsqueeze 成 (b,1,t) 喂给 Mimi encoder，拿到 Mimi 产出的全量码本（层数多于 16）后再切片 [:, :encoder_valid_num_quantizers] 只取前 16 层，最后按 padding_mask 把每条样本的有效帧数 -(-mask.sum() // encode_downsample_rate) 截断——这是把 Mimi 内部帧数对齐到 tokenizer 帧率的关键一步。

    def encode(
        self,
        input_values: torch.Tensor,
        padding_mask: Optional[torch.Tensor] = None,
        return_dict: Optional[bool] = None,
    ) -> Union[tuple[torch.Tensor, Optional[torch.Tensor]], Qwen3TTSTokenizerV2EncoderOutput]:
        """
        Encodes the input audio waveform into discrete codes.

        Args:
            input_values (`torch.Tensor` of shape `(batch_size, sequence_length)`):
                Float values of the input audio waveform.
            padding_mask (`torch.Tensor` of shape `(batch_size, sequence_length)`):
                Indicates which inputs are to be ignored due to padding, where elements are either 1 for *not masked* or 0
                for *masked*.
            return_dict (`bool`, *optional*):
                Whether or not to return a [`~utils.ModelOutput`] instead of a plain tuple.
        """
        return_dict = return_dict if return_dict is not None else self.config.return_dict

        encoded_frames = self.encoder.encode(input_values=input_values.unsqueeze(1),
                                             return_dict=True)
        audio_codes = encoded_frames.audio_codes[:, :self.encoder_valid_num_quantizers]
        audio_codes = [code[..., :-(-mask.sum() // self.encode_downsample_rate)].transpose(0, 1) for code, mask in zip(audio_codes, padding_mask)]

        if not return_dict:
            return (
                audio_codes,
            )

        return Qwen3TTSTokenizerV2EncoderOutput(audio_codes)

decode 是反向契约。输入 audio_codes (b, codes_length, num_quantizers) 即 (b, T, 16)。先 (codes[...,0] > -1).sum(1) * decode_upsample_rate 算每条目标波形采样点数（-1 是 pad 值，第 0 层有效即整帧有效）；torch.clamp(audio_codes, min=0) 把 pad 的 -1 钳到 0 再送量化器查表（避免负索引越界）；decoder.chunked_decode(audio_codes.transpose(1,2)) 把 (b,T,16) 转成 (b,16,T) 喂解码器，输出再 squeeze(1)，最后按 audio_lengths 逐条截断成 List[(samples_i,)]。transpose 这一步把通道维换到 dim=1，因为 decoder 的 RVQ/卷积都按 (b, C, T) 组织。

NPU 迁移 · 帧数对齐整除-(-mask.sum() // encode_downsample_rate) 是向上取整除法的经典写法，纯整数算子在 CPU 上算 padding_mask 的 sum。这一步发生在 encode 返回前的 list 推导里，不占 NPU 算子，迁移无风险。decode 侧 (codes[...,0] > -1).sum(1) * decode_upsample_rate 同理是 mask 级整数运算。只要 padding_mask 来自 feature_extractor 且 dtype 对，整除逻辑可直接照搬。

Encoder：复用 transformers Mimi

encoder 是整页最省事也最需要警惕迁移的一段。Qwen3TTSTokenizerV2Encoder 直接继承 transformers 的 MimiModel，__init__ 里把 upsample、decoder_transformer、decoder 三个属性显式置 None——相当于只保留 Mimi 的编码半边（卷积特征提取器 + 量化器），把解码半边废掉，因为解码走自研 decoder。Mimi 本身是 Kyutai 开源的语义+声学 RVQ 编码器，transformers 实现里含 mel/STFT 前端、encoder conv 栈、semantic/acoustic 分支 RVQ。

encode 顶层把 input_values.unsqueeze(1) 成 (b,1,t) 交给 self.encoder.encode（Mimi 的 encode），拿 encoded_frames.audio_codes。Mimi 内部产出的码本层数 ≥ 16，这里 [:, :self.encoder_valid_num_quantizers] 只取前 16 层——encoder_valid_num_quantizers=16 由 config 写死。所以 encoder 的算子风险不在本文件，而在 transformers 的 MimiModel 实现：mel/STFT 前端、encoder 的 depthwise/grouped conv、以及 RVQ 的码本距离 argmin。

NPU 迁移 · Mimi mel/STFT 前端Mimi 的音频前端含 STFT/mel 类算子（与 cosyVoice2 的 STFT 同族）。在 910B 上 CANN 的复数 STFT 路径不稳，保守方向是这一段钉在 CPU：encode 属于数据预处理/特征提取，对吞吐不敏感，与 SFT 页 mel/STFT 留 CPU 同一判断。若要整体上 NPU，需逐算子核验 transformers MimiModel 的 encode 前端，优先把 STFT 切 CPU、conv/linear 部分可留 NPU。这一结论待真机验证。
NPU 迁移 · RVQ argmin 量化Mimi encoder 内部 RVQ 在训练/编码时对每个连续向量算到各码本条目的欧氏距离并取 argmin 选码字——是 cdist+argmin 组合，算子密集。910B 上大批 argmin 与广播距离计算可能低效但可跑；保守方向是先验证数值对齐（与 CPU 码本逐帧比 max-abs diff），若性能不达标再考虑自定义 kernel 或把量化步拆到 CPU。注意：本文件的 EuclideanCodebook 只实现了 decode（查表），encode 侧 argmin 在 Mimi 内部，迁移时要去看 transformers 源码而非本文件。

Decoder：pre_transformer + RVQ + 上采样

decoder 是自研的 Qwen3TTSTokenizerV2Decoder，自下而上四段：quantizer（SplitResidualVectorQuantizer）把码本索引解码成连续表示；pre_conv 一个 1D 卷积把 codebook_dim 投到 latent_dim 并转置成 (b, latent, T)；pre_transformer（Qwen3TTSTokenizerV2DecoderTransformerModel，8 层滑窗因果 Transformer）做时序建模；upsample 逐级 ConvTranspose1d+ConvNeXt block 上采样；decoder 末段 ConvNeXt 残差块 + SnakeBeta + Conv1d 出单通道波形，最后 clamp(-1,1)。

forward(codes) 的形状流：输入 codes (b, T, 16)（经外层 transpose 后进来的）→ quantizer.decode 得 (b, latent, T) → pre_conv 转 (b, latent, T) 再 transpose(1,2) 成 (b, T, latent) → pre_transformer(inputs_embeds=…) 取 last_hidden_state (b, T, hidden) → permute(0,2,1) 回 (b, hidden, T) → 逐级 upsample block → 末段 decoder 出 (b, 1, samples) → clamp(-1,1)。注意 pre_transformer 内部又做一次 input_proj/output_proj（latent↔hidden），形状在 latent_dim 与 hidden_size 间来回切。

    def forward(self, codes):
        if codes.shape[1] != self.config.num_quantizers:
            raise ValueError(f"Expected {self.config.num_quantizers} layer of codes, got {codes.shape[1]}")

        hidden = self.quantizer.decode(codes)
        hidden = self.pre_conv(hidden).transpose(1, 2)

        hidden = self.pre_transformer(inputs_embeds=hidden).last_hidden_state
        hidden = hidden.permute(0, 2, 1)
        for blocks in self.upsample:
            for block in blocks:
                hidden = block(hidden)
        wav = hidden
        for block in self.decoder:
            wav = block(wav)
        return wav.clamp(min=-1, max=1)

chunked_decode 是工程上的关键：它把长序列切成 chunk_size=300 帧、带 left_context_size=25 帧左上下文逐块解码，输出按 total_upsample 倍数对齐裁剪再 cat。这是为了控显存：decoder 的 ConvTranspose 上采样把 T 放大 ~1920 倍成 samples，整段一次过会爆显存。分块解码对数值无影响（因果卷积 + 滑窗注意力的左上下文保证块间连续），但要求每块的左上下文正确拼接。迁移到 NPU 时这个分块逻辑可直接照搬，块内算子才是风险所在。

NPU 迁移 · 滑窗因果注意力pre_transformer 的 8 层全是 attention_type="sliding_attention"（layer_types 返回全 sliding），窗口 sliding_window=72。注意力走 Qwen3TTSTokenizerV2DecoderAttention，forward 里按 _attn_implementation 选 eager 或 ALL_ATTENTION_FUNCTIONS 里的实现。预训练权重默认 FA2（_supports_flash_attn=True）。迁移方向：改 attn_implementation="sdpa"，torch_npu 会路由到 npu_fusion_attention；需验证 sliding window 语义在 sdpa 路径下保留（create_sliding_window_causal_mask 产出的 mask 仍生效）。这一段是 decoder 在 NPU 上最可能卡的算子。
NPU 迁移 · SnakeBeta 与 ConvTransposedecoder 的激活 SnakeBeta 用 torch.sin+torch.pow 实现（x + 1/b * sin²(xα)），是逐元素算子，NPU 原生可跑但要注意 torch.exp 的数值范围。ConvTranspose1d（Qwen3TTSTokenizerV2CausalTransConvNet）是标准转置卷积，NPU 支持；但它带手算的 right_pad 裁剪（hidden[..., : len - right_pad]），属逻辑切片无风险。保守方向：这两类算子整体低风险，迁移时主要核验 SnakeBeta 的 sin/pow 在 bf16 下不溢出（建议用小样本对 max-abs diff）。

深读：SplitRVQ 的语义/声学分治

decoder 侧的量化器是 SplitResidualVectorQuantizer，把 16 层码本拆成语义层（n_q_semantic=1，第 0 层）与声学层（n_q_acoustic=15，第 1-15 层）两组，各走一个 ResidualVectorQuantizer。decode 时先对前 1 层解码、再对后 15 层解码，两者相加。这与 SFT 页讲的第 0 层 codec 走主 CE、第 1-15 层走 sub-talker MTP 是同源设计：语义层粗粒度、声学层逐层残差修正，tokenizer 的码本结构与 talker 的损失分工一一对应。

class SplitResidualVectorQuantizer(nn.Module):
    """Residual Vector Quantizer with separate projections for the first quantizer and the rest.

    Args:
        n_q (int): Number of residual vector quantizers used.
        n_semantic_q (int): Number of residual vector quantizers used for the semantic quantizer.
        **kwargs: Arguments to the constructor of `ResidualVectorQuantizer` that are shared between both.
    """

    def __init__(
        self,
        *,
        n_q: int = 8,
        n_q_semantic: int = 1,
        **kwargs,
    ):
        super().__init__()
        assert n_q > n_q_semantic, (
            f"Number of quantizers {n_q} must be larger "
            f"than the number of semantic quantizers {n_q_semantic}."
        )
        self.max_n_q = n_q
        self.n_q_semantic = n_q_semantic
        self.n_q_acoustic = n_q - n_q_semantic
        q_dropout = kwargs.pop("q_dropout", False)
        self.rvq_first = ResidualVectorQuantizer(
            n_q=n_q_semantic, force_projection=True, q_dropout=False, **kwargs
        )
        self.rvq_rest = ResidualVectorQuantizer(
            n_q=n_q - n_q_semantic,
            force_projection=True,
            q_dropout=q_dropout,
            **kwargs,
        )

    def decode(self, codes: torch.Tensor) -> torch.Tensor:
        """Decode the given codes to the quantized representation."""
        # codes is [B, K, T], with T frames, K nb of codebooks.
        quantized = self.rvq_first.decode(codes[:, : self.n_q_semantic])
        if codes.shape[1] > self.n_q_semantic:
            quantized += self.rvq_rest.decode(codes[:, self.n_q_semantic :])
        return quantized

ResidualVectorQuantizer.decode 把 (b, T, 16) transpose 成 (16, b, T)，逐层调 VectorQuantization.decode → EuclideanCodebook.decode：embedding_sum / cluster_usage.clamp(min=eps) 算出实际码本表，F.embedding(codes, embedding) 查表得 (b, T, codebook_dim)，再 project_out + transpose 成 (b, dim, T)，逐层累加。encode 侧的 argmin/EMA 更新不在本文件（EuclideanCodebook 只实现了 decode，cluster_usage/embedding_sum 是训练期 EMA 统计量），所以 decoder 这段在 NPU 上几乎全是查表 + 线性投影 + 累加，风险低。

NPU 迁移 · F.embedding 查表F.embedding(codes, embedding) 是纯索引查表，NPU 原生支持。clamp(audio_codes, min=0) 已把 pad 的 -1 钳成合法索引，查表不会越界。风险点在 embedding_sum / cluster_usage.clamp(min=eps) 这一步除法——bf16 下除法精度有限，但这里 cluster_usage 是训练期定下来的常数（推理时不变），可考虑预先在 CPU 把码本表算好再搬 NPU，避免运行时除法。属优化项而非硬卡点。

与 25Hz tokenizer 的关系

Qwen3-TTS 同族还有 25Hz tokenizer（qwen_tts/core/tokenizer_25hz/，类 Qwen3TTSTokenizerV1）。两者在帧率常数上完全相同：25Hz 的 Qwen3TTSTokenizerV1Config 同样写 input_sample_rate=24000、encode_downsample_rate=1920——即 25Hz 的实际帧率算下来也是 12.5Hz。25Hz / 12Hz 是品牌命名而非精确帧率，二者真正的差异在 decoder 结构与 decode 的条件输入。

契约差异在推理 wrapper qwen_tts/inference/qwen3_tts_tokenizer.py 里写得最清楚：encode 返回时 25Hz 给 audio_codes + xvectors + ref_mels 三件套（含说话人 x-vector 与参考 mel），12Hz 只给 audio_codes；decode 时 25Hz 必须传 xvectors 与 ref_mels（raise ValueError(...)），12Hz 只传 audio_codes 即可。形状上 25Hz 单样本 (C,)、12Hz 单样本 (C, Q)——12Hz 多一个码本维。

在 SFT 与推理流水线中的位置

在 SFT 侧，12Hz tokenizer 不是训练对象，而是数据来源。finetuning/sft_12hz.py 的 --init_model_path 默认 Qwen/Qwen3-TTS-12Hz-1.7B-Base——这个 Base 模型内部就含 12Hz tokenizer 作为 speech_tokenizer。SFT 训练的是 talker/code_predictor/speaker_encoder，tokenizer 的权重在 SFT 里冻结（SFT 页的保存段还会把 speaker_encoder 整段删掉，tokenizer 权重原样保留）。训练样本里的 audio_codes (t_codec, 16) 就是预先用 12Hz tokenizer 的 encode 产出的码本索引。

NPU 迁移 · 冻结 tokenizer 上 NPUSFT 时 tokenizer 权重冻结、只做前向（encode 在数据预处理、decode 不参与 SFT）。encode 若走 NPU，风险集中在 Mimi 前端 STFT（见 risk-mimi-stft）。保守做法是 encode 在 CPU 数据管线完成（与 SFT 页 mel/STFT 留 CPU 一致），把码本索引入库当训练样本，SFT 循环里完全不动 tokenizer——这样 SFT 阶段 tokenizer 零 NPU 风险。推理阶段才需把 decode 整体迁 NPU，那时再逐算子核验。

配置项速查与迁移要点表

Qwen3TTSTokenizerV2DecoderConfig 的默认值是 decoder 结构的契约面：码本大小 2048、16 层量化、decoder 8 层 Transformer、hidden 1024、滑窗 72、上采样率 (8,5,4,3) 与 upsampling_ratios=(2,2)。total_upsample = np.prod(upsample_rates + upsampling_ratios) = 8*5*4*3*2*2 = 1920——正好等于 decode_upsample_rate，这是 1 帧 token = 1920 个采样点 的来源。下表把迁移点收束一下，源码全部 pin 在 022e286b。
