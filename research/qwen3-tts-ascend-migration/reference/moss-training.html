MOSS 训练验证

逐段拆 OpenMOSS/MOSS-TTS 的 SFT 脚本 moss_tts_delay/finetuning/sft.py（718 行）：accelerate + AdamW 的 setup/prepare、loss-mask 与多 VQ 通道权重的 collate、forward→backward→clip→step→zero 的标准循环、save_pretrained 每轮存盘——对照 Qwen3-TTS sft_12hz.py 的 SFT 循环与 MindSpeed-MM 的 Megatron 引擎，讲清“另一条同形态 accelerate TTS 训练脚本能多大程度佐证 Ascend 可训，以及它自身留在 CUDA 语义上的那些点”。

学习目标

  • 定位 MOSS SFT 的 setup、collate、训练循环、保存四段的行号，说出每段入口
  • 说清 collate 产出的 input_ids (b,t,n_vq+1) 多通道布局、loss_mask 的 prompt 屏蔽、与 channelwise_loss_weight 怎么把文本/音频头分开加权
  • 区分 MOSS 训练在 accelerate 上与 Qwen3-TTS 的同构处（accumulate/backward/clip/step）与异构处（save_pretrained vs 手写 safetensors、多 VQ 通道 loss vs 主 CE+MTP）
  • 指出 MOSS 脚本里仍写死 CUDA 的点（torch.cuda.is_available、enable_flash_sdp、audio_tokenizer.to(cuda)），标注这些在 NPU 上要走 torch_npu patch 还是必须改

前置知识

  • 读过 target/sft-training-loop.html，知道 accelerate 的 prepare/backward/accumulate 三段
  • 读过 reference/mm-training.html，知道 Megatron forward_backward 与 accumulate 语义差异
  • 理解多 VQ（residual/vector-quantized）codec 的通道结构与 next-token label shift

总览：MOSS 的 accelerate SFT 循环

MOSS-TTS 的 SFT 脚本 moss_tts_delay/finetuning/sft.py 718 行，结构与 Qwen3-TTS 的 sft_12hz.py 高度同构：都是单个入口 + accelerate 包住一个 for-epoch 循环，accelerator.prepare 一次性接管 model/optimizer/dataloader/scheduler，accelerator.accumulate 管累积，accelerator.backward 管反向。差别在细节：MOSS 的 collate 在多 VQ 通道上构造标签并按 channelwise_loss_weight 分头加权，保存走 save_pretrained(safe_serialization=True) 而非手写 safetensors。所以本页的价值是：它是同形态 accelerate TTS 训练的第二个样本——Qwen3-TTS 那条循环的 accelerate ownership 与 bf16/clip/step 惯例在 MOSS 这里被独立复现了一次，迁移到 Ascend 的结论可互相印证。

MOSS 的 codec 不是 16 层残差而是多 VQ（vector-quantized）：一条语音压成 (T, n_vq) 的整数张量，n_vq 个并行的码本通道，每通道独立预测下一 token。这决定了 input_ids 是 (b, t, n_vq+1) 三维——最后一维是“通道”，通道 0 是文本 token，通道 1..n_vq 是各层 VQ codec。collate 产出的 labels 同形，靠 loss_mask 屏蔽 prompt 段、靠 audio_pad_code 屏蔽 delay pattern 的结构性占位。模型 forward 内部对每个通道算 CE 并按 channelwise_loss_weight 加权求和，返回单个 outputs.loss——这条与 Qwen3-TTS 的“主 CE + 0.3*MTP”是两套不同的多通道损失设计。

Setup 与 Accelerate ownership

setup 段的顺序与 Qwen3-TTS 一致：先建加速器、构造数据/模型/优化器/调度器，再一次性 accelerator.prepare。关键差异在注意力实现是探测出来的，不是写死的：MOSS 默认 --attn-implementation auto，由 resolve_attn_implementation 在运行时按“flash_attn 是否装了 + GPU 算力 major>=8 + dtype 是否 fp16/bf16”决定走 flash_attention_2 还是 sdpa。这意味着它没有 Qwen3-TTS 那种“硬绑 FA2”的字面量——但 torch.cuda.is_available() 的分支判断仍是 CUDA 语义，在 NPU 上需 torch_npu 兜底。

def configure_torch_backends() -> None:
    if torch.cuda.is_available():
        torch.backends.cuda.enable_cudnn_sdp(False)
        torch.backends.cuda.enable_flash_sdp(True)
        torch.backends.cuda.enable_mem_efficient_sdp(True)
        torch.backends.cuda.enable_math_sdp(True)


def resolve_torch_dtype(mixed_precision: str) -> torch.dtype:
    if not torch.cuda.is_available():
        return torch.float32
    if mixed_precision == "fp16":
        return torch.float16
    if mixed_precision == "bf16":
        return torch.bfloat16
    return torch.float32


def resolve_attn_implementation(requested: str, dtype: torch.dtype) -> str:
    if requested != "auto":
        return requested
    if not torch.cuda.is_available():
        return "eager"
    if (
        importlib.util.find_spec("flash_attn") is not None
        and dtype in {torch.float16, torch.bfloat16}
    ):
        major, _ = torch.cuda.get_device_capability()
        if major >= 8:
            return "flash_attention_2"
    return "sdpa"

L146-151 configure_torch_backends 在 torch.cuda.is_available() 为真时一次性开 flash_sdp/mem_efficient_sdp/math_sdp、关 cudnn_sdp——这是 PyTorch SDPA 的四个后端开关，目的是让注意力优先走 flash 路径。L154-161 resolve_torch_dtype 同样以 torch.cuda.is_available() 为前提，没 CUDA 就回落 float32（不混合精度）。L164-176 resolve_attn_implementation：非 auto 直接返回；没 CUDA 返回 eager；装了 flash_attn 且 dtype 是 fp16/bf16 且 GPU major>=8 返回 flash_attention_2，否则 sdpa。这三段是整条脚本对 CUDA 的硬依赖——在 NPU 上 torch.cuda.is_available() 默认 False，会直接走 float32 + eager，性能与精度都错。

NPU 迁移 · torch.cuda.is_available 分支L146、L154、L167 三处 torch.cuda.is_available() 是 MOSS 脚本对设备能力的唯一探测，全部走 CUDA 语义。迁到 Ascend 有两条路：(1) 走 accelerate+torch_npu——torch_npu 的 monkey-patch 会让 torch.cuda.is_available() 返回 True（NPU 被映射成 cuda 设备），这三个分支会被“骗”进走 flash/sdpa/bf16，但 enable_flash_sdp 这类 torch.backends.cuda.* 开关在 NPU 上语义不同，建议显式跳过 configure_torch_backends；(2) 显式传 --attn-implementation sdpa + --mixed-precision bf16 并把 dtype 解析改成认 torch.npu.is_available()。两条都待真机验证，但(2)更可控、不依赖 patch 的边角行为。
NPU 迁移 · attn_implementation auto 探测L170 find_spec("flash_attn") + L173 torch.cuda.get_device_capability() 探测 FA2 是否可用。NPU 上 flash_attn 包不适用（它是 CUDA kernel），探测应直接判否、落 sdpa。关键是 sdpa 在 torch_npu 下会路由到 aclnnFlashAttention/npu_fusion_attention，与 Qwen3-TTS 迁移方向一致。建议显式传 --attn-implementation sdpa 跳过 auto 探测，避免在 NPU 上意外命中某个分支。这与 Qwen3-TTS 那条“FA2 硬绑”风险同源、但 MOSS 因为是探测式反而更容易绕过——只需显式指定。
MOSS 默认 mixed_precision="bf16"（L126），与 Qwen3-TTS 一致。910B 原生支持 bf16，无需 GradScaler——这条结论在两个脚本里互相印证，是迁移里少数“零风险”的点。

Collate：多 VQ 通道与 loss_mask

collate_fn 在 moss_tts_delay/finetuning/dataset.py，把一个 batch 拼成三个对齐张量。关键事实：MOSS 的 codec 是 多 VQ，_pack_record 返回的 input_ids 形状 (T, n_vq+1)——通道 0 是文本 token，通道 1..n_vq 是各层 VQ codec（不是 Qwen3-TTS 的 16 层残差）。collate 用 processor._pad 把 batch 内样本 pad 成 (b, t, n_vq+1)，labels 是 input_ids[:,1:,:] 砍第一位（next-token label shift，与 Qwen3-TTS 同约定），再用 loss_mask 与 attention_mask 双重屏蔽成 -100。

    def collate_fn(self, batch: List[Dict[str, torch.Tensor]]) -> Dict[str, torch.Tensor]:
        input_ids_list = [item["input_ids"] for item in batch]
        padded = self.processor._pad(input_ids_list)

        full_input_ids = padded["input_ids"].to(torch.long)
        full_attention_mask = padded["attention_mask"].bool()

        loss_masks = pad_sequence(
            [item["loss_mask"] for item in batch],
            batch_first=True,
            padding_value=False,
            padding_side="left",
        )

        labels = full_input_ids[:, 1:, :].clone()
        labels = labels.masked_fill(~loss_masks.unsqueeze(-1), -100)
        labels = labels.masked_fill(~full_attention_mask[:, 1:].unsqueeze(-1), -100)
        # Audio pad code is a structural placeholder from the delay pattern, not a trainable target.
        labels[:, :, 1:] = labels[:, :, 1:].masked_fill(
            labels[:, :, 1:] == self.processor.model_config.audio_pad_code,
            -100,
        )

        return {
            "input_ids": full_input_ids[:, :-1, :].contiguous(),
            "attention_mask": full_attention_mask[:, :-1].contiguous(),
            "labels": labels.contiguous(),
        }

逐行看 collate。L163-167 processor._pad(input_ids_list) 产出 input_ids (b,t,n_vq+1) long 与 attention_mask (b,t) bool。L169-174 pad_sequence(loss_mask, left) 把各样本的 loss_mask 左 pad（因为右对齐 pad，mask 要左对齐才能对上有效位）。

L176 labels = full_input_ids[:,1:,:].clone()——标签砍输入第一位，实现“用第 t 位预测第 t+1 位”的 next-token shift，这与 Qwen3-TTS 的 labels=codec_0_labels[:,1:] 同约定，只是 MOSS 是所有通道一起 shift而非只第 0 层。L177-178 两重 masked_fill(~..., -100)：先按 loss_mask 屏蔽 prompt 段（只对 assistant 段算 loss），再按 attention_mask 屏蔽 pad 位。

L180-183 是 MOSS 独有的一步：labels[:,:,1:]（只对音频通道）把等于 audio_pad_code 的位置也填 -100。注释解释得很清楚——audio_pad_code 是 delay pattern 产生的结构性占位（VQ 各层因 delay 错位产生的填充），不是可训练目标，必须从 loss 里剔除。这是 delay-pattern 多 VQ 训练的特有处理，Qwen3-TTS 的残差 codec 没有等价物。最终 L186-188 返回 input_ids[:,:-1,:]（输入砍最后一位）、attention_mask[:,:-1]、labels——输入与标签各砍一端，对齐成 next-token。

前向时 channelwise_loss_weight（main 里 L499-502 parse_channelwise_loss_weight 解析）决定各通道 CE 的权重：默认 "1,32" 表示文本头权重 1、音频头总权重 32，均摊到 n_vq 个音频通道。这条 resolved_channelwise_loss_weight 一路传到 model(..., channelwise_loss_weight=…)（L644），模型内部对每个通道算 CE 再按权重加权。这与 Qwen3-TTS 的“主 CE（第 0 层）+ 0.3*MTP（第 1-15 层）”是两种多通道监督哲学：MOSS 是所有通道都算 CE、按权重线性组合；Qwen3-TTS 是主序列 CE + sub-talker 辅助 MTP。

NPU 迁移 · collate 全 CPUcollate 全程在 CPU long 张量上做 pad/masked_fill/索引，属 DataLoader worker 进程的数据管线，不占 NPU。L29 normalize_audio_codes 显式 .cpu().contiguous() 把 codec 张量钉在 CPU。与 Qwen3-TTS 一样，collate 段对 Ascend 迁移零风险。唯一要留意的是 processor.encode_audios_from_path（dataset.py:85，按需编码 ref 音频）会调 audio_tokenizer——若 tokenizer 在 NPU 上，编码走 NPU 算子，需核验 codec 模型的算子；但这条在 prepare_data 预编码后通常不进训练循环。

训练循环：forward→backward→clip→step→zero

进入 _training_loop（L617），结构是 for epoch: model.train(); for batch: with accumulate(model): forward; backward; clip; step; sched.step; zero——与 Qwen3-TTS 的循环逐行同构。差别只在：MOSS 的 forward 直接拿 outputs.loss（模型内部已加权组合好多通道 CE），没有 Qwen3-TTS 那条 loss = outputs.loss + 0.3*sub_talker_loss 的外部组合；MOSS 在 step 后多了一个 lr_scheduler.step()（Qwen3-TTS 那边没显式 sched.step，靠 accelerate 内部）。

NPU 迁移 · accumulate/backward/clip 在 NPU循环这五步本身不含 CUDA 算子，accelerator.backward/clip_grad_norm_/optimizer.step 在 NPU 上走 torch_npu 的 elementwise + 一次 allreduce 全局范数（多卡时），与 Qwen3-TTS 同一路径、同风险等级。lr_scheduler.step 是纯 Python。真正要验的是：(1) accelerate 是否正确路由到 NPU device（需 accelerate config 选 torch_npu 后端）；(2) 多卡时 DDP 后端是否 HCCL（accelerate 默认 nccl，需改）。这两条与 Qwen3-TTS 迁移完全同源——MOSS 作为第二个样本，不增加新的循环级风险，反而佐证了“accelerate TTS 循环迁 NPU 的风险集中在 device 路由与后端，不在循环本身”。
MOSS 不能直接证明 Ascend 可训的地方就在这里：循环本身与 Qwen3-TTS 同构，但 configure_torch_backends、resolve_torch_dtype、resolve_attn_implementation 三段都以 torch.cuda.is_available() 为前提，在原生 NPU 环境（无 torch_npu patch 兜底）会全部回落 float32+eager。MOSS 脚本本身没有一处显式写 torch_npu——它的 Ascend 可行性完全依赖 torch_npu 的 cuda 兼容层或手动改这三段。这与 MindSpeed-MM（全文 import torch_npu + allow_internal_format）形成对照。

def save_checkpoint(
    accelerator: Accelerator,
    model: MossTTSDelayModel,
    model_path: str,
    codec_path: str,
    output_dir: Path,
    train_args: Dict[str, Any],
) -> None:
    accelerator.wait_for_everyone()
    if accelerator.is_main_process:
        output_dir.mkdir(parents=True, exist_ok=True)
    state_dict = accelerator.get_state_dict(model)
    unwrapped_model = accelerator.unwrap_model(model)

    unwrapped_model.save_pretrained(
        output_dir,
        is_main_process=accelerator.is_main_process,
        save_function=accelerator.save,
        state_dict=state_dict,
        safe_serialization=True,
    )
    if accelerator.is_main_process:
        copy_support_files(output_dir)
        copy_inference_assets(model_path, codec_path, output_dir)
        with open(output_dir / "finetune_args.json", "w", encoding="utf-8") as f:
            json.dump(train_args, f, indent=2, ensure_ascii=False)
    accelerator.wait_for_everyone()

save_checkpoint（L357-383）是 MOSS 保存的核心。L365 accelerator.wait_for_everyone() 先同步所有 rank。L368 accelerator.get_state_dict(model) 取（可能分片的）state_dict，L369 unwrap_model 解包拿原始模型。L371-377 unwrapped_model.save_pretrained(output_dir, is_main_process=…, save_function=accelerator.save, state_dict=…, safe_serialization=True)——走 transformers 的 save_pretrained，safe_serialization=True 即 safetensors 格式，由 accelerate.save 落盘。这与 Qwen3-TTS 手写 save_file(state_dict) 等价但更省事。

L378-382 main process 还做三件配套：copy_support_files 把 modeling/configuration/processing/inference_utils 等 python 文件拷进输出目录（trust_remote_code 推理需要）；copy_inference_assets 拷 tokenizer/added_tokens/chat_template 等资产并写 processor_config.json（含 codec_path）；json.dump(train_args) 写 finetune_args.json。这一整套让输出目录直接可推理——比 Qwen3-TTS 的“复制骨架+改 config+注入 speaker[3000]”更标准化（Qwen3-TTS 因为有 speaker 注册机制才需要那套手写逻辑）。

L696-704 在每 epoch 末触发：checkpoint_dir = output_root / f"checkpoint-epoch-{epoch}"，调 save_checkpoint。注意是每轮都存，不是只在末轮——便于挑 best epoch。与 MindSpeed-MM 的 Megatron 分片 ckpt（含 optimizer state、按拓扑分片）相比，MOSS 这种 save_pretrained 是面向推理部署的轻量保存，不含 optimizer state，不能直接断点续训。

NPU 迁移 · save_pretrained 已安全accelerator.get_state_dict（L368）在 accelerate 内部会做跨 rank 的 state_dict 收集，落到 main process 后 safe_serialization=True 写 safetensors——全程已 .to(cpu)（accelerate.save 的标准行为），NPU 上无算子风险。唯一注意 wait_for_everyone（L365、L383）在多卡时走 HCCL barrier，与 MM 同语义。若要断点续训，MOSS 这套不够——需另存 optimizer state（参考 MM 的 save_checkpoint），本脚本未提供。

深读：与 Qwen3-TTS/MM 训练的对照

把三条路径放一起。MOSS 与 Qwen3-TTS 是同形态的 accelerate TTS SFT：都用 accelerate prepare/accumulate/backward，都 bf16 无 scaler，都 next-token label shift，都每轮存盘。差异在损失设计（多 VQ 通道加权 CE vs 主 CE+MTP）、codec 结构（多 VQ vs 16 层残差）、保存方式（save_pretrained vs 手写 safetensors+speaker 注入）。MOSS 与 MindSpeed-MM 是异形态：MM 不碰 accelerate、用 Megatron forward_backward 调度、梯度累积单位是 microbatch（语义不同）、ckpt 是 Megatron 分片含 optimizer state。MOSS 这页的意义是：它作为第二个独立实现，把“accelerate TTS 训练迁 NPU”的共性风险（device 路由、attn 实现、HCCL 后端）从 Qwen3-TTS 单点结论扩展成两点交叉验证。

上图把三条路径的“一次更新 + 一次保存”对位。MOSS 与 Qwen3-TTS 的 setup→forward→backward→clip→step→save 几乎逐段同构，是横向可比的同类；MM 是纵向的另一套引擎。所以本页 npu_risk 的结论可以直接与 Qwen3-TTS 的迁移要点表逐条比对——相同项互相加强，相异项（如 MOSS 的 auto 探测式 attn、save_pretrained）给出更省事的迁移路径。
