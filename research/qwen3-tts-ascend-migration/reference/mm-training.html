MM 训练边界

逐段拆 MindSpeed-MM 的训练主入口 mindspeed_mm/training.py（1144 行）：pretrain/train/train_step 三层编排、Megatron forward_backward 调度的真实 forward-loss-backward 顺序、梯度累积与 mixed-precision 的 loss scale 钩子、NPU 设备初始化与 allow_internal_format——对照 Qwen3-TTS sft_12hz.py 的 accelerate 循环，讲清“accelerate 训练引擎怎么换成 MM 风格的 NPU 原生训练”。

学习目标

  • 定位 pretrain/train/train_step 三层入口的行号，说出每层各负责什么（编排 / 循环 / 单步）
  • 说清 train_step 里 forward_backward_func 与 optimizer.step 的真实先后，以及 zero_grad 在最前面这一顺序
  • 区分 Megatron 的 microbatch 累积与 accelerate 的 grad_accum 两种语义，以及 loss scale / mixed precision 在两条路径里分别由谁管
  • 指出 NPU 设备初始化的 torch_npu 调用点（set_compile_mode、allow_internal_format）与源码里仍写死的 device="cuda" 字面量，标注迁移时要改的行

前置知识

  • 读过 target/sft-training-loop.html，知道 accelerate 的 prepare/backward/accumulate 三段
  • 理解 Megatron 的 pipeline/microbatch 基本概念（不需要细到 P2P 调度）
  • 知道 BF16 原生支持、FP16 才需要 loss scaler 这条混合精度常识

总览：三层训练编排

MindSpeed-MM 的训练主入口 mindspeed_mm/training.py 只有 1144 行，但它是整条迁移的主参考——Qwen3-TTS 的 SFT 循环套在 accelerate 里，而 MM 这套代码本身就是一个为昇腾调过的训练引擎。它把训练拆成三层：pretrain() 是编排层，负责初始化、建模型/优化器/数据迭代器、决定要不要进训练；train() 是循环层，跑 while iteration < train_iters 的主循环，每轮调一次 train_step 并处理日志/评估/checkpoint/退出条件；train_step() 是单步层，里面真正调 forward_backward_func 做前向+反向、再 optimizer.step()。三层各自的事权很清楚，迁移时按层对位。

与 Qwen3-TTS 的 sft_12hz.py 对照：那边是单个 train() 函数 162 行，accelerate 包住一个 for-epoch 循环，accelerator.prepare 一次性接管 model/opt/loader，accelerator.backward 接管反向，accelerator.accumulate 管累积。这边不碰 accelerate，自己用 Megatron 的 forward_backward_func（按 microbatch 调度 PP/DP）、自己的 optimizer.step、自己的 update_num_microbatches 管累积。一句话：accelerate 的“循环 ownership”在 MM 这里被拆成了 Megatron 的 forward-backward 调度 + Megatron optimizer 的 step/scale。

NPU 迁移 · NPU 设备初始化import torch_npu（L9）是整条 NPU 路径的开关：它 monkey-patch torch 的 device("cuda")/.cuda()/torch.cuda.* 到 NPU。源码里 L198-201、L554、L664-670、L769、L815、L885、L981 等大量写 device="cuda" 与 torch.cuda.empty_cache()，不是漏改，是靠这层 patch 落到 NPU。迁移 Qwen3-TTS 时若走 accelerate+torch_npu 路线，需确认 torch_npu 已 import 且版本匹配；若直接迁到 MM 引擎则天然带上。两个真正要手动开的开关是 L178 set_compile_mode(jit_compile=True) 与 L181 allow_internal_format——这两个不影响能否跑通，但影响性能与算子融合，建议先默认开、出问题再关 internal_format 排查。待真机验证。
NPU 迁移 · allow_internal_formattorch.npu.config.allow_internal_format（L181）让 CANN 用内部排布（FracZ 等）跑融合算子。对 conv/linear 密集模型通常提效；但对某些自定义算子或需要与标准 NCHW 交互的路径会触发额外转排布，反而变慢甚至算子不匹配。Qwen3-TTS 的 codec embedding 查表 + speaker 注入是 elementwise 索引，与 format 无关，风险低；真正要观察的是 talker 的 attention——若走 torch_npu 的 fusion attention，internal_format 默认配合，一般无需手动改。保守：先按 MM 默认开，性能或算子报错时再针对性关。

Train 循环层：while 迭代与 train_step

L346 train() 带 @train_step_decorator（其实是 auto_setting 的装饰器，做性能自动调优的注入，不影响主流程）。它先把 config.grad_scale_func = optimizer.scale_loss（L405）——这一行是混合精度的核心钩子：把“对 loss 做 scale”这件事委托给 optimizer。BF16 时 scale 是 no-op，FP16 时它做 loss scaling。然后设 no_sync_func/grad_sync_func（L407-422，overlap_grad_reduce 时延迟 allreduce）、param_sync_func（L423-431，overlap_param_gather 时延迟参数聚合）、finalize_model_grads_func（L432，跨 PP stage 收尾梯度）。这一组 config 注入是理解 MM 训练引擎“可插拔并行策略”的钥匙。

NPU 迁移 · loss scale 与混合精度config.grad_scale_func = optimizer.scale_loss（L405）+ optimizer.get_loss_scale()（L570）是 MM 的 mixed-precision 钩子。BF16 训练时 scale 恒为 1、无 overflow 跳步；FP16 时 optimizer 是 Float16Optimizer，内部 GradScaler 做 dynamic loss scaling，skipped_iter 记录因 inf/nan 跳过的步。迁移 Qwen3-TTS 的 mixed_precision='bf16' 时：不需要引入 torch.cuda.amp.GradScaler——直接用 MM 的 BF16 路径，optimizer.scale_loss 为 no-op，与 Qwen3-TTS accelerate 的 bf16 行为一致。若被迫走 FP16（910B 也支持），才需要 loss scale，且 scale 逻辑在 optimizer 内部，不要自己在 loss 上乘系数。
NPU 迁移 · HCCL 后端MM 的多卡通信全走 Megatron 的 mpu（model parallel group），后端是 HCCL（昇腾集合通信库），不是 NCCL。overlap_grad_reduce/grad_sync_func（L407-422）控制 allreduce 何时发，finalize_model_grads（L432，经 finalize_model_grads_wrapper 包装）做跨 stage 梯度收尾——这些在 NPU 上原生走 HCCL。对照 Qwen3-TTS：accelerate 的 DDP 后端要手动改 nccl→hccl；MM 这边不用改，引擎已按 HCCL 写。多卡迁移真正要验的是 HCCL 拓扑（HCCS 带宽）与 torch.distributed.barrier()（L696、L784）在 NPU 上的行为。

Train_step 单步层：forward_backward + optimizer.step

L735 train_step() 是真正干活的地方。顺序非常关键，和 accelerate 的 backward→clip→step→zero 不一样：MM 是 zero_grad → forward_backward → empty_cache → optimizer.step → lr.step。L744-747 先 model_chunk.zero_grad_buffer()（Megatron 的梯度 buffer，PP/overlap 用）+ optimizer.zero_grad()——清零在最前，因为一个 step 内要跑 N 个 microbatch，每个 microbatch 的梯度往同一 buffer 累加，进 step 前必须清。accelerate 那边是 step 后 zero（下一轮才清），顺序反过来的根因是 microbatch 累积语义不同。

深读：accelerate 循环怎么换成 MM 风格

把两条路径的“一次优化器更新”对位。Qwen3-TTS accelerate：进 accelerator.accumulate(model) 上下文 → forward 算 loss → accelerator.backward(loss) → 累积到 4 步时 sync_gradients 为 True → clip_grad_norm_(1.0) → optimizer.step() → optimizer.zero_grad()。MM：zero_grad_buffer → for N microbatch: forward_backward_func（内含 forward+backward，梯度进 buffer 累加）→ optimizer.step()（内含 unscale+clip+inf-check+update）→ scheduler.step(increment)。两条都是“攒一批梯度再 step”，但MM 的 clip 在 optimizer.step 内部、accelerate 的 clip 在 step 外部——迁移时不能把 accelerate 的 clip_grad_norm_ 照搬到 MM，会双重裁剪。

迁移的实质映射：accelerate 的 Accelerator(grad_accum=4, bf16) 对应 MM 的 update_num_microbatches 让 N=4 + BF16 optimizer（scale_loss no-op）；accelerate 的 accelerator.prepare(model, opt, loader) 对应 MM 的 setup_model_and_optimizer + build_train_valid_test_data_iterators；accelerate 的 accelerator.backward 对应 forward_backward_func 里的反向段；accelerate 的 clip_grad_norm_ 在 MM 里删掉（optimizer.step 内部已有）；accelerate 的 accelerator.unwrap_model().state_dict() 对应 MM 的 unwrap_model(model[0]) + save_checkpoint。

迁移最大的一处语义陷阱是梯度累积单位不同。accelerate 的 grad_accum=4 是“4 个 data step（每个 step 都 forward+backward）才 optimizer.step 一次”，每个 data step 处理 batch_size 个样本；MM 的 num_microbatches=4 是“1 个 optimizer step 内跑 4 个 microbatch 的 forward+backward”，每个 microbatch 处理 micro_batch_size 个样本。两者样本量级要核对：accelerate 一次 step 消费 4 * batch_size，MM 一次 step 消费 4 * micro_batch_size * dp。lr schedule 按“每 step 消费样本数”对齐，否则学习率有效步长会错。
