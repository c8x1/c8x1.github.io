MM 配置入口

逐段拆 MindSpeed-MM 的配置入口与参数体系：两条并存的配置链——Megatron/Deepspeed argparse（pretrain 路径）与 FSDP2 的 pydantic BaseArguments 继承树（ConfigManager 路径）。看清 arguments.py 怎么往 Megatron parser 里塞 MM 专属开关、config_manager 怎么做 YAML+CLI 合并与继承、parallel_args/training_args/device 哪些字段决定 NPU 行为（device 路由、hccl 后端、bf16、recompute、ring-attention），以及 multimodal 配置如何通过 dot-path 继承扩展。

学习目标

  • 区分 MindSpeed-MM 的两条配置链：Megatron argparse（pretrain）与 FSDP2 ConfigManager（pydantic 继承），说出各自入口文件与合并优先级
  • 说出 arguments.py 的 process_args 往 Megatron parser 注入了哪些 MM 专属开关、哪些是 NPU/Ascend 相关（use-deter-comp / allow-internal-format / recompute-skip-core-attention 等）
  • 讲清 ConfigManager 的 YAML+CLI+additional 三源合并优先级（CLI > code > YAML > default）与 dot-path 嵌套字段的继承扩展机制
  • 指出 parallel_args / device.py 里决定 NPU 行为的开关：IS_NPU_AVAILABLE 设备路由、hccl 后端、ring-attention 仅 NPU、bf16 param_dtype

前置知识

  • 读过 Qwen3-TTS 训练循环页（理解 SFT 参数从哪来）
  • 了解 argparse 与 argparse sub-group 机制
  • 知道 pydantic BaseModel 的字段继承
  • 知道 Megatron/Deepspeed 的 extra_args_provider 钩子约定

总览：两条并存的配置链

MindSpeed-MM 作为 Qwen3-TTS SFT 迁移的主参考工程，它的第一道关卡是配置体系。难点在于它同时维护两条配置链，二者职责不同、入口不同：一条是 Megatron/Deepspeed argparse，服务 pretrain_*.py 这类传统训练脚本，参数以 --tensor-model-parallel-size 这种连字符 flag 喂进 Megatron parser；另一条是 FSDP2 的 ConfigManager，服务新的 fully_shard 训练路径，参数以 YAML 文件 + dot-path 嵌套字段（如 parallel.tensor_parallel_size）表达，底层是 pydantic BaseModel 继承树。本页要让读者先看清“这两条链分别从哪进、各自怎么组织参数、哪些开关决定 NPU 行为”，再去读具体的训练循环。

上图的左半是 Megatron 链：pretrain_vlm.py 调 pretrain() 时传入 extra_args_provider=mm_extra_args_provider，training.py:125 用 extra_args_provider_decorator 把它包一层（在用户 provider 之外追加 MM 专属参数），再交给 megatron.training.arguments.parse_args。右半是 FSDP2 链：YAML 文件直接被 ConfigManager 读取，喂进继承自 BaseArguments 的 Arguments 根类，合并后存进全局 get_args()。两条链最终都落到一个全局可取的 args 对象，但结构完全不同——迁移时要先认出自己走的是哪条。

Megatron 链：arguments.py 往 parser 塞 MM 开关

先看 Megatron 链的“加料”层。mindspeed_mm/arguments.py 不自己解析参数，它只负责往 Megatron 的 argparse parser 里追加 argument group。核心是 process_args 这个函数——它把十几个 _add_* 子函数串起来，每个子函数建一个 group 并 add_argument。extra_args_provider_decorator 把用户传入的 provider 与 process_args 组合：先跑用户的、再跑 MM 的，保证 MM 开关不会被用户覆盖（conflict_handler="resolve" 容忍重名）。

def extra_args_provider_decorator(extra_args_provider):
    @wraps(extra_args_provider)
    def wrapper(parser):
        if extra_args_provider is not None:
            parser = extra_args_provider(parser)
        parser = process_args(parser)
        return parser

    return wrapper

逐行看 NPU 相关开关。L79-82 --use-deter-comp（默认 False）：开 NPU 确定性计算，对调试数值复现有用，但会牺牲吞吐。L87-90 --allow-tf32：NPU 上对应 torch.npu.aclnn.allow_hf32（见 device.py:155-163），是低精度矩阵乘的开关。L91-94 --downcast-to-bf16：加载 checkpoint 时把 fp32 权重就地降成 bf16，省显存但有精度损失。L95-98 --allow-internal-format：CANN 内部格式，device.py:21 在检测到 NPU 时会 torch.npu.config.allow_internal_format = False 把它关掉——这是 MM 与 device 层的协同点。L99-102 --virtual-pipeline-model-parallel-size 即 vpp：把 PP 的层切成多个 virtual fold（vf），让前后向流水更平滑，是 Megatron interleaved schedule 的开关。L107-113 --recompute-skip-core-attention 与 --recompute-num-layers-skip-core-attention：梯度检查点重计算时跳过 core attention 算子，减少重算开销。L135-137 --optimizer 把 choices 扩成 ['adam','sgd','muon']——Muon 是 MM 新增的优化器选项，NPU 上需核验 Newton-Schulz 正交化算子。

NPU 迁移 · allow-internal-format 与 device 层协同--allow-internal-format 控制 CANN 内部数据格式（让某些算子用 NPU 原生布局省一次转换）。但 device.py:20-21 在检测到 IS_NPU_AVAILABLE 时强制 torch.npu.config.allow_internal_format = False——即运行时被 device 层关掉。迁移时不要靠这个 flag 期待内部格式生效，应以 device 层的实际设置为准；若要开需改 device.py 而非命令行。待真机验证内部格式对 TTS 注意力算子的稳定性影响。
NPU 迁移 · vpp / virtual-fold 调度--virtual-pipeline-model-parallel-size（vpp）把 PP 的层切成多个 virtual fold，做 interleaved 前后向。这是 Megatron 的标准机制，在 NPU 上需 HCCL 支持 all-to-all 通信；vf 过大会增加通信 bubble、过小则流水不饱满。TTS 模型层数有限（Talker 一般几十层），vpp 取值要配合 TP/CP 一起算。保守方向：单机迁移先 vpp=None（不开），多机多卡再调 2-4。待真机验证 bubble 与显存权衡。
arguments.py 里没有 --device 这个 flag。NPU/CUDA 的设备选择不是 Megatron 参数，而是 device.py 在 import torch_npu 时通过 IS_NPU_AVAILABLE 运行时决定的（见下文 parallel 段）。这是迁移者常踩的坑：在命令行找 --device 是找不到的。

MM 子配置：mm-data/model/tool 的 JSON 注入

Megatron 链里还有一个子配置层：multimodal 的数据/模型/工具参数不走 Megatron flag，而是三个 JSON 文件路径 --mm-data / --mm-model / --mm-tool。mm_extra_args_provider 只负责把这三个路径 flag 加进 parser，真正读 JSON 的是 merge_mm_args——它把三个文件加载成 MMConfig 对象挂到 args.mm。这是 MM“配置继承”的 Megatron 侧体现：主干走 argparse、多模态细节走 JSON dot-path。

FSDP2 链：ConfigManager 与 pydantic 继承树

FSDP2 路径完全换了一套配置体系。入口是 config/config_manager.py 的 ConfigManager，底层是 pydantic BaseArguments（config/arguments/base_args.py）。根配置类 Arguments（fsdp/params/argument.py）用 dataclass.field(default_factory=...) 组合六个子配置：parallel / model / data / training / tools / features，每个子类又继承 BaseArguments。这是类级继承——子类自动获得父类字段（base_args.py:24-39 的 __init_subclass__ 深拷贝父类 _dynamic_fields_info）。

L20-27 Arguments 根类：六个字段用 field(default_factory=ParallelArguments) 这种工厂默认值，保证每个实例有独立的子配置对象。注意这些不是普通属性而是类型注解——parallel: ParallelArguments 告诉 ConfigManager._collect_defined_fields（config_manager.py:230-260）这是个 BaseArguments 子类，要递归收集其字段到 dot-path（如 parallel.tensor_parallel_size）。L29-30 model_post_init：pydantic 在实例化后调它，立刻执行 self.training.compute_distributed_training(self.parallel)——这是跨 section 的依赖解析：训练步数依赖 data_parallel_size，而后者来自 parallel section。所以 FSDP2 配置不是静态加载完就结束，还有一道后处理。

    def load_and_parse(self) -> BaseArguments:
        """Load and parse all configuration sources"""
        _print("[INFO] Starting configuration loading...")

        try:
            # 1. Load configuration sources
            yaml_config = self._load_yaml_config()
            cli_config = self._parse_cli_args() if self.allow_cli_override else {}

            # 2. Process YAML extensions
            self._process_yaml_extensions(yaml_config)

            # 3. Validate command-line arguments
            self._validate_cli_args(cli_config)

            # 4. Safely merge configurations
            merged_config = self._safe_merge_configs(yaml_config, cli_config, self.additional_args)

            # 5. Check required fields
            self._check_required_fields(merged_config)

            # 6. Create configuration object
            config_obj = self.config_class(**merged_config)

            # 7. Validate configuration object
            self._validate_config_object(config_obj, merged_config)

            self._config = config_obj

            self.print_summary()

            return config_obj

        except Exception as e:
            _print(f"[ERROR] Configuration loading failed: {e}")
            if isinstance(e, ConfigValidationError):
                raise
            else:
                raise ConfigValidationError(f"Configuration loading failed") from e

    def _safe_merge_configs(self,
                            yaml_config: Dict[str, Any],
                            cli_config: Dict[str, Any],
                            additional_config: Dict[str, Any]) -> Dict[str, Any]:
        """
        Safely merge all configuration sources, support nested fields
        Priority: command line > code input > YAML > default values
        """
        merged = {}

        def _merge_dict(target: Dict, source: Dict, path: str = ""):
            """Recursively merge dictionaries, support nested fields"""
            for key, value in source.items():
                full_key = f"{path}.{key}" if path else key

                if isinstance(value, dict) and key in target and isinstance(target[key], dict):
                    # Recursively merge nested dictionaries
                    _merge_dict(target[key], value, full_key)
                else:
                    # Set value directly
                    if '.' in full_key:
                        self._set_nested_value(target, full_key, value)
                    else:
                        target[key] = value

        # 1. First apply YAML configuration
        if yaml_config:
            _merge_dict(merged, yaml_config)

        # 2. Then apply code input configuration
        if additional_config:
            _merge_dict(merged, additional_config)

        # 3. Finally apply command-line configuration
        if cli_config and self.allow_cli_override:
            for key, value in cli_config.items():
                # Only allow overriding existing fields
                can_override = key in merged or self._is_field_defined(
                    key) or key in self._dynamic_fields or self._is_nested_field_defined(key)
                if can_override:

                    if '.' in key:
                        self._set_nested_value(merged, key, value)
                    else:
                        merged[key] = value
                else:
                    _print(f"[WARNING] Ignoring undefined command-line argument: {key}")

        return merged

L604-652 是合并的核心，注释在 L610 写明优先级：command line > code input > YAML > default values。L612-627 的 _merge_dict 递归合并嵌套 dict——同名 key 若两边都是 dict 则深合并，否则后值覆盖前值；dot-path key（如 data.dataset.id）走 _set_nested_value（L321-343）拆开层级写入。L629-635 先铺 YAML，再叠 additional（代码传入），最后 L637-650 叠 CLI——但 CLI 只能覆盖已存在的字段（L641 的 can_override 判断），未定义的 CLI 参数会被警告忽略（L650）。这套优先级保证了“YAML 定基线、代码做注入、CLI 做覆盖”的可预测分层。

两条链的 CLI 语义不同：Megatron 链的 CLI 是标准 --flag=value argparse；FSDP2 链的 CLI 是 key=value 且不能以 - 开头。迁移时若把 FSDP2 的 YAML 参数当成 Megatron flag 传，或反过来，都会失效。先认链再传参。

并行参数与 device 路由：哪些开关决定 NPU 行为

FSDP2 链里和 NPU 行为最相关的是 parallel_args.py 的 ParallelArguments 与 fsdp/utils/device.py。ParallelArguments 定义了 TP/CP/DP/FSDP/EP 各维并行大小，model_post_init 在实例化后用 WORLD_SIZE / TP / CP 反推 DP，并做整除校验。而 device.py 的 IS_NPU_AVAILABLE 是整个工程的设备路由开关——它在 import torch_npu 成功时置 True，get_device_type() 据此返回 "npu"，get_dist_comm_backend() 据此返回 "hccl"。

    def model_post_init(self, __context):
        self.local_rank = int(os.getenv("LOCAL_RANK"))
        self.global_rank = int(os.getenv("RANK"))
        self.world_size = int(os.getenv("WORLD_SIZE"))

        if self.fully_shard_parallel_size == "auto":
            # If -1, use all remaining processes after tensor parallelism for FSDP
            self.fully_shard_parallel_size = self.world_size // self.tensor_parallel_size
        else:
            self.fully_shard_parallel_size = int(self.fully_shard_parallel_size)

        if self.expert_fully_shard_parallel_size is None:
            self.expert_fully_shard_parallel_size = self.world_size // self.expert_parallel_size

        if (
            self.world_size
            % (
                self.tensor_parallel_size
                * self.ring_attention_size
                * self.ulysses_parallel_size
            )
            != 0
        ):
            raise ValueError(
                f"World size should be a multiple of tensor_parallel_size: {self.tensor_parallel_size}, ulysses_parallel_size: {self.ulysses_parallel_size}, ring_attention_size: {self.ring_attention_size}."
            )
        if (
            self.world_size
            % (
                self.tensor_parallel_size
                * self.fully_shard_parallel_size
            )
            != 0
        ):
            raise ValueError(
                f"World size should be a multiple of tensor_parallel_size: {self.tensor_parallel_size}, fully_shard_parallel_size: {self.fully_shard_parallel_size}."
            )

        dp_size = self.world_size // (
            self.tensor_parallel_size
            * self.ring_attention_size
            * self.ulysses_parallel_size
        )
        if self.data_parallel_size is None:
            self.data_parallel_size = dp_size

        if self.data_parallel_size != dp_size:
            raise ValueError(f"data_parallel_size should be equal to tensor_parallel_size: {self.tensor_parallel_size}, ulysses_parallel_size: {self.ulysses_parallel_size}, ring_attention_size: {self.ring_attention_size}.")

        if self.fully_shard_parallel_size < self.ring_attention_size * self.ulysses_parallel_size:
            raise ValueError("fully shard parallel size should be greater the ring_attention_size * ulysses_parallel_size.")
        if self.tensor_parallel_size != 1:
            raise ValueError("Tensor parallel size not supported yet.")
        if self.ring_attention_size != 1 and not IS_NPU_AVAILABLE:
            raise ValueError("Ring Attention only support on NPU.")

def get_device_type() -> str:
    """Get device type based on current machine, currently only support CPU, CUDA, NPU."""
    if IS_CUDA_AVAILABLE:
        device = "cuda"
    elif IS_NPU_AVAILABLE:
        device = "npu"
    else:
        device = "cpu"

    return device

class _GlobalConfig:
    """Global configuration storage"""

    def __init__(self):
        self._current_config: Optional[BaseArguments] = None

    def set(self, config: BaseArguments):
        """Set global configuration"""
        self._current_config = config

    def get(self) -> Optional[BaseArguments]:
        """Get global configuration"""
        return self._current_config

    def clear(self):
        """Clear global configuration"""
        self._current_config = None


# Global configuration instance
_global_config = _GlobalConfig()


def set_global_config(config_manager: ConfigManager) -> BaseArguments:
    """
    Set global configuration

    Args:
        config_manager: Configuration manager

    Returns:
        TrainArguments: Global configuration object
    """
    # Load and parse configuration

    # Set to global configuration
    _global_config.set(config)

    if int(os.environ.get('RANK', '0')) == 0:

    return config


def get_args() -> Optional[BaseArguments]:
    """Get global arguments"""
    return _global_config.get()

深读：multimodal 配置继承的两种形态

把“multimodal 配置继承”收束一下——它在 MindSpeed-MM 里有两种截然不同的形态。形态一是 Megatron 链的文件级继承：主干 Megatron 参数走 argparse，多模态细节走 mm-data/model/tool 三个 JSON 文件，各自成 section 挂到 args.mm，section 内部用 ConfigReader 的属性访问。形态二是 FSDP2 链的类级继承：Arguments 根类组合六个子配置类，子类继承 BaseArguments 自动获得父类动态字段，YAML 里未定义的字段还能被 _process_yaml_extensions 运行时注册。

两种形态各有取舍。形态一（文件级）的好处是多模态细节与主干 Megatron 参数解耦——加新模型只动 mm-model.json，不碰 Megatron parser；坏处是 ConfigReader 无类型校验、无 schema。形态二（类级）的好处是 pydantic 类型安全 + dot-path 嵌套 + 运行时扩展，坏处是配置类继承链深、调试时得追多层 __init_subclass__。迁移 Qwen3-TTS 时，若沿用 accelerate（如 target SFT 页）则两条 MM 链都不直接用；若迁到 MindSpeed-MM 训练引擎，必须先选定走 Megatron 链还是 FSDP2 链——这决定后续所有参数怎么写。
