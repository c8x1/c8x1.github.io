学习轨道：源码深读

MM NPU 开关

逐段拆 MindSpeed-MM 在 910B 上的 NPU/Ascend 接入与算子补丁体系：training.py 顶部的 torch_npu 引入与 deterministic/internal_format 开关、PatchesManager 的配置驱动算子补丁注册、infer_fa/ring/ulysses 的 NPU 专用注意力替换、MindSpore adaptor 对 torch.stft/scatter_add/masked_scatter 的算子级适配，并与 Qwen3-TTS 的 FA2→sdpa 选择对照——这是迁移到 NPU 真正要碰的算子层。

学习目标

  • 说清 training.py 顶部 torch_npu 接入的三处开关（jit_compile / allow_internal_format / deterministic）各自的语义
  • 复述 PatchesManager 的配置驱动注册机制：patch 配置如何映射到 megatron 原函数替换
  • 区分三类 NPU 注意力替换：infer_fa_patch 推理 FA、ring_attn/ulysses 的 npu_fusion_attention、gated_delta_net 的 AscendC 算子
  • 对照 Qwen3-TTS 的 FA2→sdpa，理解 MindSpeed-MM 为何把注意力替换做在算子补丁层而非 attn_implementation 参数

前置知识

  • 读过 Megatron-LM 的 DotProductAttention 与 Float16Module
  • 理解 torch_npu 是 PyTorch 的 NPU 后端扩展
  • 熟悉 Qwen3-TTS SFT 页的 FA2 硬绑风险

总览：NPU 开关的三层

MindSpeed-MM 是 Megatron-LM 的 Ascend 分支，整条 NPU 接入不是在一个文件里完成，而是分散在三层：引擎层 training.py 顶部 import torch_npu 后做三处全局开关（deterministic、jit_compile、allow_internal_format），随后调用 PatchesManager.apply_patches_from_config() 装配算子补丁；补丁层 patchs/ 下 18 个 patch 文件，按模型配置里的 patch 字典决定替换哪些 megatron 原函数；跨后端层 mindspore/mindspore_adaptor.py 在走 MindSpore 算子后端时补 torch 算子在 CANN 上的缺口（stft、scatter_add_、masked_scatter_）。三层各自独立开关，迁移者要按层逐个核对。

一个关键对照：Qwen3-TTS 把注意力实现写在 from_pretrained(attn_implementation="flash_attention_2") 这个模型参数里，改注意力 = 改装载参数（FA2→sdpa）。MindSpeed-MM 走的是 Megatron 的 DotProductAttention 抽象——注意力是可替换的算子插件，真正的 NPU 注意力实现（npu_fusion_attention / npu_fused_infer_attention_score）在补丁里 monkey-patch 进 DotProductAttention.forward。所以这里没有 attn_implementation 这个参数，而是 use_flash_attn + use_infer_fa + use_fusion_attn_v2 一组布尔开关决定走哪条 NPU 注意力路径。

引擎层：torch_npu 接入与三处开关

training.py 第 9 行 import torch_npu 是整个仓的 NPU 入口——这一行让 torch.npu 命名空间可用（torch.npu.config、torch.npu.current_stream）。它不是可选依赖：后续 training.py 顶部就直连 torch.npu.config.allow_internal_format，patch 文件也直接 import torch_npu。迁移者若误以为可回退 CUDA，import torch_npu 这一关就会在纯 torch 环境炸掉。

L180-181 是最隐蔽的一行：torch.backends.cuda.matmul.allow_tf32 = args.allow_tf32 设 CUDA 的 TF32（在 NPU 上语义被 torch_npu 解释为对应精度模式），torch.npu.config.allow_internal_format = args.allow_internal_format 开启 NPU 内部格式（把 NHWC/转置布局交给 CANN 自动选择）。internal_format=True 会改变张量在 NPU 上的物理布局，多数算子受益但少数自定义算子会因布局不匹配出错——这是迁移时常被忽略的隐性开关。L186 PatchesManager.apply_patches_from_config() 是进入补丁层的分水岭，下一节展开。

NPU 迁移 · internal_format 隐性布局开关torch.npu.config.allow_internal_format 默认开，让 CANN 自动选内部格式以加速 matmul/conv，但自定义算子或某些 reduce 类算子在 internal format 下会因 stride/contiguous 假设不符报错。迁移遇到“算子不支持”类报错时，先试 allow_internal_format=False 复现，确认是否布局问题而非算子本身缺失。该开关对 Qwen3-TTS 的 sdpa 路径无直接等价物——sdpa 在 torch_npu 内部已处理布局，不暴露此开关。
NPU 迁移 · deterministic 与 jit_compileuse_deter_comp 关内部非确定性优化用于复现，jit_compile 开 GE 图编译用于性能。二者都改 NPU 算子选择路径：开 deterministic 会关掉部分融合算子（性能下降）；开 jit_compile 会把算子组融合成子图（调试断点失效）。迁移复现阶段建议 deter=on / jit=off 先对齐数值，再逐个放开。这两开关在 Qwen3-TTS 路径不存在，是 MindSpeed-MM 独有的 NPU 调谐面。

补丁层：配置驱动的算子替换

PatchesManager 是 MindSpeed-MM 把 NPU/Ascend 优化做成可选补丁的中枢。它维护一个 configs 字典：键是补丁名（如 infer_fa、ae_float32），值是 (原函数全限定名, 替换函数) 元组列表。apply_patches_from_config 读模型配置 mm.model.patch 字典，对同时在配置和 configs 里出现的键且值为真的，逐个 register_patch 后统一 apply_patches。这是按模型配置动态决定打哪些补丁，不是编译期固定。

NPU 迁移 · 补丁开关在配置而非代码NPU 注意力/优化器替换全部受 mm.model.patch 字典控制，不在代码里硬编码。迁移若发现跑的还是 megatron 默认注意力（未替换为 npu_fusion_attention），第一步查的是 patch 配置字典是否开了 infer_fa/ring 等键，而不是去改 forward。这点与 Qwen3-TTS 把 FA2 写死在 from_pretrained 参数完全相反——后者改装载参数，前者改配置字典。

NPU 注意力补丁：infer_fa 与 ring/ulysses

NPU 专用注意力分两类。推理类 infer_fa_patch.py：替换 DotProductAttention.forward，在推理（有 kv_cache）且 use_flash_attn+use_infer_fa 开启时走 torch_npu.npu_fused_infer_attention_score——这是 CANN 的 Paged/增量 FA 算子，专为 decode 阶段单步 q 设计。训练/CP 类 ring_attn_patch.py 与 ulysses_patches.py：在 context parallel 场景把注意力替换为 npu_fusion_attention（v2）或 torch_npu.npu_fusion_attention（v1），支持 ring/ulysses 两种 CP 通信拓扑。两者都把 Megatron 的抽象注意力换成 NPU 原生算子。

mindspeed_mm/patchs/infer_fa_patch.py · L19–L30

def dot_product_attention_forward_infer_wrapper(fn):
    @wraps(fn)
    def wrapper(self, query, key, value, attention_mask, **kwargs):
        if not hasattr(get_args().mm.model, "generation_config"):
            raise AssertionError("This infer fa patch is only available for inference.")
        if not getattr(get_args().mm.model.generation_config, "kv_cache", False):
            raise AssertionError("Inference fa is only available when kv_cache is True.")
        if get_args().use_flash_attn and getattr(self.config, "use_infer_fa", False):
            return dot_product_attention_forward_infer(self, query, key, value, attention_mask, **kwargs)
        return fn(self, query, key, value, attention_mask, **kwargs)

    return wrapper

L19-30 的 wrapper 是补丁的标准形态：@wraps(fn) 包原函数，进函数体先做两个断言——必须有 generation_config（保证是推理）、kv_cache=True，然后 L26 if get_args().use_flash_attn and getattr(self.config, "use_infer_fa", False) 双开关命中才走 dot_product_attention_forward_infer，否则 return fn(...) 回退原实现。这种双开关 + 回退模式是补丁的常态：默认不替换，显式开两个开关才替换，方便对照数值。

NPU 迁移 · FA 算子对 stride 敏感infer_fa_patch L44-46 与 ring_patch L219 都对 q/k/v 做 transpose(...).contiguous()——NPU FA 算子不接受非连续/转置视图输入，必须物理重排。若迁移自定义注意力时省略 contiguous，会报 stride/布局错。这是 internal_format 与 FA 算子的交叉风险点：internal_format 改的物理布局可能让 contiguous 假设失效，需在算子边界处显式重排。
NPU 迁移 · FA2 与 NPU FA 数值不严格等价MindSpeed-MM 的 npu_fusion_attention 与 CUDA FlashAttention-2 在 bf16 下允许合理数值差异（pse/sparse 参数路径不同），不保证 bit-精确。对照 Qwen3-TTS 的 FA2→sdpa 也是同理：迁移复现时用小样本做 max-abs diff 设阈值（如 1e-2 量级），不要期望逐元素相等。MindSpeed-MM 提供 v1/v2 两条路径正便于对照：v1 走 torch_npu，v2 走 mindspeed.ops，数值略有不同，可二选一作 baseline。

精度补丁：AE 保 fp32 与 Float16Module

models_patches.py 的 ae_float32 补丁改的是 Float16Module.__init__ 与 forward。Megatron 默认把整个模型包进 Float16Module 做 bf16/fp16 转换，但多模态里的自动编码器（AE/VAE）部分对精度敏感，强行半精度会数值崩溃。这个补丁让 AE 子模块在整体半精度时仍保持 fp32——是 NPU 上常见的精度保命手段。

mindspeed_mm/patchs/models_patches.py · L30–L53

def float16Module_init(self, config, module):
    super(Float16Module, self).__init__(config)

    # if AEModel use fp32
    ae_config = getattr(config.mm.model, "ae", None)
    ae_float32 = False
    if ae_config is not None and getattr(ae_config, "dtype", None) == torch.float32:
        module.ae = module.ae.float()
        ae_float32 = True

    if config.fp16:
        if ae_float32:
            self.add_module("module", _transform_model_dtype(module, lambda module: module.half(), [module.ae]))
        else:
            self.add_module("module", module.half())

        def float16_convertor(val):
            return val.half()

    elif config.bf16:
        if ae_float32:
            self.add_module("module", _transform_model_dtype(module, lambda module: module.bfloat16(), [module.ae]))
        else:
            self.add_module("module", module.bfloat16())

L34-38 先取 ae_config，若 AE 的 dtype==torch.float32 则 module.ae = module.ae.float() 单独把 AE 转回 fp32，并记 ae_float32=True。L40-44（fp16 分支）/ L49-53（bf16 分支）按 ae_float32 分流：若 AE 要保 fp32，用 _transform_model_dtype(module, lambda m: m.half()/bfloat16(), [module.ae])——对除 AE 外的子模块做半精度转换，AE 跳过；否则整模块半精度。这个 ignore_child_modules=[module.ae] 的跳过逻辑是精度保护的核心。

NPU 迁移 · AE/VAE 强制 fp32图像/视频 AE 的 conv+残差结构在 bf16 下易数值溢出，NPU 上尤甚（部分 conv 算子在 bf16 下累计误差大）。ae_float32 补丁让 AE 保 fp32 是精度保命而非性能优化——fp32 conv 在 NPU 上吞吐约为 bf16 的 1/2，但换来稳定。迁移 Qwen3-TTS 时其 codec embedding/speaker_encoder 都是低频小算子无需此处理；若把 MindSpeed-MM 的视觉 AE 直接搬用，务必开此补丁。待真机验证 AE 在 910B bf16 下的具体崩溃点。

通信补丁：HCCL 的 PREMUL_SUM 缺口

MindSpeed-MM 多卡走 HCCL（华为集合通信库）替代 NCCL。HCCL 不支持 ReduceOp.PREMUL_SUM（先乘再求和的 fused 归约），而 MoE/EP 的梯度归约常用它。premul_sum_patch.py 用 wrapper 把 PREMUL_SUM 拆成 SUM+事后标量乘——这是典型的算子语义缺口补丁：后端不支持某 op，就在 Python 层用支持的 op 组合等价实现。

def hccl_premul_sum_wrapper(op, output_name):
    """
    A wrapper for distributed operations to handle ReduceOp.PREMUL_SUM which is not supported in Huawei HCCL.
    This wrapper intercepts operations using ReduceOp.PREMUL_SUM and converts them into equivalent
    ReduceOp.SUM operations followed by scalar multiplication.
    """

    def wrapper(*args, **kwargs):
        # Note:Although the sequence of operations(ReduceOp.SUM followed by multiplication) may differ from semantics,
        # we have verified that there is no problem with the performance and accuracy of this sequence.
        factor = None
        if "op" in kwargs and kwargs["op"] == ReduceOp.PREMUL_SUM:
            factor = kwargs["op"].__getstate__()[1]
            kwargs["op"] = ReduceOp.SUM
        handle = op(*args, **kwargs)
        if handle is not None:
            handle.wait()
        if factor is not None:
            output = args[0] if len(args) > 0 else kwargs[output_name]
            output.data.mul_(factor)
        return handle

    return wrapper

L18-31 的 wrapper：进函数体查 kwargs["op"] == ReduceOp.PREMUL_SUM，若是则 L23 factor = kwargs["op"].__getstate__()[1] 取出预乘因子，L24 把 op 改回 ReduceOp.SUM；L25 调原 op 完成 SUM 归约；L26-27 handle.wait() 等同步；L28-30 把结果 output.data.mul_(factor) 事后乘回因子。L36-39 把 all_reduce/reduce_scatter/reduce_scatter_tensor 三个集合通信全替换成 wrapper。注释 L19-20 明说 SUM-then-mul 与原语义顺序不同但已验证精度无误——是语义近似而非严格等价。

NPU 迁移 · HCCL 缺 PREMUL_SUM这是 NPU 通信层明确的算子缺口（非推测）：torch_npu/HCCL 公开不支持 PREMUL_SUM，补丁用 SUM+mul 等价。风险在两点：1) mul_ 是原地操作，若 output 张量被多处引用会污染；2) SUM-then-mul 与 premul-then-sum 在 fp16/bf16 下舍入路径不同，L20 注释称“已验证”但具体阈值未给——MoE/EP 大规模训练前应重验。Qwen3-TTS 单卡 SFT 不涉及此风险，仅多卡 EP/MoE 时碰。

另一处通信相关：ep_patch.py 的 finalize_model_grads_wrapper 在 EP FSDP 组内做 torch.distributed.all_reduce(local_grad, op=ReduceOp.AVG, async_op=True)，末尾 torch.npu.current_stream().synchronize() 显式同步 NPU 流。这揭示一个 NPU 通信惯例：异步集合通信后常需显式 stream 同步，因为 HCCL 与 NPU 计算流的默认协同语义和 NCCL/CUDA 不同，不显式同步可能读到未完成归约的梯度。
