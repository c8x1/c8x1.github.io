MM 检查点转换

按固定 revision mindspeed-mm-0edd553e 逐段拆 MindSpeed-MM 的检查点转换工具链：convert_cli 的 jsonargparse 自动注册、Converter/DcpConverter 双抽象、HF safetensors → torch-dcp 的分片并发改写、DCP → HF 的 partial-load 合并，以及 Qwen3-TTS 专用 dcp_to_hf 把训练后 speaker 向量注入 codec_embedding[3000] 并删 speaker_encoder 权重写 safetensors 的路径——对照 Qwen3-TTS 原生 sft_12hz.py 的 speaker[3000] 注入与 safetensors 导出。

学习目标

  • 说清 convert_cli 如何用 __init_subclass__ 自动发现所有 Converter/DcpConverter 子类并暴露成 CLI 子命令
  • 区分两条格式：torch-dcp（DCP，多 rank 分片、可 partial load）与 HF safetensors（含 .index.json 分片）
  • 讲明 hf_to_dcp 的分片并发与 _process_single_file 的 save_dict 结构，以及 dcp_to_hf 的 partial-load 线程模型
  • 对照 Qwen3-TTS 原生保存路径，解释 MM 版 dcp_to_hf 为何把 speaker 注入从训练循环搬到转换期、并删 speaker_encoder 权重

前置知识

  • 读过 target/sft-training-loop 的保存段（speaker[3000] 注入与 safetensors 导出）
  • 理解 PyTorch state_dict 与 safetensors 文件格式
  • 知道 torch.distributed.checkpoint（DCP）的 FileSystemReader/Writer 与 metadata 概念

总览：两条格式与四个转换方向

MindSpeed-MM 的检查点转换代码不在 mindspeed_mm/tools/（那里是 mem/profiler），而在仓库顶层独立的 checkpoint/ 目录，按 common/（抽象+DCP 公共工具）、vlm_model/（HF↔Megatron/MM 格式 + per-model converter）、fsdp/custom_model_converter/（FSDP 训练后 DCP↔HF，含 Qwen3-TTS 专用）组织。整个工具链围绕两种落盘格式：HF safetensors（标准 HF 目录，含 model-00001-of-00005.safetensors + model.safetensors.index.json，权重键无前缀）与 torch-dcp（PyTorch distributed checkpoint，落在 release/ 子目录，键带 model. 前缀，可按 metadata partial-load）。四个方向：HF→DCP、DCP→HF、HF→MM(Megatron)、MM→HF，Qwen3-TTS 走的是 DCP↔HF 这条 FSDP 路径。

Qwen3-TTS 在 MM 里的训练保存产物是 DCP 格式（FSDP 训练引擎保存），推理/部署要的是 HF safetensors，所以 checkpoint/fsdp/custom_model_converter/qwen3tts.py 的 Qwen3TTSConverter.dcp_to_hf 是关键路径——它把 DCP 合并回 HF，并在合并期（而非训练循环里）完成 speaker 向量注入与 speaker_encoder 权重删除。这与 Qwen3-TTS 原生 sft_12hz.py 在每轮末直接注入 + 写 safetensors 的做法是同一语义的两种实现。

CLI 自动注册：__init_subclass__ + jsonargparse

转换的命令行入口是 checkpoint/convert_cli.py，只有 24 行。它本身不认识任何具体模型，靠 jsonargparse.auto_cli 对 Commandable.subclasses 这个列表做反射，把每个已注册的 Converter/DcpConverter 子类暴露成一个子命令。子类的注册发生在 converter.py 的 Commandable.__init_subclass__：任何继承 Commandable 的类在定义时就被 append 进 subclasses，前提是它自己没有新增抽象方法（own_abstract=False），即它已经实装了父类所有 abstractmethod。

class Commandable:
    subclasses = []

    def __init_subclass__(cls, **kwargs):
        """All subclasses of Converter will be stored in the class attribute 'subclalsses'"""
        super().__init_subclass__(**kwargs)
        own_abstract = any(getattr(value, '__isabstractmethod__', False) for key, value in cls.__dict__.items())
        if not own_abstract:
            cls.subclasses.append(cls)

    @classmethod
    def add_command(cls, command: Callable):
        cls.subclasses.append(command)

L8-16 Commandable.__init_subclass__ 遍历类体，any(getattr(value,'__isabstractmethod__',False) ...) 判断子类是否还留着未实装的抽象方法；若是纯实装类（无 own_abstract）才 append。这是“import 即注册”模式——只要某个 converter 模块被 import，它的类就进表。Qwen3TTSConverter 继承 DcpConverter 并实装了 dcp_to_hf，但 hf_to_dcp 留作 pass（非 abstractmethod，是空实现），所以它会被注册。

L23-39 Converter 定义三个 abstractmethod（hf_to_mm/mm_to_hf/resplit）给 VLM 路径；L41-49 DcpConverter 定义 hf_to_dcp/dcp_to_hf 给 FSDP 路径。两类都继承 Commandable，所以 VLM converter 与 FSDP converter 会混在同一张 CLI 子命令表里。子命令名取自类名（jsonargparse 默认），方法名即子命令的 action。

HF→DCP：分片并发改写

HF→DCP 把一个标准 HF 目录（多个 .safetensors）整体转成 DCP。核心思路：沿用 HF 自身的分片边界——每个 safetensors 文件对应一个 DCP shard，逐文件 load_file(device='cpu') → 加前缀 → 包进 {'model': state_dict} → partial_save_dcp_state_dict 写一个 shard。分片边界由源文件决定，不按 tensor 重新切。

def _process_single_file(idx, safe_path, save_path, state_dict_convert_func, add_checkpoint_version):
    """
    Process a single safetensors file: load, convert, and save as DCP shard.
    This function is designed to be called in a worker thread.

    Args:
        idx: File index, used as part_idx for DCP storage prefix.
        safe_path: Path to the safetensors file to load.
        save_path: Directory path to save the DCP shard.
        state_dict_convert_func: Optional function to convert the loaded state dict.
        add_checkpoint_version: If True, add checkpoint_version to the save dict.
    """
    save_path = Path(save_path)

    # Each worker creates its own storage_writer to avoid thread-safety issues
    storage_writer = FileSystemWriter(save_path)

    # Load and convert
    state_dict = load_file(str(safe_path), device="cpu")
    if state_dict_convert_func:
        state_dict = state_dict_convert_func(state_dict)

    save_dict = {"model": state_dict}
    if add_checkpoint_version:
        save_dict["checkpoint_version"] = DCP_CHECKPOINT_VERSION

    # Save as DCP shard
    global_meta, all_write = partial_save_dcp_state_dict(save_dict, storage_writer, part_idx=idx)

    return idx, global_meta, all_write

L25-54 _process_single_file 是 worker 线程体。L40 FileSystemWriter(save_path) 每个 worker 自己建一个 writer（注释明说是“avoid thread-safety issues”——FileSystemWriter 非线程安全，并发必须 per-thread 实例）。L43 load_file(device='cpu') 强制 CPU 加载，避免把权重搬到 GPU。L47-49 save_dict={'model': state_dict}，首个 shard（i==0）额外塞 checkpoint_version=3.0（DCP_CHECKPOINT_VERSION）。L52 part_idx=idx 决定 storage 前缀 __{idx}_，避免分片文件名撞车。

L82-86 的 _StoragePrefix(f'__{i+part_idx}_') 是每个 shard 的落盘前缀，all_writes.wait() 阻塞等写完。多 worker 时各自独立建 planner/writer，最后由主线程 merge_meta_info 把各 shard 的 state_dict_metadata 用 {**a, **b} 合并——前提是各 shard 键不相交，注释明说 disjoint。这正是按 HF 文件边界分片能成立的原因：每个 safetensors 文件的键集天然不相交。

NPU 迁移 · DCP 全程 CPUload_file(device='cpu') 与 FileSystemWriter 都不碰 NPU，DCP 读写是纯 CPU+磁盘 I/O。HF→DCP 无需上 NPU，转换可在普通 CPU 节点跑。唯一注意 1.7B 权重全量驻留 CPU 内存（sharded 模式逐文件处理可降峰值），多 worker 时每个 worker 各持一份 shard，num_workers 调大受内存约束。与硬件迁移无关，纯资源规划。

DCP→HF：partial-load 分片合并

反向 DCP→HF 更精细：不是把整个 DCP 一次性 load 进内存，而是按目标 HF 分片的需求键集合 partial-load——对每个目标 safetensors 文件，只从 DCP metadata 里取它需要的那些键，partial load 后转回 HF 键名、写一个 safetensors shard。这样峰值内存只占一个 shard。merge_dcp_to_hf_sharded 是这条路径，merge_dcp_to_hf 是它的非分片简化版（全量 load）。

def _process_single_dcp_shard(
    idx,
    safetensor_file,
    selected_keys,
    load_dir,
    save_dir,
    metadata,
    state_dict_convert_func,
    hf_metadata,
):
    """
    Process a single HF safetensors shard: partial-load DCP weights, convert, and save.
    This function is designed to be called in a worker thread.

    Args:
        idx: File index, used to restore task ordering in logs/errors.
        safetensor_file: Target safetensors filename.
        selected_keys: DCP keys needed by this target safetensors shard.
        load_dir: Directory path to load the DCP checkpoint.
        save_dir: Directory path to save the HF shard.
        metadata: Full DCP metadata. It is read-only in this worker.
        state_dict_convert_func: Optional function to convert the loaded state dict.
        hf_metadata: Metadata written into the target safetensors file.
    """
    load_dir = Path(load_dir)
    save_dir = Path(save_dir)

    # Each worker creates its own storage_reader to avoid thread-safety issues.
    storage_reader = FileSystemReader(str(load_dir))

    partial_metadata = extract_metadata(selected_keys, metadata)
    partial_state_dict = partial_load_dcp_state_dict(partial_metadata, storage_reader)
    partial_state_dict = partial_state_dict["model"] if "model" in partial_state_dict else partial_state_dict

    if state_dict_convert_func:
        partial_state_dict = state_dict_convert_func(partial_state_dict)

    save_file(partial_state_dict, save_dir / safetensor_file, metadata=hf_metadata)

Qwen3-TTS 专用：speaker[3000] 注入与删 speaker_encoder

Qwen3TTSConverter.dcp_to_hf 是本页核心。它继承 DcpConverter，实装了 dcp_to_hf（hf_to_dcp 留空 pass）。相比通用 merge_dcp_to_hf，它多了三件 Qwen3-TTS 专属事：(1) 从 DCP state_dict 里重建 speaker_encoder（取 speaker_encoder.* 键去前缀 load）；(2) 用 ref 音频算 mel、过 speaker_encoder 得 speaker 向量，写进 talker.model.codec_embedding.weight[3000]；(3) 删掉所有 speaker_encoder 权重再写 safetensors。这三步与 Qwen3-TTS 原生 sft_12hz.py 保存段语义一致——只是原生版在训练循环里用首步缓存的 speaker 向量，MM 版在转换期现算。

L85-96 是 mel 计算：librosa.load(speaker_audio_path, sr=None, mono=True) 读 ref 音频（保留原采样率），转 float32，过 mel_spectrogram(n_fft=1024, num_mels=128, sampling_rate=24000, hop_size=256, win_size=1024, fmin=0, fmax=12000)——这组超参与 Qwen3-TTS 原生 dataset.py 的 mel_spectrogram 完全一致（见 target/sft-data-collate 的 risk-stft-cpu）。.transpose(1,2) 把 (1,T_mel,128) 调成模型期望的 (1,T_mel,128) 布局。

L97 state_dict['talker.model.codec_embedding.weight'][3000] = _speaker_encoder(mels).to(weight.dtype)——这一行是音色定制的落点：speaker_encoder 前向得向量，搬到 codec_embedding 表的第 3000 行。与原生版 codec_embedding.weight[3000] = target_speaker_embedding[0] 同位同语义，区别是原生版用训练首步缓存的向量、MM 版转换期现算。L99-105 改 config：tts_model_type='custom_voice'、spk_id={speaker_name:3000}、spk_is_dialect={speaker_name:False}——把新音色注册到 3000 索引。

L106-109 删 speaker_encoder 权重：keys_to_drop = [k for k in state_dict if k.startswith('speaker_encoder')] 逐个 del——推理不再需要它，留在 safetensors 里是死重。L111-114 把改后的 talker_config 写回 config.json（indent=2, ensure_ascii=False 保中文不转义）。L116-117 save_file(state_dict, save_dir/model.safetensors) 单文件落盘（非 sharded），L118 set_directory_permissions 统一权限（DIR_MODE 0o750 / FILE_MODE 0o640）。

NPU 迁移 · mel+speaker_encoder 转换期 CPU转换期的 mel_spectrogram 与 Qwen3TTSSpeakerEncoder 前向都在 CPU 跑（librosa + torch CPU）。建议保持 CPU，与 target 页 STFT 钉 CPU 同一判断：mel 的 STFT 复数路径在 CANN 上不稳，且转换期对吞吐不敏感。speaker_encoder 是 conv/linear，即便上 NPU 也无特殊算子，但没必要——ref 音频只有几秒、mel 只几帧。.to(weight.dtype) 这里 weight.dtype 是 codec_embedding 的 dtype（通常 bf16/fp32），speaker 向量 cast 到该 dtype 写入，无精度风险。待真机验证：若训练侧 weight 是 bf16 而 speaker_encoder 跑 fp32，cast 链 fp32→bf16 是一次性、可接受。
NPU 迁移 · dtype/device 一致性DCP 合并后的 state_dict 全程 CPU 张量，save_file 写 safetensors 无 NPU 鬼风险。唯一 device 一致性点：_speaker_encoder(mels).to(weight.dtype) 只 cast dtype 不迁 device——speaker_encoder 输入 mels 在 CPU、weight 在 CPU，所以隐式同 device，不会报跨 device 错。但若有人把 weight 搬到 NPU 再调 dcp_to_hf，state_dict['…weight'][3000] = cpu向量 会因跨 device 赋值报错。保守做法：转换全程保持 CPU，.to(weight.device) 而非 .to(weight.dtype) 更稳（两者在 CPU 下等价）。待真机验证转换侧不应触碰 NPU。
