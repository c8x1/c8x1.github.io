学习轨道：源码深读

LLM 多节点边界

逐段拆 MindSpeed-LLM 的多节点规模化与容错骨架：controller/processor 节点发现、ARF 进程组重建、DP replica group、弹性 scale-in/scale-out 重建、跨 rank optimizer 数据修复、通信算子的 wrapper 改写——对照 Qwen3-TTS 单进程 accelerate 循环，讲清“单机多卡训练怎么扩到多机、节点挂了怎么不重训而续上”这套工程。

学习目标

  • 定位 tft_init_controller_processor / tft_train / initialize_distributed_wrapper 三处入口行号，说出节点发现、controller/processor 启动、replica group 建立各自由谁负责
  • 说清 scale_in_rebuild_callback 重建进程组的步骤顺序（world group → dp_cp replica → 不变 group → model/optim group → num_microbatches → dataset），以及为何要 reinit_process_group(rebuild_link=True)
  • 区分 ARF（节点重启全量重建进程组）与 elastic（弹性 scale-in/scale-out 局部重建）两条容错路径的触发条件与重建粒度
  • 解释跨 rank optimizer 修复用 send/recv 点对点 + ByteTensor 序列化在 NPU 上传输的机制，以及 finalize_model_grads 的 allreduce 走 TP/embedding group 的多机通信分工

前置知识

  • 读过 reference/mm-training.html，知道 MindSpeed 用 Megatron 的 mpu 管并行组、HCCL 为通信后端
  • 理解 data_parallel / tensor_parallel / pipeline_parallel 三类进程组的概念
  • 知道 torch.distributed.new_group / reinit_process_group / barrier / send / recv 这些原生集合通信原语

总览：从单机多卡到多机的扩展路径

Qwen3-TTS 的 SFT 循环是单进程 accelerate：一个 Accelerator 包住 model/opt/loader，多卡靠 DDP，通信靠 NCCL（迁 NPU 要改 HCCL），挂了就重训——它没有节点级容错。MindSpeed-LLM 在这条路径上叠了一整层工程，把规模化与容错做进训练引擎本身：节点发现（controller/processor 两角色）、进程组管理（mpu 的 TP/CP/EP/DP/PP 全套 + HA 加的 replica group）、容错重建（ARF 全量重建、elastic 局部 scale-in/scale-out）、跨 rank 数据修复（send/recv 传 optimizer state）。这套代码主要在 mindspeed_llm/core/high_availability/ 与 mindspeed_llm/tasks/high_availability/，靠给 Megatron 的原生通信函数打 wrapper 注入，不侵入主干训练循环。

上图三条重建路径的粒度不同。ARF（node reboot）是节点重启后把所有进程组（dp/dp_cp/cp/mp/tp/pp/ep/replica）从头建一遍，靠 reinit_process_group(rebuild_link=True) 重建通信链路。elastic scale-in 是缩容（去掉若干 dp rank）后只重建受影响的 dp_cp replica group 与 model/optim 关联组，并把 num_microbatches 按新 dp_size 重算以保持总 workload。elastic scale-out 是扩容的反向。三者最终都要做跨 rank 数据修复：把存活的 rank 上的 optimizer state（含 momentum、step、loss scale）通过 torch.distributed.send/recv 点对点传给新 rank，让新 rank 接着练而不重训。Qwen3-TTS 单进程路径完全没有这一层，规模化迁移的核心增量就在这里。

节点发现：controller/processor 两角色

tft_init_controller_processor()（L40）是整个 HA 路径的启动点，被 initialize_distributed_wrapper 在标准分布式初始化内部调用（见下节 wrapper）。它把每个 rank 分成两个角色：controller（只在 rank 0 起，单点协调）和 processor（每个 rank 都起，接收 controller 指令）。两者地址从环境变量 TTP_ADDR / CONTROLLER_ADDR / PROCESSOR_ADDR 取，端口固定 8000。注意 controller 只在 MINDX_TASK_ID 未设且 rank==0 时起——这是 MindX 训练平台与裸 torchrun 两种部署的分流。

    default_ip = '127.0.0.1'
    ttp_ip = os.getenv('TTP_ADDR', default_ip)
    controller_ip = os.getenv('CONTROLLER_ADDR', default_ip)
    if controller_ip == default_ip:

    processor_ip = os.getenv('PROCESSOR_ADDR', default_ip)

    world_size = args.world_size

    enable_elastic_training = args.enable_elastic_training if hasattr(args, 'enable_elastic_training') else False

L55-57 读三个开关：enable_worker_reboot（worker 节点可重启）、enable_hbmfault_repair（HBM 显存故障修复）、enable_elastic_training（弹性缩扩容）。这三个 flag 决定后面 controller 会接哪种回调。L59 enable_mindx = os.getenv('MINDX_TASK_ID')——这是判断是不是 MindX 平台托管的标志，MindX 自己管 controller 时本进程就不再起 controller（L60 enable_mindx is None 才 tft_init_controller）。L60-62 rank0 起 controller + tft_start_controller(ip, 8000)；L63-65 所有 rank 起 processor + tft_start_processor(ip, 8000)。L66-67 若本节点是 reboot 节点（之前挂过），置 ARF reboot flag——这个 flag 会让后续通信 wrapper 短路返回。

这套 controller/processor 是 mindio_ttp 框架（Training Task Platform）的能力，本仓库只调它、不实现它。controller 负责在故障时下发 repair/rollback/rebuild 指令给各 processor，processor 收到后调本仓库注册的回调。所以真正的容错编排不在本仓库内——本仓库提供的是“被回调时该怎么重建进程组、怎么传 optimizer state”的实现。迁移读者要理解：这套机制强依赖 mindio_ttp，脱离 MindX/mindio_ttp 部署时这些回调不会被触发。

NPU 迁移 · controller 单点controller 只在 rank0 起，是协调单点。多机训练里 rank0 所在节点若先挂，整条 HA 编排就停。NPU 场景下 controller 走 TCP（ip+8000），不占 NPU 算力，但需保证 rank0 节点的网络可达性与 mindio_ttp 进程稳定。Qwen3-TTS accelerate 路径没有 controller——多机迁移若想要容错就得引入这一层，否则仍是“挂了重训”。是否值得引入取决于训练时长：12Hz 单说话人 SFT 几小时可重训，百亿 LLM 训练几天则必须 HA。待真机验证 mindio_ttp 在 910B 集群上的稳定性。

训练入口包装：tft_train 与 ARF reboot 短路

tft_train()（L112）是 Megatron 原生 train() 的 HA 包装层，带 @tft_exception_handler 装饰。它做两件事：L116 set_mindio_export_version("MindSpeed-LLM") 告知 mindio_ttp 当前是 MindSpeed-LLM 版本（mindio_ttp 内部按版本走不同修复逻辑）；L122-123 若本节点是 ARF reboot 节点，直接 raise RuntimeError("ARF FINISH") 退出本进程——让 controller 重新拉起一个干净进程，而不是在脏状态上继续。

@tft_exception_handler
def tft_train(*args, **kwarg):
    try:
        from mindspeed_llm.training.training import train
        # gitee MindSpeed-LLM/1.0.0
        set_mindio_export_version("MindSpeed-LLM")

    except ModuleNotFoundError:
        ttp_logger.error(f'Mindio only support MindSpeed-LLM!')
        raise

    if tft_is_arf_reboot_node():
        raise RuntimeError("ARF FINISH")

    return train(*args, **kwarg)

L114 from mindspeed_llm.training.training import train 延迟 import——避免循环引用（HA 模块被 training 在初始化时引用，反过来再 import training 的 train 会成环）。L118-120 若不是 mindio_ttp 环境（ModuleNotFoundError）直接报错退出，因为整套 HA 强依赖 mindio_ttp。L125 return train(*args, **kwarg) 在非 reboot 节点上把控制权交还 Megatron 原生 train。

L128-139 的 tft_train_wrapper(fn) 是给任意训练函数套壳的通用装饰器工厂：同样 set_mindio_export_version + ARF reboot 短路 + 透传 fn。它的作用是让训练循环里的子函数（如 evaluate、save_checkpoint）也能被 HA 包装，reboot 节点上短路掉避免在重建期间跑半截逻辑。注意它返回的是 wrapper 函数对象，调用方要 tft_train_wrapper(train)(args) 这样用。

ARF reboot 短路是防脏状态续训的关键。节点重启后 NPU 上的显存、HCCL 链路、进程组全是脏的，此时若 train 继续，会在已失效的 group 上发 allreduce 报错或静默 hang。所以 HA 的策略是：reboot 节点直接退进程，让 controller 拉起新进程走完整重建流程，而不是在旧进程上修修补补。这与 Qwen3-TTS accelerate 的“进程不退、靠 try/except 续”是完全不同的容错哲学。

分布式初始化包装：device set 与 replica group order

initialize_distributed_wrapper(fn)（在 tasks/high_availability/initialize_patch.py L8）包住 Megatron 的 initialize_distributed。它在原生初始化前后各插一段：前段 L11-13 torch.cuda.set_device(rank % device_count) 显式设 NPU device（字面 cuda，经 torch_npu patch 落 NPU）；中段 L14-15 调 tft_init_controller_processor()（上节那个）起 controller/processor；L16 fn(*args, **kwargs) 跑原生分布式初始化；后段 L17-28 按 tp-cp-ep-dp-pp（或 tp-cp-ep-pp-dp）order 建 replica dp group。

def initialize_distributed_wrapper(fn):
    @wraps(fn)
    def wrapper(*args, **kwargs):
        device_count = torch.cuda.device_count()
        device = get_args().rank % device_count
        torch.cuda.set_device(device)
        from mindspeed_llm.core.high_availability import tft_init_controller_processor, ttp_initialize_replica_dp_group
        tft_init_controller_processor()
        fn(*args, **kwargs)
        world_size: int = torch.distributed.get_world_size()
        args = megatron.training.get_args()
        order = 'tp-cp-ep-dp-pp' if not args.use_tp_pp_dp_mapping else 'tp-cp-ep-pp-dp'
        ttp_initialize_replica_dp_group(
            args.pipeline_model_parallel_size,
            args.tensor_model_parallel_size,
            args.context_parallel_size,
            args.expert_model_parallel_size,
            args.expert_tensor_parallel_size,
            world_size,
            order
        )
    return wrapper

L19 order = 'tp-cp-ep-dp-pp' if not args.use_tp_pp_dp_mapping else 'tp-cp-ep-pp-dp'——replica group 的 rank 编排顺序，决定哪些 rank 互为副本（互备）。默认 order 把 DP 放在 PP 前（dp 在内层），use_tp_pp_dp_mapping 时把 PP 提到 DP 前。这个 order 直接影响 scale-in 时 fault rank 落在哪个 replica group、能不能从副本恢复。L20-28 把 pp/tp/cp/ep/etp 全部传给 ttp_initialize_replica_dp_group，它在内部按 order 切分 dp rank 成 REPLICA_NUM=2 份（见 tft_replica_group.py 的 build_dp_cp_replica_group）。

L32-40 的 build_train_valid_test_data_iterators_wrapper 是数据迭代器包装：若本节点是 ARF reboot 节点，get_args().do_train = True 强制设为训练态。这是因为重启节点可能从评估态被拉起，需保证它进的是训练分支。这两个 wrapper 是通过 monkey-patch Megatron 原生函数注入的（patch 管理器统一打），所以主干训练代码里看不到 HA 调用——迁移读者若只读 training.py 看不到这层，必须来看 tasks 下的 patch。

NPU 迁移 · replica group orderorder='tp-cp-ep-dp-pp' 这个字符串决定 fault rank 的副本归属，改错会导致 scale-in 时找不到健康副本。Qwen3-TTS 迁移到多机 HA 时，必须与实际 PP/DP 拓扑对齐这个 order：单 PP（TTS 典型）默认 order 即可；多 PP 且想副本跨 PP stage 才用 use_tp_pp_dp_mapping。NPU 多机拓扑下还要确认 HCCS 带宽能支撑副本间的 optimizer state 同步（replica group 跨节点时数据量级 = optimizer state 全量）。保守：先单机多卡验 replica 逻辑，再上多机。待真机验证跨节点副本恢复延迟。

弹性 scale-in：重建进程组与重算 microbatch

scale_in_rebuild_callback(new_dp_ranks, new_world_ranks, args, params)（L34）是缩容回调——节点故障后去掉若干 dp rank，剩余 rank 重组进程组继续练。它要求只缩 DP（L46 check_scale_in_params 强校验 scale-in-strategy == "DP"），且不支持 EP/CP>1（L53-55）。流程：建新 world group → 找 fault rank 在哪个 replica group → 重建受影响的 dp_cp replica group → 重建不变组 → 更新 model/optim 关联 group → 重算 num_microbatches → 重建 dataset → barrier 同步。

def scale_in_rebuild_callback(new_dp_ranks: list, new_world_ranks: list, args, params: str):
    """
    Args:
        new_dp_ranks: List of new data parallel ranks after scale-in
        new_world_ranks: List of new world ranks after scale-in
        args: Training arguments containing model and optimizer information
        params: Scale-in strategy parameters
    This function handles the scale-in operation by rebuilding process groups,
    updating replica groups, reinitializing model and tensor groups, and preparing
    the training environment for continued execution with fewer resources.
    """
    ttp_logger.info(f"scale-in strategy params: {params}, new_dp_ranks: {new_dp_ranks}, new_world_ranks: {new_world_ranks}")
    elastic_training_common.check_scale_in_params(params)
    cur_rank = torch.distributed.get_rank()
    if len(args) <= ha_constant.OPTIM_INDEX:
        raise RuntimeError(f"args error: {args}")
    models = args[ha_constant.MODEL_INDEX]
    optimizer = args[ha_constant.OPTIM_INDEX]
    arguments = get_args()
    if arguments.expert_model_parallel_size > 1 or arguments.context_parallel_size > 1:
        raise RuntimeError(f"not support ep or cp bigger than 1, but got ep: {arguments.expert_model_parallel_size} "
                           f"cp: {arguments.context_parallel_size} ")
    elastic_training_common.SCALE_IN_WORLD_GROUP = torch.distributed.new_group(new_world_ranks, use_local_synchronization=True)
    ttp_logger.info(f"backend: {arguments.distributed_backend}, rank: {cur_rank}, world_size: "
                           f"{len(new_world_ranks)}, new_world_ranks: {new_world_ranks}")
    old_dp_ranks = torch.distributed.get_process_group_ranks(
        mpu.get_data_parallel_group(with_context_parallel=True))
    dp_cp_replica_group = tft_replica_group.ttp_get_dp_cp_replica_group()
    dp_cp_replica_ranks = torch.distributed.get_process_group_ranks(dp_cp_replica_group)
    if len(dp_cp_replica_ranks) == 0:
        raise RuntimeError(f"dp_cp_replica_ranks is empty")
    elastic_training_common.ORIGIN_DP_SIZE = len(old_dp_ranks)
    elastic_training_common.ORIGIN_NUM_MICRO_BATCHES = get_num_microbatches()
    both_replica_group_fault, changed_old_dp_ranks = get_changed_old_dp_ranks(cur_rank, old_dp_ranks, new_dp_ranks,
                                                                   dp_cp_replica_ranks)
    fault_idxs, fault_local_idxs, fault_first_group = get_fault_msgs(cur_rank, old_dp_ranks, changed_old_dp_ranks,
                                                                     new_dp_ranks,
                                                                   dp_cp_replica_ranks)
    build_scale_in_dp_cp_replica_group(fault_local_idxs, fault_first_group,
                                       both_replica_group_fault, changed_old_dp_ranks)
    rebuild_not_changed_group(cur_rank, both_replica_group_fault, arguments)
    ttp_logger.info(f"rank: {cur_rank} rebuild not changed group done")
    update_model_and_optim_related_group(models, optimizer)
    change_num_micro_batches(old_dp_ranks, new_dp_ranks, arguments)
    elastic_training_common.update_scale_in_flag(True)

L56 SCALE_IN_WORLD_GROUP = torch.distributed.new_group(new_world_ranks, use_local_synchronization=True) 建新 world 组，use_local_synchronization=True 让本 rank 不阻塞等所有 rank 同步就完成建组——这是缩容时的关键，因为被去掉的 rank 已不可达，不能等它们。L65-66 把旧 dp_size 与旧 num_microbatches 存进 elastic_training_common 全局态，后面重算用。L67-73 算出 fault rank 落在左/右哪个 replica group（get_changed_old_dp_ranks / get_fault_msgs），据此决定能不能用另一副本的健康 rank 顶替。L76 update_model_and_optim_related_group(models, optimizer) 把 optimizer 内部缓存的 data_parallel_group 指针换成新的 replica group——不换的话 optimizer.step 会往失效 group 发 allreduce。

L77 change_num_micro_batches(old_dp_ranks, new_dp_ranks, arguments) 是规模化的精髓：缩容后 dp_size 变小，但总 workload 要守恒，所以每个 rank 的 num_microbatches 要变大。L78 update_scale_in_flag(True) 置全局 scale-in 运行态，让通信 wrapper 把后续 all_reduce 的 group 从默认改到 SCALE_IN_WORLD_GROUP。L79-82 把所有 timer 的 barrier group 改到新 world group（计时器在计时结束时隐式 barrier，组错了会 hang 在失效 rank 上）。L88 build_dataset(args) 按新 dp 切分数据，L89 torch.distributed.barrier() 同步后才继续训练。

def change_num_micro_batches(old_dp_ranks, new_dp_ranks, arguments):
    """
    Args:
        old_dp_ranks: List of data parallel ranks before scale-in
        new_dp_ranks: List of data parallel ranks after scale-in
        arguments: Command line arguments
    This function recalculates the number of microbatches based on the new data parallel
    size while maintaining the total workload. It ensures even distribution of work
    across the remaining ranks and updates the global microbatch calculator.
    """
    old_dp_size = len(old_dp_ranks)
    new_dp_size = len(new_dp_ranks)
    total_num_microbatches = get_num_microbatches() * old_dp_size
    new_num_microbatches = total_num_microbatches // new_dp_size
    elastic_training_common.HAS_DATA = total_num_microbatches % new_dp_size
    if elastic_training_common.HAS_DATA and torch.distributed.get_rank() in new_dp_ranks[:elastic_training_common.HAS_DATA]:
        new_num_microbatches += 1
    ttp_logger.info(f"new num_micro_batches: {new_num_microbatches}, new_dp_size: {new_dp_size},"
                           f"_GLOBAL_NUM_MICROBATCHES_CALCULATOR:"
                           f" {num_microbatches_calculator._GLOBAL_NUM_MICROBATCHES_CALCULATOR}")
    if arguments.rampup_batch_size is not None and len(arguments.rampup_batch_size) == 3:

NPU 迁移 · rebuild_link 重建 HCCLdestroy_sub_process_group（elastic_training_common.py:100-112）在销毁失效 group 时先 torch.distributed.reinit_process_group(group, rebuild_link=True) 再 destroy_process_group——rebuild_link=True 让 HCCL 重建底层通信链路（not just 逻辑组）。NPU 上这意味着重新建 HCCS 连接、重新分配通信 buffer，是有开销的操作，但必要：失效 rank 的链路已断，不 rebuild 会让后续通信卡在失效连接上。多机 scale-in 时 rebuild_link 涉及跨节点链路重建，延迟比单机大。保守：故障检测要快（controller 心跳），rebuild 期间训练暂停，barrier 保证所有 rank 重建完再继续。待真机验证多机 rebuild_link 延迟。

def save_and_send_ckpt(dest_rank, optim_idx, train_args):
    """
    Save memory checkpoint and send to dest rank.
    """
    t1 = time.time()
    rank = torch.distributed.get_rank()
    state_dict = save_memory_ckpt(train_args[ha_constant.OPTIM_INDEX], train_args[ha_constant.SCHEDULER_INDEX], optim_idx)
    buffer = io.BytesIO()
    torch.save(state_dict, buffer)
    state_dict_bytes = buffer.getvalue()
    state_dict_tensor = torch.ByteTensor(torch.ByteStorage.from_buffer(state_dict_bytes)).to('npu')

    t2 = time.time()
    # Send tensor size first
    size_tensor = torch.tensor([state_dict_tensor.numel()], dtype=torch.long).to('npu')
    torch.distributed.send(size_tensor, dst=dest_rank, group=get_repair_group())

    # Send the serialized state_dict tensor
    torch.distributed.send(state_dict_tensor, dst=dest_rank, group=get_repair_group())

    ttp_logger.info(f"[repair] send rank {rank} save and serialize ckpt:{t2 - t1:.3f}s, "
                           f"send ckpt:{time.time() - t1:.3f}s")

NPU 迁移 · ByteTensor 走 NPU把序列化后的 state_dict 包成 torch.ByteTensor 再 .to('npu') 走 distributed.send/recv，意味着字节流要在 NPU 通信域上点对点传。这是 HA 专属的通信路径，不走集合通信（allreduce/broadcast），而是 send/recv 点对点。NPU 上 send/recv 走 HCCL 的点对点原语，行为与 CUDA 等价，但要注意：1) ByteTensor 在 NPU 上的拷贝开销（CPU 序列化→NPU 字节张量→HCCL 传→NPU 字节张量→CPU 反序列化），数据量大时这趟往返不可忽略；2) optimizer state 全量传输，大模型（百亿参数）的 FP32 momentum 是几十 GB，传输时长是分钟级。保守：修复时优先传分片后的 optimizer state（optim_idx 参数就是分片索引，按 expert/attention 分块传），减少单次传输量。待真机验证大模型 optimizer state 跨节点传输带宽。
NPU 迁移 · 多机 RNG 一致性容错续训绕不开 RNG 状态：训练里的 dropout、数据 shuffle、MoE router 都依赖 RNG，故障前各 rank 的 RNG 状态已前进若干步。重建后新 rank 若 RNG 重置，续训的样本顺序与 dropout 模式与故障前不一致，等效于换了随机种子续训，可能引入分布漂移。本仓库的修复路径传的是 optimizer/scheduler state（L121 save_memory_ckpt 含 optimizer + scheduler），未见显式传 RNG state（torch.get_rng_state / set_rng_state）。Megatron 的数据迭代器靠 consumed_samples 定位数据进度，RNG 重建靠“按 consumed_samples 重新 init sampler”而非传 state。NPU 上 RNG（Philox 等）与 CUDA 不同，多机续训后 RNG 一致性需单独核验。保守：续训后先跑几个 step 对比 loss 曲线是否与故障前平滑衔接。待真机验证 RNG 续训一致性。

def communication_wrapper(fn):
    @wraps(fn)
    def wrapper(*args, **kwargs):
        from megatron.training import get_args

        try:
            arguments = get_args()
        except AssertionError:
            return fn(*args, **kwargs)
        if arguments.enable_high_availability:
            from mindspeed_llm.core.high_availability import tft_is_arf_reboot_node

            if tft_is_arf_reboot_node():
                return None
            if arguments.enable_elastic_training:
                group_index = 2
                return torch_wrapper(fn, group_index, *args, **kwargs)
        return fn(*args, **kwargs)

    return wrapper


def barrier_wrapper(fn):
    @wraps(fn)
    def wrapper(*args, **kwargs):
        from megatron.training import get_args

        try:
            arguments = get_args()
        except AssertionError:
            return fn(*args, **kwargs)

def allreduce_layernorm_grads(model: List[torch.nn.Module], config: TransformerConfig):
    """
    All-reduce layernorm grads (for sequence parallelism).
    """

    # All-reduce layernorm parameters across model parallel nodes
    # when sequence parallelism is used
    if parallel_state.get_tensor_model_parallel_world_size() > 1 and (
        config.sequence_parallel or config.qk_layernorm
    ):
        grads = []
        for model_chunk in model:
            for name, param in get_attr_wrapped_model(model_chunk, 'named_parameters')():
                if not param.requires_grad:
                    continue
                if (
                    param.requires_grad
                    and getattr(param, 'sequence_parallel', False)
                    or 'q_layernorm' in name
                    or 'k_layernorm' in name
                ):
                    grad = param.main_grad
                    grads.append(grad.data)
        if grads:
            coalesced = _flatten_dense_tensors(grads)
            torch.distributed.all_reduce(
                coalesced, group=parallel_state.get_tensor_model_parallel_group()
            )
            for buf, synced in zip(grads, _unflatten_dense_tensors(coalesced, grads)):
                buf.copy_(synced)

边界与不确定

HA 这套机制强依赖 mindio_ttp 框架与 MindX 训练平台。本仓库只提供“被回调时怎么重建/修复”的实现（register_callbacks 注册的 repair/rollback/rebuild 回调），真正的故障探测、controller 下发指令、进程拉起都在 mindio_ttp 内，本页未逐行核验 mindio_ttp 内部。脱离 MindX/mindio_ttp 部署时，enable_high_availability/enable_elastic_training 这些 flag 即便设了，回调也不会被触发——等于关掉了 HA。Qwen3-TTS 迁移若不在 MindX 上跑，这整套容错不可用，只能靠“挂了重训”+ checkpoint 续训。
change_num_micro_batches（L208-232）直接改写 num_microbatches_calculator._GLOBAL_NUM_MICROBATCHES_CALCULATOR 的私有字段（data_parallel_size、micro_batch_times_data_parallel_size、num_micro_batches），这是戳 Megatron-Core 全局单例私有态的写法，强绑具体 Megatron 版本。本页据字段名与赋值语义推断其作用（重算 microbatch 数守恒 workload），未在 Megatron-Core 内部逐行核验这些字段的消费链路。升级 Megatron 时此函数可能因字段改名而失效。
RNG 一致性这条是本页最不确定的判断。源码可见的修复路径传 optimizer + scheduler state，未见显式传 torch.get_rng_state() / NPU RNG state。本页推断“Megatron 靠 consumed_samples 重 init sampler 来重建数据进度、RNG 不传 state”，这是据 Megatron 数据迭代器设计惯例的推测，未逐行核验 sampler 重建时 RNG 是否真的能精确衔接故障前状态。多机续训 RNG 一致性是规模化训练的隐蔽风险点，待真机专项验证。
