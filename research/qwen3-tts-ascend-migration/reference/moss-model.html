MOSS Delay 模型

按 moss_tts_delay/modeling_moss_tts.py 的真实类层次逐段拆解 MOSS Delay 模型：单一 Qwen3 骨架 + n_vq 路外挂 embedding + (1+n_vq) 个并行 lm_head，靠 Delay-Pattern 时移调度让一帧的多层 RVQ 在一次前向并行出齐。讲清 emb_ext 汇总、多头 logits 与逐通道加权损失，对照 Qwen3-TTS 的 talker/code_predictor/codec 担差结构差异（单骨架并行 vs 双塔残差、32 路并行 vs 16 层 MTP、无独立 speaker 模块）——每段标出昇腾 910B 迁移的算子风险点。

学习目标

  • 在 modeling_moss_tts.py 里定位 MossTTSDelayModel 的 language_model/emb_ext/lm_heads 三个构造点并说出 1+n_vq 个头各预测什么
  • 说清 _compute_input_embeddings 把 (b,t,1+n_vq) 的 text 通道走 Qwen embed_tokens、n_vq 路音频通道走 emb_ext 逐路相加的汇总机制
  • 区分 Delay 模型的逐通道 CrossEntropy（labels (b,t,1+n_vq) 配 -100 屏蔽）与 Qwen3-TTS 的主 CE + 0.3×MTP 双损失结构差异
  • 解释 generate 里 audio_lengths/delayed_lengths 两个状态如何驱动 Delay-Pattern 的对角线移位采样，与 Qwen3-TTS 嵌套 generate 的 sub-talker 补全路径对照

前置知识

  • 读过 Qwen3-TTS 的复合模型架构页（talker + code_predictor + speaker_encoder）
  • 熟悉 HF transformers 的 PreTrainedModel 与 Qwen3Model 骨架
  • 理解 RVQ（残差向量量化）与 next-token 的 label shift

总览：单骨架 + 多头并行 + Delay-Pattern

MOSS Delay（MossTTSDelayModel）与 Qwen3-TTS 走的是两条完全不同的多码本建模路线。Qwen3-TTS 是双塔残差：一个 talker 学第 0 层 codec 的 next-token，一个 sub-talker（code_predictor）在 talker 隐状态上补 1-15 层、靠嵌套 generate 逐层补全。MOSS Delay 则是单骨架并行：一个 Qwen3-8B 骨架同时承担文本与全部 32 路 RVQ，骨架末层接 1+n_vq=33 个线性头，一帧的多层 RVQ 在一次前向里并行出齐；层间粗粒度依赖不靠第二个 transformer，而靠数据侧的 Delay-Pattern 时移——把第 k 层 RVQ 相对第 k-1 层往后挪 1 步，靠因果 attention 让第 t 步预测第 k 层时已“看见”第 t-1 步的第 k-1 层。

这张图与 Qwen3-TTS 架构页那张 talker→codec_head + talker→code_predictor 的嵌套图形成鲜明对照。MOSS 这边没有第二个 transformer：emb_ext 只是 32 个 nn.Embedding，lm_heads 只是 33 个 nn.Linear，真正的算力全在唯一一个 self.language_model = Qwen3Model(...) 里。这也意味着迁移时只有一个骨架的 attention 要改，不像 Qwen3-TTS 要同时覆盖 talker 的 20 层与 code_predictor 的 5 层——但代价是 8B 骨架比 Qwen3-TTS 的 talker（约 1B 量级）大得多，单次前向重。

配置契约：language_config + n_vq + 音频 token id

MossTTSDelayConfig（configuration_moss_tts.py:25）是扁平结构，不像 Qwen3-TTS 那样嵌套 talker_config→code_predictor_config 三层。它只有一个子配置 language_config（一个 Qwen3Config，:80-85），其余字段直接平铺在顶层。__init__ 末尾把 self.hidden_size 与 self.vocab_size 直接指向 language_config 的同名字段（:98-99）——也就是说骨架的维度与词表完全继承自 Qwen3，MOSS 只在上面加音频维度。

NPU 迁移 · attn 继承自 Qwen3language_config 是 Qwen3Config，骨架是 Qwen3Model，所以 _attn_implementation 的派发语义完全沿用 HF Qwen3。好处是迁移经验可平移自 Qwen3-TTS 的 talker 骨架（同样是 Qwen3 系注意力）；风险是 8B 骨架在 sdpa 路径下、12.5Hz 长帧率序列的 4D mask 显存峰值可能吃紧。保守方向：装载时显式 attn_implementation="sdpa"，先关 use_cache 验证正确性，再开 KV cache 测吞吐。

模型装配：emb_ext 汇总与多头 lm_head

MossTTSDelayModel.__init__（modeling:164-194）是整个模型的装配点，结构极简：一个骨架 + 两组 ModuleList。L170 self.language_model = Qwen3Model(config.language_config) 建 8B 骨架；L175-180 self.emb_ext 建 32 个 nn.Embedding(audio_vocab_size+1, hidden_size, padding_idx=None)；L185-191 self.lm_heads 建一个 text head（Linear(hidden_size, vocab_size)）+ 32 个 audio head（Linear(hidden_size, audio_vocab_size+1)）。没有 projection、没有 sub-talker、没有 speaker encoder——这是与 Qwen3-TTS 装配段最直接的差异。

    def __init__(self, config: MossTTSDelayConfig):
        super().__init__(config)
        self.config = config

        config.language_config.torch_dtype = config.torch_dtype
        
        self.language_model = Qwen3Model(config.language_config)

        # Audio VQ Embeddings (Extra channels)
        # Note: input_ids[..., 0] uses Qwen's embedding. 
        # input_ids[..., 1:] use these extensions.
        self.emb_ext = nn.ModuleList()
        for vq_idx in range(self.config.n_vq):
            # Add +1 for potential padding/special tokens logic if strictly required by upstream data prep
            self.emb_ext.append(
                nn.Embedding(self.config.audio_vocab_size + 1, config.language_config.hidden_size, padding_idx=None)
            )

        # Multi-Head Prediction Layers
        # Head 0: Main language head
        # Head 1..N: Audio VQ heads
        self.lm_heads = nn.ModuleList([
            nn.Linear(config.language_config.hidden_size, config.language_config.vocab_size, bias=False)
        ])
        for vq_idx in range(self.config.n_vq):
            self.lm_heads.append(
                nn.Linear(config.language_config.hidden_size, self.config.audio_vocab_size + 1, bias=False)
            )

        # Initialize weights and apply final processing
        self.post_init()

注意 L173-174 的注释：input_ids[..., 0] 用 Qwen 自带的 embedding（get_input_embeddings() 返回 self.language_model.get_input_embeddings()，modeling:196-197），input_ids[..., 1:] 用 emb_ext。所以 text 与 audio 的查表维度天然一致（都是 language_config.hidden_size），不需要 Qwen3-TTS 那个 text_projection(2048→1024) 做维度对齐——MOSS 把音频嵌入直接做在骨架维度上，省了一层投影。

L194 self.post_init() 后，骨架、emb_ext、lm_heads 全部就位。对照 Qwen3-TTS：Qwen3TTSForConditionalGeneration.__init__ 里还要条件建 speaker_encoder、注册 supported_speakers、装 speech_tokenizer；MOSS Delay 这些都没有——speaker 靠对话模板与参考音频文本承载，codec 是外部 moss_audio_tokenizer 子模块（Causal Audio Tokenizer，即 Cat），在本仓库里是空目录（.gitmodules 声明但未初始化）。

    def _compute_input_embeddings(self, input_ids: torch.LongTensor) -> torch.Tensor:
        """
        Computes the combined embeddings from text and multiple audio VQ channels.

        Args:
            input_ids: Shape (Batch, Seq_Len, 1 + n_vq)
        """
        inputs_embeds = self.get_input_embeddings()(input_ids[..., 0])

        # Add VQ Embeddings
        for i, embed_layer in enumerate(self.emb_ext):
            # i corresponds to channel i+1 in input_ids
            # We assume the data pipeline ensures indices are within range
            inputs_embeds = inputs_embeds + embed_layer(input_ids[..., i + 1])
        
        return inputs_embeds

_compute_input_embeddings（:199-214）是 embedding 汇总的唯一入口。L206 inputs_embeds = self.get_input_embeddings()(input_ids[..., 0]) 先用 Qwen embed_tokens 查 text 通道，得到 (b,t,d)。L209-212 for i, embed_layer in enumerate(self.emb_ext): inputs_embeds += embed_layer(input_ids[..., i+1]) 逐路把 32 个 audio 通道的 embedding 原地累加进同一个张量。最终 (b,t,d)，d = hidden_size。这与 Qwen3-TTS 的三路相加（text+codec+speaker）形似但更扁平：Qwen3-TTS 第 0 层 codec 走 codec_embedding、1-15 走 CP 子表、还有 speaker 注入位 6；MOSS 是 1+32 路全部原地 sum。

shape 跟踪：input_ids (b,t,1+n_vq=33) → 通道 0 查表 (b,t,d) → 逐 i 查 emb_ext[i] 得 (b,t,d) 原地 += 32 次 → inputs_embeds (b,t,d)。L254 的形状断言 input_ids.shape[-1] != self.config.n_vq + 1 报错，锁死这个 33 通道契约。延迟模式不在 embedding 层体现——时移发生在数据侧的 processing（apply_delay_pattern），模型只看到移好位的 input_ids。

NPU 迁移 · 逐路 embedding 累加L212 的 inputs_embeds += embed_layer(...) 是 32 次 elementwise 累加，NPU 原生支持无算子风险。但要确认 32 个 emb_ext 与 Qwen embed_tokens 同 device 同 dtype——若迁移时分了卡或 dtype 不一致，原地 += 会广播报错。保守方向：装载后统一 .to(device, dtype) 再进前向。32 路 Embedding 查表本身是 gather 算子，CANN 支持，无特殊风险。

前向：骨架出 hidden，多头并行算 logits

shape 跟踪：inputs_embeds (b,t,d) → 骨架 → last_hidden_state (b,t,d) → 复制 33 份 → 逐头 Linear → layer_logits = [logits_0 (b,t,V_text), logits_1..32 (b,t,V_audio)] 共 33 个张量的列表。损失阶段 labels 形状 (b,t,1+n_vq=33)，L331 cur_labels = labels[..., i] 按通道 i 取 (b,t)，与第 i 个 logits 算 CE。

NPU 迁移 · logits -inf 填充L300 logits[..., -1] = float("-inf") 是原地标量填充，elementwise 无风险。但 33 个头的 logits 张量在长帧率序列下（12.5Hz，1 小时≈45000 帧）总尺寸 = 33 × b × t × V_audio，显存峰值可能很高。bf16 下 -inf 是合法值，CANN softmax 会正确忽略。保守方向：长序列段先估测 logits 总显存，必要时减小 batch 或开梯度检查点（骨架声明 supports_gradient_checkpointing=True，modeling:94）。

逐通道损失：labels (b,t,33) 与加权聚合

损失段（modeling:310-379）是 MOSS Delay 与 Qwen3-TTS 损失结构的根本差异点。Qwen3-TTS 是主 CE + 0.3×MTP 两个损失按固定权重组合，主 talker 只对第 0 层 codec 算 CE、残差层由 sub-talker 的 MTP 损失负责。MOSS Delay 则是33 个通道各算一个 CE 再聚合，没有主辅之分。L326 all_token_nums = torch.sum(labels != -100, dim=1) 按 (b,c) 统计每样本每通道的有效 token 数。

            all_sum_losses = torch.stack(all_sum_losses_list, dim=1)

            # Weighted Loss Aggregation
            if channelwise_loss_weight is not None:
                if len(channelwise_loss_weight) != n_heads:
                    raise ValueError(f"channelwise_loss_weight length {len(channelwise_loss_weight)} != {n_heads}")
                
                w_tensor = torch.tensor(channelwise_loss_weight, device=all_sum_losses.device, dtype=all_sum_losses.dtype)
                
                # Sample losses: Weighted sum over channels per sample / Total weight
                # Normalize by token count per channel
                # Avoid division by zero with epsilon or mask
                token_counts_safe = all_token_nums.float().clamp(min=1.0)
                
                normalized_losses = all_sum_losses / token_counts_safe
                sample_losses = (normalized_losses * w_tensor).sum(dim=1) / w_tensor.sum()
                
                # Channel losses: Sum over batch / Sum tokens over batch
                total_loss_per_channel = all_sum_losses.sum(dim=0)
                total_tokens_per_channel = all_token_nums.sum(dim=0).float().clamp(min=1.0)
                channel_losses = total_loss_per_channel / total_tokens_per_channel
                
                # Final scalar loss
                loss = (channel_losses * w_tensor).sum() / w_tensor.sum()
            else:
                # Default average if no weights provided
                total_tokens = all_token_nums.sum().float().clamp(min=1.0)
                loss = all_sum_losses.sum() / total_tokens
                channel_losses = all_sum_losses.sum(dim=0) / all_token_nums.sum(dim=0).clamp(min=1.0)

L354-374 是加权聚合。若传了 channelwise_loss_weight（长度必须 = n_heads=33，否则 L355-356 报错）：L358 构造权重张量，L365 normalized_losses = all_sum_losses / token_counts_safe 每通道归一化，L366 sample_losses = (normalized_losses * w).sum(1) / w.sum() 加权样本损失，L371-372 channel_losses 逐通道均值，L374 loss = (channel_losses * w).sum() / w.sum() 最终标量。L375-379 是无权重默认路径：按总 token 数平均。

对照 Qwen3-TTS 的 loss = outputs.loss + 0.3 * sub_talker_loss：那边是两个语义不同的损失项（主 next-token + 残差 MTP），权重 0.3 是官方固定配方；MOSS 这边是33 个同语义通道损失的加权平均，权重 channelwise_loss_weight 是可配置的（默认等权 1）。迁移时 MOSS 的损失链无需特殊 NPU 处理——全是 CrossEntropy + elementwise reduce，但要注意 L358 的权重张量 device/dtype 要与 all_sum_losses 对齐（代码已显式 device=...dtype=...）。

NPU 迁移 · 33 路 CE 循环L328 的 for i, logits in enumerate(layer_logits) 是 Python 层 33 次循环、每次一次 CrossEntropy + view/sum。NPU 上这 33 次都是独立 kernel launch，launch 开销叠加可能成为训练吞吐瓶颈。保守方向：可考虑把 33 个 logits 预先 stack 成 (b,t,33,V_max) 一次性 CE（需处理 V_text≠V_audio 的维度不一致，可能要分两组：1 text + 32 audio），减少 launch 数。当前实现可跑通，性能待真机测。

生成流：Delay-Pattern 状态机采样

generate（modeling:393-526）是 Delay-Pattern 的灵魂。它不用 GenerationMixin，而是手写逐 step 循环（L443 for time_step in tqdm(range(max_new_tokens))），靠两个状态张量驱动：audio_lengths（每样本已生成的音频帧数）与 delayed_lengths（每样本当前已“解冻”可采样的 RVQ 层数）。这两个量决定每步对哪些 (样本, 通道) 采样、对哪些保持 pad。

对照 Qwen3-TTS 的生成流：Qwen3-TTS 每生成一个第 0 层 codec token 就嵌套调一次 code_predictor.generate 自回归补 15 层残差（两层 KV cache）；MOSS Delay 不嵌套，单层循环里 33 个头一次性出齐当帧全部层（靠 delay mask 控制哪些通道本步真采样）。所以 MOSS 的生成步数 ≈ 音频帧数，Qwen3-TTS 的步数 ≈ 帧数 ×（1 + 15 层 sub-talker 内循环）。MOSS 因此在 README 标注“inference speed high / parallel RVQ prediction”——并行性来自多头 + delay 调度，不来自嵌套。

NPU 迁移 · 单层 KV cacheMOSS 只有一层 KV cache（骨架的 past_key_values，L450 每步更新），没有 Qwen3-TTS 那种外层+内层双 cache 的显存峰值问题。CANN 的 DynamicCache update/concat 算子只需验证一次。但 8B 骨架 + 12.5Hz 长序列（README 称支持 1 小时≈45000 帧）下，单层 cache 的 K/V 张量 (b, n_kv_heads, t, head_dim) 仍会很长。保守方向：先关 use_cache 验证采样正确性（对短样本比对 CPU eager 基线），再开 cache 测长序列吞吐与显存。
NPU 迁移 · sample_token 采样核sample_token（inference_utils:111-145）内部用 torch.topk（apply_top_k:19-26）、torch.sort+cumsum（apply_top_p_optimized:44-59）、torch.multinomial（:143）。这几个算子在 CANN 上的覆盖度需真机确认——尤其 multinomial 与带 scatter_ 的 top-p 路径。保守方向：若 NPU 上 multinomial 不稳，可把采样退回 CPU（logits .cpu() 后采样再搬回），生成阶段采样计算量相对骨架前向很小，CPU 回退对吞吐影响有限。repetition penalty（inference_utils:62-108）有 Python 层 for h in range(H) 循环，长序列下也建议监测开销。

Codec 边界：外部 audio_tokenizer 与 Delay-Pattern 数据侧

MOSS Delay 的 codec（RVQ 编解码器）是仓库外部资产。processing 里 audio_tokenizer_class = "AutoModel"（processing_moss_tts:151），from_pretrained 时 audio_tokenizer = AutoModel.from_pretrained(audio_tokenizer_name_or_path)（:236-240），默认拉 OpenMOSS-Team/MOSS-Audio-Tokenizer（:218）。本仓库 moss_audio_tokenizer/ 是空目录（.gitmodules 声明 submodule 未初始化），README 称该 tokenizer 即 Cat（Causal Audio Tokenizer）。编码（wav→RVQ 32 路）与解码（RVQ→wav 声码器）都在这个外部模型里，本仓库只管上层语言模型。

与 Qwen3-TTS 的 codec 边界对照：Qwen3-TTS 的 speech_tokenizer 也是推理时装载的外部资产（架构页 risk-speech-tokenizer-load），codec 是 16 层残差；MOSS 的 audio_tokenizer 产 32 路 RVQ。两者都把“离散化”外包给独立模型，本仓库只处理离散 token 序列。迁移时 codec 这层都是 CPU/HF 资产装载，无 NPU 算子风险，但离线 NPU 环境需提前缓存 tokenizer/audio_tokenizer 目录。

Delay-Pattern 的时移在数据侧完成，不在模型里。apply_delay_pattern（processing_moss_tts_delay_with_codec:216-233）把 (t, n_vq) 的音频 token 沿时间轴对角线挪：L220-224 建一个 (t+n_vq-1, n_vq) 的全 pad 张量，L225-230 把第 0 路原样放、末尾补 pad，L231-232 for i in 1..n_vq-1: delayed_tokens[i:i+t, i] = tokens[:, i] 把第 i 路整体后移 i 步。这样第 t 步、第 i 路的位置上放的是原序列第 t-i 步的第 i 路——因果 attention 在第 t 步看到的就是“第 t 步的第 0 路 + 第 t-1 步的第 1 路 + …”，正好实现 README 说的对角线依赖。

    @staticmethod
    def apply_delay_pattern(
        tokens: torch.Tensor, ch0_pad_id: int, pad_id: int
    ) -> torch.Tensor:
        delayed_tokens = torch.full(
            (tokens.shape[0] + tokens.shape[1] - 1, tokens.shape[1]),
            pad_id,
            dtype=torch.long,
        )
        delayed_tokens[:, 0] = torch.cat(
            [
                tokens[:, 0],
                torch.full((tokens.shape[1] - 1,), ch0_pad_id, dtype=torch.long),
            ]
        )
        for i in range(1, tokens.shape[1]):
            delayed_tokens[i : i + tokens.shape[0], i] = tokens[:, i]
        return delayed_tokens

Delay-Pattern vs Qwen3-TTS 残差结构： Qwen3-TTS 的 16 层残差靠模型结构（talker 学第 0 层、code_predictor 在隐状态上补 1-15 层、嵌套 generate 逐层补全）实现层间依赖；MOSS Delay 的 32 路靠数据时移 + 单骨架多头实现——把层间依赖“折叠”进时间轴，用因果 attention 免费获得。前者模型复杂但序列短，后者模型简单但序列长（多 31 pad 步 × 帧数）。这是两种 RVQ 建模哲学的根本权衡。
NPU 迁移 · delay 时移留 CPUapply_delay_pattern 是纯 torch 索引赋值（torch.full + 切片赋值），在 processing 的 __call__ 里跑、属 CPU 数据预处理管线。建议保持 CPU（与 Qwen3-TTS 把 mel/STFT 钉 CPU 同一判断：预处理对吞吐不敏感，且 CANN 对这种“按轴对角线切片赋值”未必有融合算子）。NPU 上若要在线做 delay，需验证切片赋值路径；保守做法是在 DataLoader worker（CPU）里移好位再喂模型。
