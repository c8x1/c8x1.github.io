MM 保存状态

按 mindspeed_mm/training.py 的真实顺序逐段拆 MindSpeed-MM 的 checkpoint 保存/恢复：Megatron save_checkpoint 写出的 model_optim_rng.pt 布局、循环里四种触发保存的时机、AE 单独的 torch.save 路径、LoRA 与 .pt 的 load wrapper、确定性 seed_all 的 RNG state；并与 Qwen3-TTS 每轮 copytree + 注入 speaker[3000] + safetensors 的做法逐点对照，标出 .to(cpu) 写盘、多卡 save 协调、resume RNG state 的昇腾风险。

学习目标

  • 说清 MM 的 checkpoint 分两层：Megatron 的 model_optim_rng.pt（model+optim+rng 一体）与 utils.save_ae_checkpoint 的轻量 .ckpt，各自由谁调用
  • 按 training.py 行号定位四种触发 save 的时机：interval、microbatches 增长、exit_signal、exit_duration/exit_interval
  • 解释 load_checkpoint_wrapper 与 training_patches 的 .pt load 如何在 resume 时改 strict、置 ckpt_format=torch、清 args.load
  • 对照 Qwen3-TTS 的 copytree+注入 speaker[3000]+safetensors，说出 MM 为什么不这么做、各代价在昇腾上的差异

前置知识

  • 读过 Megatron checkpointing 的 save/load 基本语义
  • 理解 PP/TP 切分下 per-rank 各存一份 state_dict
  • 熟悉 Qwen3-TTS 的 safetensors 保存流程

总览：MM checkpoint 的两层保存

MindSpeed-MM 的 checkpoint 不是自己从零写的——它复用 Megatron 的 save_checkpoint / save_checkpoint_and_time / load_checkpoint，自身只在 mindspeed_mm/training.py 里决定何时触发、在 utils/utils.py 里补一条AE 专用的轻量保存路径、并在 tasks/finetune/lora 与 patchs/training_patches 里包一层 load wrapper 改 resume 行为。所以本页分两层：Megatron 层产出的 model_optim_rng.pt（每个 mp_rank 一份，含 model 权重 + optimizer state + RNG state + 调度器状态），和 AE 层的 checkpoint.ckpt（epoch/step/optim/scaler/sampler/ema 一锅 torch.save）。下图是这两层的写入与 resume 走向，args.load 是 resume 的总开关。

三个关键事实先记牢：model_optim_rng.pt 的名字就说明它把 model、optimizer、rng 三类 state 打包进同一文件——这是 Megatron 一贯做法，resume 时一并恢复，保证续训数值连续；latest_checkpointed_iteration.txt 是 resume 的索引，记着该读哪个 iteration 的目录；args.load 在 training_patches 的 .pt load 路径里用完会被显式置 None（L121），防止后续 Megatron 主流程再重复 load 一次。

Megatron 层：model_optim_rng.pt 的目录布局

PP/TP 切分下，每个 model-parallel rank 各写一份自己的 state_dict，落到 mp_rank_<pp>_<tp> 子目录里。这个布局是 converter 文件头注释明写的——tp1/pp4 训练保存出来就是 4 个 mp_rank_00_000..003，每个下一颗 model_optim_rng.pt。resume 时各 rank 只读自己那颗，不需要跨 rank 拼数据（数据已经在保存时按切分写好了）。

这 13 行注释本身就是规格说明：HF 端是分片 safetensors（model-0000X-of-00005.safetensors + index.json），MM 端是 Megatron 的 mp_rank_XX_YYY/model_optim_rng.pt + latest_checkpointed_iteration.txt。hf_to_mm.py 做的就是把前者重排成后者，mm_to_hf.py 反过来。注意 MM 自己训练保存走的是 Megatron 原生 torch 格式（非 distributed checkpoint 的 dcp），args.ckpt_format 默认即 torch。

PP 切分 = 文件切分。这意味着 resume 时拓扑必须一致：用 pp4 存的 checkpoint 不能直接在 pp2 上 resume。源码里 validate_args_patch.py:753 直接提示 ‘impossible to resume training with different parallelism’。这是与 Qwen3-TTS（单卡 safetensors，无切分概念）最大的结构差异。

def judge_save_checkpoint(args, iteration):
    if not args.save or iteration == 0:
        return False
    if iteration % args.save_interval == 0:
        return False
    if os.getenv('OOTB_OPTIMIZER_PROFILING', 'FALSE') == 'TRUE':
        return False
    return True

L1084-1091 的判断很细：not args.save 或 iteration == 0 不存；iteration % save_interval == 0 也不存（因为前面 interval 路径已经存过）；只有 profiling 模式下再多拦一道。即这个兜底只补存 interval 没踩到的最后一次，不与循环内保存重复。

NPU 迁移 · 多卡 save 协调PP/TP 切分下每个 rank 各写一份 model_optim_rng.pt，共享一个 checkpoint 目录。多机时若用 NFS 类共享存储，多 rank 并发写不同子目录一般无冲突，但 latest_checkpointed_iteration.txt 这类 tracker 文件的写入要靠 Megatron 内部的 rank-0 协调（非 MM 代码可见）。昇腾多机 + HCCL 下，save 前后的 torch.distributed.barrier()（L696）必须走 HCCL backend——若 backend 误配成 nccl 会在这里挂死。建议先用单机多卡验证 save/load round-trip，再上多机。

AE 层：torch.save 一锅的 checkpoint.ckpt

AE（autoencoder，视频/图像 VAE 类）训练不走 Megatron 主循环，而是独立训练脚本直接调 utils/utils.py 的 save_ae_checkpoint。这条路径与 model_optim_rng.pt 完全无关——它就是一次 torch.save，把 epoch、step、optimizer_state、state_dict、ema_state_dict、scaler_state、sampler_state 全塞进一个 dict 写成 checkpoint.ckpt。

mindspeed_mm/utils/utils.py · L275–L301
固定源码 ↗
def save_ae_checkpoint(
    epoch,
    current_step,
    optimizer_state,
    state_dict,
    scaler_state,
    sampler_state,
    checkpoint_dir,
    filename="checkpoint.ckpt",
    ema_state_dict=None,
):
    if not os.path.exists(checkpoint_dir):
        os.makedirs(checkpoint_dir)
    filepath = os.path.join(checkpoint_dir, filename)
    torch.save(
        {
            "epoch": epoch,
            "current_step": current_step,
            "optimizer_state": optimizer_state,
            "state_dict": state_dict,
            "ema_state_dict": ema_state_dict,
            "scaler_state": scaler_state,
            "sampler_state": sampler_state,
        },
        filepath,
    )
    return filepath

L275-301 很直白：os.makedirs 建目录（exist_ok 由调用方保证，这里只判不存在就建）、拼 filepath = dir/filename、torch.save({...}, filepath) 一次性写。dict 的 7 个 key 覆盖了 AE 训练续训所需的全部状态：epoch/current_step 是进度、optimizer_state 是优化器动量、state_dict 是权重、ema_state_dict 是 EMA 影子权重、scaler_state 是 GradScaler（FP16 路径才用得到）、sampler_state 是数据采样器游标。

NPU 迁移 · .to(cpu) 写盘这里没有显式 .to(cpu)——torch.save 默认会把 tensor 序列化到 CPU 再落盘，但若 state_dict 里的张量还挂在 NPU 上，save 前会隐式做一次 H2D 拷贝，大模型下这一步可能瞬时占满 host 内存。保守做法是在调用方把要存的 state_dict 先 .to('cpu') 再传入，与 Qwen3-TTS 保存段 state_dict.to('cpu') 同判断。这点 MM 源码里是留白，需迁移时补。

EMA 类（utils/ema.py）的状态也由这条路径承载：它的 shadow dict 存的是各 param 的克隆动量均值（register 时 param.data.clone()，update 时指数加权）。save 时把 ema_state_dict 一并写进去，resume 时还原即可——EMA 的 shadow 是 CPU 可重建的纯张量，无 NPU 算子风险。

Resume：load wrapper 与 .pt load

resume 有两条 wrapper。第一条是 LoRA 场景的 load_checkpoint_wrapper（tasks/finetune/lora/lora_patch.py）：它把 Megatron 的 load_checkpoint 包起来，当 is_enable_lora() 且未指定 base model 时强制 strict=False——因为 LoRA 只想 load 基座权重，基座里没有 lora adapter 是正常的，不能因 strict 报 missing key。

mindspeed_mm/tasks/finetune/lora/lora_patch.py · L225–L248
固定源码 ↗
def load_checkpoint_wrapper(fn):
    @wraps(fn)
    def wrapper(*args, **kwargs):
        args_ = get_args()
        if is_enable_lora() and args_.load_base_model is None:
            strict = False
            kwargs['strict'] = strict

        return fn(*args, **kwargs)

    return wrapper


def state_dict_for_save_checkpoint(state_dict):
    state_dict_ = dict()
    for key in state_dict:
        args = get_args()
        if 'lora' in key and 'base_layer' not in key:
            if args.use_torch_fsdp2:
                prefix = "base_model.model."
                state_dict_[prefix + key] = state_dict[key]
            else:
                state_dict_[key] = state_dict[key]
    return state_dict_

L225-235 的 wrapper 很薄：从 get_args() 取参数，判 LoRA 开关，必要时改 kwargs['strict']，再透传给原 load_checkpoint。L238-248 是配套的 save 过滤 state_dict_for_save_checkpoint：LoRA 且不存全量时，只保留 key 含 'lora' 且不含 'base_layer' 的项——即只存 adapter 增量，不存基座。这是 LoRA checkpoint 比 full finetune 小一个数量级的原因。

第二条是 training_patches 里的 .pt load：当 args.use_dist_ckpt 且 args.load 非空时，绕过 dcp 格式，临时把 ckpt_format 置为 'torch' 调一次 load_checkpoint，调完立即置回原值并清空 args.load = None。

L91-122 是典型的 ‘改参数→调用→还原’ 模式：L97-99 读 tracker 文件拿到 iteration/release 标记，L102 算出要读的 .pt 路径，L104 检查存在性（不存在直接返回未加载的 model），L114-117 临时换 format 调 load 再还原，L121 清 args.load。清空的目的是让后面 Megatron 主流程的 load 分支不再触发——这条路径是 ‘先把权重灌进 model，剩下的 optimizer/rng resume 不走这条’。

注意这两条 wrapper 的职责边界：LoRA wrapper 改的是 load 的 strict 语义（只影响是否容忍 missing key）；training_patches 改的是 load 的格式与一次性（强制 torch 格式、用完清 load 开关）。两者可叠加，但若同时开启 LoRA + dist_ckpt + load，strict=False 会优先于 format 切换生效——源码里没有显式排序，靠 wrapper 注册顺序决定，迁移时建议先单测。
NPU 迁移 · resume RNG statemodel_optim_rng.pt 里的 rng 就是恢复续训数值连续性的关键。Megatron load 时会把存下的 CPU/NPU RNG state 拷回 torch.random 与设备 generator。昇腾上的隐患：seed_all（random.py:24）设了 HCCL_DETERMINISTIC=True、CLOSE_MATMUL_K_SHIFT=1、torch.use_deterministic_algorithms(True) 并 torch_npu.npu.manual_seed_all——这些环境变量 + seed 是绑定的。若存 checkpoint 时与 load 时 NPU 驱动/CANN 版本不同，HCCL_DETERMINISTIC 行为可能漂移，resume 后前几步 loss 仍可能与原 run 有微小偏差。待真机验证：用同一镜像存+load 做 max-abs loss diff。

确定性 seed：写盘前的 RNG 锚

checkpoint 能否续训数值连续，不只看存了什么，还看存之前的 RNG 是否可复现。MM 的 utils/random.py::seed_all 一次把五路随机性都钉死：Python random、numpy、torch CPU、torch NPU，外加三个 CANN 环境变量。这段在 training.py 里由 --use-deter-comp 触发。

mindspeed_mm/utils/random.py · L24–L34
固定源码 ↗
def seed_all(seed=1234):
    random.seed(seed)
    os.environ["HCCL_DETERMINISTIC"] = "True"
    os.environ["CLOSE_MATMUL_K_SHIFT"] = "1"
    os.environ["PYTHONHASHSEED"] = str(seed)
    np.random.seed(seed)
    torch.manual_seed(seed)
    torch.use_deterministic_algorithms(True)

    torch_npu.npu.manual_seed_all(seed)
    torch_npu.npu.manual_seed(seed)

L25-28 是三个 CANN 环境变量：HCCL_DETERMINISTIC=True 让集合通信可复现（代价是 allreduce 走确定性路径、可能慢）、CLOSE_MATMUL_K_SHIFT=1 关掉矩阵乘的 K 维 shift 抖动、PYTHONHASHSEED 固定哈希随机。L29-33 分别 seed numpy、torch CPU、torch NPU。L31 torch.use_deterministic_algorithms(True) 会强制部分算子走确定性实现——某些 CUDA-only 的非确定算子会直接 raise，NPU 上对应算子是否提供确定性实现需逐算子核验。

NPU 迁移 · device='cuda' 硬编码training.py 里仍有若干 device='cuda' 与 torch.cuda.empty_cache() 硬编码（如 L199 时间张量、L667 exit_duration 标志、L769/L815 empty_cache、L885/L1056 loss 占位张量）。这些在 torch_npu 下通常会被 torch_npu 的 monkey-patch 自动重定向到 npu（因为 import torch_npu 注入了 torch.cuda → torch.npu 别名），多数能跑。但 L199 的 start_time_tensor 紧跟 all_reduce，若 backend 未就绪会在第一步 collective 挂。建议把 device='cuda' 统一替成 device=get_device('npu')，或确认 torch_npu patch 覆盖后再用。待真机验证哪些 cuda 调用未被 patch。

反向与 step：optimizer / scheduler state 的更新点

checkpoint 里存的 optimizer state 与 scheduler state 是在 step 之后的快照。看 train_step 末尾的更新顺序：optimizer.step() 先更新参数与动量、opt_param_scheduler.step(increment) 再推 lr，然后循环回到顶部的 save 判断。所以存盘的是 ‘这一步刚 step 完’ 的状态，resume 时从下一个 iteration 接着算，无重复 step。

L784 的 torch.distributed.barrier() 是 optimizer 为 None 时的同步兜底（纯推理 forward_only 路径）。正常训练走 L781-782 的 optimizer.step()。Megatron 的分布式 optimizer 内部会做梯度 allreduce（HCCL），step 成功后 optimizer state（动量/二阶矩）已更新——这些 state 就是 model_optim_rng.pt 里 optim 部分的内容。

NPU 迁移 · 分布式 optimizer allreduceMM 用 Megatron distributed optimizer（args.use_distributed_optimizer），其梯度同步走 torch.distributed.all_reduce，backend 必须是 HCCL。save 时存的 optimizer state 是已经 allreduce + step 之后的局部副本（每个 rank 存自己那一份切分），resume 时各 rank 各读各的，不跨 rank 同步。风险点：若 save 时某 rank 因梯度 inf 走了 skipped_iter（L810），它的 optimizer state 与其他 rank 不同步——resume 后这 rank 的动量会与全局错位。建议 save 前显式检查 skipped_iter 计数，非零则跳过该次保存。

保存/恢复之外，MM 还提供了 NPU 设备探测的小工具 is_npu_available 与 get_device。它们不直接参与 checkpoint，但 AE 路径与若干训练脚本靠它们决定把张量放哪——是 .to(cpu) 写盘判断的底层依赖。

@lru_cache
def is_npu_available():
    """Checks if `torch_npu` is installed and potentially if a NPU is in the environment"""
    if importlib.util.find_spec("torch_npu") is None:
        return False
    import torch_npu
    try:
        # Will raise a RuntimeError if no NPU is found
        _ = torch.npu.device_count()
        return torch.npu.is_available()
    except RuntimeError:
        return False


def get_device(device="npu"):
    """
    only support npu and cpu device, default npu.
    device format: cpu, npu, or npu:0
    """
    if isinstance(device, torch.device):
        return device
    device = device.lower().strip()
    if device == "cpu":
        return torch.device(device)

    device_infos = device.split(":")
    device_name = device_infos[0]
    if device_name == "npu":
        if is_npu_available():
            if len(device_infos) == 1:
                return torch.device(device_name)
            if len(device_infos) == 2:
                device_id = int(device_infos[1])
                num_devices = torch.npu.device_count()
                if device_id < num_devices:
                    return torch.device(f"{device_name}:{device_id}")
                else:
                    raise ValueError(f"device_id: {device_id} must less than device nums: {num_devices}")
        else:
            raise RuntimeError("NPU environment is not available")
    raise ValueError("only support npu and cpu device. device format: cpu, npu, or npu:0")

L182-193 的 is_npu_available 用 importlib.util.find_spec('torch_npu') 探包是否装了 torch_npu，再用 torch.npu.device_count() 探真机（catch RuntimeError 当无 NPU）。@lru_cache 说明结果在进程内缓存——不能在运行中途插拔 NPU，但训练场景无此需求。

L196-222 的 get_device(device='npu') 只认 cpu / npu / npu:N 三种字符串，显式拒绝 cuda。NPU 不可用时直接 raise RuntimeError。这意味着 AE 训练脚本里 .to(get_device()) 在 CUDA 机器上会炸——这本就是 NPU 专用代码路径，迁移到 CUDA 需改默认值。对 checkpoint 的意义：AE save 前若调用方用 get_device('cpu') 显式落 CPU，就规避了上节 risk-torch-save-npu 的隐式 H2D。

对照：与 Qwen3-TTS 每轮保存的差异

Qwen3-TTS 的 SFT 保存（见 target/sft-training-loop 页）与 MM 的 checkpoint 是两种范式。Qwen3-TTS 是 accelerate 单卡脚本，每轮末做：copytree 复制原模型骨架 → 改 config 注册 speaker → state_dict.to(cpu) → 删 speaker_encoder 权重 → 注入 codec_embedding.weight[3000] → 写 safetensors。MM 是 Megatron 多卡分布式训练，按切分每 rank 各存 model_optim_rng.pt。

NPU 迁移 · NPU internal format 与 safetensorsQwen3-TTS 写 safetensors 走标准 torch 布局，NPU 无关。MM 的 torch.npu.config.allow_internal_format（training.py:184）会让某些张量在 NPU 上用 CANN 内部 5 维格式（ND→NDC1HWC0 类）。save 时Megatron 会把 state_dict 序列化成标准布局（torch.save 走 pickle + storage），internal format 的影响通常在序列化时被规整掉；但若走 distributed checkpoint (dcp) 路径，某些算子的 internal format storage 可能被原样落盘，CUDA 侧 load 不回来。MM 默认 ckpt_format=torch 规避了这点，迁移时若改用 dcp 需先把 allow_internal_format=False。待真机验证。
