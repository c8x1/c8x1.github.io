LLM 恢复文档

按 mindspeed_llm/training/checkpointing.py 的真实顺序逐段拆 MindSpeed-LLM 的 checkpoint 保存/恢复：save_checkpoint_wrapper 怎么选 torch / torch_dist 格式、generate_state_dict 怎么把 model+optimizer+scheduler+rng 一锅打包、tracker 文件与多 rank barrier 的协调、_load_base_checkpoint_wrapper 的 LoRA 合并、load_checkpoint_wrapper 的 loose 加载；并讲清确定性 seed_all 与 resume RNG state 的连续性，逐点对照 Qwen3-TTS 每轮 copytree+注入 speaker[3000]+safetensors 的做法与 MindSpeed-MM 的 model_optim_rng.pt，标出 .to(cpu) 写盘、多卡 save 协调、resume RNG 跨 CANN 版本漂移、分布式 optimizer state allreduce 的昇腾风险。

学习目标

  • 说清 LLM 的 checkpoint 复用 Megatron save_checkpoint 但被 save_checkpoint_wrapper 包了一层：它在 torch / torch_dist 两种格式间选型、把 rng_state 攘进 state_dict、再调 torch.save 或 dist_checkpointing.save
  • 按 training.py 行号定位四种触发 save 的时机：microbatches 增长、interval 到点、exit 信号、exit_duration / exit_interval，并说出 update_save_checkpoint_chmod 兜底的权限收口
  • 解释 _load_base_checkpoint_wrapper 的 LoRA 合并（base_layer 改名 + merge_dicts）与 load_checkpoint_wrapper 的 loose 加载（strict=False）各自改了什么、靠 features_manager 注册 patch 生效
  • 对照 Qwen3-TTS 的 copytree+注入 speaker[3000]+safetensors 与 MindSpeed-MM 的 model_optim_rng.pt，说出 LLM 为什么走 generate_state_dict 一体打包、三者存 optim/rng 的差异

前置知识

  • 读过 Megatron save_checkpoint / load_checkpoint 的基本语义
  • 理解 PP/TP/EP/VP 切分下每个 rank 各存一份 state_dict 的布局
  • 熟悉 Qwen3-TTS 的 safetensors 保存与 MindSpeed-MM 的 model_optim_rng.pt（见 reference/mm-checkpoint 页）

总览：wrapper 包住 Megatron 的 save/load

MindSpeed-LLM 的 checkpoint 不是自己从零写的——它复用 Megatron 的 save_checkpoint / load_checkpoint / _load_base_checkpoint，自身只在 mindspeed_llm/training/checkpointing.py 里包一层 wrapper，改保存格式选型、rng 摄取、LoRA 合并、loose 加载四件事；训练循环里何时触发在 mindspeed_llm/training/training.py，确定性 seed 在 mindspeed_llm/training/utils.py::seed_all。wrapper 不是直接覆盖函数，而是由 features_manager/finetune/lora.py / features_manager/common/training.py 通过 patch_manager.register_patch 把它们挂到 Megatron 的符号上——所以 Megatron 主流程无感知地走了 LLM 的 wrapper。下图是 save/load/resume 的走向，save_checkpoint_wrapper 是写入侧的总开关，generate_state_dict 是把四类 state 揉进一个 dict 的枢纽。

三个关键事实先记牢：save_checkpoint_wrapper 头一行 model = unwrap_model(model)（L209）先把 DDP/FSDP 剥掉再存，所以存的 state_dict 是裸模块，不含 wrapper 的冗余键；rng_state = get_rng_state(args.ckpt_format)（L217）在构造 state_dict 之前就把随机数状态抓出来，随后随 model/optimizer 一起落盘——这是 resume 续训数值连续的载体；ckpt_format = args.ckpt_format if args.use_dist_ckpt else 'torch'（L212）把格式选型压成一句三元，不开 use_dist_ckpt 就是 legacy torch 单文件，开了才走分片。

Save 入口：格式选型与 rng 摄取

save_checkpoint_wrapper 把 Megatron 原版 save 的‘搓 state_dict → 选策略 → 落盘 → 更 tracker’四步攥进一个函数。它先 unwrap_model 剥壳，再按 args.use_dist_ckpt 决定走分片 torch_dist 还是 legacy torch，然后抓 rng、算路径、搓 state_dict。下面这段是头部——格式选型与 rng 摄取。

逐行往下看。L209 model = unwrap_model(model) 剥掉 DDP/FSDP wrapper——这是 wrapper 相对裸 Megatron 的第一处改动，确保 state_dict 里是纯模块键。对 LoRA 训练这步尤其关键：unwrap 后 Peft 的 lora 层才以原始命名出现在 dict 里，后续 filter_lora_keys 才能按 'lora' 子串筛。

L212 ckpt_format = args.ckpt_format if args.use_dist_ckpt else 'torch'：不开分布式 ckpt 时强制 legacy torch，无视 args.ckpt_format 的值。这意味着 LLM 默认路径就是单文件 torch.save，与 MindSpeed-MM 默认 ckpt_format=torch 同取向。L217 rng_state = get_rng_state(args.ckpt_format) 把 CPU 与 NPU 两路 generator 的 state 抓成对象，随后塞进 state_dict。

L221-225 仅在 torch_dist 格式下额外抓 rerun_state_machine.state_dict——rerun 状态机记录的是 ‘某层在前向里被重跑过几次’ 的元数据，只有分片格式才持久化，legacy 路径丢弃。L228-231 get_checkpoint_name 算落盘路径：分布式模式返回目录（return_base_dir=True，各 rank 写自己的分片），legacy 返回单文件名。

NPU 迁移 · .to(cpu) 写盘legacy 路径 L354 torch.save(state_dict, checkpoint_name) 没有显式 .to('cpu')——torch.save 会隐式把 NPU 张量序列化到 host 再落盘，大模型下这一步可能瞬时撑满 host 内存。这与 MindSpeed-MM 的 save_ae_checkpoint 同一处留白。保守做法：保存前在调用方对 state_dict 做一次 .to('cpu')，与 Qwen3-TTS 保存段 state_dict.to('cpu')（sft_12hz.py:147）同判断。LLM 源码里这步是缺的，迁移大模型时需补，待真机验证内存峰值。

state_dict 搓合：model+optim+scheduler+rng 一锅

rng 抓完就搓 state_dict。核心是 Megatron 的 generate_state_dict——LLM 把 model、optimizer、opt_param_scheduler、rng_state、iteration 一起传进去，它返回一个嵌套 dict，四类 state 揉进同一结构。这是 LLM checkpoint 与 Qwen3-TTS（只存权重）最根本的差异：这里存的 optimizer 动量/二阶矩与 scheduler 的 lr 游标是为续训，不是为推理。

NPU 迁移 · 分布式 optimizer state allreduceLLM 用 Megatron distributed optimizer（args.use_distributed_optimizer），其梯度同步走 torch.distributed.all_reduce，backend 必须是 HCCL。L272-276 在分布式 save 时把 optimizer state 也分片（sharding_type='fully_sharded_model_space' 或 'dp_zero_gather_scatter'）——每个 rank 存自己那份切分。风险：若 save 时某 rank 因 inf/nan 梯度走了 skipped_iter，它的 optimizer state 与全局错位，resume 后这 rank 的动量与全局不同步。建议 save 前显式检查 skipped_iter 计数，非零则跳过该次保存。待真机验证：单机多卡先做 save/load round-trip 的 loss diff。
opt_param_scheduler 进 state_dict 意味着 lr 调度器游标被持久化。LLM 的 scheduler 按样本数推进（increment = num_microbatches * micro_batch_size * data_parallel_size，Megatron 语义），所以 resume 时不仅 iteration 要对，consumed_train_samples 也要对——state_dict 里同时存了这两者。

NPU 迁移 · async_save 与 HCCLargs.async_save 只在 use_dist_ckpt + torch_dist 下支持（L243-247，legacy 直接 raise NotImplementedError）。异步写盘时训练循环不阻塞，但 finalize 回调里仍有 torch.distributed.barrier 路径——backend 必须是 HCCL。风险点：异步队列若有未完成的写请求（L198-202 的 pending_async 警告），下一次 save 会叠在未完成的写上，NPU 大模型下可能撑爆临时存储。建议迁移时先关 async_save 走同步验证，确认 round-trip 后再开异步。待真机验证。

def update_save_checkpoint_chmod(save_path, permission=0o640):
    """
    Update file permissions for saved checkpoint files.

    This function modifies the file permissions of saved checkpoints for security
    when high availability mode is disabled.

    Args:
        save_path (str): Path to the saved checkpoint file.
        permission (int, optional): File permission mode. Defaults to 0o640.

    Note:
        Permission updates are skipped when high availability is enabled
        to avoid conflicts with HA mechanisms.
    """
    args = get_args()
    if args.enable_high_availability:
        return
    if os.path.exists(save_path) and os.path.isdir(save_path):
        for root, _, files in os.walk(save_path):
            for file in files:
                file_path = os.path.join(root, file)

                try:
                    os.chmod(file_path, permission)
                except PermissionError:
                    logging.warning(f"permission error: {file_path}")
                except Exception as ee:
                    logging.warning(f"failed to change permission: {file_path}: {ee}")

L122-123 开 HA 时直接 return，不收口——因为 HA（mindio_acp）有自己的权限管理。L124-130 os.walk 遍历目录下所有文件 os.chmod(file_path, 0o640)，PermissionError 与其它异常都只 warning 不中断。注意：异步 save 时这步被跳过——L813/833/852/865 都有 if not args.async_save 前置，因为异步写盘还没完成就 chmod 会改到半成品文件，留到 L889-890 异步全部 finalize 后再统一收口。

NPU 迁移 · chmod 与共享存储os.chmod 在 NFS/共享存储上通常正常，但若 checkpoint 目录在不同 rank 的节点上挂载点不同（非共享存储），rank0 改的权限其它 rank 看不到——LLM 用 rank0 单写 tracker + 各 rank 各写自己分片的模型，权限一致性靠‘各改各的文件’保证。风险：HA 模式下 update_save_checkpoint_chmod 直接 return（L122），HA 自身的权限策略需单独核验。昇腾多机共享存储下建议先验证 chmod 不阻塞 save 主路径（os.walk 大目录可能慢），必要时关掉或挪到异步。待真机验证。
