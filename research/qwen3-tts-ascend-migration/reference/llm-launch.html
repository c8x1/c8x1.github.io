LLM 启动形态

逐段拆 MindSpeed-LLM 的启动形态：pretrain_gpt.py 顶层入口怎么把 model_provider/forward_step/data provider 三个回调交给 pretrain、pretrain 编排层（initialize_megatron 的 torch_npu 初始化、set_jit_fusion_options、device='cuda' 字面量）、train 循环复用 Megatron 自身的 train_step、与 MindSpeed-MM training.py 的关键差异（MM 重写 train_step、LLM 直接 import）、torchrun 启动脚本与 NPU 环境变量/启动配置注入——把 LLM 定位为 Qwen3-TTS 迁移的规模化训练卫星，解释分布式并行/启动/通信但不表述为 TTS 实现。

学习目标

  • 定位 pretrain_gpt.py 的 main→pretrain 调用链（L297-305）与三个回调（model_provider/forward_step/train_valid_test_datasets_provider）各自行号
  • 说清 pretrain 编排层的启动顺序：initialize_megatron → set_jit_fusion_options → build_train_args（setup_model_and_optimizer + build iterators）→ train
  • 区分 LLM 与 MM 的核心结构差异：LLM 直接 import megatron.training.training 的 train_step，MM 在本文件重写 train_step——以及这一差异对迁移的含义
  • 指出 NPU 设备初始化的真实调用点（import torch_npu、set_compile_mode、use_ascend_mc2、tp_comm_overlap）与 torchrun 启动脚本里 device='cuda' 字面量靠 torch_npu patch 落到 NPU 的模式

前置知识

  • 读过 target/sft-training-loop.html，知道 accelerate 的 prepare/backward/accumulate 三段
  • 读过 reference/mm-training.html，知道 MM 的 pretrain→train→train_step 三层编排
  • 理解 Megatron 的 model_provider/forward_step_func/data provider 回调约定，以及 microbatch 累积的基本语义

总览：从入口到训练循环的启动链

MindSpeed-LLM 是 Qwen3-TTS 迁移的规模化训练卫星——它解释的是分布式并行、启动形态、通信后端、性能融合与断点恢复，而不是 TTS 模型本身。本页只看“启动形态”：一条 pretrain_gpt.py 顶层脚本，怎么经 pretrain() 编排层把模型/数据/前向三件回调喂进 Megatron 训练引擎，最终落到 train() 主循环。整条启动链的关键事实：pretrain_gpt.py 的 main()（L297）只做一件事——把三个 provider 交给 pretrain（L302-305）；pretrain（L430）做初始化+编排+决定进不进 train；train（L581）跑 while iteration < train_iters，每轮调 train_step——但这个 train_step 是 从 megatron.training.training import 的（L51），不在本文件里，这是 LLM 与 MM 最大的结构差异。

和 reference/mm-training.html 对照，两套代码骨架高度同源（都是 Megatron 的 pretrain/train/train_step 三层），但结构重心不同。MM 把 train_step 在本文件（mindspeed_mm/training.py）里重写，逐行展开了 forward_backward→optimizer.step→lr.step 的调度细节；LLM 则直接复用 Megatron 上游的 train_step，在本文件里只保留 pretrain 与 train 两层 + 一圈包装（build_train_args、get_model_provider_func、各种 wrapper 装饰器）。换句话说：读 MM 能看清“一个 optimizer step 内部到底怎么跑”，读 LLM 看到的是“启动怎么把三件回调接进引擎、以及引擎本身（Megatron 上游）在哪”。这对迁移的意义是——若要把 Qwen3-TTS 迁到 LLM 引擎，真正要改的是 pretrain_gpt.py 这一层（provider 回调）和启动脚本，train_step 内部不动。

顶层入口：main 与三个 provider 回调

pretrain_gpt.py 是面向 GPT 系（含 Qwen3）的通用训练入口，本身不含任何 TTS 语义。它把三件事写成三个函数，交给 pretrain：model_provider（L42）按 spec 建 GPTModel（mcore 或 legacy）；forward_step（L216）取 batch、跑前向、返回 output_tensor 与偏函数化的 loss_func；train_valid_test_datasets_provider（L269）用 BlendedMegatronDatasetBuilder 建 train/valid/test 三套数据集。这是 Megatron 的标准约定：引擎不认识具体模型与数据，全靠调用方把函数传进来。

L297 @auto_coverage 是 MindSpeed-LLM 的装饰器（来自 mindspeed_llm.training.utils），做算子覆盖/补丁注入，不影响主流程逻辑。L300 train_valid_test_datasets_provider.is_distributed = True 是 Megatron 的约定标记：数据集按 TP rank 分布构建，只在 is_dataset_built_on_rank（L242，即 TP rank 0）上真正建。L302-305 pretrain(...) 四个参数就是上面三个 provider + ModelType.encoder_or_decoder——注意没有 forward_step 之外的前向逻辑，模型怎么算 loss 全在 forward_step（L216-239）里：取 batch、model(tokens, position_ids, attention_mask, labels=…)、返回 (output_tensor, partial(loss_func, loss_mask))。迁移 Qwen3-TTS 时，forward_step 这一格就是要填“embedding 汇总 + talker 前向 + CE+MTP”的位置。

三个 provider 是迁移 Qwen3-TTS 到 LLM 引擎的三个落点：model_provider 填 talker/code_predictor/speaker_encoder 的 Megatron 模型定义；forward_step 填前向+损失；train_valid_test_datasets_provider 填 collate 后的 batch 迭代器。引擎本身的 pretrain/train/train_step 不需要改——这是“卫星”而非“实现”的含义：LLM 提供的是可复用启动框架，不是 TTS 算子。

Pretrain 编排层：初始化与 JIT 融合

pretrain（L430）接五个参数：四个 provider + model_type。它的真实启动顺序是 initialize_megatron（L467，建进程组/mpu/设全局 args）→ set_jit_fusion_options（L478，JIT 与 torch_npu 编译开关）→ build_train_args（L502，建模型/优化器/数据迭代器，返回 train_args）→ 按 args.skip_train 决定进 train 或 tft_train（高可用分支，L524-540）。与 MM 的 pretrain 几乎同构，但 LLM 把 set_jit_fusion_options 调用放在 pretrain 里（L478），而 MM 把它放在 initialize_megatron 内部——调用时机略有前后，但都在模型装载前。

逐行看启动关键点。L467-468 initialize_megatron 是入口——它在 mindspeed_llm/training/initialize.py L70 定义，内部 import torch_npu（initialize.py L21）是整条 NPU 路径的总开关，随后 parse_args_decorator 解析参数、validate_args 校验、set_global_variables 建 mpu 与 tokenizer，最后 finish_mpu_init 调 _initialize_distributed 建进程组。L478 set_jit_fusion_options() 设 JIT 层融合：它在 initialize.py L160 定义，开 PyTorch JIT profiling，并在 args.jit_compile 为真时调 torch_npu.npu.set_compile_mode(jit_compile=True)（initialize.py L173）——这是 NPU 的 GE 图编译开关，把图编译成昇腾算子融合图。

L484-489 又是经典的 device='cuda' 字面量：torch.tensor([_TRAIN_START_TIME], dtype=torch.float, device='cuda') 做 all_reduce(MIN) 取各 rank 的最小启动时间。和 MM 一样，这里字面写 cuda 不是 bug——initialize_megatron 内部已 import torch_npu，torch_npu 的 monkey-patch 把 device('cuda') 重定向到 NPU。整条 LLM 启动链里这种“写 cuda、靠 patch 落 NPU”的模式反复出现（L486、L959、L1181 等），迁移读者看到不要以为是漏改。

NPU 迁移 · torch_npu import 是总开关import torch_npu（initialize.py L21）是 LLM 全链路 NPU 化的开关：它 monkey-patch torch 的 device('cuda')/.cuda()/torch.cuda.* 到 NPU。pretrain 里 L486、L959、L1181 等大量 device='cuda' 字面量，不是漏改，是靠这层 patch 落到 NPU。迁移 Qwen3-TTS 走 accelerate+torch_npu 路线时，需确认 torch_npu 已在进程启动早期 import 且版本与 CANN 匹配；直接迁到 LLM 引擎则天然带上（initialize_megatron 强制走）。两个要手动开的开关是 initialize.py L173 set_compile_mode(jit_compile=True)（影响 GE 融合）与 use_ascend_mc2/tp_comm_overlap（initialize.py L127/L153，影响 TP 通信重叠）——这些影响性能与算子融合，建议先按启动脚本默认开、出问题再针对性关。待真机验证。
NPU 迁移 · set_jit_fusion_options 时机set_jit_fusion_options()（L478，定义 initialize.py L160-173）必须在模型装载之前调：它先开 PyTorch JIT profiling/融合标志（L163-168），再 _warmup_jit_function()（L170），最后才按 args.jit_compile 调 torch_npu.npu.set_compile_mode(jit_compile=True)（L173）。LLM 把这次调用放在 pretrain 的 L478（initialize 之后、build_train_args 之前），时机正确。对照 MM 是在 initialize_megatron 内部 L192 调。迁移时若自定义启动顺序，必须保证“JIT 融合开关 → 模型 provider → setup_model_and_optimizer”这个先后，否则装载出的模型拿不到改过的融合路径。

initialize_megatron：分布式与 NPU 配置

initialize_megatron（initialize.py L70）是 LLM 自定义版的 Megatron 初始化，签名与上游一致但内部塞了昇腾特定逻辑。它先 ensure_valid(torch.cuda.is_available())（L89，经 torch_npu patch 等价于 NPU 可用性检查）、parse_args_decorator(parse_args)（L92，给 megatron 参数解析加 LLM 扩展）、validate_args（L106）、set_global_variables（L110，建 mpu/timer/tokenizer），然后进 finish_mpu_init（L118）调 _initialize_distributed 建进程组。与 MM 同源的 initialize_megatron 相比，LLM 多了 use_deter_comp（L113，确定性计算）、use_ascend_mc2（L127，昇腾 MC2 通信配置）、tp_comm_overlap（L153，TP 通信重叠）三个昇腾专属开关——但没有 MM 那个显式的 allow_internal_format 全局设置。

NPU 迁移 · use_ascend_mc2 / tp_comm_overlapinitialize_cfg_from_args(args)（initialize.py L128，use_ascend_mc2 触发）与 _initialize_tp_communicators()（L154，tp_comm_overlap 触发）是 LLM 的两个昇腾通信优化开关。MC2 让 TP allreduce 走昇腾融合通信，通信重叠让计算与 reduce overlap。这两个影响的是多卡 TP 性能，不是能否跑通：单卡迁移（Qwen3-TTS SFT 的典型场景）两者都不触发，零风险；多卡 TP 时建议先按启动脚本默认开，若出现通信 hang 或算子不匹配再针对性关 tp_comm_overlap。启动脚本里 --use-ascend-coc（COC 通信重叠）是同一族开关。待真机验证 MC2/overlap 在 910B 上的实际收益。
NPU 迁移 · 无 allow_internal_format 开关与 MM 不同，LLM 的 initialize_megatron 与 arguments.py 里没有显式的 allow_internal_format 参数（grep 全 training/ 无命中）。这不代表 CANN internal format 不生效——torch_npu 默认行为仍可能启用内部排布；只是 LLM 没有像 MM 那样把它做成一个可配开关。迁移 Qwen3-TTS 走 LLM 引擎时，若遇到算子因排布不匹配报错（典型：自定义算子或与标准 NCHW 强交互的路径），需在 torch_npu 层面手动调 torch.npu.config.allow_internal_format = False 排查，而不是指望启动参数。对照 MM 那条“默认开、算子报错再关”的策略，LLM 这里要更主动地从代码侧设。

NPU 迁移 · setup_model_and_optimizer 的设备路由setup_model_and_optimizer（L360，megatron.training.training）内部会把 CPU 上的 vanilla 模型搬到当前 rank 的 device。经 torch_npu patch，这个 device 是 NPU。迁移 Qwen3-TTS 时，它的 Qwen3TTSModel.from_pretrained(bf16, FA2) 这一步在 LLM 引擎里要改成 model_provider 返回 mcore GPTModel（或自定义 TTS 模型 wrapper），FA2 改 --use-flash-attn（启动脚本 L40，LLM 的 flash-attn 实现走 torch_npu 路由）。不要把 accelerate 的 from_pretrained 语义照搬进 model_provider——LLM 引擎自己管设备/dtype/wrapper，provider 只返回 CPU vanilla 模型。

L721-734 的 forward pre-hook 启停是 LLM 的启动期保护：首步（iteration == start_iteration）禁用 forward pre-hook（L680 disable_forward_pre_hook），等第一个 train_step 成功跑完（非 skipped_iter）再 enable_forward_pre_hook(model)（L732）恢复 param_sync。注释（L676-678）说明这是为了防止 checkpoint 加载或随机初始化的错误在首次 all-gather 传播到所有 rank。对照 MM 没有这段——LLM 在启动期多了一层“先验证一步成功再开 pre-hook”的保守保护。

NPU 迁移 · _enable_npu_datadump_step_end_enable_npu_datadump_step_end()（L716，定义 L87）是 LLM 的 NPU 侧 dump 钩子，在每步末触发。这是诊断/调试用的昇腾数据 dump，不影响训练正确性，但开启会有开销。迁移 Qwen3-TTS 时若不需要诊断，确认该钩子默认不触发实 dump（按 args 开关控制）即可；真机调试通信/算子问题时可主动开它抓 NPU 侧中间数据。具体 dump 内容与 CANN 的 dump 机制绑定，待真机验证开关路径。

torchrun 启动脚本与 NPU 环境注入

LLM 的启动不是 python pretrain_gpt.py，而是 torchrun ... pretrain_gpt.py <一堆参数>（启动脚本 L122）。启动脚本（examples/mcore/qwen3/pretrain_qwen3_8b_4K_ptd.sh）分四段注入配置：环境变量段（L3-8，HCCL/超时/NPU 内存/调度）、分布式参数段（nproc_per_node/nnodes/master）、训练参数段（lr/bf16/clip-grad/micro-batch/global-batch）、模型并行参数段（TP/PP/CP）。迁移 Qwen3-TTS 时，启动脚本是“启动配置/参数注入”的主要落点——引擎代码不动，靠脚本参数适配。

逐行看 NPU 环境变量。L3-4 HCCL_CONNECT_TIMEOUT=3600 / HCCL_EXEC_TIMEOUT=3600——HCCL（昇腾集合通信库）的连接与执行超时，单秒级，多机/大 PP 时常要调大避免假超时。L5 CUDA_DEVICE_MAX_CONNECTIONS=1——字面 CUDA 但经 torch_npu patch 落 NPU，限制设备最大连接数为 1（配合 PP 调度）。L6 PYTORCH_NPU_ALLOC_CONF=expandable_segments:True——NPU 显存分配策略，开 expandable_segments 减少碎片化，这是昇腾专属显存管理开关，CUDA 无等价。L7 NPU_ASD_ENABLE=0、L8 TASK_QUEUE_ENABLE=2——NPU 的 ASD（异常检测）与任务队列调度，是 CANN 运行时的调优开关。

训练参数段的关键几行：脚本里 --bf16（BF16 训练，910B 原生支持、无 GradScaler）、--clip-grad 1.0（全局范数裁剪，在 optimizer.step 内部）、--initial-loss-scale 4096（FP16 的初始 loss scale，BF16 时 no-op）。优化参数段 --use-flash-attn（昇腾 flash attention，替代 FA2）、--use-ascend-coc（COC 通信重叠）、--use-distributed-optimizer（ZeRO-1/2 分布式优化器）。这些就是 Qwen3-TTS accelerate 路径里“FA2→sdpa、accelerate prepare→setup_model_and_optimizer、nccl→hccl”在 LLM 引擎里的等价参数化形态。

L122 torchrun $DISTRIBUTED_ARGS pretrain_gpt.py——torchrun 起 N 个进程，每进程一个 rank，分别 import torch_npu（在 initialize_megatron 内）后落到对应 NPU。L132 --distributed-backend nccl 字面写 nccl，但经 torch_npu patch 后端实际是 HCCL——与 MM 一样，写 nccl 不是 bug，是约定字面量。整条启动链的“NPU 化”不在脚本里显式声明 npu，而是靠 torch_npu 的 monkey-patch 把 cuda 字面量重定向。这是 LLM/MM 共有的启动哲学：源码与脚本尽量保持与上游 Megatron 一致，NPU 适配收敛到 import torch_npu + 一组环境变量。

NPU 迁移 · NPU 环境变量PYTORCH_NPU_ALLOC_CONF=expandable_segments:True（脚本 L6）是昇腾显存碎片化治理开关，对长序列（Qwen3-TTS 的 codec 序列）尤其有用，建议保留。HCCL_CONNECT/EXEC_TIMEOUT（L3-4）多卡时按拓扑调大。CUDA_DEVICE_MAX_CONNECTIONS=1（L5）配合 PP 调度，改 TP 时可考虑放开。NPU_ASD_ENABLE/TASK_QUEUE_ENABLE（L7-8）是 CANN 运行时调优，出问题可先关 ASD 抓异常。这些环境变量不改源码、只改启动环境，是迁移时最便宜的旋钮——优先在脚本层调，再动模型/引擎层。待真机验证各开关对吞吐/稳定性的实际影响。
NPU 迁移 · distributed-backend nccl 字面量启动脚本 L132 --distributed-backend nccl 字面是 nccl，但经 torch_npu patch 实际走 HCCL。不要手动把它改成 hccl——megatron/torch_npu 的兼容层按 nccl 字面量路由到 HCCL，改字面反而可能绕过 patch 触发真 nccl 调用失败。对照 Qwen3-TTS accelerate 路径要手动把 DDP 后端 nccl→hccl，LLM 引擎这里不改字面量，靠 import torch_npu 自动落 HCCL。多卡迁移真正要验的是 HCCL 拓扑（HCCS 带宽）与 torch.distributed.barrier() 在 NPU 上的行为，不是 backend 字面量。
