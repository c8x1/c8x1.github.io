LLM 并行维度

逐段拆 MindSpeed-LLM 的并行状态初始化 mindspeed_llm/core/parallel_state.py：以 initialize_model_parallel_decorator 包住 Megatron，按 order='tp-cp-ep-dp-pp' 的 rank 布局逐组 new_group，补齐 EP/Tensor-EP/DP-modulo-EP/CP 三类 ring 组；CP 在 context_parallel/ 下分 megatron ring / ulysses / kv-allgather 三种 attention 算法；ZeRO 与 SP 走 Megatron 分布式 optimizer + layernorm allreduce。对照 Qwen3-TTS 的单进程 accelerate 循环，讲清‘单卡 SFT 要长成规模化训练，靠的是这一层并行工程’。

学习目标

  • 定位 initialize_model_parallel_decorator 的 rank 布局循环，说出 DP/CP/EP/Tensor-EP/DP-modulo-EP 五种 group 各自由哪段代码建
  • 说清 order='tp-cp-ep-dp-pp' 的 rank 排布约定与 DP*CP 必须被 EP 整除这条约束的来源
  • 区分 CP 的 megatron ring attention、ulysses attention、kv-allgather 三条算法路径在 attention 算子上的不同，以及各自走的 torch_npu 调用
  • 指出 get_nccl_options_wrapper 把 NCCL ProcessGroup 替成 HCCL 的注入点，并对照 Qwen3-TTS accelerate 的 DDP(nccl)→hccl 这条迁移线

前置知识

  • 读过 target/sft-training-loop.html，知道 Qwen3-TTS 是 accelerate 单进程、DDP 多卡要改 hccl
  • 读过 reference/mm-training.html，了解 Megatron 的 mpu 与 forward_backward 调度
  • 理解 TP/PP/DP/ZeRO 的基本概念，知道 MoE 的 expert 维度

总览：六个并行维度的工程职责

MindSpeed-LLM 是这条迁移链上的规模化训练卫星——Qwen3-TTS 的 sft_12hz.py 默认单进程、靠 accelerate 包一个训练循环；要把它长成多机多卡的规模化训练，拼的不是模型本身，而是并行维度工程。MindSpeed-LLM 在 Megatron 的 mpu 之上加了一层 decorator（mindspeed_llm/core/parallel_state.py 的 initialize_model_parallel_decorator），补齐 Megatron 原生不完整的 EP、Tensor-EP、DP-modulo-EP、以及 CP 的三类 ring 通信组。核心维度六个：TP 张量并行（层内切）、PP 流水并行（层间切）、CP 上下文并行（序列切，长序列用）、EP 专家并行（MoE 专家切）、SP 序列并行（layernorm 梯度 allreduce，常随 TP 启用）、DP 数据并行（含 ZeRO 分布式 optimizer 切参数/梯度/状态）。

六个维度不是并列罗列，而是嵌套在一个 rank 网格里：一个全局 rank 同时属于一个 TP 组、一个 CP 组、一个 EP 组、一个 DP 组、一个 PP 组，由 order='tp-cp-ep-dp-pp' 决定 rank 在网格里的排布顺序（最内维是 TP，最外维是 PP）。这种嵌套意味着通信组之间有共享 rank，torch.distributed.new_group 建组时要按依赖顺序遍历。最关键的一条约束在 L122：data_parallel_size * context_parallel_size % expert_model_parallel_size == 0——DP×CP 必须被 EP 整除，否则 EP 组无法在 DP×CP 的 rank 池里均匀切分，直接抛错。

Decorator：包住 Megatron 的 rank 布局

initialize_model_parallel_decorator（L35）是个装饰器工厂：它接受 Megatron 原生的 initialize_model_parallel 作为参数，返回一个 wrapper。wrapper 先原样调 initialize_model_parallel(...)（L92-112）让 Megatron 把 TP/PP/CP/DP 的基础组建好，然后在它建完之后用自己的逻辑补建 EP/Tensor-EP/DP-modulo-EP 这几组 Megatron 原生没有或不完整的组。这个‘先调原函数再补建’的顺序很关键——因为 EP 组的 rank 池要从 DP×CP 组里再切，必须等 DP/CP 组就位。注意 Megatron 原版里 EP 被设成 1，所以这里 L146 注释说‘Regenerate ep related groups’。

wrapper 的签名（L64-84）一口气列出全部并行维度的尺寸：tensor_model_parallel_size、pipeline_model_parallel_size、context_parallel_size、expert_model_parallel_size、num_distributed_optimizer_instances（ZeRO 分几实例）、expert_tensor_parallel_size、以及 order: str = 'tp-cp-ep-dp-pp'。这个 order 默认值决定了 rank 在多维网格里的排布方向——它是字符串约定，不是数值，Megatron 内部按这个顺序做 stride 计算。迁移读者只需记住：改 order 会改变所有组的 rank 成员，跨拓扑时通常保持默认，只在特殊场景（如 encoder/decoder 分裂）才动。

L88-90 有一处 VP（virtual pipeline，interleaved PP）的前置：pipeline_model_parallel_size == 2 and virtual_pipeline_model_parallel_size is not None 时手动把 Megatron 的 _VIRTUAL_PIPELINE_MODEL_PARALLEL_RANK 设为 0、_WORLD_SIZE 设为传入的 VP size——这是 interleaved 调度（一个 rank 持多个 PP stage 的层块）的前置状态。对照 Qwen3-TTS：它完全没有这些维度，单进程跑，连 torch.distributed.init_process_group 都不显式调（accelerate 隐式管）。从 Qwen3-TTS 迁到规模化，第一件事就是承认‘现在要管 rank 网格了’。

NPU 迁移 · rank 布局与多机拓扑order='tp-cp-ep-dp-pp' 决定 TP/CP/EP/DP/PP 各自落在哪些全局 rank 上。在 910B 多机场景，TP/CP/EP 这类高频通信维度应尽量落同一节点（8 卡 HCCS 互联带宽高），PP/DP 落跨节点。MindSpeed-LLM 不强制改 order，但若拓扑与默认 order 错配，TP 落跨节点会因 HCCS 带宽不足严重拖慢 allreduce。迁移时建议：先按默认 order 在单机 8 卡验证通信能通，再上多机观察 TP allreduce 带宽；若带宽不够再考虑调 order 或用 tp_x/tp_y 的 2D TP（见 L263-280 tp_2d）。待真机验证拓扑带宽。

Rank 布局循环：DP/CP/EP 组怎么切

L114-120 先算四个计数：num_tensor_model_parallel_groups = world_size // tp_size、num_pipeline_model_parallel_groups = world_size // pp_size、data_parallel_size = world_size // (tp * pp * cp)。注意 DP 是除掉 TP/PP/CP 之后剩下的——所以 world_size = tp * pp * cp * dp，EP 是从 DP×CP 池里再切，不占新的 rank 维度。这是理解整段布局循环的算术基础：EP 不能凭空多出 rank，它复用 DP×CP 的 rank。

L136-144 建 all_data_parallel_group_ranks 与 all_data_parallel_group_ranks_with_cp 两个 rank 列表。外层 for 遍历 PP stage（i in range(pp_size)），每个 stage 内按 stride 取 DP 组成员。两条列表的区别是 stride 步长：不带 CP 的用 cp * tp 步长（CP/TP 维度内的 rank 视作同一 DP 组的‘列内’），带 CP 的用 tp 步长（CP 不同 rank 也算同一 DP 组）。这两个列表后续喂给 EP 组构建——EP 组从带 CP 的 DP rank 池里切，所以 EP 本质上是‘在 DP×CP 的 rank 上对专家做再切分’。

shape 跟踪一下：假设 world_size=16, tp=2, pp=2, cp=1, ep=2。则 dp = 16//(2*2*1) = 4，dp*cp = 4，4 % ep(2) == 0 通过 L122 约束。num_pipeline_model_parallel_groups = 16//2 = 8。对 PP stage 0（i=0），start_rank=0, end_rank=8；j in range(cp*tp=2) 取 range(0, 8, 2) 和 range(1, 8, 2) 两条 DP 组，即 rank {0,2,4,6} 与 {1,3,5,7}。EP=2 再把每条 DP 组 {0,2,4,6} 切成 {0,2} 与 {4,6} 两个 EP 组——这就是 EP 复用 DP rank 的样子。

NPU 迁移 · EP all-to-all 与拓扑EP 组（L181-192）在 MoE 前向触发 expert all-to-all：token 按路由表发往对应 expert 所在 rank，算完再发回。all-to-all 是对 HCCL 拓扑最敏感的集合通信——它本质是全连接置换，跨节点时延迟和带宽都吃紧。910B 单节点 8 卡 HCCS 带宽尚可，多节点 EP 跨机 a2a 易成瓶颈。迁移 Qwen3-TTS 时它不涉及 MoE（Qwen3-TTS talker 是 dense），EP 这维用不上，可设 expert_model_parallel_size=1 跳过整段 L146-192。若未来训 MoE 版 TTS，才需验 EP a2a 的 HCCL 拓扑与 all_ep_groups 的 rank 分布。

EP / Tensor-EP / DP-modulo-EP 三组

L146-192 建三组 MoE 相关通信组，是 decorator 相对 Megatron 原版的核心增量。Tensor-EP 组（L151-163）：rank 范围是 tp * ep 大小的小块，连续取——它把 TP 与 EP 合到一个组里，用于 expert 内部再做张量切。DP-modulo-EP 组（L165-179）：在 Tensor-EP 组内再按 tp * ep 步长取 DP rank，还额外建了一个 gloo 后端的同 rank 组（L176）——gloo 用于 EP 的 CPU 同步协调（如负载均衡统计），HCCL 不擅长这类小数据控制流。EP 组（L181-192）：从带 CP 的 DP rank 池里按 ep_size 切，这是 MoE 路由 token 实际发送的组。三组层层嵌套，缺一不可。

NPU 迁移 · 通信组爆炸与 HCCL 资源六个维度嵌套 + EP 三组 + CP 三子组，一个 rank 上可能挂十几个 ProcessGroup。每个 HCCL ProcessGroup 在 CANN 侧占通信域资源（stream、notifier）。910B 上通信域数量有上限，维度开太满（如 tp+pp+cp+ep+sp 全开）可能触顶或建组变慢。迁移 Qwen3-TTS 时它只需 TP+DP（甚至单 DP），组数极少；规模化若贪多维度，建议先小规模验证建组耗时与通信域余量。保守：按需开维度，PP/CP/EP 能不开就不开。待真机验证通信域上限。

CP 三算法：ring / ulysses / kv-allgather

上下文并行把序列沿长度维度切到 CP 个 rank，每个 rank 只算自己那段序列的 attention。难点是 causal attention 里每个 token 要看它之前所有 token，跨 rank 的 KV 要靠通信补。MindSpeed-LLM 在 context_parallel/get_batch_utils.py 的 get_batch_on_this_cp_rank 里按 args.context_parallel_algo 分四条路：megatron ring（megatron_cp_algo）、ulysses（ulysses_cp_algo，含 mamba）、hybrid（hybrid_cp_algo）、kv-allgather（kvallgather_cp_algo）。Qwen3-TTS 的长序列 attention 是 CP 最直接的受益点——它单卡要把整个 codec 序列塞进 attention，显存吃紧。

四条路里 megatron ring 与 kv-allgather 复用同一段 batch 切分（_get_batch_on_this_cp_rank_in_megatron_cp），区别在 attention 算子：ring 走 P2P 环传递 KV、每 rank 算局部 attention 再聚合 softmax 统计量；kv-allgather 直接 allgather 全部 KV 再算，通信粗但实现简单，CP 小时划算。ulysses 走另一套切分（_get_batch_on_this_cp_rank_in_ulysses_cp），它沿 head 维度切、配合 all-to-all 换头，适合 head 多的模型。L19-29 的注释说清了 megatron ring 的负载均衡技巧：序列切成 2*CP 段，chunk_i 与 chunk_{2CP-1-i} 配给同一 rank，让前后段均衡（否则靠后的 chunk 负载远大于靠前的）。

ring attention 的真正实现在 context_parallel/ring_context_parallel.py 的 AttentionWithCp（torch.autograd.Function）。L100-216 是 forward 的双层 ring 循环：外层 for j in range(outer_size) 走外环换 KV 块（L104 outer_ring.async_send_recv），内层 for i in range(inner_size) 走内环换（L108 inner_ring.async_send_recv）。每次拿到对端的 KV 块后调 torch_npu.npu_fusion_attention（L130-142 / L156-166）算局部 attention，把 softmax max/sum 累进 global_attn_outs，最后统一归约。这里用的是昇腾原生 npu_fusion_attention 而不是 FlashAttention-2——CP ring 与 NPU fusion attention 是绑定的，这对 Qwen3-TTS 的 FA2 硬绑是直接对照。

HCCL 注入：get_nccl_options_wrapper

整条并行通信能落到 HCCL，靠的是 get_nccl_options_wrapper（L373-384）。它包住 Megatron 的 get_nccl_options：每次 Megatron 建组调 get_nccl_options(pg_name, cfgs) 时，wrapper 先检查 torch_npu._C._distributed_c10d.ProcessGroupHCCL.Options 是否有 hccl_config 属性——有就构造一个 HCCL Options 并设 hccl_config = {'group_name': pg_name} 返回，把 Megatron 本要建的 NCCL 组直接偷换成 HCCL 组；没有才 fallback 回原 NCCL options。这是一处‘把 NCCL 全局偷梁换柱成 HCCL’的注入点。

def get_nccl_options_wrapper(get_nccl_options):
    @wraps(get_nccl_options)
    def wrapper(pg_name, nccl_comm_cfgs):
        if hasattr(torch_npu._C._distributed_c10d.ProcessGroupHCCL.Options, "hccl_config"):
            try:
                options = torch_npu._C._distributed_c10d.ProcessGroupHCCL.Options()
                options.hccl_config = {"group_name": str(pg_name)}
                return options
            except Exception:
                return get_nccl_options(pg_name, nccl_comm_cfgs)
        return get_nccl_options(pg_name, nccl_comm_cfgs)
    return wrapper

group_name 用 pg_name（如 'tp_exp'/'dp_modulo_exp'/'exp'）——CANN 侧据此区分不同通信域的聚合策略。try/except 包裹是为了兼容老版 torch_npu（没有 hccl_config 属性时不崩）。这个 wrapper 与 initialize_model_parallel_decorator 一样，是通过 patch_manager 注册到 megatron.core.parallel_state.get_nccl_options 生效的（注册点见 features_manager/ 下各 feature，本 checkout 只提供 wrapper 定义）。对照 Qwen3-TTS：它 accelerate 的 DDP 默认 NCCL，迁移要手动 backend='hccl' 或走 torch_npu patch；MindSpeed-LLM 这套是工程化、按组精细化的 HCCL 注入，比 accelerate 的粗粒度 backend 设置更可控。

NPU 迁移 · NCCL→HCCL 后端MindSpeed-LLM 全部集合通信经 get_nccl_options_wrapper 落 HCCL，不需手动设 backend='hccl'——这与 Qwen3-TTS accelerate 路线（需手动改 DDP backend 或靠 torch_npu patch）是工程量差异的核心。迁移 Qwen3-TTS 到规模化若直接接 MindSpeed-LLM，HCCL 注入天然就位；若留在 accelerate，则要自己保证 init_process_group(backend='hccl') 且 DDP 包裹时 backend 匹配。风险：HCCL 版本与 torch_npu/CANN 版本要三件套匹配，版本错配会建组失败或通信 hang。保守：先验单节点 TP=2 的 allreduce 通断，再上多节点。待真机验证版本矩阵。

DP/ZeRO 与 SP：分布式 optimizer + layernorm allreduce

DP 维度在 MindSpeed-LLM 里有两层增强：一是ZeRO 分布式 optimizer（core/optimizer/distrib_optimizer.py 的 distributed_optimizer_init，L22-34），它把 optimizer state / 梯度 / 参数按 DP 组切分（对应 ZeRO-1/2/3 的不同切法），num_distributed_optimizer_instances（decorator 签名 L74）还能把 ZeRO 切成多实例做分级聚合；二是SP 序列并行（core/distributed/finalize_model_grads.py 的 allreduce_layernorm_grads，L27-45），TP>1 且开 sequence_parallel 时，layernorm 的梯度沿序列维分布，反向要 allreduce 聚合。两者都落在 DP/TP 组上，通信走 HCCL。

对照：Qwen3-TTS 单进程怎么长成规模化

把 Qwen3-TTS 的 sft_12hz.py 与 MindSpeed-LLM 这套并行工程对位。Qwen3-TTS：Accelerator(grad_accum=4, bf16) 单进程，from_pretrained(FA2, bf16)，DataLoader(bs=2)，accelerator.prepare 接管——零并行维度，连 torch.distributed 都隐式。要长成规模化，要补的正是本页这六维：序列长就上 CP（ring attention + npu_fusion_attention，天然替掉 FA2 硬绑），卡多就上 TP（层内切）+ DP（数据并行 + ZeRO-1 切状态），模型更深就上 PP（层间切 + P2P）。每补一维，就要在 initialize_model_parallel 的 rank 网格里多开一组通信组，并承受相应 HCCL 通信开销。
