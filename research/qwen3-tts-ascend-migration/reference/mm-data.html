学习轨道：源码深读

MM 数据分派

按 MindSpeed-MM 的 data 包逐段拆开多模态数据管线：build_mm_dataset 按 dataset_type 分派到 13 种 dataset、build_mm_dataloader 按 dataloader_mode 选 base/sampler/variable 三种装载、collate 由 DATA_COLLATOR 注册表按 model_name 选 collator、bucket sampler 在 rank0 建桶再 broadcast 给各卡——每段标出 CPU 数据管线、host 端建桶/分桶、多卡广播在 910B 上的性能与正确性风险，并与 Qwen3-TTS 的 TTSDataset/collate_fn 做单模态 vs 多模态的对照。

学习目标

  • 定位 build_mm_dataset 与 build_mm_dataloader 两个分派入口，说出 dataset_type→dataset 类、dataloader_mode→prepare_* 函数的映射
  • 说清 DATA_COLLATOR 注册表如何按 model_name 选 collator，以及 Qwen2VL 路径为何委托给 transformers 的 DataCollatorForSeq2Seq + mm_plugin
  • 区分三类 sampler：StatefulDistributedSampler（朴素分片）、LengthGroupedSampler（按长度 megabatch）、BucketBatchSampler（rank0 建桶 broadcast），并指出它们在 910B 多卡下的正确性前提
  • 对照 Qwen3-TTS 的 collate_fn（单模态 16 层 codec 固定布局）与 MindSpeed-MM 的多模态 collate（变长 pixel_values 拼接 + mm_plugin 注入），说出两者 batch 张量形状的根本差异

前置知识

  • 读过 target/sft-training-loop.html 与 target/sft-data-collate.html，了解 Qwen3-TTS 的 TTSDataset/collate_fn 与 16 层 codec 布局
  • 理解 PyTorch DataLoader / Sampler / collate_fn 三件套的职责分工
  • 理解分布式训练里 rank/world_size 与 data parallel 分片

总览：两层分派 + 三种装载

MindSpeed-MM 的 mindspeed_mm/data 包共 41 个 py 文件，但分派逻辑收敛在两个入口：build_mm_dataset(dataset_param) 按 dataset_type 把配置路由到 13 种 dataset 类（t2v/i2v/t2i/multimodal/huggingface/deepseekvl2/audio/feature/lumina/bagel/…），build_mm_dataloader(dataset, dataloader_param, …) 按 dataloader_mode 选 base/sampler/variable 三种装载策略。两者都定义在 mindspeed_mm/data/__init__.py，是迁移到 NPU 时数据侧唯一要先看懂的两块。分派靠字符串键（dataset_type、dataloader_mode、collate_param.model_name），不靠类型系统——这意味着新增一个模型要同时注册三处：dataset 类、DATA_COLLATOR 项、sampler 路径。

贯穿整条管线的张量是 batch 字典。Qwen3-TTS 的 collate 产出 input_ids (b,t,2) + codec_ids (b,t,16) 这种固定通道布局——因为 TTS 只有一条音频模态、16 层 codec 结构固定。而 MindSpeed-MM 的多模态 collate 产出的是变长拼接：input_ids (b,t) long、pixel_values (N_total,d) float（N_total 是 batch 内所有图像 patch 数之和，每条样本 patch 数不同）、labels (b,t) long 用 -100/IGNORE_INDEX 屏蔽。本质差异：TTS 的通道维是模态结构（文本+codec 双通道、16 残差层），MM 的通道维被压平成一维 token 流，模态边界靠 image_grid_thw 等元数据张量重建。

Dataset 分派：build_mm_dataset 的字符串路由

build_mm_dataset 在 mindspeed_mm/data/__init__.py，先把 dataset_param 标准化成 dict，校验三个必填键 dataset_type / basic_parameters / preprocess_parameters，再用一串 if-elif 按 dataset_type 路由。这是典型的字符串键分派——没有基类多态，新增类型就是加一个 elif 分支。迁移到 NPU 时，这段本身无算子风险，但它决定了后续走哪条 dataset 类，而不同 dataset 类的 __getitem__ 含不同的 host decode 路径。

逐段看路由表。t2v/i2v/t2i/dt2v 是视频/图像生成类（DynamicVideoTextDataset 是 dt2v 动态分辨率变体）；multimodal 是 InternVL 风格的对话多模态，basic_param 可为 list 用 ConcatDataset 拼接多源数据；huggingface 走 get_qwen2vl_dataset（Qwen2VL/Qwen3-Omni 路径，用 datasets.load_dataset + .map 预处理）；audio/lumina/bagel 各有专门 dataset。注意 repeat_time 在 multimodal/deepseekvl2 分支被透传，控制数据复制/截断倍率。

迁移对照：Qwen3-TTS 没有「分派」概念——finetuning/dataset.py 只有一个 TTSDataset 类，sft_12hz.py 直接 TTSDataset(data, processor, config) 构造。MindSpeed-MM 之所以要分派，是因为它要服务十几个不同模型（InternVL/Qwen2VL/Qwen3-Omni/OpenSoraPlan/DeepSeekVL/Bagel…），每个模型的样本结构不同。把 Qwen3-TTS 接进 MindSpeed-MM 训练引擎时，需要在 build_mm_dataset 加一个 elif dataset_type == 'qwen3_tts' 分支，并在 DATA_COLLATOR 注册对应 collator——这是「单模态脚本」与「多模态框架」对接的第一个落点。
NPU 迁移 · image/video decode 在 hostMultiModalChatDataset.multi_modal_get_item（datasets/multimodal_dataset.py:145-169）调 self.img_video_processor(image_path=…) 读图、过 torchvision/processor。视频路径 video_get_item 还要 VideoReader 解码帧。这些都在 DataLoader worker 进程的 host CPU 上，NPU 不参与。风险是 host 吞吐：当 num_workers 不够、或单样本解码重（高分辨率图、长视频）时，host decode 成瓶颈，NPU 空等。昇腾机一般配 ARM host，单核弱于 x86，num_workers 要开更大（建议 8+，按 CPU 核数调）。这与 Qwen3-TTS 的 mel_spectrogram 钉 CPU 同属「host 算力不足」类，但 MM 这边压力更大（图/视频 decode 远重于 mel）。待真机验证 num_workers 上限与 worker 间内存。

Dataloader 分派：base / sampler / variable

build_mm_dataloader 同文件，按 dataloader_mode 分三路：base 走 prepare_base_dataloader（最朴素的 torch.utils.data.DataLoader，不挂分布式 sampler）；sampler 走 prepare_sampler_dataloader（按 sampler_type 再二次分派到 6 种 sampler，是多卡训练的主力路径）；variable 走 prepare_variable_dataloader（给变长视频用 VariableVideoBatchSampler）。注意它会从 megatron 的 args 强制注入 batch_size=args.micro_batch_size、num_workers=args.num_workers、seed=args.seed——这三个值以命令行参数为准，不以 data.json 为准，迁移时容易在配置来源上踩坑。

进入 prepare_sampler_dataloader（主力路径）。它先处理 IterableDataset 例外（这类数据 num_workers=0、不挂 sampler），再按 sampler_type 二次分派。六个分支里与多卡正确性强相关的有四个：stateful_distributed_sampler（朴素 StatefulDistributedSampler 分片）、LengthGroupedSampler（按样本长度做 megabatch 聚类减少 padding）、BucketBatchSampler（rank0 建桶再 broadcast）、BaseRandomBatchSampler（bucket 分桶前的随机批采样）。每个分支末尾都 collate_param.pop('model_name') 从 DATA_COLLATOR 取 collator——sampler 选型与 collator 选型是解耦的，由两个独立配置键控制。

NPU 迁移 · dp group 与 HCCL 后端process_group = mpu.get_data_parallel_group() 取的是 megatron 维护的 data parallel 组。StatefulDistributedSampler（sampler.py:361-390）内部用 num_replicas=process_group.size()、rank=process_group.rank() 做朴素分片，indices[self.rank::self.num_replicas]。这在 NPU 上本身无算子，但 process_group 的后端必须是 HCCL——若 megatron 初始化时 backend 留成 nccl，后续 allreduce/broadcast 会失败。BucketBatchSampler 还会 dist.broadcast_object_list(…, src=0) 把 rank0 建好的桶 index 广播给所有卡（见 sampler.py:618），这一步是显式集合通信，910B 上走 HCCL 的 broadcast_object_list，需验证其与 mindspore/torch_npu 的 object list 序列化兼容。单卡迁移不受影响，单机多卡/多机段才会碰到。

Collate：DATA_COLLATOR 注册表与 mm_plugin

collate 的分派核心是 DATA_COLLATOR 这个 dict（dataloader/data_collator.py:681-694），键是 model_name，值是 collator 类。prepare_*_dataloader 里统一用 collate_fn = DATA_COLLATOR[data_collate_type](**collate_param, dataset_param=dataset_param) 取。这张表是「新增模型要改的第二处」：给 Qwen3-TTS 加 MindSpeed-MM 训练支持，除了 dataset 分派，还得在这里加 'qwen3_tts': SomeCollator。

看 Qwen2VL 这条最常用路径。DataCollatorForQwen2vl（L182-196）本身很薄：__init__ 里从 dataset_param.preprocess_parameters 装载 tokenizer/processor/template，再 new 一个 MultiModalDataCollatorForSeq2Seq（继承自 transformers 的 DataCollatorForSeq2Seq），__call__ 直接转发。真正干活的是 MultiModalDataCollatorForSeq2Seq.__call__（func_utils/collator.py:43-149）：它把每条样本里的 images/videos/audios pop 出来收集成跨样本的大 list，再交 template.mm_plugin.get_mm_inputs 处理成 pixel_values、image_grid_thw 等张量，最后 super().__call__（即 DataCollatorForSeq2Seq）做 input_ids/labels 的 padding。

class DataCollatorForQwen2vl:
    def __init__(self, ignore_pad_token_for_loss: bool, dataset_param=None, **kwargs):
        process_args = ProcessorArguments(**dataset_param.preprocess_parameters.to_dict())
        tokenizer_module = load_tokenizer(process_args)
        tokenizer = tokenizer_module.get('tokenizer')
        template = get_template_and_fix_tokenizer(tokenizer, dataset_param.basic_parameters.template)
        self.data_collator = MultiModalDataCollatorForSeq2Seq(
            template=template,
            pad_to_multiple_of=8,  # for shift short attention
            label_pad_token_id=IGNORE_INDEX if ignore_pad_token_for_loss else tokenizer.pad_token_id,
            **tokenizer_module,
        )

    def __call__(self, *args, **kwargs):
        return self.data_collator(*args, **kwargs)

MultiModalDataCollatorForSeq2Seq.__call__ 有一个对 zero3/fsdp 很关键的兜底：当整个 batch 没有图像/视频（sum(batch_imglens)==0）但 template 配了 image_token，它会造一个 64x64 白图 + fake input_ids 塞进 batch（L58-72）。注释写明「avoid process hanging in zero3/fsdp case」——因为 zero3 下所有 rank 都要参与参数分片的 forward，若某 rank batch 无图像，processor 不产生 pixel_values，会导致集合通信死等。这个假图兜底在 NPU 上同样适用：MindSpeed 的 ZeRO 等价物（FSDP/TP）也会有「各 rank 必须有同形输入」的约束。

NPU 迁移 · mm_plugin 算子链template.mm_plugin.get_mm_inputs（collator.py:102-111）把原始 PIL/numpy 图像喂进 processor（如 AutoProcessor.from_pretrained(qwen2vl)），其内部含 smart_resize、patchify、归一化——底层是 torchvision/Pillow CPU 算子。建议留 CPU（与 Qwen3-TTS mel 钉 CPU 同一判断：这是预处理，对 NPU 吞吐不敏感且 CANN 的图像算子未必覆盖）。唯一要验证的是若 mm_plugin 内部有 .to(device) 把 pixel_values 提前搬 NPU 再 patchify，则要走 NPU 算子路径核验；当前源码看 collate 输出仍是 CPU 张量，搬到 NPU 发生在训练循环的 to(device)，不在 collate 内。

样本构造：MultiModalChatDataset.getitem 的模态分派

样本返回 dict 的 key 在 _init_return_dict（L127-134）固定为 pixel_values / image_flags / input_ids / labels / attention_mask。注意 image_flags 标记哪些 patch 是真图（L166 torch.tensor([1]*num_image_patches)）——collate 阶段 DataCollatorForInternvl 用 torch.concat（不是 stack）合并 pixel_values/image_flags，正是因为各样本 patch 数不同。这与 Qwen3-TTS __getitem__ 返回 {text_ids (1,t), audio_codes (t,16), ref_mel} 形成对照：TTS 样本是定结构的（每条都有 text+16 层 codec+1 个 ref mel），MM 样本是变结构的（图/多图/视频，patch 数各异）。

NPU 迁移 · 重试换样本的多卡一致性getitem 的失败重试用 random.randint(0, len-1)（multimodal_dataset.py:267）。在 DataLoader 多 worker 下，get_seed_worker(seed)（utils.py:641）给每个 worker 派不同 seed 保证可复现。但坏样本的分布若与 rank 相关（如数据已分片），重试换的样本会让各 rank batch 数量/内容发散，进而触发集合通信对齐失败（DDP allreduce 要求各 rank batch 形状一致）。NPU 多卡下要验证：1) num_workers 相同时各 rank 见到的坏样本集是否一致；2) 若不一致，考虑在 dataset 层先过滤坏样本而非运行时重试。Qwen3-TTS 无此机制（无重试），迁移 TTS 时若引入 MM 框架要注意别因重试破坏 DDP 形状对齐。

深读：Bucket sampler 的 rank0 建桶 + broadcast

MindSpeed-MM 数据侧最复杂、也最易在多卡下出正确性问题的是 BucketBatchSampler（dataloader/sampler.py:485-674）。它的设计动机：多模态样本的图像 token 数差异巨大（小图几十 token，大图几千），若随机凑批，同 batch padding 到最长会浪费大量算力。所以它按「图像 token 数」把样本分桶，桶内样本 token 数相近，padding 浪费小。桶由 BucketManager_qwen2vl（bucket_manager.py:358）管理。

BucketBatchSampler.__iter__（L532-674）的关键约束：只有 rank0 建桶（if self.rank == 0 L550），其余 rank self.bucket_manager = None（L604）。建好的 idx_range_total 由 dist.broadcast_object_list(idx_range_total_b, src=0)（L618）广播给所有 rank。这是显式的 rank0-centralized 设计——避免每张卡都跑一遍 PIL.Image.open 算 token 数（那是 O(N) 的 host 重活），只在 rank0 跑一次。代价是：若 rank0 算出的桶与别的 rank 期望不一致（如数据分片不均），所有 rank 拿到的 index 列表相同但取自己分片 idx_range[rank::num_replicas]。

    def create_buckets(self):
        merged_buckets = {}
        # Calculate tokens by image size
        resize_image_resolution = round(self.image_size / self.factor) * self.factor
        max_tokens = int(resize_image_resolution / self.patch_size) ** 2

        # Determine the bucket range based on bucket_interval
        bucket_range = tuple(range(0, max_tokens + 1, self.bucket_interval))

        # Ensure max_tokens is included in bucket_range
        if max_tokens % self.bucket_interval != 0:
            bucket_range = bucket_range + (max_tokens,)

        # Convert bucket_range to (start, end) interval pairs
        result_bucket_range = list(zip(bucket_range[:-1], bucket_range[1:]))

        for bucket_range in result_bucket_range:
            sorted_bucket_range = tuple(sorted(bucket_range))
            if bucket_range not in merged_buckets:
                merged_buckets[sorted_bucket_range] = BucketManager.Bucket(sorted_bucket_range, self.num_groups)
        return list(merged_buckets.values())

generate_index（L329-355）把桶内 package 重排成最终 index 列表：先 shuffle package 序，再对每个桶 refresh_index 桶内 shuffle，最后按 package 取样本拼成一维 list。返回的 index_list 就是广播对象。各 rank 拿到后按 priority_mode 分两种取法：data_bucketing_img（L609-629，桶序优先，按 idx_range[rank::num_replicas] 取）和 data_reordering_img（L631-664，按 global_batch_size 在桶内重排）。

NPU 迁移 · 建桶 host CPU 单点rank0 建桶用 multiprocessing.Pool(processes=suggest_thread_count(dataset))（bucket_manager.py:499,511），suggest_thread_count（L195-212）按数据量与 os.cpu_count() 给线程数（数据量 <50000 取 1/12 核，否则 1/6 核，最小 8）。风险点：昇腾机 host 常是 ARM、核数与单核性能弱于 x86，PIL.Image.open 单线程吞吐有限，大数据集首次建桶可能要几十分钟且只有 rank0 在跑，其余 NPU 全程空等。保守方向：1) 评估是否换 data_reordering_img（用 process_calculate_images_token 同样 host 重，但只算 token 数不落桶，略轻）；2) 增大 processes_num 上限（当前受 1/6 核 cap）；3) 对小数据集干脆退回 stateful_distributed_sampler 省掉建桶。待真机测建桶耗时与内存。
NPU 迁移 · broadcast_object_list 序列化dist.broadcast_object_list(idx_range_total_b, src=0)（sampler.py:618）把一个可能上百万长度的 Python list 广播给所有 rank，底层是 pickle + HCCL broadcast。风险：1) 大 list 的 pickle/broadcast 耗时与内存峰值在首步明显；2) torch_npu/mindspore 的 broadcast_object_list 对超长 list 的实现兼容性需验证（曾有 object list 大小上限的报告）。保守方向：把 bucket index 持久化到文件，各 rank 直接读文件而非 broadcast（部分 MindSpeed 版本支持 data_sharding=True 走文件分片路径 L620-625）。这是 910B 多卡数据侧最该压测的一环。
与 Qwen3-TTS 的根本对照：TTS 的 collate_fn 用 DataLoader(shuffle=True, collate_fn=dataset.collate_fn)，没有任何 sampler/bucket——因为 TTS 样本长度差异主要来自文本长度，且 SFT 数据量小（定制单说话人），padding 浪费可接受，无需建桶。MindSpeed-MM 要服务海量多模态预训练/SFT，图像 token 数方差极大，才上 bucket。把 Qwen3-TTS 接进 MindSpeed-MM 时，若数据量大可借用 bucket 机制减少 padding，若仍是小规模定制 SFT 则直接用 stateful_distributed_sampler 即可，不必承担建桶开销。

样本如何分派到不同模型：三键解耦

把贯穿全篇的「分派」收束一下。MindSpeed-MM 的数据管线被三个字符串键解耦：dataset_type（选 dataset 类，决定 __getitem__ 怎么造样本）、model_name（选 collator，决定 batch 张量布局）、sampler_type/dataloader_mode（选 sampler，决定 index 顺序与多卡分片）。三键独立配置，可以「InternVL dataset + Qwen2VL collator + BucketBatchSampler」自由组合——只要 dataset 产出的样本 dict 含 collator 要的 key。这是框架级灵活性的代价：每加一个模型要碰这三处注册。

Qwen3-TTS 这边没有这套解耦——finetuning/dataset.py 里 dataset、collate、sampler 三者同属一个 TTSDataset 类（collate_fn 是它的方法），sampler 用 DataLoader 默认 shuffle。这意味着 TTS 的样本结构与 batch 布局是强绑定的（16 层 codec 布局写死在 collate_fn 里）。迁移到 NPU 时，TTS 这套强绑定反而简单（少配置项、少分派歧义），但牺牲了换数据源/换采样策略的灵活性。下表把两者的分派哲学摆在一起。
