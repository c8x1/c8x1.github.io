MOSS 包边界

以固定 SHA moss-tts-ad99ec5f 的仓库根为坐标，逐目录拆解 MOSS-TTS 的包边界：四个 TTS 变体子包（local / delay / local_v1.5 / realtime）与一个 git submodule 形式的音频编解码器子包（moss_audio_tokenizer）如何分工，各自基于哪条 Qwen3 主干、用哪种 RVQ 调度策略、依赖何种 audio tokenizer，以及哪些子包是 torch-free 边界。本页是语音/codec 卫星页：对照 Qwen3-TTS 的 token/声码器/说话人差异讲清 TTS 特有的模块边界，但 MOSS-TTS 不被表述为 Ascend 训练已验证实现——只标出 audio_tokenizer/realtime 部署段在昇腾上的静态迁移注意点。

学习目标

  • 按目录说出四个 TTS 变体子包的职责：local（Temporal+Depth 双 transformer）、delay（单 8B 主干 + 33 并行头 + delay-pattern）、local_v1.5（48kHz 立体声升级）、realtime（流式低延迟）
  • 识别 moss_audio_tokenizer 是 git submodule 而非本仓源码——它是受限外部资产，本页不嵌入其内容，只标边界
  • 区分各变体基于的 Qwen3 主干规模（1.7B / 4B / 8B）与 RVQ 层数（12 / 16 / 32），对照 Qwen3-TTS 的 16 层残差 codec
  • 识别 llama_cpp 子包提供的 torch-free 推理边界，以及 audio tokenizer 走 ONNX/TensorRT 的部署形态对昇腾的影响

前置知识

  • 读过 Qwen3-TTS 包结构与 SFT 训练循环
  • 理解 RVQ 残差量化与多码本语言建模
  • 理解 git submodule 与 Python namespace 包的差别

总览：一个仓库、五条边界

MOSS-TTS 仓库根的 pyproject.toml 把整包命名为 moss-tts，但仓库实际由四个独立的 TTS 变体子包加一个外部编解码器 submodule组成。这不是单一 monolithic 包，而是一组并列实现：同一团队、同一套 Qwen3 系列主干、四种 RVQ 调度策略。理解边界前先记住五个顶层目录：moss_tts_local/（双 transformer 时序+深度）、moss_tts_delay/（单 8B 主干 + delay-pattern + 33 并行头）、moss_tts_local_v1.5/（48kHz 立体声升级版）、moss_tts_realtime/（流式多轮低延迟）、moss_audio_tokenizer/（git submodule，外部 codec）。

local vs delay：两种 RVQ 建模范式

moss_tts_local/ 与 moss_tts_delay/ 是同一团队对多码本语音建模的两种正交解法，这是 MOSS-TTS 包边界的核心张力。两者都把语音压成 RVQ 离散整数序列（同一 Cat 音频 tokenizer、24kHz、12.5Hz 帧率），但如何在一步内产出多层码本走了两条路：local 用“时序 transformer 主干 + 深度 transformer 局部模块”的嵌套自回归，每个时间步逐层预测；delay 用单个大主干 + 32 个并行线性头，靠 delay-pattern 的时间偏移让因果性自动成立。

moss_tts_local/README.md · L11–L18
固定源码 ↗
## 1. Overview: Temporal + Depth Modeling

Unlike the "Delay-Pattern" architecture which shifts codebooks across time steps, the **MossTTSLocal** architecture performs **Time-Synchronous Generation**. In each time step $t$, the model predicts the entire sequence of Residual Vector Quantization (RVQ) tokens for that specific frame.

### Key Components
*   **Global Transformer (Temporal Backbone):** Responsible for long-range dependencies, linguistic context, and global style/prosody modeling.
*   **Local Transformer (Depth Transformer):** A lightweight module that models the coarse-to-fine dependencies between RVQ codebook layers within a single time step.
*   **Cat (Causal Audio Tokenizer):** The underlying discrete interface that provides high-fidelity audio compression and reconstruction at 24kHz.

local 的 README（moss_tts_local/README.md:11-18）把范式说得很直白：Time-Synchronous Generation——每个时间步 t 产出该帧全部 RVQ 层。Global transformer（时序主干，基于 Qwen3-1.7B）负责长程依赖与韵律，Local transformer（4 个 block，hidden 1536 / FFN 8960）负责单个时间步内 RVQ 码本层间的由粗到细依赖。Cat（Causal Audio Tokenizer）是底层离散接口。这种范式天然流式友好，但每步要多跑一次深度模块。

moss_tts_delay/README.md · L11–L20
固定源码 ↗
## 1. Overview: Parallel Heads + Delay Pattern

Unlike the **MossTTSLocal** architecture which uses a hierarchical "Temporal + Depth" approach, **MossTTSDelay** integrates all modeling into a single large-scale Transformer. It achieves efficient multi-codebook modeling by shifting the RVQ layers in the time domain, allowing the model to predict all codebook layers for a given step simultaneously through multiple linear heads.


### Key Components
*   **Unified Transformer Backbone:** A large-scale language model (based on the **Qwen-8B** scale) that handles text encoding, prosody modeling, and audio token prediction in a single forward pass.
*   **Multi-Head Output Layer:** The backbone is equipped with **$1 + N_q$** (where $N_q=32$) prediction heads. One head manages the primary sequence logic, while the other 32 heads parallelly predict the RVQ codebook layers.
*   **Delay-Pattern Scheduling:** A specialized data formatting technique that introduces a 1-step offset between successive RVQ layers. This enables causal dependency modeling across codebook depths without requiring an additional "Depth Transformer."

delay 走相反方向：把所有建模塞进单个大 transformer（Qwen-8B 规模），主干末端接 1 + N_q = 33 个线性头——1 个管主序列逻辑，32 个并行预测各 RVQ 层。多层间的因果依赖不靠额外模块，而靠 delay-pattern 调度：相邻 RVQ 层在时间上错开 1 步，于是预测第 k 层时第 k-1 层的历史已在上下文里。这是用数据排布换结构复杂度——主干少一层嵌套循环，单步前向就能出整帧 32 层码本，推理更快、长文本更稳。

对照 Qwen3-TTS：Qwen3-TTS 的 16 层残差 codec 用的是第三种——主 talker 只算第 0 层 next-token CE，第 1-15 层交给独立的 code_predictor + sub-talker 的 MTP 辅助损失。三者共享“残差量化 + 多码本语言建模”的大框架，但 Qwen3-TTS 把残差层监督放在损失侧（MTP），local 放在结构侧（深度 transformer），delay 放在数据侧（delay-pattern 偏移）。所以 MOSS 不是 Qwen3-TTS 的等价实现，而是同族的姊妹架构——本页只读边界，不把 MOSS 当作 Ascend 训练金标准。

NPU 迁移 · 包边界多与设备无关local vs delay 的范式选择是算法结构层面的差异，与张量落 NPU 还是 CUDA 无关——主干都是 transformers 的 Qwen3/Qwen 系列模块，迁昇腾时只要把注意力实现与 device 路由换掉即可，不影响“双 transformer vs 单主干+并行头”的结构。真正与设备耦合的是并行头数量：delay 的 33 个线性头在单步前向里对同一个 hidden states 做 33 次 matmul，NPU 上算子可跑但要注意显存/算子融合；local 的深度模块是 4 个小 transformer block，单步内串行跑多次 attention，对 sdpa 的数值对齐要求更高。这些属静态推断，待 910B 真机验证。

local_v1.5 与 realtime：升级与流式

moss_tts_local_v1.5/ 是 local 范式的升级版，不是新架构。它仍是 MossTTSLocal 时序+深度双 transformer，但把主干从 Qwen3-1.7B 换到 Qwen3-4B、深度模块缩成 1 个 block（hidden 2560 / FFN 9728）、RVQ 层从 32 降到 12，并把音频格式升到 48kHz 立体声、audio tokenizer 换成 OpenMOSS-Team/MOSS-Audio-Tokenizer-v2。这是“更大主干 + 更少码本层 + 更高音质 codec”的取舍组合。

moss_tts_realtime/ 面向流式多轮对话。README（moss_tts_realtime/README.md:5-8）说明它由 1.7B 主干 + 200M local transformer 组成，主干把隐状态暴露给 local transformer 去自回归生成 RVQ 音频 token；文本 token 与语音 token 在不同层级处理，天然支持流式文本输入 + 流式音频输出。这是为语音 agent 场景设计的——与 local 的“单段高质量合成”、delay 的“长文本稳定生成”定位不同。RVQ 16 层、24kHz，codec 仍是外部 MOSS-Audio-Tokenizer。

## 1. Overview
MOSS-TTS-RealTime is built on a scalable architecture consisting of a 1.7B-parameter backbone and a 200M-parameter local Transformer, designed to deliver high-quality, low-latency speech synthesis. The backbone encodes linguistic and contextual information and exposes its hidden states to the local Transformer, which autoregressively generates RVQ-based audio tokens. Audio reconstruction is performed using the MOSS-Audio-Tokenizer, enabling efficient and high-fidelity waveform synthesis.

The model adopts a hierarchical input design in which text tokens and speech tokens are processed at different levels, allowing the system to naturally support streaming text input and streaming audio output. This design enables incremental generation while preserving long-range context, making it well suited for real-time conversational scenarios.

realtime 的推理入口 moss_tts_realtime/infer.py 里有和 local README 逐字相同 的 resolve_attn_implementation() 函数——先试 flash_attn + major>=8 选 FA2，CUDA 兜底 sdpa，CPU 兜底 eager。这说明四变体共享同一套CUDA 优先的注意力选择逻辑，是 MOSS 包内重复出现的硬编码 CUDA 分支。注意 L41 有一行注释：local transformer 模块用了 StaticCache，与 FlashAttention 实现冲突，要用 FA 必须换 DynamicCache——这是 MOSS 特有的、跨注意力实现与 KV cache 结构的耦合点，迁移昇腾时要一并处理。

    device = "cuda" if torch.cuda.is_available() else "cpu"
    dtype = torch.bfloat16 if device == "cuda" else torch.float32

    def resolve_attn_implementation() -> str:
        # Prefer FlashAttention 2 when package + device conditions are met.
        if (
            device == "cuda"
            and importlib.util.find_spec("flash_attn") is not None
            and dtype in {torch.float16, torch.bfloat16}
        ):
            major, _ = torch.cuda.get_device_capability()
            if major >= 8:
                return "flash_attention_2"

        # CUDA fallback: use PyTorch SDPA kernels.
        if device == "cuda":
            return "sdpa"

        # CPU fallback.
        return "eager"


    attn_implementation = resolve_attn_implementation()
    print(f"[INFO] Using attn_implementation={attn_implementation}")
    
    # Because the local transformer module uses StaticCache, which causes errors when using the FlashAttention implementation. If FlashAttention is required, replace StaticCache with DynamicCache.
    model = MossTTSRealtime.from_pretrained(model_path, attn_implementation=attn_implementation, torch_dtype=torch.bfloat16).to(device)
NPU 迁移 · 四变体共享 CUDA 注意力分支resolve_attn_implementation() 在 local README 与 realtime infer.py 里逐字重复：判 torch.cuda.is_available() + flash_attn 包 + get_device_capability major>=8 才选 FA2，否则 CUDA 上 sdpa、CPU 上 eager。在 NPU 上 torch.cuda.is_available() 为 False，会直接落到 eager——白丢 sdpa 的高效路径。迁昇腾需把这套逻辑替换成判 torch_npu 可用性、显式选 sdpa（由 torch_npu 路由到 aclnn 融合注意力）。另：realtime 的 StaticCache 与 FA 冲突那条注释，若改 sdpa 也需核验 cache 类型是否仍兼容。均属静态推断，待真机验证。

moss_audio_tokenizer：git submodule 边界

moss_audio_tokenizer/ 是五个顶层目录里最特殊的一个：它不是本仓写的源码，而是一个 git submodule。本 checkout（固定 SHA moss-tts-ad99ec5f）里该目录是空目录——submodule 未被 populate。要读它的真实源码得单独 clone https://github.com/OpenMOSS/MOSS-Audio-Tokenizer，本页不跨边界嵌入其内容，只标清边界：它是受限外部资产，四个 TTS 变体都“外部加载”它，而非把 codec 实现打进 moss-tts 本包。

.gitmodules 明确声明：[submodule "moss_audio_tokenizer"]，path = moss_audio_tokenizer，url = https://github.com/OpenMOSS/MOSS-Audio-Tokenizer。这意味着 codec（即各变体 README 反复提到的 Cat Causal Audio Tokenizer，以及 v1.5 用的 MOSS-Audio-Tokenizer-v2）的实现、权重、训练脚本都不在本仓的 SHA 范围内——引用时不能写 moss_audio_tokenizer/... 的行号，因为目录是空的。这与 Qwen3-TTS 把 codec 留在外部 HuggingFace checkpoint（qwen3tts.processor.audio_tokenizer）的做法精神一致：TTS 系统普遍把声码器/codec 当独立组件，TTS 模型仓只管 token 建模。

NPU 迁移 · codec 是独立外部部署单元audio tokenizer（Cat / v2）与 TTS 主干是两个模型、可分属两卡。在昇腾上这意味着 codec 的算子兼容性要单独验证：codec 内部若含卷积/反卷积/转置卷积（声码器常见）或 ISTFT，CANN 的复数路径与反卷积实现需逐算子核验。主干换 sdpa 后 codec 不受影响——它走自己的前向。保守方向：codec 优先用 ONNX Runtime 的 NPU EP（若提供）或钉 CPU，主干上 NPU；不要把 codec 的声码器算子混进主干 sdpa 迁移。属静态推断，待真机验证。

llama_cpp：torch-free 推理边界

moss_tts_delay/llama_cpp/ 是整仓唯一与 torch 解耦的推理路径，也是唯一注册成 console script 的入口。它的 README 第一句就定性：torch-free（或 torch-optional）的端到端 TTS 推理——用 llama.cpp 跑 Qwen3 主干（GGUF 格式）、NumPy 跑 embedding/LM 头/delay 状态机/采样、ONNX Runtime 或 TensorRT 跑音频 tokenizer。PyTorch 可选地把 LM 头加速约 30×。这是 MOSS 在包边界上划出的一条部署态防火带：把主干从 transformers+torch 抽离成 C++ 推理引擎。

# MOSS-TTS-Delay: llama.cpp Inference Backend

[English](README.md) | [简体中文](README_zh.md)

This package provides a **torch-free** (or torch-optional) end-to-end TTS inference pipeline for MOSS-TTS-Delay using:

- **llama.cpp** for the Qwen3 backbone (GGUF format, GPU/CPU)
- **NumPy** for embeddings, LM heads, delay state machine, and sampling
- **ONNX Runtime** or **TensorRT** for the audio tokenizer

When PyTorch is available, LM heads can optionally be GPU-accelerated (~30x faster).

pyproject.toml 给这条路配了三组互斥可选依赖：llama-cpp-onnx（torch-free + ONNX audio）、llama-cpp-trt（+ TensorRT audio，最高性能）、llama-cpp-torch（可选 torch 加速 LM 头）。依赖清单里没有 torch、没有 transformers——只有 numpy/pyyaml/tokenizers/soundfile 加上 onnxruntime-gpu 或 tensorrt/cuda-python。也就是说走这条路，整个 Qwen3 主干经 llama.cpp 的 C++ 实现，绕过了 torch_npu 的注意力算子问题。

这条边界对昇腾迁移有两面性。好处：主干不再是 transformers 的 Qwen3Model，sdpa/FA2 的硬绑问题自然消失——llama.cpp 自己管 KV cache 与注意力内核，可针对 NPU 编译。代价：llama.cpp 的 NPU 后端成熟度不如 CUDA/Metal，且 audio tokenizer 仍依赖 ONNX Runtime 或 TensorRT，这两条在昇腾上都要找对应 EP/插件。所以 torch-free 不是“免迁移”，而是把迁移焦点从 torch_npu 注意力挪到了 llama.cpp 编译后端 + ONNX/TRT EP 选型。

NPU 迁移 · llama.cpp NPU 后端待验证moss_tts_delay/llama_cpp 把主干交给 llama.cpp，理论上绕开了 torch_npu 注意力。但 llama.cpp 对昇腾 NPU 的 CANN 后端支持度需独立核验（公开的 ggml NPU 后端不如 CUDA/Metal 成熟），且 audio tokenizer 仍要 ONNX Runtime NPU EP 或 TRT 等价物——若 codec 用了 CANN 不稳的反卷积/复数 STFT，这条仍卡。保守方向：昇腾上若走此路，先验证 llama.cpp 能否在 NPU 上跑通 Qwen3 主干，否则退回 transformers + torch_npu + sdpa 走 delay 变体的 torch 路径。属静态推断，待真机验证。

与 Qwen3-TTS 包结构对照

把 MOSS 包边界与金标准 Qwen3-TTS 放一起看，TTS 系统的“主干 + codec + speaker”三件边界在两边都清晰，但切法不同。Qwen3-TTS 是单一命名空间 qwen3tts，一个 model 对象下挂 talker、code_predictor、speaker_encoder、codec_embedding，codec 留在外部 checkpoint 由 processor 持有；MOSS 是并列多变体，每个变体自成子包，codec 全部走外部 submodule，speaker 不显式建表（靠 reference 音频过 speaker_encoder，realtime 在推理时 inline 算）。

维度Qwen3-TTS（金标准）MOSS-TTS（本页）
包形态单一命名空间 qwen3ttspy-modules=空 + 四并列子包
变体数14（local/delay/v1.5/realtime）
主干Qwen3 Talker + code_predictorQwen3-1.7B/4B 或 Qwen-8B，按变体
RVQ 层数16 层残差12/16/32，按变体
残差层建模损失侧 MTP（sub-talker x0.3）结构侧（local 深度模块）/ 数据侧（delay-pattern）
codec 归属外部 HF checkpoint，processor 持有外部 git submodule，独立 AutoModel 加载
speaker注入 codec_embedding 表第 3000 行reference 音频过 speaker_encoder，推理时 inline
torch-free 路径无有（moss_tts_delay/llama_cpp）
Ascend 训练定位金标准 SFT 循环语音/codec 卫星，非已验证训练实现

最重要的一条定位差异：Qwen3-TTS 在本站是Ascend 迁移的金标准训练目标（有逐段 SFT 循环、speaker 注入、safetensors 导出），MOSS-TTS 是语音/codec 卫星——读它是为了看清 TTS 特有的模块边界与 codec/speaker 切法，而不是把它当昇腾上已验证的训练实现。所以本页的 npu_risk 都落在部署态/推理态（注意力分支、codec 独立部署、llama.cpp 后端），不涉及 MOSS 的训练循环迁移。MOSS 各子包确有 finetuning/ 子目录，但那是 MOSS 自己的训练脚本，不在本页金标准范围内。

本页所有源码引用固定在 OpenMOSS-Team/MOSS-TTS@ad99ec5f（仓库根、各子包 README、__init__.py、infer.py、pyproject.toml、.gitmodules）。moss_audio_tokenizer/ 因 submodule 未 populate 为空目录，不跨仓引用其内容。本页“昇腾方向”是据源码与 torch_npu/llama.cpp 公开语义的静态推断，不是 910B 真机结果——MOSS 不被表述为 Ascend 训练已验证实现，部署态风险均待真机验证。
