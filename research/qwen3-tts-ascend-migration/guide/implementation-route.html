学习轨道：实施路线

实施路线总览

把未来 910B 硬件验证组织成六个阶段：环境、单卡、单机多卡、多机、性能、验收。除固定源码引用外，每一步都是 pending_hardware 的实验计划——只定义观测量与通过条件，不写成硬件运行声明。

学习目标

  • 按阶段记录验证输入、观察项与通过条件
  • 避免把计划表写成硬件运行声明
  • 把每个阶段的源码锚点与硬件验证分开

前置知识

  • 已完成迁移边界课并能区分静态证据与运行结论
  • 读过 SFT 循环与硬件映射页

六阶段实施路线

以下六阶段是验证顺序，不是已完成结果。固定源码只提供待检查的入口和配置锚点；任何环境、设备、吞吐、数值或验收结论均待真机验证。

每个阶段的“通过条件”是执行前定义的数值门限/日志检查项，不是本站声称已通过。所有阶段整体标 pending_hardware。

1. 环境

固定环境清单：NPU 驱动、CANN 8.5.2、PyTorch/torch-npu 2.7.1、依赖解析与最小 import torch_npu。通过条件：在固定环境清单中保存可复查日志。

2. 单卡

受控样本上检查装载、前向、loss、dtype 与错误日志。通过条件：预设断言和数值门限均有记录。

3. 单机多卡

检查 launcher、rank、collective、数据分片与 loss 一致性。通过条件：每个 rank 的预设检查均通过。DDP 后端必须从 nccl 换 hccl（accelerator.prepare 注入），参考 MindSpeed-MM 的并行组初始化。

4. 多机

检查节点发现、通信、恢复和失败日志。通过条件：受控作业的跨节点一致性与恢复检查有证据。参考 MindSpeed-LLM 的 high_availability 与多机扩展路径。

5. 性能

在固定工作负载、版本和测量窗口下记录吞吐、显存、profiling 与数值稳定性。没有测量前不比较性能——本站不声称任何 NPU 性能数字。

6. 验收

汇总环境、单卡、单机多卡、多机和性能的日志，检查保存与重载，并保留失败条件。通过条件由执行前的验收表定义。
