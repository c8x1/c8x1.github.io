学习轨道：源码深读

迁移：分布式

迁移映射总线页——把 Qwen3-TTS 单进程 accelerate 的 accelerator.prepare（DDP/HCCL 后端）映射到 MindSpeed-MM/LLM 的 Megatron mpu 并行组、HCCL 注入、CP ring/ulysses、多机扩展。对照单进程的‘零并行维度’与 MM/LLM 的 TP/PP/CP/EP/SP/ZeRO 六维工程，讲清规模化靠的是并行工程而非换模型，每处对照标出昇腾风险。

学习目标

  • 定位 Qwen3-TTS accelerator.prepare 这一行（sft_12hz.py:62-64）并说出它把‘循环 ownership’一次性交给了 accelerate，对应 MM 的 setup_model_and_optimizer 与 LLM 的 initialize_model_parallel_decorator 哪两段
  • 说清 TP/PP/CP/EP/SP/ZeRO 六个并行维度各自切的对象与主要通信，以及 Qwen3-TTS 单进程在每一维上的现状（基本是‘无’）
  • 区分两条 HCCL 落地路径：Qwen3-TTS accelerate 路线要手动改 DDP backend，MindSpeed-LLM 靠 get_nccl_options_wrapper 偷换 NCCL→HCCL
  • 指出 CP ring attention 用 npu_fusion_attention 替掉 Qwen3-TTS FA2 硬绑的对照点，以及多机扩展优先扩 DP 而非 TP 的带宽依据

前置知识

  • 读过 target/sft-training-loop.html，知道 Qwen3-TTS 是 accelerate 单进程、accelerator.prepare 接管 model/opt/loader、DDP 后端需改 hccl
  • 读过 reference/mm-training.html，了解 Megatron 的 mpu 与 forward_backward 调度、config 注入式并行策略
  • 读过 reference/llm-parallel.html 与 reference/llm-scale.html，了解 TP/PP/CP/EP/SP/ZeRO 六维与多机容错
  • 知道 torch.distributed.new_group / allreduce / send-recv 等集合通信原语与 NCCL/HCCL 后端概念

总览：单进程 accelerate → 六维并行工程

这是迁移映射的总线页：它不逐行重读某个文件，而是把 Qwen3-TTS 的 sft_12hz.py 单进程 accelerate 循环，逐维度对位到 MindSpeed-MM / MindSpeed-LLM 的分布式并行工程上。Qwen3-TTS 的训练循环由一个 Accelerator(grad_accum=4, bf16) 包住，accelerator.prepare(model, optimizer, train_dataloader) 一行就把设备路由、DDP 包裹、梯度累积、反向全部托管——零显式并行维度，连 torch.distributed.init_process_group 都由 accelerate 隐式调。要把它长成多机多卡的规模化训练，拼的不是模型本身，而是 MM/LLM 在 Megatron 的 mpu 之上补齐的六个并行维度：TP 层内切、PP 层间切、CP 序列切、EP 专家切、SP 序列并行、DP/ZeRO 数据与状态切。

这张映射图的核心是‘ownership 的拆解’。accelerate 把‘循环 ownership’攥在一个对象里——prepare 接管 model/opt/loader，backward 接管反向，accumulate 管累积，unwrap_model 管取回权重。MM/LLM 不这么攥：它把这套 ownership拆成并行组与策略回调——initialize_model_parallel 建 mpu 的 TP/CP/EP/DP/PP 组（LLM 加 decorator 补 EP 三组与 CP 三子组），setup_model_and_optimizer 建三件套并套 DDP/PP wrapper，config 上的 grad_scale_func/no_sync_func/finalize_model_grads_func 等函数指针把并行策略注入引擎。一句话：accelerate 的‘一个对象管一切’在 MM/LLM 里是‘一组通信组 + 一组回调指针’。

accelerator.prepare → Megatron mpu：ownership 的拆解

先看迁移的起点：Qwen3-TTS 把 ownership 一次性交出去的那一行。sft_12hz.py 的 setup 段先逐个构造 model/config/dataset/loader/optimizer，最后 L62-64 accelerator.prepare(qwen3tts.model, optimizer, train_dataloader) 三件套一起包裹，返回新的 (model, optimizer, train_dataloader)。后续循环只用返回值，不再碰原始对象——这是 accelerate 的 ownership 约定：prepare 之后，设备的路由、DDP 的包裹、loader 的分布式切片都由 accelerator 接管。对照 MM/LLM，这一行对应两个动作的合体：initialize_model_parallel 建通信组 + setup_model_and_optimizer 建三件套。

L44 Accelerator(grad_accum=4, bf16, tensorboard) 把梯度累积、混合精度、日志三件事写死在构造里——这对应 MM 的 config.grad_scale_func=optimizer.scale_loss（BF16 时 no-op）与 update_num_microbatches，但 accelerate 是声明式（构造即定策略），MM 是注入式（往 config 挂函数指针）。L48-52 from_pretrained(FA2, bf16) 发生在 prepare 之前，所以 FA2 是原始模型语义，不是 accelerate 包裹后的——这也是后面 FA2 硬绑风险的源头。L62-64 的 prepare 才是‘循环 ownership 转移’的分水岭。

对应到 MM 侧：pretrain()（training.py:86）先 initialize_megatron 建进程组与 mpu（这一步等价于把 accelerate 隐式做的 init_process_group 显式化、并补齐 TP/PP/CP/DP 组），再 setup_model_and_optimizer 建三件套（等价于 prepare 的 model/opt 部分），再 build_train_valid_test_data_iterators 建数据迭代器（等价于 prepare 的 loader 部分）。注意顺序与 accelerate 相反：accelerate 是先构造好 model/opt/loader 再一起 prepare，MM 是先建组、再建模型、最后建数据——因为模型 wrapper 要按组（TP/PP 切法）来套，组没建好模型没法包。

MM 在 train() 里把并行策略挂成 config 的函数指针，而不是写死在构造里。L405 config.grad_scale_func = optimizer.scale_loss 是混合精度钩子（BF16 时 no-op），L407-422 no_sync_func/grad_sync_func 控制 allreduce 何时发（overlap_grad_reduce 时延迟同步），L423-431 param_sync_func 控制 overlap_param_gather 时延迟参数聚合，L432 finalize_model_grads_func 做跨 PP stage 梯度收尾。这一组 config 注入是‘用函数指针把策略注入引擎’的设计，与 accelerate 把策略写死在 Accelerator 构造里完全不同——迁移时要理解：accelerate 的 grad_accum 在这边对应的是 config 这组回调 + update_num_microbatches。

NPU 迁移 · prepare 的设备路由accelerator.prepare 默认按 CUDA 语义取 device、在多卡时注入 DDP。迁移到 NPU 有两条路：路线 A 留在 accelerate，accelerate config 选 torch_npu 后端、device 设 npu:0、DDP backend 手动改 hccl——改动小但要自己保证版本匹配；路线 B 直接迁到 MindSpeed 训练引擎，prepare 这一行整个删掉，ownership 交给 Megatron 的 mpu 与 setup_model_and_optimizer，HCCL 注入天然就位（见下节 get_nccl_options_wrapper）。路线 B 工程量大但并行维度全开，是规模化的正经路径；路线 A 适合先单卡/单机多卡验证语义。两条路线都待真机验证 device 路由与 prepare 的 ownership 行为。
ownership 拆解的实质：accelerate 的 Accelerator(grad_accum, bf16) = MM 的 update_num_microbatches（N=4）+ BF16 optimizer（scale_loss no-op）；accelerator.prepare = initialize_model_parallel + setup_model_and_optimizer + build_train_valid_test_data_iterators；accelerator.backward = forward_backward_func 的反向段；accelerator.accumulate = num_microbatches 梯度 buffer 累加。语义不完全对等（累积单位见 mm-training 页的陷阱说明），迁移时按样本量对齐 N。

rank 网格：从‘无 rank’到 order 布局

Qwen3-TTS 单进程没有 rank 网格——它不显式调 torch.distributed.init_process_group，world_size 概念由 accelerate 隐式管（单进程时为 1）。迁到规模化的第一件事是承认‘现在要管 rank 网格了’：一个全局 rank 同时属于一个 TP 组、一个 CP 组、一个 EP 组、一个 DP 组、一个 PP 组，由 order='tp-cp-ep-dp-pp' 决定 rank 在多维网格里的排布顺序（最内维 TP，最外维 PP）。MindSpeed-LLM 的 initialize_model_parallel_decorator（L63 wrapper 签名）一口气列出全部维度尺寸，这个 order 是字符串约定、不是数值，Megatron 内部按它做 stride 计算。

wrapper 的‘先调原函数再补建’顺序是关键（L92-112 先让 Megatron 建 TP/PP/CP/DP 基础组，再补建 EP/Tensor-EP/DP-modulo-EP）。因为 EP 组的 rank 池要从 DP×CP 组里再切，必须等 DP/CP 组就位。L114-120 是布局循环的算术基础：data_parallel_size = world_size // (tp * pp * cp)——DP 是除掉 TP/PP/CP 之后剩下的，所以 world_size = tp * pp * cp * dp。EP 不占新 rank 维度，它复用 DP×CP 的 rank 再切。L122 的硬约束 data_parallel_size * context_parallel_size % expert_model_parallel_size == 0 保证 EP 能在 DP×CP 池里均匀切分，否则直接抛错。

NPU 迁移 · rank 布局与多机拓扑order='tp-cp-ep-dp-pp' 决定 TP/CP/EP/DP/PP 各自落在哪些全局 rank 上。在 910B 多机场景，TP/CP/EP 这类高频通信维度应尽量落同一节点（8 卡 HCCS 互联带宽高），PP/DP 落跨节点。MindSpeed-LLM 不强制改 order，但若拓扑与默认 order 错配，TP 落跨节点会因 HCCS 带宽不足严重拖慢 allreduce。迁移建议：先按默认 order 在单机 8 卡验证通信通断，再上多机观察 TP allreduce 带宽；不够再考虑调 order 或 2D TP（tp_x/tp_y，L263-280）。Qwen3-TTS 单进程无此问题，迁多机后才碰到。待真机验证拓扑带宽。

HCCL 注入：两条后端落地路径

整条并行通信能落到 HCCL，是迁移的第一硬前置。Qwen3-TTS 的 accelerate 路线里，accelerator.prepare 多卡时会注入 DDP wrapper，后端默认 NCCL——要手动改 backend='hccl' 或靠 torch_npu 的 monkey-patch。MindSpeed-LLM 走的是工程化、按组精细化的另一条路：get_nccl_options_wrapper（L373）包住 Megatron 的 get_nccl_options，每次 Megatron 建组时它检查 ProcessGroupHCCL.Options 有无 hccl_config 属性——有就构造 HCCL Options 并设 hccl_config = {'group_name': pg_name} 返回，把 Megatron 本要建的 NCCL 组直接偷换成 HCCL 组；没有才 fallback 回原 NCCL options。

mindspeed_llm/core/parallel_state.py · L373–L384
固定源码 ↗
def get_nccl_options_wrapper(get_nccl_options):
    @wraps(get_nccl_options)
    def wrapper(pg_name, nccl_comm_cfgs):
        if hasattr(torch_npu._C._distributed_c10d.ProcessGroupHCCL.Options, "hccl_config"):
            try:
                options = torch_npu._C._distributed_c10d.ProcessGroupHCCL.Options()
                options.hccl_config = {"group_name": str(pg_name)}
                return options
            except Exception:
                return get_nccl_options(pg_name, nccl_comm_cfgs)
        return get_nccl_options(pg_name, nccl_comm_cfgs)
    return wrapper

group_name 用 pg_name（如 'exp'/'tp_exp'/'dp_modulo_exp'）——CANN 侧据此区分不同通信域的聚合策略。try/except 包裹兼容老版 torch_npu（无 hccl_config 属性时不崩）。这个 wrapper 与 initialize_model_parallel_decorator 一样，经 patch_manager 注册到 megatron.core.parallel_state.get_nccl_options 生效。对照 Qwen3-TTS：它 accelerate 的 DDP 默认 NCCL，迁移要手动 init_process_group(backend='hccl')；LLM 这套是全量、按组名注入的 HCCL，每个通信组都带 group_name，比 accelerate 的粗粒度 backend 设置更可控。

NPU 迁移 · NCCL→HCCL 后端两条路径本质都是把 NCCL 换 HCCL，但工程量与可控度不同。路径 A（accelerate）需手动设 backend='hccl' 且 DDP 包裹时 backend 匹配，粗粒度；路径 B（MindSpeed-LLM）不需手动设，wrapper 天然注入，每个组带 group_name 可按 CANN 侧精细化调聚合。风险共性：HCCL 版本与 torch_npu/CANN 版本要三件套匹配，版本错配会建组失败或通信 hang。保守：不论哪条路径，先验单节点 TP=2 的 allreduce 通断（最小可验证单元），再上多节点。Qwen3-TTS 单卡迁移不受影响，单机多卡/多机段才会真正碰到 HCCL 拓扑。待真机验证版本矩阵。

CP ring attention：FA2 硬绑的 NPU 替代

这是迁移最直接受益的一处对照。Qwen3-TTS 装载时写死 attn_implementation="flash_attention_2"（CUDA 专属），talker 的整个 codec 序列要塞进单卡 attention、显存吃紧。MindSpeed-LLM 的 CP（上下文并行）把序列沿长度切到 CP 个 rank，每个 rank 只算自己那段——它的 ring attention 算子用的是昇腾原生 torch_npu.npu_fusion_attention，不是 FlashAttention-2。所以 CP ring 与 npu_fusion_attention 是绑定的，这对 Qwen3-TTS 的 FA2 硬绑是直接对照：单卡时直接 sdpa（路由到 npu_fusion_attention），长序列上 CP 时用这套 ring attention。

多机扩展：从‘挂了重训’到 HA 续训

Qwen3-TTS 的 accelerate 路径没有节点级容错——单进程或单机多卡挂了就重训，靠 checkpoint 续。MindSpeed-LLM 在这条路径上叠了一整层 HA（高可用）工程：节点发现（controller/processor 两角色）、进程组管理（mpu 全套 + replica group）、容错重建（ARF 全量重建、elastic 局部 scale-in/scale-out）、跨 rank 数据修复（send/recv 传 optimizer state）。这套代码在 mindspeed_llm/core/high_availability/ 与 tasks/high_availability/，靠给 Megatron 原生通信函数打 wrapper 注入，不侵入主干训练循环。Qwen3-TTS 单进程完全没有这一层，多机迁移的核心增量就在这里。

三条重建路径粒度不同。ARF（node reboot）是节点重启后把所有进程组从头建一遍，靠 reinit_process_group(rebuild_link=True) 重建底层 HCCL 链路，分钟级开销。elastic scale-in 是缩容（去掉若干 dp rank）后只重建受影响的 dp_cp replica group，并把 num_microbatches 按 total // new_dp_size 重算以守恒总 workload。scale-out 是扩容反向。三者最终都要做跨 rank 数据修复：把存活 rank 的 optimizer state（momentum、step、loss scale）序列化成 ByteTensor 搬到 NPU，用 torch.distributed.send/recv 点对点传给新 rank，让新 rank 接着练而不重训。

NPU 迁移 · HA 强依赖 mindio_ttp这套 HA 强依赖 mindio_ttp 框架与 MindX 训练平台：controller/processor 协调、故障探测、进程拉起都在 mindio_ttp 内，本仓库只提供‘被回调时怎么重建/修复’的实现。脱离 MindX/mindio_ttp 部署时，enable_high_availability/enable_elastic_training 即便设了回调也不触发，等于关掉 HA。NPU 场景下 ARF 全量重建在多机上是分钟级开销（依赖 rank 数与拓扑），重建期间训练完全暂停；scale-in 后 num_microbatches 变大意味着单 rank 显存峰值上升，可能 OOM——这是容错引发的新风险。保守：对训练时长几小时内的任务（如 Qwen3-TTS SFT）不值得上 ARF，挂了重训更划算；几天到几周的 LLM 预训练才必须 HA。待真机验证 mindio_ttp 在 910B 集群稳定性。
NPU 迁移 · 多机 RNG 一致性容错续训绕不开 RNG：训练里的 dropout、数据 shuffle、MoE router 都依赖 RNG，故障前各 rank 的 RNG 状态已前进若干步。重建后新 rank 若 RNG 重置，续训的样本顺序与 dropout 模式与故障前不一致，等效换了随机种子续训，可能引入分布漂移。本仓库修复路径传的是 optimizer + scheduler state，未见显式传 RNG state（torch.get_rng_state / set_rng_state）——Megatron 靠 consumed_samples 重 init sampler 来重建数据进度，RNG 不传 state。NPU 上 RNG（Philox 等）与 CUDA 不同，多机续训后 RNG 一致性需单独核验。保守：续训后先跑几个 step 对比 loss 曲线是否与故障前平滑衔接。待真机验证 RNG 续训一致性。
