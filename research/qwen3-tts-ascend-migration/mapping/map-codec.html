学习轨道：源码深读

迁移：Codec

把 Qwen3-TTS 的音频 codec/tokenizer（12Hz 的 Mimi 残差 RVQ、16 层、codebook 2048；25Hz 的 Whisper 编码器 + GRVQ、codebook 32768）逐段映射到 MOSS-TTS 的 codec/audio_tokenizer（外部 MOSS-Audio-Tokenizer 即 Cat，32 路 RVQ、codebook 1024+1、24000/1920 帧率链）。讲清 STFT/mel 提取在 NPU 上钉 CPU 的判断、FSQ/VQ 的 argmax/argmin 与 EMA 更新算子、Whisper encoder 的 varlen FA2 attention、以及 codec 作为独立外部资产部署的边界——每段标出昇腾 910B 的算子级风险。源码固定在 QwenLM/Qwen3-TTS@022e286b 与 OpenMOSS/MOSS-TTS@ad99ec5f。

学习目标

  • 对照三套 codec 的量化结构：Qwen3-TTS 12Hz 的 16 层残差 RVQ（语义1+声学15）、25Hz 的 GRVQ（单组单层、codebook 32768）、MOSS 的 32 路 RVQ（codebook 1024+1），说出层间依赖机制差异
  • 说清 STFT/mel 提取在三个 codec 里的位置（12Hz 在 Mimi 前端、25Hz 在 Whisper log_mel_spectrogram、MOSS 在外部 audio_tokenizer 内部）与钉 CPU 的统一判断
  • 区分 VQ encode 的 argmax/argmin 距离计算、F.embedding 查表、以及训练期 EMA/one_hot 更新路径在推理时不走的边界，标出 910B 上的保守方向
  • 定位 Whisper encoder 的 flash_attn_varlen_func 硬绑与 manual fallback 路径，以及 codec 作为外部 AutoModel 资产独立部署的装载边界

前置知识

  • 读过 12Hz tokenizer 与 25Hz tokenizer 契约页
  • 理解残差向量量化（RVQ）与分组 RVQ（GRVQ）的多码本结构
  • 熟悉 STFT 复数路径、argmin 最近邻、F.embedding 查表的基本语义

总览：三套 codec 的映射对照

Qwen3-TTS 与 MOSS-TTS 都把“连续语音 ↔ 离散 token”的离散化外包给独立的 codec/tokenizer，但三套实现在编码主干、量化器类型、码本规模、层间依赖四条轴上各走各路。Qwen3-TTS 的 12Hz tokenizer（Qwen3TTSTokenizerV2Model）encoder 复用 transformers 的 MimiModel，16 层残差 RVQ、codebook 2048、帧率 24000/1920=12.5Hz；25Hz tokenizer（Qwen3TTSTokenizerV1）encoder 用 Whisper 风格主干（conv1/conv2 + 32 层 ResidualAttentionBlock）+ GRVQ，codebook 32768。MOSS-TTS 的 codec 是仓库外部资产 MOSS-Audio-Tokenizer（即 Cat，Causal Audio Tokenizer），由 AutoModel.from_pretrained 装载，产 32 路 RVQ、codebook 1024+1。本页把这些差异收成一张映射图与一张对照表。

一个贯穿全页的事实先记住：三套 codec 在 SFT/训练侧几乎都不占 NPU 算子风险——12Hz/25Hz 的 encode 属数据预处理（在 DataLoader worker 里跑），MOSS 的 encode 也只在 processor 的 encode_audios_from_wav 里调外部模型。真正上 NPU 的是 decode（推理时把 talker 生成的码本还原成波形），那一步才需逐算子核验。所以映射的主轴不是“哪个 codec 更易迁”，而是“encode 钉 CPU、decode 上 NPU”这条分界线两边各有什么算子。

NPU 迁移 · encode/decode 分界三套 codec 的 encode 都属数据预处理（12Hz/25Hz 在 __getitem__/processor 里、MOSS 在 encode_audios_from_wav 里），对吞吐不敏感。保守做法是把 encode 整段钉在 CPU（码本索引入库当训练样本），SFT 循环里完全不动 codec——这样 SFT 阶段 codec 零 NPU 风险。只有推理阶段的 decode（talker 产出码本后还原波形）才需把 codec 整体迁 NPU，那时再逐算子核验 STFT-free 的 conv/attention/RVQ-decode 路径。这条分界线是整页迁移判断的总纲。

量化结构对照：残差 RVQ vs GRVQ vs 32 路

三套 codec 的量化结构差异是迁移语义的根。Qwen3-TTS 12Hz 的 SplitResidualVectorQuantizer 把 16 层码本拆成语义层（n_q_semantic=1，第 0 层）与声学层（n_q_acoustic=15，第 1-15 层），decode 时前 1 层与后 15 层各走一个 ResidualVectorQuantizer 再相加。这对应 talker 侧第 0 层走主 CE、1-15 层走 sub-talker MTP 的损失分工。

class SplitResidualVectorQuantizer(nn.Module):
    """Residual Vector Quantizer with separate projections for the first quantizer and the rest.

    Args:
        n_q (int): Number of residual vector quantizers used.
        n_semantic_q (int): Number of residual vector quantizers used for the semantic quantizer.
        **kwargs: Arguments to the constructor of `ResidualVectorQuantizer` that are shared between both.
    """

    def __init__(
        self,
        *,
        n_q: int = 8,
        n_q_semantic: int = 1,
        **kwargs,
    ):
        super().__init__()
        assert n_q > n_q_semantic, (
            f"Number of quantizers {n_q} must be larger "
            f"than the number of semantic quantizers {n_q_semantic}."
        )
        self.max_n_q = n_q
        self.n_q_semantic = n_q_semantic
        self.n_q_acoustic = n_q - n_q_semantic
        q_dropout = kwargs.pop("q_dropout", False)
        self.rvq_first = ResidualVectorQuantizer(
            n_q=n_q_semantic, force_projection=True, q_dropout=False, **kwargs
        )
        self.rvq_rest = ResidualVectorQuantizer(
            n_q=n_q - n_q_semantic,
            force_projection=True,
            q_dropout=q_dropout,
            **kwargs,
        )

    def decode(self, codes: torch.Tensor) -> torch.Tensor:
        """Decode the given codes to the quantized representation."""
        # codes is [B, K, T], with T frames, K nb of codebooks.
        quantized = self.rvq_first.decode(codes[:, : self.n_q_semantic])
        if codes.shape[1] > self.n_q_semantic:
            quantized += self.rvq_rest.decode(codes[:, self.n_q_semantic :])
        return quantized

逐行看 12Hz 的 SplitRVQ。L818 quantized = self.rvq_first.decode(codes[:, :self.n_q_semantic]) 解第 0 层（语义），L819-820 若码本多于语义层则 quantized += self.rvq_rest.decode(codes[:, self.n_q_semantic:]) 解后 15 层（声学）累加。两个 RVQ 各自有 input_proj/output_proj（force_projection=True，L806/810），是 Conv1d(1×1) 投影。decode 全是查表 + 线性投影 + 累加，encode 侧的 argmin/EMA 不在本文件（在 transformers MimiModel 内部）。

MOSS 侧的 codec 量化结构不在本仓库里——它是外部 MOSS-Audio-Tokenizer 的实现细节。但 MOSS 的语言模型层对 codec 产出的 32 路 RVQ 有明确契约：n_vq=32（configuration_moss_tts.py:66）、audio_vocab_size=1024（实际 Embedding 用 1024+1，多 1 位给 pad/special）、audio_pad_code=1024。32 路的层间依赖不靠第二个 transformer 或 sub-talker，而靠数据侧的 Delay-Pattern 时移——把第 k 层相对第 k-1 层后移 1 步，用因果 attention 免费获得层间粗粒度依赖。这与 Qwen3-TTS 12Hz 的“talker 学第 0 层 + sub-talker 补 1-15 层”是两条完全不同的 RVQ 建模哲学。

    @staticmethod
    def apply_delay_pattern(
        tokens: torch.Tensor, ch0_pad_id: int, pad_id: int
    ) -> torch.Tensor:
        delayed_tokens = torch.full(
            (tokens.shape[0] + tokens.shape[1] - 1, tokens.shape[1]),
            pad_id,
            dtype=torch.long,
        )
        delayed_tokens[:, 0] = torch.cat(
            [
                tokens[:, 0],
                torch.full((tokens.shape[1] - 1,), ch0_pad_id, dtype=torch.long),
            ]
        )
        for i in range(1, tokens.shape[1]):
            delayed_tokens[i : i + tokens.shape[0], i] = tokens[:, i]
        return delayed_tokens

MOSS 的 Delay-Pattern 时移（L220-232）把 (t, n_vq=32) 的音频 token 沿对角线挪成 (t+31, 32)：第 0 路原样放、末尾补 pad，第 i 路整体后移 i 步。这样第 t 步看到的是“第 t 步的第 0 路 + 第 t-1 步的第 1 路 + …”，因果 attention 一次前向就能让 32 路在同一骨架里并行出齐。对照 12Hz 的 SplitRVQ：那边层间依赖靠模型结构（双塔 + sub-talker 嵌套 generate），序列短但模型复杂；MOSS 靠数据时移 + 单骨架多头，模型简单但序列长（多 31 pad 步 × 帧数）。这个权衡与 codec 算子风险无关，但决定了 decode 时的序列长度——MOSS 的 decode 输入更长，ConvTranspose 上采样的显存压力更大。

NPU 迁移 · RVQ decode 查表累加三套 codec 的 decode 侧量化都是 F.embedding 查表 + 投影 + 逐层累加，NPU 原生支持（gather + elementwise）。12Hz 的 EuclideanCodebook.decode（modeling_…_v2.py:676-679）用 embedding_sum / cluster_usage.clamp(min=eps) 算码本表再查——这里有个 bf16 除法精度问题，但 cluster_usage 是训练期定下的常数（推理不变），可预先在 CPU 把码本表算好再搬 NPU避免运行时除法，属优化项非硬卡点。MOSS 的 decode 在外部 Cat 模型内部，迁移时需把那个外部模型整体核验，但查表 gather 本身低风险。

STFT/mel 提取：钉 CPU 的统一判断

三套 codec 的编码主干都含 STFT/mel 前端，且这一段在 NPU 上的判断完全一致：钉 CPU。判断依据有二：第一，CANN 的复数 STFT 路径（torch.stft(return_complex=True) + .abs()**2 + torch.view_as_real）在多版 torch_npu 上不稳；第二，STFT/mel 属数据预处理，对吞吐不敏感，encode 在 DataLoader worker 里跑不占 NPU。三处源码逐一对照。

Qwen3-TTS 25Hz 的 STFT 在 log_mel_spectrogram（whisper_encoder.py:62-107）：L97-98 window = torch.hann_window(N_FFT)、stft = torch.stft(audio, N_FFT=400, HOP_LENGTH=160, window=window, return_complex=True)，再 L99 magnitudes = stft[..., :-1].abs() ** 2、L101-102 filters @ magnitudes、L104-106 log10+clamp+归一。N_FFT=400、HOP=160、n_mels=128（16kHz 标定的 mel_filters，24kHz 路径同理）。这是 Whisper 标准前端，return_complex=True 是 NPU 不稳的根因。

Qwen3-TTS 12Hz 的 STFT 不在本文件——12Hz encoder 直接继承 transformers 的 MimiModel（class Qwen3TTSTokenizerV2Encoder(MimiModel)，modeling_…_v2.py:899），把 upsample/decoder_transformer/decoder 置空只保留编码半边。所以 12Hz 的 mel/STFT 前端与 RVQ argmin 都在 transformers 的 MimiModel 实现里，本仓库不可见。迁移 12Hz 时要去看 transformers 源码而非本文件——这是与 25Hz 最大的不同：25Hz 的 STFT 全在本仓库可见可改，12Hz 的藏在依赖里。

NPU 迁移 · STFT 复数路径钉 CPU三套 codec 的 STFT/mel 前端统一建议保持 CPU：与 cosyVoice2/SFT 页 mel/STFT 钉 CPU 同一判断。25Hz 的 torch.stft(return_complex=True) 在 CANN 上不稳，是 encode 上 NPU 的硬卡点；12Hz 与 MOSS 的 STFT 藏在依赖/外部模型里，核验成本更高。保守做法统一是 encode 在 CPU 数据管线完成、码本索引入库，SFT/推理的 decode 路径才整体上 NPU（decode 不含 STFT，只有 conv/attention/RVQ-decode，无复数路径风险）。mel 算完再 .to(device) 搬 NPU 不冲突。注意 25Hz 的 mel_filters 用 @lru_cache 缓存 npz（whisper_encoder.py:43-59），device 切换时缓存失效需重建。

FSQ/VQ 量化算子：argmax、argmin、EMA

VQ 量化的核心算子分两类：encode 侧的最近邻选择（argmax/argmin over codebook）与训练期的 EMA 码本更新（one_hot + scatter_add_）。三套 codec 这两类的可见度不同。25Hz 的 EuclideanCodebook.quantize（core_vq.py:186-194）把两者都写在本仓库里，是分析的最佳样本；12Hz 的 argmin 在 transformers MimiModel 内部、MOSS 的在外部 Cat 里，不可见。

qwen_tts/core/tokenizer_25hz/vq/core_vq.py · L186–L216

    def quantize(self, x):
        embed = self.embed.t()
        dist = -(
            x.pow(2).sum(1, keepdim=True)
            - 2 * x @ embed
            + embed.pow(2).sum(0, keepdim=True)
        )
        embed_ind = dist.max(dim=-1).indices
        return embed_ind

    def dequantize(self, embed_ind):
        quantize = F.embedding(embed_ind, self.embed)
        return quantize

    def encode(self, x, buffers):
        self.inited, self.cluster_size, self.embed, self.embed_avg = buffers

        shape = x.shape
        # pre-process
        x = preprocess(x)
        # quantize
        embed_ind = self.quantize(x)
        # post-process
        embed_ind = postprocess_emb(embed_ind, shape)
        return embed_ind

    def decode(self, embed_ind, buffers):
        self.inited, self.cluster_size, self.embed, self.embed_avg = buffers

        quantize = self.dequantize(embed_ind)
        return quantize

逐行看 25Hz 的最近邻。L187 embed = self.embed.t() 把码本表转置成 (dim=1280, codebook_size=32768)。L188-192 dist = -(x.pow(2).sum(1, keepdim=True) - 2*x@embed + embed.pow(2).sum(0, keepdim=True)) 是欧氏距离的展开（负号是因为要取 max 等价取 min 距离）。L193 embed_ind = dist.max(dim=-1).indices 取 argmax——这就是把每个向量映射到码本里最近的那一项。关键算子：x @ embed 是 (T, 1280) @ (1280, 32768) 的大 matmul，dist.max(dim=-1).indices 是 codebook 维度的 argmax reduce。

shape 跟踪：输入 x (T, 1280) → x.pow(2).sum(1, keepdim=True) 得 (T,1) → x @ embed 得 (T, 32768) → embed.pow(2).sum(0) 得 (32768,) → 广播相减得 dist (T, 32768) → dist.max(dim=-1).indices 得 embed_ind (T,)。dequantize（L196-198）反向用 F.embedding(embed_ind, self.embed) gather 回 (T, 1280)。12Hz 的对应逻辑在 Mimi 内部、结构同构（cdist+argmin），MOSS 的在外部 Cat 内。

训练期的 EMA 更新在 forward（core_vq.py:218-248）里：L231 embed_onehot = F.one_hot(embed_ind, self.codebook_size).type(dtype) 造大稀疏矩阵（32768 维），L236-238 ema_inplace 更新 cluster_size 与 embed_avg，L239-244 laplace_smoothing 归一再 self.embed.data.copy_(embed_normalized)。但这些训练路径在 25Hz 推理时不走——speech_vq._do_quantize 训练时直接 raise NotImplementedError（speech_vq.py:255-259），所以推理/编码只走 encode→quantize→dequantize，无 EMA/one_hot。12Hz 的 EMA 也在 Mimi 训练期、推理不走。

NPU 迁移 · argmax/argmin 与大 matmuldist.max(dim=-1).indices（25Hz）/对应 argmin（12Hz/MOSS）是 codebook 维度的 reduce，NPU 原生支持。风险点在 x @ embed 这个大 matmul：25Hz 是 (T,1280)@(1280,32768)，bf16 下 NPU 可跑但 x.pow(2).sum(1, keepdim=True) 数值范围有限（平方和易溢出），保守方向是这一步临时升 fp32 算距离再降回。推理路径无 EMA/one_hot/scatter_add_，风险集中在 matmul 与 argmax；若未来要训练 GRVQ 或重训 Mimi，F.one_hot(...,32768) 的大稀疏矩阵、ema_inplace 的原地 mul_/add_、scatter_add_ 需逐个核验 torch_npu 支撑。结论待真机验证。

Whisper encoder attention：varlen FA2 与 manual fallback

三套 codec 里只有 Qwen3-TTS 25Hz 的编码主干含一个可见的 attention 硬绑——Whisper 风格的 MultiHeadAttention（whisper_encoder.py:161）默认 use_flash_attention=True，调 flash_attn_varlen_func。12Hz 的 attention 在 transformers MimiModel 内部（sliding-attention 风格，12Hz 页详述），MOSS 的 attention 在外部 Cat 里。所以本节聚焦 25Hz 这条 FA2 硬绑，以及它和 MOSS 语言模型骨架 FA2 的对照。

Whisper encoder 的 varlen attention 把多段音频拼成一个长序列，用 cu_seqlens（累积长度，由 itertools.accumulate 算）区分各段边界。这是 varlen FlashAttention 的标准接口。硬卡点：flash_attn 包在昇腾 910B 上不可用，会落到 flash_attn_varlen_func is None 分支，自动退到 qkv_attention_manual（手写 softmax+matmul）。手写路径含 torch.matmul + F.softmax + 一个 Python for 循环逐 batch 填充 padding，NPU 上能跑但效率低。

对照 MOSS 语言模型骨架：MOSS 的 language_model = Qwen3Model（modeling_moss_tts.py:170）走 HF Qwen3 自带的 ALL_ATTENTION_FUNCTIONS[_attn_implementation] 派发，MossTTSDelayPreTrainedModel 声明 _supports_flash_attn_2 = True / _supports_sdpa = True，所以装载时传 attn_implementation="sdpa" 即可让 torch_npu 路由到 npu_fusion_attention。这是两条不同的 attention 迁移路径：Whisper encoder 要么保留 manual fallback 可跑、要么改成 npu_fusion_attention 提速（但需转换 cu_seqlens 语义）；MOSS 骨架只需一行 sdpa 配置。

NPU 迁移 · Whisper varlen FA225Hz Whisper encoder 的 flash_attn_varlen_func（whisper_encoder.py:208）是 CUDA 专属，910B 上 flash_attn 包不可用。保守方向：保留 qkv_attention_manual fallback 让语义编码器在 NPU 上跑通（manual 是 torch.matmul+F.softmax+for 循环 padding，NPU 原生可跑，只是效率低）。若要提速，把 varlen 逻辑改成 torch_npu 的 npu_fusion_attention 并去掉 for 循环，但需确认其 varlen 接口（actual_seq_lens）与这里 cu_seqlens（累积长度 [0,L1,L1+L2,...]）语义一致——二者易错位，是 manual fallback 之外最值得核验的一处。注意 L185 对非 fp16/bf16 输入强制走 manual，bf16 装载不触发。
NPU 迁移 · codec 独立部署三套 codec 在推理时都是独立部署的外部资产：Qwen3-TTS 的 speech_tokenizer 与 MOSS 的 audio_tokenizer 都通过 from_pretrained 装载。MOSS processor（processing_moss_tts.py:236-240）显式 audio_tokenizer = AutoModel.from_pretrained(audio_tokenizer_name_or_path)，默认拉 OpenMOSS-Team/MOSS-Audio-Tokenizer；本仓库 moss_audio_tokenizer/ 是空目录（submodule 未初始化）。离线 NPU 环境需提前缓存 tokenizer/audio_tokenizer 目录，否则联网拉取失败。codec 装载本身是 CPU/HF 资产，无 NPU 算子风险；但 MOSS processor 调 audio_tokenizer.batch_encode(wav_list_, num_quantizers=n_vq)（processing:829）时，若 audio_tokenizer 已搬 NPU，则其内部算子才需核验。保守做法：codec 留 CPU 数据管线。

NPU 迁移 · decode 上 NPU 的顺序推理阶段 decode（码本→波形）整体上 NPU 时，三套 codec 的核验顺序建议：先 STFT-free 路径（decode 不含 STFT，只有 RVQ-decode 查表 + conv + attention），这部分 12Hz/25Hz/MOSS 都是低-中风险原生算子；再 attention 硬绑（25Hz Whisper encoder 的 varlen FA2，退 manual 或换 npu_fusion_attention）；最后 SnakeBeta/ConvTranspose这类 elementwise/卷积，bf16 下小样本对 max-abs diff 核验即可。12Hz 与 MOSS 的 attention 藏在依赖/外部模型里，核验成本高于 25Hz——这决定了 25Hz 是三套里迁移可见度最高、最适合先验证的样本。
