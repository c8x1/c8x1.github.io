迁移：910B 验证

把 Qwen3-TTS 的 CUDA/硬件假设（FA2 硬绑、device='cuda' 语义、bf16）逐条映射到 910B/NPU 的验证门：FA2→sdpa、device→torch_npu、bf16 原生无 scaler、HCCL、CANN 算子支持，并对照 MindSpeed-MM 的两条 NPU 化路线（参数级 vs 算子补丁级）。整页是待真机验证的清单，不是运行结论。

学习目标

  • 把 Qwen3-TTS 的 FA2 硬绑、accelerate device 路由、bf16 三个 CUDA 假设逐一映射到 910B 上的对应验证门
  • 说清 Qwen3-TTS 的 attn_implementation 参数级路线与 MindSpeed-MM 的算子补丁级路线在迁移改法上的差别
  • 复述 910B 真机验证六阶段（环境/单卡/单机多卡/多机/性能/验收）每步的观测量与通过条件
  • 区分 bf16 原生无 scaler 与需要升精度兜底的算子（masked_scatter/matmul 混合精度）

前置知识

  • 读过 Qwen3-TTS SFT 训练循环的 setup 段（FA2 装载与 accelerate.prepare）
  • 读过 Qwen3-TTS 复合模型架构的 from_pretrained 注入路径
  • 读过 MindSpeed-MM NPU 开关页（torch_npu 接入与 PatchesManager）

总览：CUDA 假设 → NPU 验证门

Qwen3-TTS 的 finetuning/sft_12hz.py 假设它跑在 CUDA 上——这不是显式声明，而是藏在三处：FA2 是 CUDA 专属注意力；accelerate 默认按 CUDA 语义取 device 与同步；bf16 在 CUDA 上需要 torch.cuda.amp 的 GradScaler 配套（虽然这里没显式用）。迁到 910B 不是“换个 device 字符串”，而是要把这三条假设逐一映射到 NPU 的对应验证门。本页是这条映射的总线：每条 CUDA 假设对应一个 NPU 验证门，配源码位置与保守方向，全部待真机验证。

读这张图要注意两条轴：左侧是 Qwen3-TTS 源码里写死的硬件假设（FA2 装载、accelerate 路由、bf16 精度），右侧是 910B 上每个假设对应的验证门（sdpa 路由、torch_npu device、原生精度、HCCL、CANN 算子）。两个对照源在底层起支撑作用：torch_npu 的公开语义（device 命名空间、npu_fusion_attention）与 MindSpeed-MM 已在 910B 上实践过的算子补丁体系（参数级 vs 补丁级两条路线）。本页不重复这些源的内部细节，只取其迁移改法作对照。

FA2 硬绑 → sdpa 验证门

第一个 CUDA 假设在装载阶段就出现。sft_12hz.py:48-52 的 from_pretrained 把 attn_implementation="flash_attention_2" 写死——FlashAttention-2 是 CUDA 专属，在 NPU 上这一行会让模型装载阶段直接报算子不支持。这是整条迁移的第一个硬卡点：模型还没进循环就在装载阶段炸掉。保守方向是改这一个字符串参数为 "sdpa"（或临时用 "eager"）。

NPU 迁移 · FA2→sdpa 数值对齐sdpa 在 torch_npu 内部会路由到 aclnnFlashAttention / npu_fusion_attention，与 CUDA FlashAttention-2 在 bf16 下允许合理数值差异，不保证 bit-精确。验证条件：1) 小样本对 talker 主前向与 sub-talker 的 forward_sub_talker_finetune 各做一次 max-abs diff，设阈值（建议 1e-2 量级）；2) output_hidden_states 仍能取到隐状态；3) 两组注意力（talker 20 层 + code_predictor 5 层）都命中 sdpa。该映射待真机验证。

accelerate device 路由 → torch_npu

第二个 CUDA 假设更隐蔽：accelerate 默认按 CUDA 语义取 device 与做同步。sft_12hz.py:44 建加速器、sft_12hz.py:62-64 的 accelerator.prepare 三件套包裹，返回的 model/optimizer/loader 后续只用返回值。在 NPU 上，model.device、accelerator.unwrap_model、梯度同步全都走 accelerate 的 device 路由——若路由没切到 torch_npu，ref_mels.to(model.device) 这类调用会按 CUDA 语义取 device 而失败。

L44 Accelerator(...) 与 L62-64 accelerator.prepare(model, optimizer, train_dataloader) 之间夹着 from_pretrained 与 DataLoader 构造。注意 from_pretrained 发生在 prepare 之前，所以装载时还没被 accelerate 包裹——device 由 accelerate 在 prepare 时决定。在 NPU 上有两条保守路线：要么 accelerate config 选 torch_npu 后端并把 device 设为 npu:0，要么放弃 accelerate 直接迁到 MindSpeed 训练引擎。单卡迁移可先验证 accelerate + torch_npu 的 device 路由，多卡再换 HCCL。

MindSpeed-MM 的对照给出了第三条线索：它不修改 accelerate 配置，而是直接 monkey-patch accelerate 的 PartialState。mindspore/mindspore_adaptor.py 给 accelerate.state.PartialState._prepare_backend 与 set_device 加 wrapper，赋值 LOCAL_RANK 环境变量取 rank 并避开 set_device 报错——这是“accelerate 原生不支持 torch_npu 时强制它认 NPU”的具体实现位置。对 Qwen3-TTS 而言，若 accelerate config 仍不识别 torch_npu，这是可借鉴的兜底路径。

NPU 迁移 · accelerate device 路由accelerate 默认按 CUDA 语义取 device/sync。验证条件：1) model.device 在 prepare 后是 npu:0；2) ref_mels.to(model.device) 把 mel 搬到 NPU 成功（mel 已算完，不涉及 STFT）；3) accelerator.unwrap_model().state_dict() 在保存段能解包。若 accelerate config 不识别 torch_npu，兜底是 patch PartialState（参考 mindspore_adaptor.py:82-86）或直接迁 MindSpeed 引擎。该映射待真机验证。

bf16 原生：无需 scaler 的验证门

第三个假设是精度。sft_12hz.py:44 的 mixed_precision="bf16" 与 L50 的 torch_dtype=torch.bfloat16 都是 bf16。这条在 910B 上原生支持，是迁移里少有的“无风险”点——但它的无风险是有前提的：910B 的 bf16 矩阵乘与 CUDA 一样不需要 loss scaler。不要因为习惯性 CUDA bf16 配 GradScaler 而引入 scaler，那是 FP16 路径才需要的。

bf16 原生不等于所有算子在 bf16 下都稳。MindSpeed-MM 的实践给出对照：models_patches.py 的 ae_float32 补丁让视觉 AE/VAE 在整体 bf16 时强制保 fp32，因为 conv+残差结构在 bf16 下易数值溢出；mindspore_adaptor.py 的 masked_scatter_ 与 matmul wrapper 在低精度或混合精度时先升 fp32 再算。Qwen3-TTS 的 talker/codec embedding 都是低频小算子，speaker_encoder 是 conv1d 但推理即删，所以整体不需要 ae_float32 这类精度保命补丁——但若迁移时引入了视觉/STFT 侧的额外 AE，必须留意 bf16 conv 的崩溃点。

NPU 迁移 · bf16 无 scaler910B 原生支持 bf16，mixed_precision="bf16" 无风险，无需引入 GradScaler（与 FP16 路径不同）。验证条件：1) 训练 loss 不溢出/不 NaN；2) 若后续引入视觉 AE/conv 重的子模块，单独对 conv 类算子查 bf16 数值崩溃点，必要时按 ae_float32 思路保 fp32；3) masked_scatter/matmul 若遇混合精度报错，先统一升 fp32。该映射的“无 scaler”部分待真机确认 loss 稳定。

HCCL 与 CANN 算子：两条对照路线

多卡时第四、第五个假设浮现。accelerator.prepare 在多卡会把 model 包成 DDP，backward 触发 allreduce 梯度同步——后端必须是 HCCL 而非 NCCL。同时 talker/code_predictor 的全部注意力都依赖 CANN 是否提供对应算子。这两条在 Qwen3-TTS 源码里没有显式写（accelerate 默认 NCCL、FA2 默认 CUDA 算子），迁移时才暴露。MindSpeed-MM 的算子补丁体系恰好是这两条在 910B 上的实践范本。

对照两条 NPU 化路线。Qwen3-TTS（参数级）：注意力实现是装载参数 attn_implementation，FA2→sdpa 改一个字符串，sdpa 在 torch_npu 内部自动路由 NPU 算子，迁移者不碰算子边界；通信走 accelerate 的 DDP。MindSpeed-MM（补丁级）：注意力是 Megatron 的 DotProductAttention 抽象，NPU 化靠 PatchesManager 配置驱动的算子补丁替换 forward，迁移者要懂 use_flash_attn/use_infer_fa/use_fusion_attn_v2 一组开关；通信走 HCCL 且有 PREMUL_SUM 缺口补丁。前者简单但黑箱，后者复杂但可控。

mindspeed_mm/patchs/patch_manager.py · L98–L108
固定源码 ↗
    @staticmethod
    def apply_patches_from_config():
        cfg = get_args().mm.model
        if hasattr(cfg, "patch"):
            cfg = cfg.patch.to_dict()
            for key in cfg.keys() & PatchesManager.configs.keys():
                if not cfg.get(key):
                    continue
                for orig_func_name, new_func in PatchesManager.configs[key]:
                    PatchesManager.register_patch(orig_func_name, new_func)
        PatchesManager.apply_patches()

看补丁级的核心机制。L100-103 cfg = get_args().mm.model 取模型配置，读 cfg.patch 子字典，for key in cfg.keys() & PatchesManager.configs.keys() 做交集——配置里写了但 configs 里没有的键被静默忽略，configs 里有但配置没开的补丁不打。L106-107 对每个键的元组列表逐个 register_patch，L108 一次性 apply。这就是“注意力/通信替换不在代码里硬编码，而在配置字典里”的具体形态——与 Qwen3-TTS 把 FA2 写死在装载参数完全相反。Qwen3-TTS 迁移走参数级即可，不必引入这套补丁机制；但若要复刻 MindSpeed-MM 的可控性，这套配置驱动补丁是范本。

对照引擎层接入。training.py:172-186 是 MindSpeed-MM 的 NPU 入口：L173-175 use_deter_comp 开确定性计算，L177-178 jit_compile 开 GE 图编译，L180-181 allow_internal_format 开 CANN 自动布局。这三处开关在 Qwen3-TTS 路径不存在——torch_npu 内部已处理 sdpa 的布局，不暴露 internal_format。但 HCCL 的算子缺口是共通的：MindSpeed-MM 的 premul_sum_patch 用 SUM+事后 mul 补 HCCL 不支持的 PREMUL_SUM，Qwen3-TTS 多卡若用到 MoE/EP 的 fused 归约会碰同一缺口，标准 allreduce 则不涉及。

NPU 迁移 · DDP/HCCL 后端多卡时 prepare 注入 DDP，backward 触发 allreduce，后端必须 HCCL。验证条件：1) accelerate config 选 torch_npu 后端且通信后端设 hccl；2) 标准 allreduce（ReduceOp.SUM/AVG）能跑通；3) 若引入 MoE/EP 的 PREMUL_SUM 归约，走 SUM+事后 mul 兜底（参考 premul_sum_patch.py）。单卡迁移不受影响，仅单机多卡/多机段碰 HCCL。该映射待真机验证。
NPU 迁移 · CANN 算子支持清单talker/code_predictor 的 conv/linear/RoPE/attention 都依赖 CANN 提供对应算子。已确认的缺口：torch.stft 在 CANN 不支持（mindspore_adaptor.py:87 注释写明），但 Qwen3-TTS 的 mel/STFT 在 CPU 数据管线，不上 NPU，不触发。验证条件：1) 单卡前向无“算子不支持”报错；2) masked_scatter_/scatter_add_ 若出现，保证 index 连续（CANN8.5 要求）；3) matmul 不接受混合精度输入，显式对齐 dtype。该映射待真机验证算子覆盖度。

910B 真机验证清单：六阶段

把上面散落的验证门收成一份真机验证清单，按环境→单卡→单机多卡→多机→性能→验收六阶段推进。每阶段给观测量与通过条件，全部待真机验证——本页是计划，不写运行结论。前一阶段不通过不进下一阶段，避免在多机才发现单卡的 device 路由问题。

环境阶段确认 torch_npu 与 CANN 可用，是后续所有阶段的硬前置。import torch_npu 成功、torch.npu 命名空间可用、CANN 版本与卡型匹配。这一步失败说明环境本身没装好，不要贸然进单卡——MindSpeed-MM 的 training.py:9 也是这一行作为整个仓的 NPU 入口。

单卡阶段是验证 FA2→sdpa 与 bf16 的主战场：装载阶段改 attn_implementation="sdpa" 后模型能进前向、loss 不 NaN、数值与 FA2 基线在容差内。这里要同时验 talker 主前向与 sub-talker 的 forward_sub_talker_finetune 两条路径——嵌套结构下两组注意力必须一起验，不能只测主路径。单机多卡阶段才真正碰 HCCL：accelerate config 选 torch_npu 后端、通信设 hccl、backward 的 allreduce 同步通过、多卡 grad 与单卡对齐。

性能阶段关注 sdpa 在 torch_npu 的融合路径（是否落到 npu_fusion_attention 融合算子而非退化为非融合路径）与推理嵌套 generate 的显存峰值——外层 talker 的 KV cache 与内层 code_predictor 的 cache 同时驻留，长序列下 NPU 显存可能吃紧，保守方向先关 use_cache 验正确性再测吞吐。验收阶段回归训练目标：loss 收敛、speaker 注入位 6 的音色定制在推理时生效（codec_embedding.weight[3000] 取到训练得到的 speaker 向量）。六阶段全过才算迁移成立，任一阶段不达标回退重验。

NPU 迁移 · 六阶段验证闭环迁移成立需六阶段全过：环境→单卡→单机多卡→多机→性能→验收，前一阶段不通过不进下一阶段。验证条件逐阶段：环境 import torch_npu 成功；单卡 FA2→sdpa 前向无报错且 loss 合理；单机多卡 HCCL allreduce 通过且 grad 对齐单卡；多机跨机 HCCL 拓扑通；性能吞吐达标且显存不 OOM；验收 loss 收敛且 speaker 注入位 6 音色定制生效。关键不可跳过：单卡必须同时验 talker 主前向与 sub-talker forward。该闭环待真机逐阶段验证。
