学习轨道：源码深读

迁移：检查点

把 Qwen3-TTS 每轮 copytree+改 config+注入 speaker[3000]+safetensors 的保存范式，逐点映射到 MindSpeed-MM/LLM 的 save_checkpoint/resume/dcp/hf_to_dcp 转换与 optimizer state 持久化。本页是检查点迁移总线：左栏是 Qwen3-TTS 的训练循环保存五步（单卡、纯权重、不续训），右栏是 MM 的 model_optim_rng.pt 与 LLM 的 save_checkpoint_wrapper——多卡一体打包、为续训、含 optim/rng/scheduler，中间是 MM 版 dcp_to_hf 转换期重新注入 speaker[3000] 的桥接路径。每段标出 .to(cpu) 写盘、多卡 save 协调、resume RNG 跨 CANN 版本、权重格式转换 device 一致性的昇腾风险。

学习目标

  • 说出 Qwen3-TTS 保存五步（copytree/config/unwrap+to_cpu/删 speaker_encoder/注入 codec_embedding[3000]/save_file）各自映射到 MM/LLM 的哪个 save/resume/转换环节，以及哪些步在参考侧根本不存在
  • 区分两类 checkpoint 范式：Qwen3-TTS 纯权重不续训 vs MM/LLM model+optim+rng+scheduler 一体为续训，并解释为什么 Qwen3-TTS 不存 optimizer state、MM/LLM 必须存
  • 讲清 MM 版 Qwen3TTSConverter.dcp_to_hf 如何把 speaker[3000] 注入从训练循环搬到转换期、与原生 sft_12hz.py 注入行逐字对照
  • 标出检查点迁移在 910B 上的四个硬风险：保存 .to(cpu) 写盘、多卡 save 协调 barrier 走 HCCL、resume RNG 跨 CANN 版本漂移、权重格式转换 device/dtype 一致性

前置知识

  • 读过 target/optimizer-checkpoint-export 页（Qwen3-TTS 保存五步与 speaker[3000] 注入机制）
  • 读过 reference/mm-checkpoint 页（MM 的 model_optim_rng.pt 两层保存）
  • 读过 reference/llm-checkpoint 与 reference/mm-conversion 页（LLM save_checkpoint_wrapper、HF↔DCP 转换）

总览：保存侧对参考侧的映射

Qwen3-TTS 的检查点是一个 accelerate 单卡脚本在每轮末做的一套五步保存，产物是给推理用的定制音色权重，单文件 safetensors、纯权重、不续训。MindSpeed-MM 与 MindSpeed-LLM 是 Megatron 多卡分布式训练的检查点，产物是给续训用的完整训练状态，每个 mp_rank 一份、含 model+optimizer+rng+scheduler。这两套范式在“保存”二字下指的东西完全不同：前者存的是权重的终态，后者存的是训练过程的中间快照。本页的迁移任务是搞清：Qwen3-TTS 的哪些保存动作在 NPU 多卡训练里要被替换成什么、哪些动作要搬到转换期、哪些动作根本不该照搬。下图是保存侧（左）对参考侧 save/resume/转换（右）的映射。

三条映射线是本页骨架：① Qwen3-TTS 的 state_dict.to(cpu)（A3）映射到 MM/LLM 保存写盘前的缺失环节——参考侧的 torch.save 没有显式 .to(cpu)，迁移时要把 Qwen3-TTS 这步补回去，否则大模型在 NPU 上隐式 H2D 会撑爆 host 内存。② Qwen3-TTS 的 codec_embedding.weight[3000] = target_speaker 注入（A5）在参考侧不存在于训练循环保存里——它被搬到转换期，由 MM 版 dcp_to_hf 在合并 DCP 回 HF 时现算 speaker 向量注入（B8），是同位同语义的两种实现。③ Qwen3-TTS 的 save_file safetensors 终产物（A6）正是转换期的格式目标，MM/LLM 训练存的是 DCP/.pt，最终都要转成这一格式的 safetensors 才能推理。

Qwen3-TTS 保存步参考侧对应环节映射方向昇腾风险
copytree 复制骨架转换期 copytree(qwen3tts.py:58)搬到转换期无，纯文件 IO
改 config spk_id=3000转换期改 config(qwen3tts.py:99-105)搬到转换期无，CPU json
unwrap+state_dict.to(cpu)save_checkpoint_wrapper unwrap补 .to(cpu) 写盘中，隐式 H2D 内存
删 speaker_encoder 权重转换期删(qwen3tts.py:106-109)搬到转换期无，dict del
注入 codec_embedding[3000]转换期注入(qwen3tts.py:97)搬到转换期现算中，device 一致
save_file safetensorstorch.save/.pt + dcp_to_hf格式目标=转换输出低，CPU 写盘
（无 optimizer state）generate_state_dict 存 optim参考侧多存中，allreduce HCCL
（无 RNG state）get_rng_state 存 rng参考侧多存高，跨 CANN 版本

格式映射：safetensors ↔ model_optim_rng.pt ↔ torch_dist

先把三家的落盘格式对齐。Qwen3-TTS 走 safetensors 单文件（model.safetensors），权重键无前缀、纯权重无元数据；MM 走 Megatron legacy torch 单文件（mp_rank_XX_YYY/model_optim_rng.pt），键带 model. 前缀、model+optim+rng 一体打包；LLM 默认也是 legacy torch，但开了 use_dist_ckpt 后走 torch_dist 分片（落在 release/ 目录、可 partial load）。注意 MM 的 model_optim_rng.pt 这个名字本身就说明它把三类 state 揉一锅，而 Qwen3-TTS 的 safetensors 只揉权重——这是“终态 vs 快照”分歧在文件名上的体现。

NPU 迁移 · NPU internal format 与格式转换Qwen3-TTS 写 safetensors 走标准 torch 布局，NPU 无关。MM 的 torch.npu.config.allow_internal_format（training.py:184）会让某些张量在 NPU 上用 CANN 内部 5 维格式（ND→NDC1HWC0 类）。save 时Megatron legacy torch.save 会把 state_dict 序列化成标准布局、通常被规整掉；但若走 LLM 的 torch_dist 分片路径，某些算子的 internal format storage 可能被原样落盘，CUDA 侧 load 不回来。保守做法：走 torch_dist 前显式 torch.npu.config.allow_internal_format = False，或先用 legacy torch 格式验证 round-trip 再上分片。待真机验证。

保存映射：Qwen3-TTS 五步对参考侧 save

NPU 迁移 · .to(cpu) 写盘Qwen3-TTS L148 已 .to("cpu")，safetensors 写盘无 NPU 风险——这是要把参考侧补回来的判断。参考侧 LLM 的 legacy 路径 L354 torch.save(state_dict, checkpoint_name) 与 MM 的 save_ae_checkpoint（utils.py:275）都没有显式 .to(cpu)——torch.save 会隐式把 NPU 张量序列化到 host 再落盘，大模型下这一步可能瞬时撑满 host 内存。迁移时在调用方对 state_dict 做一次 .to('cpu')，与 Qwen3-TTS 同判断。LLM/MM 源码里这步是缺的，需补，待真机验证内存峰值。

桥接：转换期重新注入 speaker[3000]

Qwen3-TTS 在 NPU 上若走 MM 的 FSDP 训练引擎，训练保存产物是 DCP 格式（不是 safetensors），推理/部署要的是 HF safetensors——中间隔一道转换。Qwen3TTSConverter.dcp_to_hf 就是这道桥：它把 DCP 合并回 HF，并在合并期（而非训练循环里）完成 speaker 向量注入与 speaker_encoder 权重删除。这与 Qwen3-TTS 原生 sft_12hz.py 在每轮末直接注入 + 写 safetensors 是同一语义的两种实现。下图是桥接的数据流，codec_embedding.weight[3000] 是两侧共同的落点。

逐行对照原生版。L85-96 librosa.load 读 ref 音频、mel_spectrogram(n_fft=1024, num_mels=128, sampling_rate=24000, hop_size=256, win_size=1024, fmin=0, fmax=12000)——这组超参与 Qwen3-TTS 原生 dataset.py 的 mel_spectrogram 完全一致。L97 state_dict["talker.model.codec_embedding.weight"][3000] = _speaker_encoder(mels).to(weight.dtype) 就是注入落点，与原生版 state_dict['talker.model.codec_embedding.weight'][3000] = target_speaker_embedding[0].detach().to(weight.device).to(weight.dtype)（sft_12hz.py:156）同位同语义：都写到 codec_embedding 表的第 3000 行、都 cast 到 weight.dtype。区别是原生版用训练首步缓存的 speaker 向量、MM 版转换期从 DCP 重建 speaker_encoder 现算。

L99-105 改 config（tts_model_type='custom_voice'、spk_id={speaker_name:3000}、spk_is_dialect={speaker_name:False}）与原生版 L134-142 逐字一致。L106-109 删 speaker_encoder 权重与原生版 L150-153 一致。L116-117 save_file 写单文件 safetensors（非 sharded）与原生版 L157-158 一致。结论：原生版在训练循环里做的五步保存，MM 版把它们原封不动搬到转换期，唯一新增的是“从 DCP 重建 speaker_encoder 现算向量”这一步——因为 FSDP 训练时 speaker_encoder 的权重被存进了 DCP，转换期需要把它取出来跑一次前向。

NPU 迁移 · 权重格式转换 device 一致DCP 合并后的 state_dict 全程 CPU 张量，save_file 写 safetensors 无 NPU 风险。但 L97 _speaker_encoder(mels).to(weight.dtype) 只 cast dtype 不迁 device——speaker_encoder 输入 mels 在 CPU、weight 在 CPU，隐式同 device 不报错。若有人把 weight 搬到 NPU 再调 dcp_to_hf，state_dict['…weight'][3000] = cpu向量 会因跨 device 赋值报错。保守做法：转换全程保持 CPU，用 .to(weight.device) 而非 .to(weight.dtype) 更稳（CPU 下两者等价）。原生版 sft_12hz.py:156 用的是 .to(weight.device).to(weight.dtype)，比 MM 版多一层 device 迁移更稳——迁移时建议沿用原生写法。待真机验证转换侧不应触碰 NPU。
NPU 迁移 · 转换期 mel/speaker_encoder 留 CPU转换期的 mel_spectrogram 与 speaker_encoder 前向都在 CPU 跑（librosa + torch CPU）。建议保持 CPU，与 Qwen3-TTS 训练侧 STFT 钉 CPU 同一判断：mel 的 STFT 复数路径在 CANN 上不稳，且转换期对吞吐不敏感。ref 音频只有几秒、mel 只几帧，speaker_encoder 是 conv/linear 即便上 NPU 也无特殊算子，但没必要。待真机验证：若训练侧 weight 是 bf16 而 speaker_encoder 跑 fp32，cast 链 fp32→bf16 是一次性、可接受。

Resume 映射：Qwen3-TTS 不续训 vs 参考侧全 state 恢复

保存映射之外还有 resume 映射。Qwen3-TTS 根本不续训——它的产物是给推理用的定制音色权重，保存即终态，下一次训练是另一个 speaker、另一份 copytree，不存在“从上次断点接着练”。所以 Qwen3-TTS 的 safetensors 没有 optimizer state、没有 RNG state、没有 scheduler 游标、没有 consumed_train_samples 计数。参考侧恰恰相反：MM/LLM 的整个 checkpoint 机制就是为续训设计的，权重+动量+二阶矩+lr 游标+RNG state+FLOPs 计数一个都不能少。下表把这条分歧落到字段级。

optimizer state 映射：不存 vs 一体打包

把 optimizer state 这条单独拎出来，因为它是 Qwen3-TTS 与参考侧分歧最大、迁移时最容易被忽略的一环。Qwen3-TTS 的 AdamW(qwen3tts.model.parameters(), lr=args.lr, weight_decay=0.01)（L60）只优化 .model 子树，speaker_encoder 不在内、且被 .detach() 切断计算图——保存时还会整段删除。Qwen3-TTS 不存 optimizer state，因为它的产物是推理权重、不续训。参考侧 MM/LLM 必须存 optimizer 动量/二阶矩，否则续训时动量从零起步、loss 跳变。

AE 层映射：torch.save 一锅对照

MM 还有一条 AE（autoencoder）专用保存路径 save_ae_checkpoint，与 model_optim_rng.pt 完全无关——它是一次 torch.save，把 epoch/step/optimizer/state_dict/ema/scaler/sampler 全塞进一个 dict 写成 checkpoint.ckpt。这条路径的形态其实更接近 Qwen3-TTS（都是一锅 dict + torch.save），区别是 AE 一锅里含 optimizer/scaler/sampler（为续训），Qwen3-TTS 一锅里只有权重（为推理）。理解这条路径有助于把 Qwen3-TTS 的保存模式“对位”到参考侧最像它的那一支。

def save_ae_checkpoint(
    epoch,
    current_step,
    optimizer_state,
    state_dict,
    scaler_state,
    sampler_state,
    checkpoint_dir,
    filename="checkpoint.ckpt",
    ema_state_dict=None,
):
    if not os.path.exists(checkpoint_dir):
        os.makedirs(checkpoint_dir)
    filepath = os.path.join(checkpoint_dir, filename)
    torch.save(
        {
            "epoch": epoch,
            "current_step": current_step,
            "optimizer_state": optimizer_state,
            "state_dict": state_dict,
            "ema_state_dict": ema_state_dict,
            "scaler_state": scaler_state,
            "sampler_state": sampler_state,
        },
        filepath,
    )
    return filepath

L289-300 torch.save({epoch, current_step, optimizer_state, state_dict, ema_state_dict, scaler_state, sampler_state}, filepath) 一次性写。dict 的 7 个 key 覆盖续训所需全部状态。对比 Qwen3-TTS：Qwen3-TTS 的 state_dict 只有 model 权重键，AE 的 state_dict 是这些 key 之一。Qwen3-TTS 用 safetensors 的 save_file，AE 用 torch.save pickle——前者跨框架安全、后者依赖 pickle 协议。迁移时若想把 Qwen3-TTS 风格的“一锅保存”用于 NPU 续训，AE 这条路径是最直接的参考：补上 optimizer/scaler/sampler key 即可。

NPU 迁移 · AE torch.save 隐式 H2DQwen3-TTS L148 已 .to("cpu") 规避。AE 这条路径没有显式 .to(cpu)——torch.save 默认把 tensor 序列化到 CPU 再落盘，但若 state_dict 里的张量还挂在 NPU 上，save 前会隐式做一次 H2D 拷贝，大模型下这一步可能瞬时占满 host 内存。保守做法是在调用方把要存的 state_dict 先 .to('cpu') 再传入，与 Qwen3-TTS 保存段 state_dict.to('cpu')（sft_12hz.py:147）同判断。MM 源码里这步是留白，需迁移时补。待真机验证内存峰值。
