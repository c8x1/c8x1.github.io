学习轨道：源码深读

迁移：数据

以 Qwen3-TTS 的数据生命周期（prepare_data 离线编码 → TTSDataset.__getitem__ 抽 mel → collate_fn 拼 16 层 codec 双通道布局）为目标侧，逐段映射到 MindSpeed-MM 的 build_mm_dataset/DATA_COLLATOR 三键分派、MindSpeed-LLM 的 fsdp2 data_factory，给出每段迁移方向：dataset 类怎么接进分派表、collate 的双通道布局怎么转成 MM 的 1D token 流、audio I/O 与 mel/STFT 留 CPU、bucketing 与 broadcast_object_list 的多卡正确性、pin_memory 语义差异——并标出哪些段逻辑照搬、哪些段需改写。

学习目标

  • 把 Qwen3-TTS 的数据生命周期切成离线编码/取数/collate/装载四段，并说出每段在 MM 侧的对应落点（build_mm_dataset、DATA_COLLATOR、build_mm_dataloader）
  • 说清 TTSDataset.collate_fn 的 (b,t,2)/(b,t,16) 双通道布局与 MM 多模态 collate 的 (b,t) 1D token 流 + image_grid_thw 重建的根本差异，以及迁移时的取舍
  • 判断 audio I/O（librosa.load）与 mel/STFT 在 910B 上的迁移方向（留 CPU），并对照 MM 的 image/video decode 留 host 的同一判断
  • 区分数据管线的四类 NPU 风险：算子级（几乎无）、host 吞吐（建桶/decode 瓶颈）、多卡同步正确性（broadcast_object_list/bucket）、边界语义（pin_memory），并说出保守方向

前置知识

  • 读过 target/sft-data-collate.html，理解 Qwen3-TTS 的 TTSDataset/collate_fn 与 16 层 codec 双通道布局
  • 读过 reference/mm-data.html，理解 MindSpeed-MM 的 build_mm_dataset/build_mm_dataloader 三键分派与 bucket sampler
  • 理解 PyTorch Dataset/DataLoader/collate_fn/sampler 四件套职责分工与分布式训练的 rank/world_size

总览：Qwen3-TTS 数据生命周期的四段映射

本页是数据侧迁移总线。目标侧是 Qwen3-TTS 的整条数据生命周期，按源码顺序分四段：离线编码（prepare_data.py 把音频过 12Hz tokenizer 压成 16 层 codec 写 JSONL）、取数（TTSDataset.__getitem__ 在 CPU 抽 24kHz mel、文本分词）、collate（collate_fn 把一个 batch 拼成 7 个对齐张量，核心是 (b,t,2) 双通道 + (b,t,16) codec 双通道布局）、装载（DataLoader + accelerator.prepare）。前两段是语义搬运（音色条件怎么来、codec 结构怎么存），后两段是工程接驳（怎么把样本送进训练循环）。迁移方向不是一对一——Qwen3-TTS 是单模态脚本（一个 TTSDataset 类包揽全部），MM 是多模态框架（dataset/collate/sampler 三键解耦），LLM 的 fsdp2 是另一套（DataCollatorForSeq2Seq + 4D attention mask）。下面逐段给出映射落点与取舍。

贯穿全图的判断：Qwen3-TTS 数据管线几乎全程在 CPU/host，NPU 只在 batch 被搬进模型前向时才介入。这与 MM 数据侧（image/video decode 在 host、collate 在 CPU 张量上 padding）是同一性质，与 LLM fsdp2（collate 在 CPU、DistributedSampler 在 host 算分片）也一致。所以数据侧的 NPU 迁移风险不在算子兼容（几乎没有 NPU 算子），而在三处：host 吞吐（mel 抽取 / image decode / bucket 建桶成为瓶颈，NPU 空等）、多卡同步正确性（bucket 的 broadcast_object_list、DistributedSampler seed 一致）、边界语义（pin_memory 在 NPU 与 CUDA 行为不同）。这与前向侧那种「FA2 硬绑」的算子级风险性质完全不同。

离线编码：prepare_data 的 device 映射

第一段是离线编码。prepare_data.py 读 input JSONL，对每 32 条音频一批调 tokenizer_12hz.encode 把波形压成离散 codec，写回 audio_codes 字段。注意 --device 默认 cuda:0——这步编码本是为 GPU 设计，enc_res.audio_codes 出来后还要 .cpu().tolist() 才能写 JSON。迁移到 NPU 时这步是一次性离线产物，不在训练热路径上，可选两条路：改 --device npu:0 让 12Hz tokenizer 在 NPU 跑（需核验其 conv/attention 算子在 CANN 支持），或直接用 CPU 跑完（编码量不大、吞吐不敏感）。无论哪条，产出的 JSONL 与硬件无关——训练时只用产物，不再碰编码器。

NPU 迁移 · 离线编码 device 可换--device cuda:0 只是装载 12Hz tokenizer 的位置。这步是一次性离线产物，不在训练热路径。建议先用 CPU 跑完离线编码，避免训练前再调 NPU 编码器——12Hz tokenizer 内部是 conv/attention 组合，CANN 支持度需逐算子核验，而离线编码对吞吐不敏感。若 CPU 太慢再评估 --device npu:0。产出的 JSONL 与硬件无关，训练时只用产物，所以这一步的 device 选择不影响后续迁移。待真机验证 CPU 编码耗时。
MM/LLM 侧无对应概念。MM 的 build_mm_dataset 不做离线编码——它直接读预处理好的数据文件（datasets.load_dataset 或自定义 JSONL）。Qwen3-TTS 的离线编码是模型特定预处理（12Hz codec 是 Qwen3-TTS 独有），迁移时这步保留在 Qwen3-TTS 自己的工具链里，不进 MM 分派表。

取数：__getitem__ 的 mel 抽取与分词映射

第二段是取数。TTSDataset.__getitem__ 处理一条 JSONL 样本，返回三个张量：text_ids（分词后文本 token）、audio_codes（16 层 codec）、ref_mel（参考音频 mel）。关键运行时调用是 extract_mels：用固定参数（24kHz / n_fft=1024 / hop=256 / 128 mel / fmax=12000）把 ref 音频变 mel，这个 mel 后面过 speaker_encoder 得 speaker 向量，是音色定制的输入。映射到 MM 侧：MM 的每个 dataset 类（如 MultiModalChatDataset、AudioDataset）各有自己的 __getitem__，干的是同样的事——读数据、过 processor、产出样本 dict——只是模态不同。

形状跟踪：text_ids 经 _tokenize_texts 得 (1, t_text)，最后 [:,:-5] 砍掉模板尾部 5 token 得 (1, t_text-5)；audio_codes = torch.tensor(audio_codes, long) 是 (t_codec, 16)；ref_mel 经 extract_mels 内部 torch.from_numpy(audio).unsqueeze(0) 升到 (1, n_samples)，过 mel_spectrogram 得 (1, 128, t_mel)，.transpose(1,2) 得 (1, t_mel, 128)，t_mel ≈ n_samples/256。注意 mel 的 t_mel 与文本/codec 的 t 不同——mel 只过 speaker_encoder 出一个 speaker 向量，不参与序列对齐。

NPU 迁移 · mel/STFT 留 CPUextract_mels（dataset.py:103-116）调 mel_spectrogram，底层是 torch.stft + 复数谱取能量。CANN 的 STFT 复数路径在历史版本里不稳（window/hann/reflect pad 组合易报算子不支持或精度偏差）。这条调用在 __getitem__ 里、属 DataLoader worker 的 CPU 数据管线，建议保持 CPU（与 cosyVoice2 STFT 钉 CPU 同一判断）。具体做法：保证 extract_mels 的输入 audio 是 CPU 张量（torch.from_numpy 已是 CPU），mel_basis/hann_window 的 .to(device) 中 device 即 CPU 张量的 device。不需上 NPU，mel 抽取对吞吐不敏感（每条样本只抽一次 ref mel）。
NPU 迁移 · librosa.load 纯 CPU_load_audio_to_np（dataset.py:43-50）用 librosa.load(sr=None, mono=True) 读音频，底层走 soundfile/audioread，是纯 CPU 库，无 NPU 路径。这是 DataLoader worker 进程里的调用，本就不该碰加速器。迁移时无需改动，保持 CPU。注意 sr=None 保留原始采样率，再由 extract_mels 的 assert sr==24000 强制 24kHz——所以数据源音频必须已是 24kHz。这条与 MM 的 image/video decode 留 host 是同一性质判断：预处理算子留 CPU/host，NPU 只接前向。
对照 MM：MultiModalChatDataset.multi_modal_get_item 调 self.img_video_processor(image_path=…) 读图、过 torchvision/processor，同样在 DataLoader worker 的 host CPU 上。两者都属「host 算力不足」类风险，但 MM 那边压力更大（图/视频 decode 远重于 mel），昇腾机 ARM host 单核弱，num_workers 要开更大。把 TTS 接进 MM 框架时，mel 抽取就是 TTS 的「host decode」等价物。

Collate：双通道布局 → MM 三键分派

第三段是collate，也是迁移最需要写新代码的一段。Qwen3-TTS 的 collate_fn 把一个 batch 拼成 7 个对齐到同一长度 t 的张量，核心是 input_ids (b,t,2) 双通道 + codec_ids (b,t,16) 16 层 codec 双通道布局。这段逻辑强绑定在 TTSDataset 类里（collate_fn 是它的方法），不是独立 collator。映射到 MM 侧：MM 的 collate 由 DATA_COLLATOR 注册表（data_collator.py:681-694）按 model_name 选 collator，prepare_*_dataloader 里统一 collate_fn = DATA_COLLATOR[data_collate_type](**collate_param, dataset_param=…) 取。要接 TTS 进 MM，除了 dataset 分派，还得在 DATA_COLLATOR 加 'qwen3_tts': TTSCollator 一项。

DATA_COLLATOR 表是「新增模型要改的第二处」（第一处是 dataset 分派）。这张表的键是 model_name，值是 collator 类。看现有项：internvl/whisper/qwen2vl/qwen3vl/qwen3omni/llm_pretrain/open_sora_plan/deepseekvl2/videoalign/glm4v_moe/bagel——注意有 qwen3omni（Qwen3-Omni，多模态含语音）和 llm_pretrain（纯 LLM）。Qwen3-TTS 的双通道 codec 布局与这些都不一样，所以要写一个新的 collator 类注册进去。这个 collator 可以直接复用 TTSDataset.collate_fn 的实现逻辑（把它从方法提成独立类），因为 collate 全程是 CPU 索引赋值，无 NPU 算子风险。

根本差异对照：Qwen3-TTS 的 collate 产出 input_ids (b,t,2) + codec_ids (b,t,16) 这种固定通道布局——因为 TTS 只有一条音频模态、16 层 codec 结构固定。而 MM 的多模态 collate（如 DataCollatorForQwen2vl 委托 MultiModalDataCollatorForSeq2Seq）产出变长拼接：input_ids (b,t) long（1D token 流）、pixel_values (N_total,d)（N_total 是 batch 内所有图像 patch 数之和，各样本 patch 数不同）、labels (b,t) 用 -100/IGNORE_INDEX 屏蔽。本质差异：TTS 的通道维是模态结构（文本+codec 双通道、16 残差层），MM 把模态压成一维 token 流，靠 image_grid_thw 等元数据张量重建模态边界。

位置不变量：speaker 注入位与双通道错位

NPU 迁移 · dp group 与 HCCL 后端process_group = mpu.get_data_parallel_group()（data/__init__.py:126）取 megatron 维护的 data parallel 组。StatefulDistributedSampler 内部用 num_replicas=process_group.size()、rank=process_group.rank() 做朴素分片 indices[self.rank::self.num_replicas]。这在 NPU 上本身无算子，但 process_group 的后端必须是 HCCL——若 megatron 初始化时 backend 留成 nccl，后续 allreduce/broadcast 会失败。Qwen3-TTS 的 accelerate DDP 同理：prepare 会把 model 包成 DDP，backward 触发 allreduce，后端必须 HCCL 而非 nccl。单卡迁移不受影响，单机多卡/多机段才会真正碰到 HCCL 拓扑与 accelerate launch 配置。
NPU 迁移 · 三键配置漂移三键解耦的副作用是配置漂移：若 dataset_type 选了 InternVL（产出 pixel_values/image_flags）但 model_name 配成 qwen2vl（期望 images/videos 由 mm_plugin 处理），collate 会因 key 不匹配报错或静默用默认值。这种问题在 NPU 迁移时更隐蔽——first error 可能不是在数据侧，而是 batch 进了模型前向才 shape 不符。保守方向：迁移任一模型时先写单卡 dry-run，print(next(iter(loader)).keys()) 确认 batch dict 的 key 集合与模型 forward 签名一致，再上多卡。Qwen3-TTS 强绑定无此漂移风险，但接进 MM 框架后要确保 dataset_type='qwen3_tts' 与 model_name='qwen3_tts' 配对。

深读：bucketing 与 broadcast_object_list 的多卡正确性

MindSpeed-MM 数据侧最复杂、最易在多卡下出正确性问题的是 BucketBatchSampler。它的设计动机：多模态样本的 token 数差异巨大（小样本几十 token，大样本几千），若随机凑批，同 batch padding 到最长会浪费大量算力。所以它按 token 数把样本分桶，桶内样本长度相近，padding 浪费小。关键约束：只有 rank0 建桶，建好的 index 列表由 dist.broadcast_object_list 广播给所有 rank——这是显式的 rank0-centralized 设计，避免每张卡都跑一遍 host 重活（算 token 数是 O(N) 的 host 操作）。

对照：MindSpeed-LLM fsdp2 的 data_factory

NPU 迁移 · pin_memory 语义Qwen3-TTS 的 DataLoader 未显式设 pin_memory（走 accelerate 默认），MM 的 prepare_base_dataloader 透传 pin_memory（默认 False，dataloader.py:116,154），LLM fsdp2 由 training_args.dataloader_pin_memory 控制。CUDA 下 pin_memory 用 cudashareable pinned memory 加速 H2D。NPU 上 torch_npu 的 pin_memory 语义不同（走 ACL 的 pinned memory 或直接忽略），未必有加速、甚至可能报错。保守方向：NPU 上先把 pin_memory=False，H2D 拷贝靠 to(device) 非阻塞，压测后再决定是否启用。这是数据→NPU 边界的易被忽略开关。待真机验证。
