Architecture Note

Control Manifest：让 TTS 生成变成可评分、可回滚的过程

如果没有结构化控制清单，multi-agent TTS 会退化成一堆 prompt。control manifest 的作用是把每次语音生成的目标、约束和评估项固定下来。

对象：产品架构 核心：可观测控制意图 适用：长内容、实时 voice agent、角色语音
最小 schema agent 分工 修复路径 评估项

最小 schema

scene_id: "chapter_03_scene_07"
speaker_profile: { role, voice_id, language, accent, identity_boundary }
prosody_plan: { emotion, intensity, pace, pitch, pauses, nonverbal_tags }
fidelity_rules: { verbatim_spans, paraphrase_allowed_spans, forbidden_claims }
model_route: { preferred, fallback, latency_budget_ms, cost_ceiling }
review: { asr_backcheck, emotion_score, speaker_similarity, policy_flags }

agent 分工

1

Intent Planner

把用户目标和业务约束写入 manifest。

2

Script Agent

产出可朗读文本和 turn-level 标注。

3

Voice Agent

选择声音、校验授权、绑定身份边界。

4

Prosody Agent

把风格目标转成标签、SSML 或模型 instructions。

5

Critic Agent

听回输出，按 manifest 逐项评分。

局部修复路径

失败现象 优先修哪里 避免什么
文本漏读或改写过度 fidelity_rules 和脚本分段。 不要只换 TTS 模型。
情绪过强或过弱 prosody_plan 的 intensity 与 reference。 不要整段重写脚本。
声音不像同一角色 speaker_profile、voice_id、相似度阈值。 不要让情绪标签覆盖身份约束。
实时延迟过高 model_route 和分句流式策略。 不要用高表现力离线模型硬撑实时场景。

评估项

忠实度

ASR 回译、关键词覆盖、verbatim span 检查。

身份一致

speaker embedding、voice card、人工抽检。

表现力

情绪分类、速度/停顿统计、LLM 感知批评。

安全

授权声音、敏感内容、合成水印、审计日志。

时延

time-to-first-audio、分句延迟、重试预算。

成本

模型单价、重生成率、缓存命中率。

Control manifest 是本文的架构建议，不是某篇论文的原始贡献；它综合了 AgentSteerTTS、PodAgent、DialogueAgents 和工业 TTS 接口趋势。
