Project Brief · 2025

PodAgent：把播客生成拆成主持人、嘉宾、编剧和声音匹配

PodAgent 的价值在于说明：长音频内容不是把摘要读出来，而是要规划角色、轮次、信息密度、语气和声音资产。

对象：自动播客生成 结构：Host-Guest-Writer + voice pool 适用：访谈、课程、企业内训、有声资讯
摘要 结构 产品说明 局限 来源

30 秒摘要

它解决什么

把输入材料转成播客时，系统不仅要抽取信息，还要决定谁来讲、怎么接话、哪里追问、哪里总结，以及每个角色该用什么声音。

为什么值得做详情页

它把 TTS 放进了内容工作流：脚本 agent 决定知识结构，角色 agent 决定对话关系，声音匹配决定听感身份。这个模式比“摘要 + 单说话人 TTS”更接近真实内容产品。

结构拆解

模块 职责 可迁移设计
Host Agent 开场、控场、追问、总结，保证节奏。 适合产品里的“导演/主持”角色，管信息密度和听众体验。
Guest Agent 承担观点、案例、专业解释或反方问题。 可映射成专家、客户、学生、旁白等不同 persona。
Writer Agent 把材料组织成可朗读脚本，而不是书面摘要。 应输出 turn-level 脚本，包含停顿、插话、过渡语。
Voice Pool 为角色匹配声音资产。 需要角色声音卡：年龄感、权威感、亲和力、语言和口音。

产品说明扩充

面向播客

重点是“节目感”：开场 hook、主持人追问、嘉宾例子、段落总结、结尾行动项。TTS 只是最后一步，真正难的是对话结构。

面向课程

可把老师、助教、学生疑问拆成多角色。语音表现要服务认知节奏：关键概念慢一点，例子轻一点，复盘更明确。

面向企业知识库

RAG 提供材料，Writer Agent 负责口语化，Host Agent 负责追问，Critic 检查是否虚构政策或误读文档。

局限与注意事项

不要让角色化掩盖事实忠实

播客越自然，越容易让模型补桥段、补笑点、补因果。生产系统应把“可改写”和“必须逐字忠实”的内容分开，并用引用回链或 ASR 回译做校验。
