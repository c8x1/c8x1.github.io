重计算与显存策略

目标：讲清四本显存账与 recompute/offload 的取舍。前置：EP10 通信与计算掩盖。

你将获得：① 看懂显存=参数+梯度+优化器状态+激活四本账，FSDP2 砍前三、recompute 砍第四 ② 能据卡数在 A2(recompute True) 与 A3(recompute False) 之间做吞吐/显存取舍，并定位 async_offload/chunkloss 两个进阶开关

你已经熟悉的情况 桥接开场

你在 PyTorch 单卡或 DDP 下训练时省显存的第一招就是 torch.utils.checkpoint（gradient checkpointing / activation recompute）：前向只留少量边界张量，反向时重跑一遍前向把激活重新算回来，用一次额外前向换掉整段激活显存。这一招的核心 trade-off 你很熟——它省的是激活这本账（随序列长 × 层数线性涨，长序列下动辄几个 G），代价是多算一遍前向，吞吐掉一截。换句话说，checkpoint 不是"白省"，而是"用计算换显存"：显存够时就该关，显存不够时才开。

在 MindSpeed-LLM 的 fsdp2 backend 里这同一招同样存在，但形态变了：它不再是你手写 use_checkpoint=True 的零散调用，而是声明式的——你在 yaml 里写 recompute: True 和 recompute_modules: [model.layers.{*}]，引擎在挂并行时按这份"清单"对模型自动包一层重计算。声明式的好处是：清单可以用通配符 {*} 表达"所有 decoder 层"，不必逐层写；关掉也只需把总开关置 False，清单留不留都无所谓。更关键的是：FSDP2 本身已经把"参数/梯度/优化器状态"这三本账砍掉了大半（分片到各卡，all_gather 时才拼回），所以 recompute 在这里专门用来砍第四本账——激活，和分片策略是正交关系，可以独立开关。这和你在 DDP 下的直觉一脉相承，只是分片把前三本账的压力接走了，recompute 只需专心对付激活。

此外，NPU 侧的显存语义也和 CUDA 不太一样：它有自己的内存池配置开关（PYTORCH_NPU_ALLOC_CONF=expandable_segments，控制缓存分配器按需扩段而非整块预留，缓解碎片化）、自己的多流复用开关（MULTI_STREAM_MEMORY_REUSE，让旁路 stream 与主流共享池内存）。比 recompute 更激进的两招也只在这套环境里跑得起来：一是"把激活异步换页到 CPU 锁页内存"的 async_offload（借助 torch 原生的 saved_tensors_hooks 在不改模型代码的前提下拦截激活），二是"把 loss 沿序列维切块算以压 logits 峰值"的 chunkloss（大 vocab 下 logits 瞬时显存是另一只显存怪兽）。本篇就把这四本账和这几把刀一次性理清。

本项目怎么做 原理

把训练显存拆成四本账：参数(weights，模型本体)、梯度(grads，与参数等量)、优化器状态(optimizer states，Adam 的 m/v 通常是参数量的 2 倍)、激活(activations，随 batch×序列长×层数线性涨)。DDP 下这四本账全部每卡一份，最先爆的就是激活（尤其长序列，激活可以远超前三账之和）；FSDP2 把前三本账按 world_size 分片，每卡只留 1/N，all_gather 时才临时拼回做计算、用完再丢。但激活那本账 FSDP2 不动——它由序列长度 × 层数决定，和参数分片无关。recompute 就专门收拾这本账：把每层中间激活丢掉，反向时重算。所以 FSDP2 和 recompute 各管一半：前者砍 P/G/O 三账，后者砍 A 一账，互不重叠。下面这张柱状图把 DDP vs FSDP2(关 recompute) vs FSDP2(开 recompute) 三种配置的四本账高度摆在一起。

柱高为相对示意，非真实比例；优化器状态在 Adam 下约为参数量的 2 倍，激活在长序列下可超过前三账之和。

有了这张图，取舍就清楚了：FSDP2 把 P/G/O 三本账压到 1/N 后，剩下唯一的显存大头就是激活 A。如果你卡数够、序列长但层数适中，关掉 recompute（A3 路线）能省掉那一遍重算前向、直接拿吞吐——因为 A 账的峰值已经装得下，没必要再花一次计算把它换掉；如果卡少、序列极长，A 账逼近上限甚至 OOM，就开 recompute（A2 路线）把 A 压下去再谈吞吐——显存是硬约束，吞吐可以之后再找补。A2 与 A3 的差异，本质就是"显存够不够"这一个变量的两个取值：卡数 ↑ → 单卡分到的参数/梯度/优化器状态 ↓ → 留给激活的余量 ↑ → 关掉 recompute 换吞吐。同理，A3 还顺手把专家并行（ep_modules）也去了——卡多到连 MoE 的专家分片都不需要，纯 FSDP 拉满。所以 A2 是"小卡 + MoE + recompute"的省显存组合，A3 是"大卡 + 纯 FSDP + 关 recompute"的吞吐组合，读这两份 yaml 的差异就读出了项目的显存哲学。

把 recompute、async_offload、chunkloss 三招排成一个"省显存阶梯"会更清楚：第一阶 recompute——激活丢掉、反向重算，代价是 NPU 算力，最便宜也最先用；第二阶 async_offload——激活留住但迁去 CPU 锁页内存并预取，代价是 PCIe 带宽和 CPU 内存，比 recompute 温和（不用重算）但需要 prefetch 掩好否则反吃吞吐；第三阶 chunkloss——专治 logits 峰值（大 vocab 下 logits 那一刻的显存可比激活还高），沿序列切块逐块算梯度，代价是多次小 kernel 启动。三者并非互斥：recompute 砍激活、chunkloss 砍 logits 峰值，砍的是不同账本可以同开；async_offload 和 recompute 都砍激活账、代价不同（算 vs 搬），通常二选一按瓶颈定。NPU 侧的 expandable_segments 不在这阶梯里——它是基础设施层的池策略，给上面三招分配显存时少一些碎片化阻力，不改任何账本总额。

async_offload：用 saved_tensors_hooks 把激活 D2H 到 pin_memory 的 CPU 段，反向 H2D 取回
class SwapTensor:
    def __init__(self, tensor, key):
        self.tensor = tensor
        self.size = tensor.size()
        self.storage_size = tensor.storage().size()
        self.tensor_cpu = torch.empty(tensor.shape, dtype=tensor.dtype, pin_memory=True, device='cpu')
        ...
    # device to host
    def launch_d2h(self, stream):
        if self.stat != "device":
            return
        ...
        with torch_npu.npu.stream(stream):
            stream.wait_event(forward_event)
            if self.is_slice_tensor:
                self.tensor_cpu.copy_(self.tensor, non_blocking=True)
            else:
                self.tensor_cpu.storage().copy_(self.tensor.storage(), non_blocking=True)
            self.stat = "host"
→ mindspeed_llm/fsdp2/features/async_offload.py:53-80（SwapTensor：CPU 暂存段 + D2H 非阻塞拷贝）
关键在于 pin_memory=True（:58）开锁页内存以支持 non_blocking 拷贝——锁页内存让 PCIe DMA 不经操作系统分页中断，否则 non_blocking 退化为阻塞，offload 就没意义了。launch_d2h 把 NPU 激活迁到 CPU 后，self.tensor.storage().resize_(0)（:89）把 NPU 段缩成 0 字节，腾出显存——这一步才是真正"省下显存"的动作，光迁过去不释放等于没省；反向时 launch_h2d 再 resize 回原 storage_size（:99）拷回。is_slice_tensor（:60，storage 大小 ≠ numel）走 storage 级 copy、否则走张量级 copy，区分切片视图与连续张量。SwapTensor 用 NPU Event（:64 h2d_event）做流间同步，确保 H2D 完成后主流才读。
异步机制：pack 时 D2H、unpack 时 H2D + 预取下一 block
class async_save_on_cpu(saved_tensors_hooks):
    def __init__(self, h2d_stream, d2h_stream, block_idx, depth, custom_check_fn=None, prefetch=True):
        def _pack_to_cpu(tensor):
            ...
            swap_tensor = SwapTensor(tensor, key)
            if block_idx < depth - 1:
                working_stream = torch_npu.npu.current_stream()
                d2h_stream.wait_stream(working_stream)
                swap_tensor.launch_d2h(d2h_stream)
            OffloadManager().put(key, swap_tensor)
            return swap_tensor
        def _unpack_from_cpu(swap_tensor) -> torch.Tensor:
            ...
            swap_tensor.launch_h2d(h2d_stream, True, working_stream)
            if prefetch:
                block_idx, tensor_idx = swap_tensor.key.split("_")
                OffloadManager().prefetch_get(int(block_idx), int(tensor_idx), h2d_stream, d2h_stream)
            return swap_tensor.tensor
        super().__init__(_pack_to_cpu, _unpack_from_cpu)
→ mindspeed_llm/fsdp2/features/async_offload.py:261-310（基于 torch.autograd.graph.saved_tensors_hooks 的 pack/unpack）
这是 torch 原生的"被 autograd 保存的张量"拦截接口：前向保存激活时改走 pack_to_cpu（迁 CPU），反向取回时走 unpack_from_cpu（迁回 NPU），从而在不改模型代码的前提下把激活换页到 CPU。它的妙处在于——你只要 with async_save_on_cpu(...): 包一段前向，框架就自动把这段前向里 autograd 决定要"留到反向"的张量拦截下来转存 CPU，反向时再迁回。比手写 offload 优雅得多，和 recompute 的"丢掉重算"是两种思路：recompute 是"省掉激活、用时重算"（费 NPU 算力），async_offload 是"留住激活、迁去 CPU"（省 NPU 显存、费 PCIe 带宽 + CPU 内存）。prefetch_get（:227）按 block/tensor 索引提前把下一层的激活 H2D 拉回，掩盖拷贝延迟——和 EP10 的 prefetch 同一思路：让数据搬运与计算重叠，否则 offload 就退化成纯串行换页，吞吐掉得比 recompute 还狠。这也是为什么 OffloadManager 是单例（:173 metaclass=SingletonMeta）——跨层共享换页表，预取才能跨层算对索引。

chunkloss：把 lm_loss 沿序列维切块，逐块 grad_and_value 累计，省 logits 峰值显存
class ChunkLoss(torch.autograd.Function):
    @staticmethod
    def forward(ctx, hidden_states, head_weight, head_bias, loss_forward, loss_kwargs_chunks, chunk_size):
        ...
        accumulated_loss = torch.tensor(0.0, device=device)
        grad_inputs = torch.empty_like(hidden_states)
        grad_weight = torch.zeros_like(head_weight)
        grad_inputs_chunks = torch.split(grad_inputs, chunk_size, dim=1)
        hidden_states_chunks = torch.split(hidden_states, chunk_size, dim=1)
        for hidden_states_chunk, grad_inputs_chunk, loss_kwargs in zip(
                hidden_states_chunks, grad_inputs_chunks, loss_kwargs_chunks):
            (chunk_grad_input, chunk_grad_weight), (per_chunk_loss, _) = torch.func.grad_and_value(
                loss_forward, argnums=(0, 1), has_aux=True
            )(hidden_states_chunk, head_weight, None, **loss_kwargs)
            accumulated_loss.add_(per_chunk_loss)
            grad_inputs_chunk.copy_(chunk_grad_input)
            grad_weight.add_(chunk_grad_weight)
        ctx.save_for_backward(grad_inputs, grad_weight)
        return accumulated_loss
→ mindspeed_llm/fsdp2/features/chunkloss.py:48-75（ChunkLoss.forward：切块 + grad_and_value 逐块累计）
省的不是激活，而是输出头 logits 的峰值显存：vocab 通常数万维（Qwen3 约 15 万），整段 logits（bs×seq×vocab×fp32）会瞬时撑爆显存，比激活还凶——它在 loss 计算那一刻才 materialize，算完就丢，但峰值顶在那。这里沿 dim=1（序列维）按 chunk_size 切，每块只 materialize 一次 logits，算完梯度立刻丢；torch.func.grad_and_value(has_aux=True)（:64）一次拿到 (chunk 梯度, chunk loss)，loss 累加（accumulated_loss.add_ 行 69）、grad 拷回预分配的 grad_inputs（行 70-71）——梯度容器在循环前一次性 torch.empty_like 预分配，循环内只 copy_ 不新建，杜绝峰值反复出现。backward（:77-95）只回放已算好的 grad，不在 loss 侧重算。这种"前向就算好梯度、反向只回放"的写法相当于把 loss 的反向也前移进了 forward，省掉一次反传图。head_bias 显式不支持（:45 抛 NotImplementedError，因为切块后 bias 的归约语义难定义）。chunk_loss_size 开关见 arguments.py:699-701，设 >0 才启用。

多流复用MULTI_STREAM_MEMORY_REUSE=2（env_config.sh:6）配合 offload 旁路 stream
小结　心智模型：显存=参数/梯度/优化器状态/激活四本账。FSDP2 把前三本账分片到 1/N（all_gather 时拼回、用完即丢），剩下激活这本账由 recompute 兜底——A2(开)用一次重算前向换激活显存，A3(关)用足卡显存换吞吐；取舍变量只有一个：显存够不够。进阶两招各有专攻：async_offload 用 saved_tensors_hooks 把激活异步换页到 pin_memory CPU 段并预取（和 recompute 是"留 vs 算"的两条路），chunkloss 把 lm loss 沿序列维切块以压 logits 峰值（大 vocab 下的另一只显存怪兽）。NPU 侧另配 expandable_segments 池（缓解碎片化、不改总额）与多流内存复用（配合 offload 旁路 stream）。读 A2 vs A3 的 yaml 差异，就读出了这套显存哲学：卡少+MoE+recompute 省显存，卡多+纯 FSDP+关 recompute 换吞吐。
常见误区：①以为 recompute"总是省"——它省的是激活显存，代价是一次额外前向的计算开销，吞吐会掉；卡数够时应关掉换吞吐（A3），不是越省越好。②以为 async_offload=CPU offload 参数/梯度——它 offload 的是激活（saved_tensors_hooks 拦的就是 autograd 保存的中间张量），参数/梯度/优化器状态仍由 FSDP2 分片管，CPU 只暂存激活副本。③把 recompute 当成 recompute_modules 的同义词——recompute 是总开关(bool)，recompute_modules 是清单(List[str]，默认 model.layers.{*})，开关关了清单就不生效；A3 两份都写了清单但开关 False，清单只是不触发的摆设。④以为 expandable_segments 省总显存——它只让分配器按需扩段、缓解碎片化，总额上限不变，不会凭空多出显存。⑤把 async_offload 和 recompute 同时当"省激活"的等价手段叠加——两者目标重叠但代价不同（一个费 PCIe+CPU 内存、一个费 NPU 算力），叠加前先想清瓶颈是显存还是带宽还是算力，否则可能两边开销都吃。⑥以为 chunkloss 省的是激活——它省的是 logits 峰值，和 recompute 砍的不是同一本账，二者可共存但动机不同。
