专家并行与 2D 组合

目标：理清 FSDP×EP 二维 DeviceMesh 与为何稠密模型 ep_size=1。前置：EP08 FSDP2 分片落地。

你将获得：① 一条从 YAML 的 ep_modules / ep_fsdp_modules / ep_dispatcher: eager 三行声明、经 model_factory 装成 EPPlanConfig、由 ParallelState 建出第二张 DeviceMesh（edp/efsdp/ep）、再由 apply_ep_modules 拿专家 DeviceMesh 对 model.layers.{*}.mlp.experts 做 all-to-all 路由 + distribute_tensor(Shard(0)) 切权重——而稠密 Qwen3-8B 因 ep_size=1 让这两个 guard 全部短路、整条 EP 路径不触发的完整落地链路 ② 弄懂"专家参数走 EP 轴、共享参数走 FSDP 轴"的二维分工、以及为何 dense 模型虽然 YAML 里也写了 ep_modules 却永远走不到这条路径——配 ep_dispatcher: eager 的 all-to-all dispatch→grouped_matmul→all-to-all combine 三段式专家前向

你已经熟悉的情况 桥接开场

EP08 你看清了稠密模型怎么分片：一张 DeviceMesh 的 fsdp 轴上，fully_shard 把每层参数切成 1/N 片，前向 all-gather 临时拼、反向 reduce_scatter——这条 axis 是"同一份参数被切成 N 份"。但 MoE（混合专家）模型有个新麻烦：专家太多了。Qwen3-MoE 一层有几十个 expert，每个 expert 是一组独立的 FFN 权重，8B 那种"每层一个 MLP"的稠密结构在 MoE 这里变成"每层 N 个 expert MLP 堆在一起"。把 N 个 expert 全摆到每张卡上，显存爆；但用 FSDP 把它们像稠密参数那样"全切成 N 份"也不太对——因为 MoE 的精髓是每个 token 只去找 top-k 个 expert 算，天然就只有一部分 expert 被激活，把不激活的 expert 整块切散反而丢了"按需路由"的好处。

专家并行（EP）的解法是换一种切法：不切参数，切"专家归属"。把 E 个 expert 沿 EP 轴整块划分到 ep_size 张卡上，每卡只持 E/ep_size 个 expert 的完整权重（不是切碎成片），前向时 token 按路由表 all-to-all 发到"该 expert 所在的那张卡"算完再 all-to-all 收回来。这和 FSDP 的"参数切片 + all-gather"是两种完全不同的并行——FSDP 切的是同一份参数，EP 切的是不同份参数（不同 expert）。于是 MoE 训练里两条 axis 同时存在：专家权重走 EP 轴（按 expert 整块划分），共享权重（attention、embed、norm、router gate）走 FSDP 轴（按参数切片）。这就是"2D 组合"——EP08 的单轴 fsdp 升维成 fsdp×ep 二维。但 Qwen3-8B 是稠密模型，根本没有 expert，于是 ep_size=1 让 EP 轴塌缩成单卡、整条 EP 代码路径短路——本篇要讲清这条短路为什么发生，以及一旦换成 MoE 这条路径会怎么走。

本项目怎么做 原理

EP 的核心模型是"两张 DeviceMesh 共享同一组物理 rank"。第一张是 EP08 讲过的共享参数 mesh，维度名 ('dp','fsdp','tp')——attention/embed/norm 这类每个 token 都要走的参数挂在这张 mesh 的 fsdp 轴上，被切成片。第二张是专家 mesh，维度名 ('edp','efsdp','ep')——专家权重挂在这张 mesh 的 ep 轴上，被整块划分。两张 mesh 形状不同、维度名不同、get_fsdp_device_mesh() 与 get_ep_device_mesh() 取的是不同子组，但它们的底层物理 rank 是同一批 NPU——每张卡既持共享参数的 1/N 片（fsdp 角色），又持某几个 expert 的完整权重（ep 角色）。这就是"2D"的真义：不是一张二维 mesh，而是对同一组卡建两张不同视角的 mesh，按参数类别分挂。

专家前向走 ep_dispatcher: eager 这条路时，是经典的三段式 all-to-all：① dispatch_preprocess 用 bincount 数每卡本地各 expert 收到多少 token、gather_along_first_dim 跨 EP 组汇总成全局分布、算出每张卡要发/收多少 token 的 split size；② alltoall_dispatch + experts_computation 先 permute 把 token 按 expert 重排、all_to_all 把 token 派发到对应 expert 所在的卡、再 permute 按本卡 local expert 重排、最后 eager_grouped_matmul 一次算完本卡所有 expert 的 FFN；③ alltoall_combine 反向 unpermute + all_to_all 把结果收回原 rank、再 unpermute 按原 token 顺序拼回 + 乘路由权重。三段式的通信原语是两次 all-to-all（dispatch + combine），不是 FSDP 的 all-gather/reduce_scatter——这是 EP 与 FSDP 在通信模式上的根本分野。下面的 SVG 把两张 mesh 与三段式画在一起。

这张图的三个关键：① "2D"是两张 mesh 不是一张二维 mesh——parallel_state.py:66 建 ('dp','fsdp','tp')，:76 另建 ('edp','efsdp','ep')，两次独立 init_device_mesh、维度名互不重叠，靠的是同一 world_size 让物理 rank 重合；② EP 的通信是 all-to-all 不是 all-gather——三段式两次 all_to_all（dispatch 行 86 + combine 行 110）把 token 在卡间派发/收回，与 FSDP 的 all-gather/reduce_scatter 是两套原语；③ dense 短路发生在 engine 的 guard（行 59/63），不在 yaml——A2 yaml 明明写了 ep_modules，但 CLI --parallel.ep_size 1 让 expert_parallel_size 不大于 1，两个 if 都假，整条 EP 路径连 expert_parallelize_modules 都不调。

源码下钻 file:line

本篇下钻六个文件，按"YAML 声明 → factory 装 EPPlanConfig → ParallelState 建专家 mesh → engine guard 短路 → expert_parallel 切权重+换 forward → dispatcher 三段式 → expert_fully_shard 旁证"的顺序。先贴 MoE 与 dense 两份 yaml 的 ep_* 段做对照，再贴 factory 装配、state 建第二张 mesh、engine 的 >1 guard，最后贴专家切权重/换 forward 与三段式 dispatcher，并以 expert_fully_shard_parallel.py 作 dense 路径看不到的 fully_shard 原语旁证（EP08 已点过）。

def alltoall_dispatch(ep_group, hidden_states, top_k_index, indices, split_sizes):
    local_indices, global_indices = indices
    input_split, output_splits = split_sizes
    hidden_states, unpermute_indices1 = permute(hidden_states, top_k_index)
    torch.accelerator.current_stream().synchronize()
    hidden_states = all_to_all(ep_group, hidden_states, output_splits.numpy(), input_split.numpy())
    hidden_states, unpermute_indices2 = permute(hidden_states, global_indices)

def alltoall_combine(ep_group, hidden_states, top_k_weights, unpermute_indices, split_sizes):

    hidden_states = unpermute(hidden_states, unpermute_indices2)
    hidden_states = all_to_all(ep_group, hidden_states, input_split.numpy(), output_splits.numpy())
    hidden_states = unpermute(hidden_states, unpermute_indices1, top_k_weights)
    return hidden_states

这是 MoE 与 dense 在建模层的分野，与 EP 在并行层的分野呼应。Qwen3MoeExperts（行 174）是 dense Qwen3ForCausalLM（qwen3.py:21，无此类）所没有的。gate_up_proj（行 180）的 shape 把 num_experts 乘在 dim0——这正是 distribute_tensor(Shard(0)) 能按 expert 划分的原因：dim0 就是 expert 维。ep_forward（行 201）是 EP 专用的前向，view(num_local_experts, ...)（行 204）只取本卡那几个 expert。Qwen3MoeSparseFusedMoeBlock.forward 行 250 的 if ep_size > 1 是建模层的 EP 开关——与 engine 行 59 的并行层开关是两道独立的闸：并行层决定"专家参数怎么切/前向函数怎么换"，建模层决定"前向里调 experts 的哪个签名"。dense Qwen3-8B 用的是 Qwen3ForCausalLM，根本没有 Qwen3MoeSparseFusedMoeBlock，所以建模层根本没有 ep_size>1 这个分支可言——又一道安全阀。gate（行 231，路由器）是共享参数（每个 token 都要走），走 fsdp 轴而非 ep 轴——二维分工在建模层就有体现：gate 与 experts 分属不同 mesh。

GPU / 已知（一维 DDP/FSDP）

并行维度单维：fsdp 轴一刀切参数
Mesh一张 DeviceMesh（dp/fsdp/tp）
切法fully_shard：参数本身切片
通信all-gather 拼 / reduce_scatter 散（NCCL）
专家稠密无 expert，ep_size 概念不适用
路由无 token 路由，所有参数全用

NPU / 本项目（二维 FSDP×EP）

并行维度二维：fsdp 轴切共享参数 + ep 轴切 expert
Mesh两张 DeviceMesh（dp/fsdp/tp 与 edp/efsdp/ep）共享物理 rank
切法共享 fully_shard 切片；expert distribute_tensor(Shard(0)) 整块划分
通信专家前向两次 all-to-all（HCCL）：dispatch + combine
专家MoE 才有；dense ep_size=1 让 EP 轴塌缩、整路径短路
路由top-k 路由 + all-to-all 把 token 派到 expert 所在卡
小结　心智模型：EP 是"对同一组卡建两张不同视角的 DeviceMesh，按参数类别分挂"——共享参数（attention/embed/norm/gate）挂 fsdp 轴被切片、专家参数挂 ep 轴被整块划分，两张 mesh 维度名不重合（('dp','fsdp','tp') vs ('edp','efsdp','ep')）但底层物理 rank 重合。专家前向是 all-to-all 三段式（preprocess→dispatch+compute→combine），通信原语是两次 all-to-all 而非 FSDP 的 all-gather/reduce_scatter——EP 切的是"不同 expert"不是"同一参数"。稠密 Qwen3-8B 无 expert，ep_size=1 让 engine 行 59/63 两个 guard 短路、专家 mesh 退化为单卡、整条 EP 代码路径连 expert_parallelize_modules 都不调——这不是限制而是语义正确，强行开 ep_size>1 会在 get_ep_modules 行 93 因匹配不到 experts 抛错。
常见误区：① 8B 稠密 → ep_size=1，无需调 EP——这不是"没开 EP"而是"不该开 EP"：稠密模型无 mlp.experts 子模块，ep_modules: model.layers.{*}.mlp.experts 匹配落空，ep_size>1 会直接 raise RuntimeError('No module named')（expert_parallel.py:93）。② 以为 dense 的 yaml 写了 ep_modules 就是配了 EP——不是，A2 yaml 行 18-22 与 MoE 逐字相同，但 CLI --parallel.ep_size 1 让 expert_parallel_size=1，engine 行 59 guard 短路；ep_modules 是模板默认值，开关在 ep_size。③ 以为"2D"是单张二维 mesh——是两张独立 mesh（行 66 与 76 两次 init_device_mesh），维度名集合不重合，get_fsdp_device_mesh 与 get_ep_device_mesh 取不同对象。④ 以为 EP 也用 all-gather 拼回参数——不，EP 用 all-to-all 拼 token（dispatcher.py:86/110），参数切完不拼回，每卡只算自己那 E/ep_size 个 expert。⑤ 以为 efsdp 与 ep 是同一回事——ep 轴按 expert 整块划分（distribute_tensor(Shard(0))），efsdp 轴对切完的专家权重再按参数切片（fully_shard），二者正交，dense 时 efsdp_size=1 也关。⑥ 以为随附 MoE 30B 示例默认开了 EP——核实：pretrain_qwen3_30b_4k_fsdp2_A3.sh:28 也传 ep_size 1，出厂态走 fsdp_size=16 单轴，二维分工是代码能力非默认行为，要真开需改 ep_size>1 且满足 ep*efsdp*edp==world_size（以源码与真实联调为准）。
