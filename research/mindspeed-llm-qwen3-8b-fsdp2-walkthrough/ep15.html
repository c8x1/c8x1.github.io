Trainer 主循环精读

目标：讲清 FSDP2 下训练一步的完整时序。前置：EP14 微调 vs 预训练差异。

你将获得：① trainer.py 主循环逐方法精读——train 入口、training_step 前向反向、no_sync 梯度累积、clip+step+zero、save，每段贴真实 file:line ② 一个反直觉点——Trainer 只调标准 loss.backward()，FSDP2 的 ReduceScatter 梯度同步是反向时由 hook 隐式触发的，无需手动 all_reduce 梯度；外加 quick_start 里那行 iteration 1/2000 | consumed tokens ... | lm loss ... | max_memory_allocated 日志的来源——TrainMonitor 的模板拼装

你已经熟悉的情况 桥接开场

你写过单卡训练循环，最熟的骨架是 for batch in loader: loss = model(batch); loss.backward(); optimizer.step(); optimizer.zero_grad()。这四行已经把"前向→反向→更新参数→清梯度"讲完了，梯度是模型自己算、自己累加到 .grad，优化器直接读 .grad 更新权重。单卡世界里没有"同步梯度"这回事——因为只有一份梯度。

到了多卡 DDP，骨架不变，但反向时多了一步：每个 rank 的梯度算完后，DDP 的 hook 会自动 all_reduce 梯度，让你看到的 .grad 已经是全量平均。你不用改训练循环代码——同步藏在 loss.backward() 里面。MindSpeed-LLM 的 FSDP2 后端把这件事又往前推了一层：不是 all_reduce 整个梯度张量，而是 ReduceScatter 把梯度切成分片只留在本 rank，省显存。但对外接口完全一样——Trainer 仍然只写 loss.backward()，分片同步是 FSDP2 在反向时由注册在参数上的 hook 隐式触发的。下面把 trainer.py 这个 40KB 文件按方法拆开读，看"标准训练循环 + 隐式分片同步 + TrainMonitor 统一日志"是怎么拼起来的。

读这个文件最大的认知冲击是：你以为分片训练会很复杂，结果发现训练循环本身朴素得像教程。三层 for 循环（epoch / update_step / micro-batch）、一个 loss.backward()、一个 optimizer.step()——和 PyTorch 官方教程几乎逐行对应。所有"分片"的复杂度都被关进了两个黑盒：一是 FSDP2 在反向时的 ReduceScatter hook（不在 trainer.py 里），二是 ckpt_manager 在 save 时的 sharded dcp 写盘（不在 trainer.py 里）。Trainer 只负责"循环骨架 + 调用顺序 + 日志聚合"，把分片细节全委托出去。这种分层让你能读懂训练循环而不必先懂 FSDP2 内部——但也埋了最大的坑：以为梯度同步在 trainer.py 里找不到就是没同步，其实它藏在 loss.backward() 这一行的 hook 里。

本项目怎么做 原理

FSDP2 训练一步的时序，表面看和单卡一样是 forward→backward→step，但中间夹了三层"隐式并行"：第一层是梯度累积——一个 update_step 内跑多个 micro-batch，前 N-1 个用 fsdp_root.no_sync() 包住，让反向只在本地累积梯度、不触发分片同步，最后一个 micro-batch 才放出同步；第二层是反向时的隐式 ReduceScatter——loss.backward() 一调，FSDP2 注册在 sharded 参数上的 backward hook 自动把本 rank 的梯度贡献 reduce-scatter 给组内各 rank，结果 .grad 是分片形式（这部分在仓库内的 fsdp_turbo/外部 hook 实现，trainer.py 里看不到同步代码）；第三层是分片优化器状态——optimizer.step() 读写的是分片参数与分片 m/v 状态，全量优化器状态从不物化。clip、lr_scheduler.step、zero_grad、save/log 都还是标准写法。下图把这一步的时序摊开。

这三层"隐式"有一个共同特征：Trainer 调的都是标准 PyTorch 接口，分片逻辑全部藏在接口背后。loss.backward() 是标准 autograd 入口，optimizer.step() 是标准优化器入口，fsdp_root.no_sync() 是 FSDP 暴露的 context manager——没有一个自定义的同步函数被显式调用。这正是 FSDP2 相对 mcore 的设计差异：mcore 在 train_step 里手写 forward/backward/optimizer/clip 一整套并显式编排通信，FSDP2 则把这些委托给 torch 原生的 hook 机制，循环骨架回归"朴素教程"风格。代价是可读性反直觉：你想在 trainer.py 里找"梯度在哪里同步"会找不到，得知道它埋在 backward hook 里——这也是本篇反复强调"无需手动 all_reduce 梯度"的原因。

def training_step(self, inputs, num_items_in_batch):
    # 1. Set model to train mode
    self.model.train()
    if hasattr(self.optimizer, "train") and callable(self.optimizer.train):
        self.optimizer.train()
    # 2. Forward pass
    loss_all = self._compute_loss(inputs, return_outputs=False, num_items_in_batch=num_items_in_batch)
    if self.args.stage == 'pt':
        loss_all.loss = loss_all.loss / self.current_gradient_accumulation_steps
        loss_all.lm_loss_output = loss_all.lm_loss_output / self.current_gradient_accumulation_steps
        loss_all.mtp_loss_output = loss_all.mtp_loss_output / self.current_gradient_accumulation_steps
    loss = loss_all.loss
    del inputs
    # 4. Multi-device parallelism: Average loss across devices (if not using FSDP internal handling)
    if torch.cuda.device_count() > 1:
        loss = loss.mean()
    # 5. Backward pass
    loss.backward()
    # 6. Return detached loss
    return loss.detach(), loss_all.lm_loss_output.detach(), loss_all.mtp_loss.detach()

单卡 / DDP（已知）

loss.backward()算梯度 + 累到 .grad；DDP 下 hook 做 all_reduce 整张量（梯度是全量）
optimizer.step()读写全量参数 + 全量 m/v 状态（单卡全物化）
梯度累积需手动 (loss/accum).backward() 累加，或用 DDP no_sync()
save ckpttorch.save(model.state_dict()) 全量字典落盘
梯度同步DDP：反向 hook all_reduce 整张量；Trainer 不写

FSDP2 / NPU（本项目）

loss.backward()同一行——但 FSDP2 hook 触发 ReduceScatter，.grad 落地为分片（只持本 rank 一片）
optimizer.step()读写分片参数 + 分片 m/v 状态（全量从不物化，省显存）
梯度累积fsdp_root.no_sync()（:277）包前 N-1 个 micro-batch，只在最后放出 RS，通信从每微批一次压到每 update 一次
save ckptckpt_manager.save（:380）走 dcp sharded，每 rank 写自己那片；HF 全量另走 :435 get_model_state_dict(full_state_dict=True, cpu_offload=True)
梯度同步FSDP2：反向 hook ReduceScatter 分片；无需手动 all_reduce 梯度——Trainer 全程没写梯度同步代码
小结　心智模型：Trainer = 标准训练循环（epoch→update→micro-batch 三层嵌套，forward/clip/step/scheduler/zero/save 全是熟悉写法）+ FSDP2 隐式分片同步（loss.backward() 一行同时算梯度和跨 rank ReduceScatter，no_sync() 压通信到每 update 一次，优化器读写分片状态）+ TrainMonitor 统一日志（5 段模板拼出 quick_start 那行 iteration/consumed tokens/lm loss/grad norm/max_memory_allocated）。对照 mcore 的 train step：mcore 在 megatron 核心里自己编排 forward/backward/optimizer/clip 一整套（train_step 函数），FSDP2 这里则贴近 HuggingFace Transformers 的写法（文件头 docstring 自述"Strictly follows the gradient accumulation and loop logic found in HuggingFace Transformers"），把分片并行的脏活留给 FSDP2 hook，循环本身保持朴素。
常见误区：① 以为要手动 all_reduce 梯度——FSDP2 在 backward 自动 ReduceScatter。trainer.py 全文搜不到一行 all_reduce 梯度的代码，loss.backward()（:482）就是同步触发点，ReduceScatter 由注册在 sharded 参数上的 hook 执行；:322 的 all_reduce 聚合的是 loss/grad_norm 四个标量用于日志，不是梯度。② 以为 optimizer.step() 会读全量参数——FSDP2 下参数和 m/v 状态都是分片的，全量从不物化。③ 以为梯度累积要自己写累加逻辑——no_sync() 上下文 + loss/accum_steps（:465）已处理，前 N-1 个 micro-batch 不触发 RS、最后放出。④ 以为 quick_start 那行日志是 mcore 风格手写的——其实是 train_monitor.py 的 5 段模板（base/optimizer/grad_norm/npu_memory 等）拼出来的，logging.py 只提供 INFO [asctime] >> 前缀。⑤ 以为 save 存的是全量 state_dict——ckpt_manager.save（:380）存的是 dcp sharded ckpt，model/optimizer 对象直接塞进 state 字典（:368）由 dcp 分片写盘；HF 全量权重是 :433 另一条路径。
