Qwen3 建模精读

目标：讲清 fsdp2 的 Qwen3 建模如何适配分片与 NPU 融合算子。前置：EP11 重计算与显存策略。

你将获得：① 看懂 fsdp2 后端 dense Qwen3 是一层薄壳——它直接复用 HF 的 transformers.Qwen3Model，只在外层包 LMHead 与三处融合算子 patch ② 能对照 HF transformers 实现，定位三开关 use_fused_rmsnorm/use_fused_rotary_pos_emb/use_flash_attn 各自落在哪条 patch / 配置链上，并理解 MoE 变体多出的 grouped-GEMM 与 token permute 路径

你已经熟悉的情况 桥接开场

你在 HuggingFace transformers 里用 Qwen3，打开 modeling_qwen3.py，看到的是一套自洽的标准 decoder 堆：每一层 = Qwen3RMSNorm → Qwen3Attention（带 RoPE）→ 残差 → Qwen3RMSNorm → Qwen3MLP（SwiGLU）→ 残差。RMSNorm 是平方-均值-归一化三步手写算式（x.pow(2).mean(-1, keepdim=True) 再加 epsilon 开方再乘 weight），RoPE 是 rotate_half + 复数乘法的手写实现（对 q/k 各做一次 cos/sin 旋转），MLP 是 gate_proj/up_proj/down_proj 三个 nn.Linear 加 SiLU 激活（silu(gate) * up 再过 down_proj），attention 默认走 nn.functional.scaled_dot_product_attention（SDPA，PyTorch 会按后端选 FlashAttention 或 math fallback）。这套实现 GPU 友好、可读，但每个小算子都是拆开的——RMSNorm 的三步是三次 kernel launch + 一个中间平方张量落盘，RoPE 的 rotate_half 要 concat-rotate 两次，MLP 的 SiLU+乘是两个算子串行。在 NPU 上这些拆开的算子会带来大量小 kernel launch 与中间张量落盘，吞吐吃亏；Ascend 侧把这些 fused 成单算子（npu_rms_norm 把三步合一、npu_rotary_mul 把 cos/sin 旋转合一、npu_swiglu 把 gate/up 拼接 + SiLU + 乘合一），省的就是中间张量往返与 launch 开销。

在 MindSpeed-LLM 的 fsdp2 后端里，Qwen3 的建模文件 qwen3.py 看起来出奇地短——只有一百多行。原因正是：它不是把 HF 的 modeling_qwen3 拷贝重写一遍，而是用继承 + 运行时 patch 的方式"借壳"。稠密版的 Qwen3ForCausalLM 直接 transformers.Qwen3PreTrainedModel 做父类，decoder 堆整个交给 self.model = transformers.Qwen3Model(config)——attention / RoPE / MLP / RMSNorm 的前向逻辑仍走 HF 原版。fsdp2 这层只做两件事：一是把输出头 lm_head 换成支持 DTensor 的 LMHead（分片落地的那一钩），二是用一个静态方法 register_patches 把 HF 原版的 RMSNorm.forward 和 apply_rotary_pos_emb 在运行时替换成 NPU 融合算子版本。换句话说，fsdp2 的 Qwen3 = HF 结构 + NPU 融合算子 patch + 输出头分片适配，三件事都在这薄薄一层里收口。这种"借壳"思路的妙处在于：HF 升级 modeling_qwen3（比如改 attention 实现、修 bug），fsdp2 这层基本不用动——它 patch 的是符号名（Qwen3RMSNorm.forward），HF 内部怎么改 RMSNorm 的实现都无所谓，只要那个符号还在，patch 就接得上。代价是：你得清楚"哪些符号被换、哪些没换"，否则会以为某段是 HF 原版、其实是 NPU 融合版——这正是本篇要逐符号厘清的。

这三处融合算子切换的开关，就摆在启动脚本 pretrain_qwen3_8b_4k_fsdp2_A3.sh 末尾三行 CLI 参数里——这也是本篇要逐行核对的入口。需要先打个预防针：三个开关里只有前两个（fused_rmsnorm / fused_rotary_pos_emb）真在 dense Qwen3 的 patch 链上落地；第三个 use_flash_attn 在 dense Qwen3 里并不走 qwen3.py 的 patch，它的作用点是另一条配置链——下面会专门讲清这个 nuance，免得你以为三开关同源。另外要分清"融合算子的 Python 调用包装"与"融合算子的 C++ kernel 实现"：前者本仓 common/fusions.py 里有（就是 torch_npu.npu_rms_norm 那几行调用），后者在 torch_npu/Ascend 仓里（本仓不可见）——别把"本仓只见开关"当成"融合算子全在外部"。

本项目怎么做 原理

把 Qwen3 的一层 decoder 摊开看，融合算子作用在三处热路径上：RMSNorm（每层前后各一次）、RoPE（attention 内部对 q/k 各一次）、attention score（SDPA / FlashAttention）。fsdp2 的策略不是重写这三处，而是用 mindspeed.patch_utils 的 MindSpeedPatchesManager（下称 pm）做运行时方法替换：HF 模型对象照常构造、照常前向，但在前向真正跑到 RMSNorm.forward 之前，pm 已经把 transformers.models.qwen3.modeling_qwen3.Qwen3RMSNorm.forward 这个符号指向了 fused_rmsnorm_forward——于是同一份 HF 代码，跑出来的算子已经换成 torch_npu.npu_rms_norm。RoPE 同理：apply_rotary_pos_emb 被换成调用 torch_npu.npu_rotary_mul 的版本。这种 patch 是符号级替换而非装饰器或子类——pm.register_patch(目标符号全名字符串, 新函数) 直接改写 module 里那个符号的绑定，HF 代码里所有 self.forward() / apply_rotary_pos_emb(...) 调用都会无感地落到新函数上。timing 也很关键：register_patches 在 model_factory.py:106-107 于模型实例化之前被调（hasattr(model_cls,'register_patches') 判定），pm.apply_patches() 落地后 HF 模型对象才构造——所以 HF 的 __init__ 跑到哪、前向跑到哪，算子都已是 NPU 版。attention 的融合则更隐蔽：dense Qwen3 不改 attention 代码，而是在加载 HF config 时把 attn_implementation 设好，让 HF 的 SDPA 路径在 NPU 上自动落到 FlashAttentionScore 算子。下面这张图把一层 Qwen3 的数据流与三处融合点标出来。

图示：★ 标的三个融合点是本篇主线。RMSNorm 与 RoPE 走 pm.register_patch 运行时替换 HF 符号；attention 的 flash 走 config-load 时设 attn_implementation（不在 qwen3.py 内）。LMHead 是 fsdp2 唯一"改写"过的层——其余 decoder 堆全复用 HF。

common/fusions · 两处融合算子的真实实现（NPU 算子调用点）
import torch
try:
    import torch_npu
except ImportError:
    pass


def fused_rmsnorm_forward(self, x):
    return torch_npu.npu_rms_norm(x, self.weight, epsilon=self.variance_epsilon)[0]


def apply_rotary_pos_emb(q, k, cos, sin, position_ids=None, unsqueeze_dim=1):
    cos = cos.unsqueeze(unsqueeze_dim)
    sin = sin.unsqueeze(unsqueeze_dim)
    q_embed = torch_npu.npu_rotary_mul(q, cos, sin)
    k_embed = torch_npu.npu_rotary_mul(k, cos, sin)
    return q_embed, k_embed


def fused_mlp_forward(self, x):
    gate_up_w = torch.cat([self.gate_proj.weight, self.up_proj.weight], dim=0)
    gate_up_output = torch.matmul(x, gate_up_w.t())

    swiglu_output = torch_npu.npu_swiglu(gate_up_output, dim=-1)
    down_proj = self.down_proj(swiglu_output)
    return down_proj

Qwen3LMHead：DTensor.to_local() 分片适配钩子（行 7-32）
class Qwen3LMHead(nn.Linear):
    def forward(self, hidden_states: torch.Tensor, loss_ctx: callable = None):
        # Handle distributed tensor (DTensor) weights and biases by converting to local tensors.
        if isinstance(self.weight, DTensor):
            w = self.weight.to_local()
            if self.bias is not None:
                if not isinstance(self.bias, DTensor):
                    raise TypeError(
                        f"Expected bias to be a DTensor when weight is a DTensor, "
                        f"but got bias of type {type(self.bias)}."
                    )
                b = self.bias.to_local()
            else:
                b = None
        else:
            w = self.weight
            b = self.bias

        if loss_ctx is None:
            logits = F.linear(hidden_states, w, b)
            return logits, None
        else:
            return None, loss_ctx(hidden_states, w, b)

小结　心智模型：fsdp2 的 Qwen3 建模 = HF 结构 + NPU 融合算子 patch + 输出头分片适配。dense 版的 qwen3.py 是一层薄壳——decoder 堆整段复用 HF transformers.Qwen3Model（attention/RoPE/MLP/RMSNorm 全不改），fsdp2 自己只动手两处：把输出头换成支持 DTensor 的 LMHead（分片落地的钩子），并用 register_patches 在运行时把 HF 的 RMSNorm.forward / apply_rotary_pos_emb 符号替换成调 torch_npu.npu_rms_norm / npu_rotary_mul 的版本。三开关里，use_fused_rmsnorm、use_fused_rotary_pos_emb 走这条 patch 链，use_flash_attn 在 dense Qwen3 走的是 config-load 时的 attn_implementation 默认链（NPU 上 SDPA→FlashAttentionScore），不是 qwen3.py 里的 if——这是最容易看走眼的一点。MoE 变体（qwen3_moe.py）同思路但多两组 patch（grouped GEMM 专家块、activation_offload 模型前向），融合点从"RMSNorm/RoPE/attention"扩到再加"grouped GEMM + token permute + swiglu"，全靠 torch_npu/Ascend 算子。一句话总结：fsdp2 不重造模型，它"借 HF 的壳、patch 进 NPU 的核、再给输出头接上 DTensor"——三招合一，就完成了从 CUDA/HF 到 Ascend/分片的建模迁移。
常见误区：①以为 fsdp2 的 Qwen3 是把 HF 的 modeling_qwen3.py 整份拷贝重写——不是直接拷 HF 模型，需改写以接 fully_shard 与 NPU 融合算子，但改写极薄：decoder 堆原样复用 HF，只动 LMHead + 两处 patch。②以为三开关同源、都在 qwen3.py 里 if——前两个（fused_rmsnorm/fused_rotary）真在 register_patches（qwen3.py:108/111），use_flash_attn 在 dense Qwen3 不走 qwen3.py 的 if，走 model_factory.py:79 的 attn_implementation 默认链，别在 qwen3.py 里找它。③以为融合算子实现在本仓外、本仓只见开关——fusions.py 实际给出了 npu_rms_norm/npu_rotary_mul/npu_swiglu 的 Python 调用点（核实：C++ kernel 仍在 torch_npu/Ascend 侧，本仓只见调用包装与算子选择）。④以为 fused_mlp_forward 也是 dense Qwen3 的 patch——它定义在 fusions.py:20，但 dense 的 register_patches 没挂它，dense MLP 仍走 HF 原版；fused_mlp 主要服务 MoE 路径（qwen3_moe.py 内联 npu_swiglu）。⑤以为 LMHead 就是个普通 Linear——它的核心是 isinstance(weight, DTensor) 时 .to_local()（modules.py:10-11），这是 FSDP2 分片后输出头能正常前向的前提，外加 loss_ctx 分支对接 chunkloss（EP11）。⑥把 dense Qwen3 与 MoE 的 patch 数记混——dense 2 条（RMSNorm/RoPE），MoE 4 条（多 grouped_gemm + activation_offload），MoE 多出来的复杂度全在专家前向，attention 部分两者同源。⑦以为 Qwen3-8B 的层数在本仓某处硬编码——num_hidden_layers 来自外部 HF config（A3.sh:22 指向 /home/data/Qwen3-8B/），本仓镜像未收录该 config.json（核实：Qwen3-8B 层数与超参，以 HF config.json 为准，本仓 examples 仅引用路径）。
