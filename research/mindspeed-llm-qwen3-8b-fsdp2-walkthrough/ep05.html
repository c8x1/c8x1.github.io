入口与启动链路

目标：理清 env→torchrun→train_fsdp2→工厂装配 的启动链。前置：EP04 项目全景。

你将获得：① 一条能从 bash A2.sh 一路追到 Trainer.train() 的完整启动时序 ② 弄懂"一份 YAML + 一串 CLI"如何被合并成同一套五段 dataclass 参数、且 CLI 永远赢的合参机制

你已经熟悉的情况 桥接开场

你最熟的那种启动方式大概是 python train.py config.yaml——一个进程、一份配置、if __name__ == "__main__" 里读 yaml、建模型、开训。要多卡就再补一句 torchrun --nproc_per_node=8 train.py ...，进程由 torchrun 拉、配置还是那一份 yaml，参数该怎么传还怎么传。

MindSpeed-LLM 的 fsdp2 路线骨架和这条"熟悉的路"几乎一样：入口仍是 torchrun 拉多进程、if __name__ == "__main__" 仍在 train_fsdp2.py 末尾。但有两处关键不同：一是配置同时来自 YAML 和 CLI 两路，且二者不冲突——CLI 优先，YAML 兜底；二是入口脚本本身几乎不写训练逻辑，它把活儿全交给一个叫 MindSpeedAutoTrainer 的"依赖注入容器"：这个容器按固定顺序（解析参数→初始化分布式→造零件→把零件注入 Trainer）一口气把整条链装配好。这一篇就是顺着 bash A2.sh 这一记回车，把 env、torchrun、解析、工厂、Trainer 这五环一节节拆给你看。

本项目怎么做 原理

fsdp2 后端的启动哲学是"环境先行、进程其次、参数合参、容器装配"四段式：先用 env_config.sh 把昇腾相关的运行时开关（HCCL 超时、TaskQueue、显存复用、PYTHONPATH…）一次性 export 进环境；再用 torchrun 按 NPUS_PER_NODE/NNODES 拉起 N 个对等进程，每个进程注入 RANK/WORLD_SIZE/LOCAL_RANK 等环境变量；进程拉起后执行 train_fsdp2.py，其入口 main() 只做一件事——构造 MindSpeedAutoTrainer() 并调 .train()。构造过程内部按 4 步推进：① _parse_args 把 YAML+CLI 合并解析成五段 dataclass；② _initialize 选后端（NPU→hccl / CUDA→nccl）、设设备、设种子、建进程组；③ _build_* 系列工厂按参数造出 model/tokenizer/data/optimizer/scheduler/checkpoint/monitor；④ 把这些零件作为具名参数注入 Trainer(...)，由 Trainer 跑主循环。下面的时序图把这条链画清楚了。

这张图刻意把"参数合参"画成一个回弯：YAML+CLI 在 fsdp2_parse_args 里合并出五段 dataclass 后，结果回流到 train_fsdp2.py 的 _initialize/_build_*，每一步都从这些 dataclass 里取值——所以"一份配置"在物理上是 YAML，在程序内是五个 dataclass 实例。

逐块拆：

  • 第 1 行 source：source examples/fsdp2/env_config.sh 把那一坨昇腾运行时环境变量灌进当前 shell——这一步必须在 torchrun 之前，因为这些变量要被 torchrun 拉起的子进程继承。其中最关键的是 TRAINING_BACKEND=mindspeed_fsdp（行 2），它决定 AutoTrainer 走哪条后端分支；其余如 HCCL_CONNECT_TIMEOUT=1800（行 3，HCCL 建链超时 1800 秒）、TASK_QUEUE_ENABLE=2（行 4，昇腾 TaskQueue 调度）、MULTI_STREAM_MEMORY_REUSE=2（行 6，多流显存复用）、PYTHONPATH=$PYTHONPATH:$(pwd)（行 10，把仓库根加进搜索路径，保证 import mindspeed_llm 能找到）都是 NPU 训练的"标配开关"，EP02 已逐条讲过。
  • 第 3–8 行拓扑变量：NPUS_PER_NODE=8 是单机 NPU 数（Qwen3-8B 走 8 卡）；MASTER_ADDR=localhost / MASTER_PORT=6499 是进程组 rendezvous 地址（单机所以 localhost）；NNODES=1 / NODE_RANK=0 表示单节点本机；WORLD_SIZE 由 NPUS_PER_NODE × NNODES 算出，最终等于 8。多机时只需改 NNODES 并在各机设不同 NODE_RANK 与同一 MASTER_ADDR，脚本结构不变——这是 torchrun 多机协议的标准用法。
  • 第 10–16 行 DISTRIBUTED_ARGS：把上面五个拓扑变量拼成 torchrun 的参数串：--nproc_per_node 8 --nnodes 1 --node_rank 0 --master_addr localhost --master_port 6499。注意这里用的是 torchrun（PyTorch≥1.9 的统一启动器），不是老式的 python -m torch.distributed.launch。torchrun 会按 --nproc_per_node 起 8 个对等子进程，每个进程注入 RANK / WORLD_SIZE / LOCAL_RANK 三个环境变量——这三个变量后面 _initialize 里会用到。
  • 第 19 行的注释 + 第 20–28 行的调用：第 19 行注释写得很直白——CLI args take precedence over the YAML when both are set（CLI 与 YAML 同时设置时 CLI 优先）。第 20 行才是真正的启动：torchrun $DISTRIBUTED_ARGS train_fsdp2.py examples/fsdp2/qwen3/pretrain_qwen3_8b_4k_fsdp2_A2.yaml --model.model_name_or_path /home/data/Qwen3-8B/ --parallel.fsdp_size 8 …。这里同时给了 YAML 文件路径（位置参数，sys.argv[1]）和一串 --段.字段 值 形式的 CLI 覆盖。YAML 里装的是"不太常调的全量配置"（recompute/optimizer/lr_scheduler/max_steps 等），CLI 上挂的是"每个环境都要改的"（模型路径、数据集、batch、output_dir、fsdp_size）。--parallel.fsdp_size 8 这一行尤其重要——它把 FSDP 分片组大小设成 8（等于全部 8 卡全分片），而 YAML 里并没有这个键，所以这里 CLI 不是"覆盖"而是"补充"。

现在跳到 Python 侧。torchrun 把 8 个进程拉起后，每个进程都执行 train_fsdp2.py，末尾这段是真正的"入口的入口"。

train_fsdp2.py 入口段（main 门面 → AutoTrainer 派发 → MindSpeedAutoTrainer 装配）
class MindSpeedAutoTrainer:
    """
    AutoTrainer: Dependency Injection Container.
    Based on FSDP2 Arguments (HfArgumentParser style).
    """

    def __init__(self):
        # 1. Parse arguments
        self._parse_args()

        # 2. Initialize distributed environment
        self._initialize(seed=self.training_args.seed)

        self.rank = torch.distributed.get_rank()
        self._print_parsed_args()

        # 3. Build components
        self.model = self._build_model()
        self.tokenizer = self._build_tokenizer()
        self.template = get_template_and_fix_tokenizer(self.tokenizer, self.data_args)
        self.data_manager = self._build_data_manager(self.tokenizer, self.template)
        self.train_dataloader = self.data_manager.create_train_dataloader()
        self.optimizer = self._build_optimizer(self.model)
        self.lr_scheduler = self._build_scheduler(self.optimizer)
        self.checkpoint_manager = self._build_checkpointer()
        self.train_monitor = self._build_monitor()
        self.model.apply_optimizer_hook(self.optimizer)

        # 4. Dependency Injection
        self.trainer = Trainer(
            model=self.model,
            optimizer=self.optimizer,
            lr_scheduler=self.lr_scheduler,
            train_dataloader=self.train_dataloader,
            args=self.training_args,
            parallel_args=self.parallel_args,
            optimization_args=self.optimization_args,
            data_args=self.data_args,
            ckpt_manager=self.checkpoint_manager,
            monitor=self.train_monitor,
            tokenizer=self.tokenizer,
        )

这段 __init__ 就是"依赖注入容器"的本体——它自己不实现任何训练算法，只做按序取件 + 具名交付。四个注释把顺序标得很清楚：① _parse_args（行 68，解析参数）；② _initialize（行 71，初始化分布式）；③ _build_model/_build_tokenizer/_build_data_manager/_build_optimizer/_build_scheduler/_build_checkpointer/_build_monitor（行 77–85，造零件）；④ Trainer(...)（行 89，注入）。注意行 79 的 get_template_and_fix_tokenizer 夹在 tokenizer 和 data 之间——它要在拿到 tokenizer 之后、喂给 data 之前，把对话模板和特殊 token 修正好（EP13 数据流水线详讲）。第 ④ 步里 Trainer 接收的全是具名参数，这就是"依赖注入"的字面体现：Trainer 不自己去 new 任何东西，它需要什么，容器就递什么。装配完，AutoTrainer.train()（见下方入口段）只需调一句 self.trainer.train(resume_from_checkpoint=...)。

class AutoTrainer:
    """
    Unified entry point for Training.
    Dispatches to MindSpeedAutoTrainer (New) or McoreAutoTrainer (Old) based on configuration.
    """

    def __init__(self):
        # Strategy Dispatch: Prioritize environment variable TRAINING_BACKEND
        # To run MindSpeed FSDP code, set: export TRAINING_BACKEND=mindspeed_fsdp
        logger.info_rank0(">>> [AutoTrainer] Initializing MindSpeed FSDP backend...")
        self.trainer = MindSpeedAutoTrainer()

    def train(self):
        """Delegate to the implementation"""
        self.trainer.train()


@auto_coverage
def main():
    trainer = AutoTrainer()
    trainer.train()


if __name__ == "__main__":
    main()

这里有个值得注意的设计：AutoTrainer 是一层"门面/派发器"（Facade/Dispatcher），它的 __init__ 注释明确写着"Prioritize environment variable TRAINING_BACKEND"（行 277），并提示"要跑 MindSpeed FSDP 就 export TRAINING_BACKEND=mindspeed_fsdp"（行 278）——这正好和 env_config.sh:2 对上。换句话说，AutoTrainer 留了"未来切到 mcore 后端"的口子，只是当前实现里直接 self.trainer = MindSpeedAutoTrainer()（行 280）。main()（行 288–290）被 @auto_coverage 装饰（行 287，作用 EP04 讲过：按白名单注入覆盖测试），它只做两件事：trainer = AutoTrainer()、trainer.train()。最后行 296–297 的 if __name__ == "__main__": main() 是 torchrun 拉起进程后真正执行的第一行 Python。

现在回答本篇最核心的问题：YAML 和 CLI 到底怎么合并、谁优先？答案在 arguments.py 的 fsdp2_parse_args。

fsdp2_parse_args 七步合参（YAML 兜底、CLI 优先的关键证据）
def fsdp2_parse_args(rootclass: TypeVar) -> TypeVar:
    """
    Parses the root argument class from CLI or YAML input.
    Supports complex types like List[Dict] and Dict via JSON serialization.
    """
    # 1: Create ArgumentParser and register all fields
    parser = _create_argument_parser(rootclass)

    # 2: Parse command-line args
    cmd_args = sys.argv[1:]
    if not cmd_args:
        raise ValueError("No arguments provided.")

    # 3: Load YAML config if the first arg is a .yaml/.yml file
    yaml_config = _load_yaml_if_provided(cmd_args)

    # 4: Merge YAML into CLI args (CLI has higher priority)
    final_cmd_args = _merge_yaml_into_cmd_args(cmd_args, yaml_config)

    # 5: Parse arguments
    args, remaining_args = parser.parse_known_args(final_cmd_args)
    if remaining_args:
        logger.info_rank0(f"Some specified arguments are not used by the ArgumentParser: {remaining_args}")

    # 6: Post-process fields that require JSON deserialization
    parse_result = _postprocess_json_fields(args, parser.dict_fields)

    # 7: Build dataclass instances
    result = _build_dataclass_instances(rootclass, parse_result)

    # 8: Cross-dataclass validation
    _validate_cross_args(result)

    return result
见 ../mindspeed_llm/fsdp2/utils/arguments.py：:787（def fsdp2_parse_args）、:796（sys.argv[1:]）、:803（CLI 优先注释）、:804（_merge_yaml_into_cmd_args）、:807（parse_known_args）

第 1 步先用 _create_argument_parser(rootclass) 把五段 dataclass（model/data/parallel/training/optimization）的所有字段动态注册成一个 argparse.ArgumentParser，每个字段注册成 --段.字段 形式的参数——这就是为什么 CLI 里写的是 --parallel.fsdp_size 8 而不是 --fsdp_size 8。第 2 步取 sys.argv[1:]（注意：YAML 文件名也在里面，作为 cmd_args[0]）。第 3 步 _load_yaml_if_provided 检查 cmd_args[0] 是否以 .yaml/.yml 结尾，是就用 yaml.safe_load 读成 dict（这就是为什么 YAML 必须是第一个位置参数）。第 4 步是关键——注释写得明明白白：# 4: Merge YAML into CLI args (CLI has higher priority)（行 803），然后调 _merge_yaml_into_cmd_args（行 804）。第 5 步把合并后的串喂给 parser.parse_known_args。后续 6/7/8 是 JSON 反序列化、构造 dataclass 实例、跨段校验。

_merge_yaml_into_cmd_args 合参实现（CLI 已出现的字段被跳过）
def _merge_yaml_into_cmd_args(cmd_args, yaml_config):
    """Convert YAML config to CLI-style arguments (lower priority than explicit CLI)."""
    working_args = cmd_args[1:] if yaml_config else cmd_args[:]
    cmd_args_string = "=".join(working_args)
    result = working_args[:]

    for base, arg_dict in yaml_config.items():
        if not isinstance(arg_dict, dict):
            continue
        for arg_name, arg_value in arg_dict.items():
            if arg_value is None:
                continue
            if f"--{base}.{arg_name}=" in cmd_args_string:
                continue  # Skip if overridden by CLI

            result.append(f"--{base}.{arg_name}")
            if isinstance(arg_value, list):
                result.extend(str(item) for item in arg_value)
            elif isinstance(arg_value, dict):
                result.append(json.dumps(arg_value, ensure_ascii=False))
            else:
                result.append(str(arg_value))

    return result
见 ../mindspeed_llm/fsdp2/utils/arguments.py：:993（def _merge_yaml_into_cmd_args）、:994（lower priority 注释）、:1006（Skip if overridden by CLI）

这一段把"CLI 优先"落到了代码上：它先把 YAML 转成等价的 --段.字段 值 串，但只要某个字段已经在 CLI 里以 --段.字段=... 形式出现过，YAML 的那份就被 continue 跳过（行 1006 注释 Skip if overridden by CLI）。注意第 1000 行（按文件实际行号，即上方 cmd_args_string = "=".join(working_args)）把 CLI 串按 = 拼起来做"是否已出现"的子串匹配——这隐含一个约定：CLI 覆盖必须用 --key=value 形式才能被识别为"已出现"（脚本里 --parallel.fsdp_size 8 是空格分隔，会被 argparse 转成 = 形式后再比对，（核实：空格分隔的 --k v 与 --k=v 在此子串匹配下是否等价，最终以 arguments.py 运行时行为为准））。合参完成后，YAML 与 CLI 在程序内已无区别，统一是一串 CLI 风格参数，由同一个 ArgumentParser 解析。这就是为什么"CLI 与 YAML 不冲突"——它们不是两套并行配置，而是 YAML 先被翻译成 CLI 串、再被 CLI 同形覆盖。

装配MindSpeedAutoTrainer DI 容器：_parse_args→_initialize→_build_*→Trainer(具名注入)
小结　心智模型：fsdp2 后端的启动是一条"torchrun 启多进程 + 一份 YAML/CLI 合参驱动工厂装配"的链——env_config 灌运行时开关，torchrun 按 NPUS_PER_NODE 拉对等进程并注入 RANK/LOCAL_RANK，train_fsdp2.py 的 main() 把球交给 MindSpeedAutoTrainer 这个依赖注入容器，容器按"解析→初始化分布式→造零件→注入 Trainer"四步把整条训练链一次装配好。读这一路代码的姿势是"顺着 __init__ 的四个注释往下追"，而不是被五段 dataclass 吓住——参数再多，入口只有一个 AutoTrainer().train()。
常见误区：① 以为 CLI 与 YAML 冲突——其实二者不冲突，_merge_yaml_into_cmd_args 先把 YAML 翻译成 CLI 串、CLI 已出现的字段被跳过（行 1006 Skip if overridden by CLI），所以CLI 优先、YAML 兜底，二者最终是同一串参数。② 以为 YAML 文件可以放任意位置——它必须是 sys.argv[1]（第一个位置参数），否则 _load_yaml_if_provided 不会识别。③ 以为 --parallel.fsdp_size 8 是"覆盖"YAML——其实 A2 的 YAML 里压根没 fsdp_size 这键，这里 CLI 是"补充"而非"覆盖"；真正"覆盖"的是 YAML 里已设、CLI 又改了同一字段的情况。④ 以为 MindSpeedAutoTrainer 就是 Trainer——前者是装配/注入容器，后者是被注入后跑主循环的引擎，二者不是一回事。
