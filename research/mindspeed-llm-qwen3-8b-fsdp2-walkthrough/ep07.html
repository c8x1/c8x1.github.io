并行状态与引擎

目标：理清 DeviceMesh + 自定义 engine 如何在 torch.distributed 之上叠加。前置：EP06 参数体系精读。

你将获得：① 一张能从 torch.distributed.init_process_group 一路追到 ParallelState 三个 DeviceMesh、再到 MindSpeedParallelEngine 六个 apply_* 的完整并行初始化链路图 ② 弄懂 LLM 层 mindspeed_llm/fsdp2/distributed/ 与核心库 _deps_mindspeed_core/mindspeed/fsdp/ 那两个同名的 parallel_state.py / mindspeed_parallel_engine.py 到底是"包装、子类、还是重写"——并以两边同函数逐行对照给出证据

你已经熟悉的情况 桥接开场

你写过多卡训练脚本的话，并行初始化大概都长这样：开头一行 torch.distributed.init_process_group(backend='nccl', rank=rank, world_size=world_size)，把 NCCL 通信组拉起来；然后用 torch.distributed.new_group(ranks) 按需切几个子通信组（一个 data-parallel 组、一个 tensor-parallel 组），DistributedDataParallel(model, process_group=dp_group) 把模型包一层；前向自动 all-reduce 梯度。这套的心智模型是"一个全局 process group + 几个手切子组 + DDP 包一层"——扁平、过程式，rank/world_size/group 三个量你到处 get_rank() / get_world_size() 取。

MindSpeed-LLM 的 fsdp2 后端在这套之上叠了两层东西：第一层是 DeviceMesh——来自 PyTorch 2.3+ 的 torch.distributed.device_mesh，它把"几个手切子组"升级成"一个有形状的多维网格"，每根轴是一个命名的子组（dp / fsdp / tp / cp / ep），mesh[dim_name].get_group() 直接拿到该轴对应的 process group；第二层是 自定义并行引擎 MindSpeedParallelEngine——它不是 DDP 那种"只管梯度 all-reduce"的薄壳，而是一个 torch.nn.Module，在 __init__ 里按顺序把 FSDP/TP/EP/CP/recompute/quantization 六种"模块变换"逐个 apply 到模型上，forward 直接转调被变换后的模型。这两层加在一起，就是本篇要讲的"在 torch.distributed 之上叠加 DeviceMesh + 自定义 engine"。

本项目怎么做 原理

fsdp2 的并行初始化哲学是"DeviceMesh 是并行拓扑的唯一真相源，engine 是按拓扑 apply 变换的编排器"。ParallelState（单例 dataclass）在 __post_init__ 里干三件事：① 若 torch 分布式没初始化就 init_process_group(backend='hccl')（NPU 的 HCCL 对应 GPU 的 NCCL）；② 按 (fsdp_size, tp_size) / (ep_fsdp_size, ep_size) / (dp_fsdp_size, cp_size) 三组形状调 init_device_mesh 建三个 DeviceMesh，分别覆盖 dense 部分、expert 部分、DP×FSDP×CP 联合组；③ 用 setattr 给单例动态挂上 get_fsdp_device_mesh / get_tp_device_mesh / get_cp_device_mesh / get_ep_device_mesh … 一批方法，下游 engine 按名字取。然后 MindSpeedParallelEngine.__init__ 先 init_parallel_state(config) 拿到这个单例，再依次 apply_quantization_modules → apply_tp_modules → apply_ep_modules → apply_cp_modules → apply_recompute_modules → apply_fsdp_modules，每一步从 parallel_state 取对应的 DeviceMesh 喂给具体的分片函数（fully_shard_parallel_modules / tensor_parallel_modules / …）。下面的拓扑图画清了 1D 与 2D 两种 DeviceMesh 形状，以及 engine 的六步 apply 顺序。

这张图的两个关键：① 1D vs 2D 不是装饰——dense 模型（Qwen3-8B）只有 fsdp 一根轴（8 卡成一条线），ep_size=1 时 EP 那条轴退化为长度 1；MoE 模型才把 fsdp 与 ep 拆成两根正交轴，同一张卡既在 fsdp 组（纵向）又在 ep 组（横向）。② apply 顺序里 FSDP 必须最后——前面 tp/ep/cp/recompute 改的是模块内部结构（切 qkv、切 expert、切序列），FSDP 是把已经切好的整模型再按 fsdp 组分片参数，必须最外层包，否则前面切的模块边界会被 FSDP 重写。

ParallelState：单例 dataclass，__post_init__ 里 init_process_group + 建三组 DeviceMesh
@dataclass
class ParallelState(metaclass=Singleton):
    data_parallel_size: int = -1
    fully_shard_parallel_size: int = -1
    tensor_parallel_size: int = -1
    context_parallel_size: int = -1
    expert_parallel_size: int = -1
    expert_fully_shard_parallel_size: int = -1
    expert_data_parallel_size: int = -1
    device_mesh_map: dict[str, DeviceMesh] = None

    def __post_init__(self):
        if not torch.distributed.is_initialized():
            torch.distributed.init_process_group(backend='hccl')
        if self.fully_shard_parallel_size < 0:
            raise Exception("ParallelState has not been initialized .")
        if self.device_mesh_map is None:
            self.device_mesh_map = dict()

        # create DP/CP/Ulysses/TP groups
        mesh_dim_names = ('dp', 'fsdp', 'tp')
        mesh_shape = (self.fully_shard_parallel_size, self.tensor_parallel_size,)
        self.data_parallel_size = get_last_mesh_dim(mesh_shape)
        mesh_shape = (self.data_parallel_size,) + mesh_shape
        self.add_device_mesh_groups(mesh_dim_names, mesh_shape)

        # create EP_DP/EP groups
        mesh_dim_names = ('edp', 'efsdp', 'ep')
        mesh_shape = (self.expert_fully_shard_parallel_size, self.expert_parallel_size,)
        self.expert_data_parallel_size = get_last_mesh_dim(mesh_shape)
        mesh_shape = (self.expert_data_parallel_size,) + mesh_shape
        self.add_device_mesh_groups(mesh_dim_names, mesh_shape)

        # create DP_FSDP
        mesh_dim_names = ('dp_fsdp', 'cp')
        mesh_shape = (self.fully_shard_parallel_size * self.data_parallel_size // self.context_parallel_size,
                      self.context_parallel_size)
        self.add_device_mesh_groups(mesh_dim_names, mesh_shape)

        if self.data_parallel_size != 1:
            raise Exception("Currently, only `data_parallel_size` = 1 is supported ...")
        if self.fully_shard_parallel_size != torch.distributed.get_world_size():
            raise Exception("Currently, `fully_shard_parallel_size` must be equal to the distributed world size.")

这段是 LLM 层并行初始化的核心。两个硬约束值得记住：① data_parallel_size != 1 抛异常（行 88）——fsdp2 当前不支持纯 DDP data-parallel，dp_size = world_size // (tp*fsdp*cp) 必须算成 1，即 tp*fsdp*cp == world_size；② fully_shard_parallel_size != world_size 抛异常（行 91）——FSDP 组必须铺满全部卡。合起来即 fsdp_size == world_size 且 tp_size==cp_size==1（Qwen3-8B 默认情形）。注意行 66 的注释 # create DP/CP/Ulysses/TP groups 与实际 ('dp','fsdp','tp') 三维不一致——这是从核心库 fork 时没改干净的痕迹（核实：注释措辞沿用上游，以代码元组为准）。

拿 Qwen3-8B 在 8 卡 A2 上跑（EP02/EP05 的默认场景）走一遍这个算术：world_size=8，tp_size=1、fsdp_size=8、cp_size=1。第一组 mesh 调 get_last_mesh_dim((fsdp_size=8, tp_size=1))：从 world_size=8 出发，先除以 8 得 1，再除以 1 得 1，所以 data_parallel_size=1；于是 mesh_shape=(1, 8, 1) 即 (dp, fsdp, tp)，init_device_mesh 在这个形状上建出 fsdp 轴对应的 8 卡 all-gather 组。EP 组那边 ep_size=1, ep_fsdp_size=1，get_last_mesh_dim((1,1)) 得 edp=8，mesh_shape=(8,1,1)=(edp,efsdp,ep)——但 Qwen3-8B dense 模型没有 expert，这组 mesh 实际不会被 apply_ep_modules 取用（expert_parallel_size>1 才进，见 engine 行 59）。第三组 ('dp_fsdp','cp')：fsdp_size * dp_size // cp_size = 8*1//1 = 8，cp_size=1，mesh_shape=(8,1)=(dp_fsdp,cp)——cp 轴长度 1，apply_cp_modules 行 73 的 cp_size>1 不满足，跳过。最终只有 fsdp 这一根轴真正激活，8 卡全走 FSDP 分片——这就是 Qwen3-8B 的并行拓扑全貌。

get_last_mesh_dim（行 27-34）的逻辑是：把 world_size 依次除以 mesh_shape 每一维，最后剩下的余项当作"未指定的最后一维"（即 dp）。所以 (fsdp_size, tp_size) 给进去，dp = world_size // fsdp_size // tp_size。这正是 model_factory 行 120 的 dp_size = world_size // (tp*fsdp*cp) 在 state 内部的等价表达——但 state 这里 cp 不在 fsdp mesh 里（cp 被拆到独立轴行 83），所以两边 cp 的处理方式不同，下面对照会讲清。

还要看一眼 add_device_mesh_groups（行 151-184）那个 setattr 机制——它是 parallel_state.get_fsdp_device_mesh() 这类调用的来源。函数体里定义了一个闭包 get_methods(name)，对每个 mesh_name 返回五个闭包方法（is_enable / get_group / get_size / get_rank / get_mesh，行 153-169），每个闭包捕获当前 name 再转调外层的 self.is_group_enable(name) 等通用方法。然后在行 180-184 用 setattr(self, 'is_{}_enable'.format(mesh_name), types.MethodType(...)) 把这五个方法动态挂到单例实例上，名字按 mesh_name 拼：fsdp → is_fsdp_enable / get_fsdp_group / get_fsdp_group_size / get_fsdp_rank / get_fsdp_device_mesh。所以你在 engine 里看到的 self.parallel_state.get_fsdp_device_mesh()（engine 行 50）不是手写方法，是 __post_init__ 建第一组 mesh 时 setattr 挂上去的。同理 tp/ep/efsdp/cp/dp_fsdp 各自一组。types.MethodType 把闭包绑成实例方法（带 self），否则闭包拿不到 self。这套"按 mesh_name 动态生成访问器"的写法两边逐字相同（LLM 行 151-184 / core 行 138-171），是 fork 共享祖先的又一证据。

注意一个容易看错的间接层：device_mesh_map（行 54）这个 dict 的 value 全是同一个 DeviceMesh 对象——行 178 的 self.device_mesh_map[mesh_name] = device_mesh 在循环里把刚建的整张 mesh 对每个 dim name 都存一遍。所以 device_mesh_map['fsdp']、device_mesh_map['dp']、device_mesh_map['tp'] 指向同一张 (dp,fsdp,tp) 三维 mesh，只是 key 不同。取轴时 get_device_mesh('fsdp')（行 145-149）返回的是 self.device_mesh_map['fsdp']['fsdp']——先拿整张 mesh（dict value），再 [mesh_name] 取该轴对应的子 DeviceMesh（PyTorch 的 DeviceMesh 支持按 dim name 索引）。get_group('fsdp')（行 127-131）则再进一步 .get_group('fsdp') 取该轴对应的 ProcessGroup。三层间接——map[name] → 整张 mesh → [name] 子 mesh → .get_group(name) ProcessGroup——读源码时别被绕晕：dict 存整张 mesh 的多份引用，按 key 取出的是整张，再按 dim name 切到轴，再 get_group 才是 torch 的 ProcessGroup。engine 的 apply_* 取的是中间那层（子 mesh / DeviceMesh），分片函数 fully_shard_parallel_modules 内部自己再 .get_group()。

另外两个查询方法也走同一 dict：is_group_enable(name)（行 121-125）判 get_group_size(name) > 1，get_rank(name)（行 139-143）调 device_mesh_map[name].get_local_rank(name) 取"本卡在该轴的局部 rank"。所以 get_fsdp_rank() = "我在 fsdp 组里是第几号"，get_tp_rank() = "我在 tp 组里是第几号"——这是 DDP 时代你手写 rank % dp_size 算的东西，现在 DeviceMesh 按 dim name 直接给。

MindSpeedParallelEngine：__init__ 拿 parallel_state 后六步 apply，forward 转调模型
class MindSpeedParallelEngine(torch.nn.Module):
    def __init__(self, config, model, init_device="cpu", weights_path=None, hf_config=None):
        super().__init__()
        self.config = config
        self.model = model
        self.init_device = init_device
        self.weights_path = weights_path
        self.hf_config = hf_config

        self.parallel_state = init_parallel_state(self.config)
        self.apply_quantization_modules()
        self.apply_tp_modules()
        self.apply_ep_modules()
        self.apply_cp_modules()
        self.apply_recompute_modules()
        self.apply_fsdp_modules()

        if self.init_device == "meta":
            logger.info_rank0("> Loading weights after FSDP wrapping...")
            WeightLoader.load(model=self.model, weights_path=self.weights_path, device=None, hf_config=self.hf_config)

    def apply_fsdp_modules(self):
        self.model = fully_shard_parallel_modules(
            self.model, self.parallel_state.get_fsdp_device_mesh(), self.config.fsdp_plan)

    def apply_cp_modules(self):
        VALID_CP_TYPES = ("ulysses", "ring")
        cp_size = self.config.context_parallel_size
        cp_type = self.config.context_parallel_type
        if cp_size > 1:
            if cp_type not in VALID_CP_TYPES:
                raise ValueError(...)
            if cp_type == "ulysses" and self.model.config.num_attention_heads % cp_size != 0:
                raise ValueError(...)
            apply_context_parallelize_modules(self.model, self.config.cp_plan)

    def forward(self, *args, **kwargs):
        return self.model(*args, **kwargs)

这段是 engine 的全貌。三个 LLM 层独有的点：① __init__ 多了 init_device（"cpu"/"meta"）、weights_path、hf_config 三参（行 20-26），并因此在 FSDP 包完后对 meta 设备触发 WeightLoader.load（行 44-46）——支持"先在 meta 上建空壳模型、FSDP 包完再灌权重"的延迟加载路径；② 多了 apply_cp_modules（行 68-78），校验 cp_type ∈ {ulysses, ring}、ulysses 下 num_attention_heads % cp_size == 0，再 apply_context_parallelize_modules；③ apply_ep_modules 把 expert_parallelize_modules 与 expert_fully_shard_modules 拆成两个独立 if（行 58-66），可只开其中一个。forward（行 107）只有一行 return self.model(*args, **kwargs)——engine 不改前向语义，所有并行都靠 __init__ 里那六步 apply 把模型结构改完了。

小结　心智模型：fsdp2 后端的并行初始化是在 torch.distributed 之上叠两层——第一层 DeviceMesh（ParallelState 单例，__post_init__ 里 init_process_group(backend='hccl') + 三组 init_device_mesh：dense (dp,fsdp,tp)、expert (edp,efsdp,ep)、CP (dp_fsdp,cp)，并 setattr 动态挂 get_*_device_mesh 方法）；第二层 MindSpeedParallelEngine（nn.Module，__init__ 里 init_parallel_state 拿单例后六步 apply_quant→tp→ep→cp→recompute→fsdp，forward 纯转调）。入口是 model_factory.py 算出 dp_size=world_size//(tp*fsdp*cp)、装 ParallelEngineConfig、实例化 engine。LLM 层 mindspeed_llm/fsdp2/distributed/ 与核心库 mindspeed/fsdp/ 的同名 parallel_state.py / mindspeed_parallel_engine.py 是同源 fork + 差异化改写，不是薄包装也不是子类——辅助方法逐字相同，但 LLM 版删 ulysses、CP 拆独立轴、加两条硬约束（dp=1、fsdp=world）、device_type 自适应、engine 多出 apply_cp 与 meta 加载、并把分片实现换成 fsdp_turbo 包。读这层的姿势是"抓 mesh 形状 + apply 顺序 + 五处 fork 分叉"：mesh 决定通信组、apply 顺序决定包壳层次、fork 分叉决定 fsdp2 与 mcore 的并行能力边界。
常见误区：① 以为还是裸 DDP group——其上是 DeviceMesh + 自定义 engine（即"它不是裸 DDP group——其上是 DeviceMesh + 自定义 engine"）：fsdp2 不再 new_group 手切子组、不再 DDP(model, process_group=...)，而是 ParallelState 建三组 DeviceMesh、MindSpeedParallelEngine 在 __init__ 里六步 apply 改模型结构，forward 只转调。别拿 DDP 的"group 句柄到处传"心智来读这套。② 以为 LLM 层 engine 是核心库 engine 的子类/薄包装——不是：两边都 class MindSpeedParallelEngine(torch.nn.Module) 互不继承，LLM 版从 fsdp_turbo 导入分片实现、core 版从 mindspeed.fsdp 导入，是 fork 后各自改写。"在核心库之上叠加"≠ import 核心库 engine，而是 fork 一份定制。③ 以为 data_parallel_size 可以 >1——fsdp2 当前硬约束 data_parallel_size==1（行 88），dp_size=world_size//(tp*fsdp*cp) 必须算成 1；想纯 data-parallel 得等后续版本。Qwen3-8B 默认就是 fsdp_size==world_size、tp=cp=1，即 8 卡全走 FSDP。④ 以为 1D 与 2D DeviceMesh 只是图好看——dense 模型（Qwen3-8B）只有 fsdp 一根轴（ep_size=1 退化），MoE（qwen3_moe/30B+）才把 fsdp 与 ep 拆成两根正交轴，get_ep_device_mesh()['ep'] 横向取 expert 组、get_efsdp_device_mesh()['fsdp'] 纵向取分片组。⑤ 以为 apply 顺序可调——固定 quant→tp→ep→cp→recompute→fsdp，FSDP 必须最后包（最外层），否则前面切的模块边界会被 FSDP 重写；行 36-41 的顺序就是契约。⑥ 以为行 66 注释 # create DP/CP/Ulysses/TP groups 描述了实际 mesh——注释与 ('dp','fsdp','tp') 三维元组不一致，是 fork 没改干净的痕迹，以代码元组为准（核实：注释措辞沿用上游核心库）。
