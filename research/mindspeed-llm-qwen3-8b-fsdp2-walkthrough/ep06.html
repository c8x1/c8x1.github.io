参数体系精读

目标：掌握五段 dataclass 即全部可调旋钮。前置：EP05 入口与启动链路。

你将获得：① 一张能从 --parallel.fsdp_modules 这类 --段.字段 CLI 一路对应到 field(default=…) 声明、再到被各工厂消费的完整参数地图 ② 弄懂为什么 fsdp2 只用五段（model/data/parallel/training/optimization）而不是 mcore 那数百项——以及 model.layers.{*} 这种模块路径通配符到底匹配什么

你已经熟悉的情况 桥接开场

你写训练脚本时配参数大概都走过这条路：parser = argparse.ArgumentParser()，然后一行行 parser.add_argument("--lr", type=float, default=1e-5)，最后 args = parser.parse_args() 拿到一个 Namespace，到处 args.lr 取值。Namespace 是个扁平对象，lr / batch_size / output_dir 全堆在一个平面上，--lr 1e-5 就 args.lr，简单直白。这种写法的问题是：参数一多就糊成一团，没有分类，add_argument 写到第三十个就分不清哪些是模型相关、哪些是训练相关。

MindSpeed-LLM 的 fsdp2 后端把这套换成了 HfArgumentParser 风格——它不再 add_argument 一个个加，而是先写五个 @dataclass：ModelArguments / DataArguments / ParallelArguments / TrainingArguments / OptimizationArguments，每个字段用 field(default=…, metadata={"help": "…"}) 声明类型与默认值；然后 fsdp2_parse_args 用反射（dataclasses.fields + get_type_hints）把这几个 dataclass 动态展开成一个 argparse.ArgumentParser，每个字段注册成 --段.字段 形式的参数（如 --parallel.fsdp_size）。解析后再 _build_dataclass_instances 把 Namespace 反射回五个 dataclass 实例。这一篇就把这五个 dataclass 按段逐字段拆给你看——参数再多，也就这五段。

本项目怎么做 原理

fsdp2 的参数哲学是"dataclass 即 schema、五段即全部旋钮、CLI 键名带段前缀"。五个 dataclass 各管一摊：ModelArguments 管"加载什么模型"（路径、是否从零训、tokenizer、量化配方）；DataArguments 管"喂什么数据"（模板、cutoff、数据集、packing）；ParallelArguments 管"怎么并行"（FSDP/TP/EP/CP 大小 + 分片模块清单）；TrainingArguments 管"怎么训"（lr/scheduler/batch/save/profile）；OptimizationArguments 管"用哪些 NPU 融合算子"（fused rmsnorm / fused rotary / flash attn）。五个 dataclass 在顶层被包进一个根 dataclass，fsdp2_parse_args 反射展开后，MindSpeedAutoTrainer 把它们具名注入下游各工厂——model 工厂读 model_args、parallel 工厂读 parallel_args、optimizer/scheduler 工厂读 training_args。下面的关系图画清了这条"五段 → 聚合 → 工厂消费"的数据流。

这张图的关键是中间那个 root Arguments 聚合点：五个 dataclass 不是一个扁平 Namespace，而是各成一段被装进根 dataclass；下游每个工厂只取自己那一段，互不串味。底部的 --parallel.fsdp_size 8 提示了 CLI 键名永远带段前缀——这也是为什么你 grep 不到裸 --lr，因为它叫 --training.lr。

见 ../mindspeed_llm/fsdp2/utils/arguments.py：:389（fsdp_modules）、:415（ep_modules）、:432（recompute_modules）、:133（quant_apply_modules）；文档 ../docs/zh/pytorch/features/fsdp2/arguments.md::438（默认值写法 model.layers.{}）

这四处都用了 model.layers.{*} 这个写法，含义需要单独说清：

  • 它不是 Python 通配符，是模块路径模式串。model.layers.{*} 描述的是 HuggingFace 模型里 model.model.layers.0 / model.model.layers.1 / ... / model.model.layers.N-1 这一串带数字下标的子模块。{*} 是个占位符，表示"匹配所有数字下标层"——运行时被替换展开成一层一层的 module 对象。文档（arguments.md:438）把它写成 model.layers.{}（占位写法略异，语义一致）。
  • 与之对比的是 *. 前缀通配。tp_colwise（行 407）用 ['*.q_proj', '*.k_proj', ...]——这里的 * 是前缀通配，匹配"任意父模块下的 q_proj"。两套写法语义不同：model.layers.{*} 是枚举数字层（要展开成 0/1/2…），*.q_proj 是按名字后缀匹配（不关心是第几层）。EP08 分片落地会讲运行时如何把它们解析成真实 nn.Module。
  • 下钻语法：model.layers.{*}.mlp.experts（ep_modules，行 415）表示先展开数字层、再沿 .mlp.experts 路径下钻——即"每一层 transformer 的 mlp 下的 experts 子模块"。这为 MoE 专家分片提供了精确到子模块的钩子。
  • 为什么 fsdp_modules 默认要列三个：model.layers.{*}（所有层）、model.embed_tokens（embedding，参数量大）、lm_head（输出头，参数量也大）——这三类是参数占比最大的模块，FSDP 分片优先切它们收益最大。其余小模块留给 ignored_modules（行 393）排除。这套清单就是 EP08 真正"切谁不切谁"的依据。

def fsdp2_parse_args(rootclass: TypeVar) -> TypeVar:
    parser = _create_argument_parser(rootclass)          # 反射展开五段成 --段.字段
    cmd_args = sys.argv[1:]
    yaml_config = _load_yaml_if_provided(cmd_args)      # 第一个 .yaml 位置参数
    final_cmd_args = _merge_yaml_into_cmd_args(cmd_args, yaml_config)  # CLI 优先
    args, remaining_args = parser.parse_known_args(final_cmd_args)
    parse_result = _postprocess_json_fields(args, parser.dict_fields)  # dict/JSON 反序列化
    result = _build_dataclass_instances(rootclass, parse_result)       # 反射回五个实例
    _validate_cross_args(result)                        # 跨段约束校验
    return result

小结　心智模型：fsdp2 后端的全部可调旋钮就是五段 dataclass——ModelArguments（加载什么）、DataArguments（喂什么）、ParallelArguments（怎么并行 + 分片模块清单）、TrainingArguments（怎么训 + NPU profiling）、OptimizationArguments（用哪些融合算子）。它们经 fsdp2_parse_args 反射展开成 --段.字段 CLI、合参后反射回五个实例、再被 MindSpeedAutoTrainer 具名注入各工厂。读这套参数的姿势是"按段读 + 抓清单"：五段各管一摊，每段内最该盯的是 List[str] 形式的模块路径清单（fsdp_modules / ep_modules / recompute_modules / quant_apply_modules）——它们用 model.layers.{*} 通配数字层、用 *.q_proj 匹配后缀，是 EP08 分片落地的真正依据。
常见误区：① 参数与 mcore 不通用——独立体系：fsdp2 这五段 dataclass 是为 FSDP2 后端专门设计的，与 mcore（Megatron-Core）那数百个 --num-layers / --lr-warmup-fraction CLI 不是一套；docs arguments.md:992–1076 虽列了对应表，但 fsdp2 不直接吃 mcore 的键名，二者只在 data_manager_type="mg" 时共享一小撮数据段旋钮。别拿 mcore 的 --lr 往 fsdp2 上套，这里叫 --training.lr。② 以为 model.layers.{*} 是正则通配——它不是 re，是 fsdp2 自定义的模块路径模式串，{*} 展开数字层下标、* 作前缀/后缀匹配，运行时由分片逻辑解析（EP08）。③ 以为不设 fsdp_size 也会分片——fsdp_size 默认 1，不分片；fsdp_modules 清单写了也只在 fsdp_size>1 时真正生效。④ 以为各段校验互不影响——段内 __post_init__ 管自己，跨段冲突（reset_attention_mask×batch、qwen3_next×cp_type、DSA×model_id）由 _validate_cross_args 兜底，五段都建好才验。⑤ 以为 OptimizationArguments 是推理参数——docstring 写 "Inference hyperparameters" 但字段全是训练融合算子开关，以字段 help 为准（核实：疑似 docstring 从 InferenceArguments 复制未改）。
