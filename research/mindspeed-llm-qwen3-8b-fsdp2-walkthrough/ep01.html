你已经熟悉的情况 桥接开场

作为算法工程师，你大概率已经习惯这样的世界：一块 NVIDIA GPU、一个 PyTorch 进程，用 torch.cuda 管显存、torch.distributed 配 NCCL 跑 DDP——每张卡一份完整模型副本、各自算一份 Batch、梯度 All-Reduce 对齐。模型再大就上 FSDP1 把参数打平成一个大向量切分，或叠一层 Megatron 的 TP/mcore。这套心智已经足够你在单机或小集群里跑通大多数训练。

本项目要带你走的，是从这条"左岸"出发，跨过三座桥，站到另一侧：华为 Ascend 910B NPU 上、用 FSDP2 后端、跑在多机多卡 DeviceMesh 拓扑里的 Qwen3-8B 训练链路。这条链路的入口就在仓库根目录的 train_fsdp2.py——它在 README 目录树里被标注为"FSDP2 训练流程"（README.md:96），而在文件第一段 import 里就出现了 torch_npu 与 transfer_to_npu（train_fsdp2.py:8 起）——三座桥的右岸，从入口文件就能一眼看到影子。本篇不深入任一座桥——那是 EP02–EP18 的事——只把地图画清楚，让你在后续每一篇都能回来对一眼"我现在站在哪座桥上"。

本项目怎么做 原理

本系列的脊柱是三座桥。每一座桥的左端是你已经熟悉的 PyTorch/GPU 单卡世界，右端是本项目实际运行的 NPU/FSDP2 多机世界。三座桥并非独立——它们叠加在一起，才构成"在 Ascend 上用 FSDP2 训 Qwen3-8B"这件事的全部坐标。

注意三座桥的叠加关系：右岸不是三个独立终点，而是同一套运行时的三个侧面——你最终跑的是"②的硬件上 + ③的拓扑里 + ①的分片范式下"的同一个训练循环。换一种说法：①决定"参数怎么切、梯度怎么同步"，②决定"这些切分与同步在哪种芯片、用哪套通信库上发生"，③决定"这一切跨几张卡、跨几台机、按什么拓扑组织"。三者只要有一个对不上，链路就跑不通或跑不出预期显存/吞吐。这就是为什么本系列要先花 EP01–03 把地图立起来，再按篇章逐层下钻。

docs/zh/pytorch/features/fsdp2/fsdp2_basic_features.md 开篇定义——桥①右岸"FSDP2"到底是什么

# PyTorch FSDP2 后端特性介绍
FSDP2 是 PyTorch 分布式并行的下一代范式 ... 它不再通过 Python 类包装模型，
而是通过 torch.distributed.fsdp.fully_shard API 对模型进行原位（In-place）的并行化处理。

docs/zh/pytorch/features/fsdp2/fsdp2_basic_features.md:1（In-place API 见同文件第 3 行）

把这段当本系列的术语锚点：FSDP2 不是"把 FSDP1 升级一下"，而是换了一条用 fully_shard 原位分片、以 DTensor 为底层基石的新范式（DTensor 见 docs/zh/pytorch/features/fsdp2/fsdp2_basic_features.md:36）。EP03 会把 FSDP1→FSDP2 的范式跃迁讲透。

有了这三段事实，三座桥的右岸就不再是抽象名词：①的右岸是 fully_shard+DTensor，②的右岸是 torch_npu+HCCL，③的右岸是 DeviceMesh 多维拓扑。本篇到此为止——入口文件 train_fsdp2.py 内部如何把这三者串成主循环，是 EP05 的下钻范围。

小结　心智模型：三座桥即学习路径。把"在 Ascend 上用 FSDP2 训 Qwen3-8B"拆成三条可独立理解、又能叠加回同一运行时的迁移轴——并行范式 / 硬件运行时 / 规模拓扑。每读一篇 EP，先问"它在过哪座桥"，再看"左右两岸各自对应什么"。三座桥的右岸加起来，才是本项目真正的运行时全貌，任何一座单独看都会失真。
常见误区：把 FSDP2 当成 Megatron mcore 的"升级版"或"FSDP1 的下一个版本号"。实际上 FSDP2 是 PyTorch 上游的另一套后端范式（fully_shard 原位 API + DTensor），与 mcore 的 FlatParameter 包装器是并行的两套思路，不是同一棵树的版本递进。本系列里 Qwen3-8B 走的是 FSDP2 这条线，而非 mcore 那条线。
