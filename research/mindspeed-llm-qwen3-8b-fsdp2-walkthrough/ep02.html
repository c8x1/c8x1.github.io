硬件与运行时：GPU vs NPU(910B)

目标：建立 CUDA 栈↔CANN/NPU 栈的分层对照。前置：EP01 系列导览。

你将获得：① 一张把 GPU 四层栈逐层映到 NPU(910B) 四层栈的分层对照图 ② env_config.sh 每个环境变量在 NPU 侧的真实语义（含哪些是 GPU 遗留名、需核实）

你已经熟悉的情况 桥接开场

在 GPU 世界里，你的运行时栈大约是四层：最底是 驱动（NVIDIA driver + 固件），之上是 CUDA Runtime/cuBLAS/cuDNN（计算库），通信走 NCCL，框架层 PyTorch 通过 torch.cuda 调度——混合精度用 torch.cuda.amp，显存用 cudaMalloc，调试靠 CUDA_LAUNCH_BLOCKING。这套心智高度自洽，以至于你几乎意识不到它"分层"。

本篇要做的，是把这四层逐一映到 Ascend 910B 那一侧：驱动→Ascend driver/固件，CUDA/cuDNN→CANN（含 Ascend 算子库），NCCL→HCCL，PyTorch→torch_npu（一个让 torch.cuda 的调用落到 NPU 的适配层）。关键认知：NPU 不是"换了张显卡"，而是另一套同构但不同源的栈——同一个 PyTorch 算子在两侧可能由完全不同的库实现，环境变量名也常常是 GPU 遗留名被 NPU 侧"沿用或忽略"。EP01 的桥②"torch.cuda → torch.npu"，本篇就把它走到底。

本项目怎么做 原理

把两张栈并排画出来，按"驱动 / 计算 / 通信 / 框架"四层对齐。左列是你熟的 GPU 栈，右列是本项目 910B 上的 NPU 栈；中间一列箭头标"对应关系"，注意对应不等于"同一套东西换个名字"——而是"承担同等职责、但实现与命名都换了"。

图 2.1　GPU 栈 vs NPU 栈 分层对照：框架（PyTorch↔torch_npu）、通信（NCCL↔HCCL）、计算/算子（CUDA/cuDNN/cuBLAS↔CANN/Ascend 算子）、驱动（NVIDIA driver↔Ascend driver）。箭头表示职责对应，不表示同名同源——很多 GPU 环境变量名在 NPU 侧只是"沿用外壳"。

这张图把 EP01 桥②的右岸拆成了四块砖。本系列后续每一篇遇到"某算子/某通信/某显存操作"时，都可以回到这里问一句"它落在哪一层、对应 GPU 哪个东西"。下面进入源码，看一个具体文件把这四层栈的旋钮都摆在哪。

源码下钻 file:line

本篇主要走读 examples/fsdp2/env_config.sh——这是 FSDP2 后端的公共环境变量文件，每个变量都是某层栈的一个旋钮。整份文件就 9 行，逐个看。

examples/fsdp2/env_config.sh 全文——FSDP2 后端的公共环境变量旋钮

export TRAINING_BACKEND=mindspeed_fsdp
export HCCL_CONNECT_TIMEOUT=1800
export TASK_QUEUE_ENABLE=2
export CPU_AFFINITY_CONF=1
export MULTI_STREAM_MEMORY_REUSE=2
export PYTORCH_NPU_ALLOC_CONF=expandable_segments:True
export TORCH_COMPILE_DEBUG=0
export CUBLAS_WORKSPACE_CONFIG=:4096:8

examples/fsdp2/env_config.sh:1（全文见 :2–:9）

逐行：① 后端选择——本项目走 mindspeed 的 FSDP2 实现，而非 torch 原生 fully_shard

export TRAINING_BACKEND=mindspeed_fsdp   # FSDP2 后端在 MindSpeed 侧的实现标识

examples/fsdp2/env_config.sh:2

TRAINING_BACKEND=mindspeed_fsdp 是栈框架层的旋钮：它告诉训练脚本用 MindSpeed 提供的 FSDP2 后端（与 EP01 提到的 train_fsdp2.py 配套），而非 PyTorch 上游裸跑的 fully_shard。对应到栈图，这一行的"作用点"在 torch_npu 与 MindSpeed 加速库之间（install_guide.md:196–205 安装 MindSpeed 加速库）。

逐行：② 通信层——HCCL 建链超时拉到 1800 秒

export HCCL_CONNECT_TIMEOUT=1800   # HCCL（NPU 的 NCCL）建链超时

examples/fsdp2/env_config.sh:3

这是栈通信层的旋钮，也是图 2.1 第二行"NCCL↔HCCL"的一个具体落点：HCCL_* 前缀就是 NPU 侧对应 NCCL 调优变量的命名族。1800 秒（30 分钟）建链超时，是为大规模多机拓扑下首帧建链留足余量——GPU 侧你更熟悉的是 NCCL_*_TIMEOUT。

逐行：③④⑤⑥ NPU 侧 CANN/显存/调度的四把旋钮（均为 torch_npu/CANN 专有前缀）

export TASK_QUEUE_ENABLE=2                  # CANN 任务队列调度
export CPU_AFFINITY_CONF=1                  # NPU 侧 CPU 亲和性绑定
export MULTI_STREAM_MEMORY_REUSE=2          # 多流显存复用
export PYTORCH_NPU_ALLOC_CONF=expandable_segments:True   # NPU 显存分配策略

examples/fsdp2/env_config.sh:4（续 :5 :6 :7）

这四行是栈计算层 + 显存语义的旋钮。TASK_QUEUE_*、MULTI_STREAM_* 是 CANN/Ascend 侧的概念（任务队列、多流复用），GPU 侧无直接同名对应；PYTORCH_NPU_ALLOC_CONF 的命名则直接对应 GPU 侧的 PYTORCH_CUDA_ALLOC_CONF——这是 NPU 借用 GPU 显存分配器接口语义的典型：把 cudaMalloc 那套"缓存分配 + 段扩展"照搬过来，expandable_segments:True 即启用可扩展段分配以缓解显存碎片（语义与 CUDA 侧同名选项一致）。

逐行：⑦ 编译调试——关掉 torch.compile 调试输出

export TORCH_COMPILE_DEBUG=0   # torch.compile 调试日志关闭

examples/fsdp2/env_config.sh:8

这一行不是 NPU 专有：TORCH_COMPILE_DEBUG 是 PyTorch 上游变量，两侧通用，对应栈框架层（torch.compile 在 NPU 侧后端是 Triton-Ascend，见 install_guide.md:182–190）。

逐行：⑧ 一个"GPU 遗留名"变量——需核实其在 NPU 侧是否仍被读取

export CUBLAS_WORKSPACE_CONFIG=:4096:8   # GPU cuBLAS 工作区配置（遗留变量名）

examples/fsdp2/env_config.sh:9

（核实：NPU 侧是否仍读此变量，最终以 CANN 文档/源码为准）　CUBLAS_WORKSPACE_CONFIG 是 GPU cuBLAS 的工作区配置变量，变量名带 CUBLAS_ 前缀，理论上属于图 2.1 第三行左列（CUDA/cuBLAS）。它出现在 NPU 训练脚本里，有两种可能：一是 torch_npu/CANN 兼容性地读取同名变量以配 NPU 侧 BLAS 工作区；二是脚本从 GPU 模板沿用、NPU 侧实际忽略。本篇不臆断其 NPU 效果，留作核实项。凡 NPU 侧出现 GPU 命名前缀（CUBLAS_ / CUDA_* 等）的变量，均按此处理。

docs/zh/pytorch/models/supported_models.md 硬件要求——单 NPU 片上内存 ≥ 64GB

仓库支持 Ascend 950、Atlas A3、Atlas A2 训练系列产品 ... 要求单 NPU 的片上内存为 64GB 及以上

docs/zh/pytorch/models/supported_models.md:21

这是 NPU 侧显存语义的硬约束：不是"装得下模型权重"那么简单——FSDP2 分片后的参数、梯度、优化器状态、激活、通信 buffer 都要落在单卡 64GB 这块片上内存里（与 GPU 侧 HBM 同概念，但叫法与边界不同）。Qwen3-8B 在 FSDP2 下走 1x16 拓扑（supported_models.md:777–778，【Test】），恰好把 8B 模型分摊到 16 张卡，单卡显存压力才落进 64GB 预算。

这一段把"框架层 PyTorch→torch_npu"的桥②落到了安装动作：torch_npu-2.7.1rc1 就是图 2.1 右列"torch_npu"那块砖的来源；CANN 环境（:147–161 的 set_env.sh）则是"计算层 CANN/Ascend 算子"那块砖被加载进运行时的方式——GPU 侧你不需要这一步，因为 CUDA runtime 跟驱动一起装好了；NPU 侧必须 source set_env.sh，否则"业务代码将无法执行"（install_guide.md:149）。这是两套栈最直观的运行时差异之一。

把这八行变量和三段安装事实叠到图 2.1 上：框架层有 TRAINING_BACKEND / TORCH_COMPILE_DEBUG / torch_npu whl；通信层有 HCCL_CONNECT_TIMEOUT；计算层有 CANN set_env.sh + TASK_QUEUE_* / MULTI_STREAM_*（以及存疑的 CUBLAS_WORKSPACE_CONFIG）；显存语义有 PYTORCH_NPU_ALLOC_CONF 与单卡 64GB 硬约束。同一个文件的九行，正好把四层栈的旋钮都点了一遍——这就是 EP01 桥②的"可落地处"。

小结　心智模型：NPU 不是"换了张显卡"，是另一套同构但不同源的栈。把 GPU 四层栈（驱动 / CUDA+cuDNN+cuBLAS / NCCL / PyTorch）逐层映到 NPU 四层栈（Ascend driver / CANN+Ascend 算子 / HCCL / torch_npu），就完成了 EP01 桥②的过桥。关键在于"职责对应"而非"同名同源"：同一个 PyTorch 算子在两侧由不同库实现、同一个环境变量名可能只是 GPU 遗留外壳（如 CUBLAS_WORKSPACE_CONFIG），而 NPU 专有旋钮（HCCL_* / PYTORCH_NPU_ALLOC_CONF / TASK_QUEUE_*）才是真正在拧 910B 那侧的螺丝。环境配置这一层如此，算子与通信原语层只会更彻底。
常见误区：以为"拿一份 GPU 上跑通的 PyTorch 训练脚本，换上 torch_npu 就能直接跑"。实际上：(1) 必须先 source CANN set_env.sh，否则算子库根本没加载（install_guide.md:149）；(2) torch.cuda.* 的调用需经 torch_npu 适配（EP01 已见 transfer_to_npu），并非零成本；(3) GPU 遗留变量名（CUBLAS_* 等）在 NPU 侧的实际效果未经核实，不能照搬 GPU 的语义断言。
