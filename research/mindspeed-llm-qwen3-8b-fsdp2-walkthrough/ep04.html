项目全景与目录地图

目标：建立 fsdp2 后端的模块依赖地图。前置：EP03 并行范式。

你将获得：① 一张能说清"train_fsdp2.py 到底依赖哪些工厂与子模块"的全景图 ② 区分 fsdp2 后端与 mcore 后端、以及它对外部 MindSpeed Core 仓库的依赖边界

你已经熟悉的情况 桥接开场

你大概写过或读过的那种"小项目"长这样：根目录一个 train.py、一个 model.py，再加个 dataset.py 就齐活了——if __name__ == "__main__" 里直接 model = MyModel()、optim = Adam(...)、for batch in loader: loss.backward(); optim.step()。整个工程的结构就是文件名本身，依赖关系一眼到底。

而 MindSpeed-LLM 的 fsdp2 路线不是这种"平铺"工程。它是一个面向多模型（Qwen3、DeepSeek、GLM5.2、Mamba3…）、多并行策略（TP/PP/CP/EP/SP…）、多后端的训练框架——同一个 train_fsdp2.py 入口要能根据命令行参数，装配出"模型 + 数据 + 优化器 + 调度器 + 检查点 + 训练器"这一整套组合，而不是写死任何一条路径。所以它的目录不是"几个脚本"，而是一组按职责切分的子包 + 一层"工厂（Factory）"胶水。这一篇就是先把这张地图摊开给你看。

本项目怎么做 原理

fsdp2 后端的组织原则是工厂装配式：入口脚本 train_fsdp2.py 自己几乎不写训练逻辑，它只负责"按参数去工厂里取零件，再把零件交给 Trainer 组装运行"。各零件由 mindspeed_llm/fsdp2/ 下的子包提供，每个子包对应一个职责域；而真正和昇腾硬件、和 PyTorch FSDP2 机制打交道的那一层（HCCL 通信补丁、随机种子、分片原语）则下沉到外部依赖 mindspeed（即 MindSpeed Core 仓库）里，fsdp2 后端只通过 from mindspeed.fsdp.utils.* import ... 来借用。

下面的依赖图把这条链路画清楚了：从入口脚本出发，经六类工厂/管理器，落到 fsdp2 包的十个子模块，最终触达 MindSpeed Core。

这张图只标"谁依赖谁"的方向，不展开各工厂内部实现——那是 EP07 起逐模块下钻的内容。

逐块拆：

  • 设备桥（行 8–9）：import torch_npu 把"昇腾 NPU 后端"注册进 PyTorch；from torch_npu.contrib import transfer_to_npu 则把默认张量设备从 CUDA 切到 NPU——之后代码里写 .to("npu") 或干脆不写设备都默认落在 NPU 上。这一段是所有 NPU 脚本的固定开场，不是 fsdp2 专属。被 try/except ImportError 包住，意味着这份脚本在纯 CUDA 环境也能 import（只是跑不起来），方便做静态分析/单测。
  • 六大装配件（行 14–19）：六行 from mindspeed_llm.fsdp2.* import 正好对应依赖图 L2 的六个格子——ModelFactory（行 14）、OptimizerFactory（行 15）、SchedulerFactory（行 16）、CheckpointManager（行 17）、Trainer（行 18）、DataFactory（行 19）。入口脚本靠这六个名字 + 参数就能拼出一条完整训练链，而不用 if model_name == "qwen3": ... elif ... 的硬编码分支。注意 Trainer 不叫 Factory——它是"已装配好的零件被送进去运转"的那个引擎，其余五个才是"造零件"的工厂。
  • 数据与参数配套（行 20–34）：TokenizerFactory（行 20）、get_template_and_fix_tokenizer（行 21）归 data 子包管；arguments（行 23–30）把所有命令行参数拆成五类 dataclass（Model/Data/Parallel/Training/Optimization），这是 EP06 的重头戏；logging/global_vars/train_monitor/device/coverage（行 22、31–34）是 utils 子包提供的横切设施。
  • 外部核心库借用（行 36–37）：这两行是跨仓库依赖——from mindspeed.fsdp.utils.random import set_seed 与 from mindspeed.fsdp.utils.torch_patch import apply_hccl_premul_sum_patch。这里 mindspeed 不是本仓库（mindspeed_llm），而是另一个独立仓库 MindSpeed Core，pip 装好后以 mindspeed 为包名存在。set_seed 管可复现性，apply_hccl_premul_sum_patch 是给 HCCL（昇腾集合通信库，对应 GPU 的 NCCL）打"premul-sum"补丁——EP03 讲过 CP/SP 这类对通信原语有特殊语义，这个补丁就是让 FSDP2 的 reduce-scatter 在昇腾上行为正确。fsdp2 后端不自己实现这些，只借用。

这段 README 明确了分层：MindSpeed Core 是"昇腾大模型加速库"底座（mindspeed/ 核心代码目录），在其之上才叠了 MindSpeed-LLM（大语言模型套件，即本仓库）。fsdp2 后端的 from mindspeed.fsdp.utils.* 就是这个分层在 import 层的体现——底层能力（HCCL 补丁、种子、FSDP2 原语）在 Core，套件层只负责"模型/数据/训练流程"的装配。这也解释了为什么本仓库还放了一份 _deps_mindspeed_core/ 镜像：走读时要随时能跳进 Core 源码核对外部 API。

GPU / 已知（小项目范式）

入口单文件 train.py：main 里直接 new 模型/优化器/数据
结构平铺几个脚本，依赖关系＝文件名顺序，无工厂
启动python train.py；多卡用 torchrun --nproc_per_node …
通信后端torchrun 默认 backend=nccl（NVIDIA 集合通信库）
设备.to("cuda")，torch.cuda 默认张量设备

NPU / 本项目（fsdp2 后端）

入口train_fsdp2.py：DI 容器式装配，按参数向工厂取件
结构十个子包 + 六类 Factory/Manager，职责按目录切
启动仍是 torchrun（脚本同形），但进程组 backend=mindspeed_fsdp
通信后端mindspeed_fsdp 经 HCCL；需 apply_hccl_premul_sum_patch 修正语义
设备transfer_to_npu 把默认设备切到 "npu"，torch_npu 注册后端

一句话定位 mcore vs fsdp2：本仓库同时维护两条后端——mcore（Megatron-Core 路线，靠 mindspeed_llm/tasks/ 那套 Megatron 训练栈）与 fsdp2（PyTorch 原生 FSDP2 路线，靠本篇讲的 mindspeed_llm/fsdp2/ 子包 + train_fsdp2.py 入口）。两者是独立后端，不是同一套代码加个开关。本系列只走 fsdp2，mcore 略过。

小结　心智模型：fsdp2 后端是一个"工厂装配式"训练框架——入口脚本只取件不写逻辑，十个子包按职责供件，硬件/FSDP2 原语下沉到外部 MindSpeed Core；Qwen3-8B 是 models/qwen3/ 这一个供件格里的具体型号。读这一路代码的姿势是"顺着 import 找工厂，再进工厂找实现"，而不是从头到尾线性读 train_fsdp2.py。
常见误区：① 别把 fsdp2 当 mcore 的配置分支——它是独立后端，有自己的入口、子包、参数体系，和 mcore 共用的只是"同属一个仓库"这层外壳。② 别以为 mindspeed.fsdp.* 在本仓库里——它在另一个仓库 MindSpeed Core，本仓库只是镜像了一份到 _deps_mindspeed_core/ 方便走读。③ 别把"六个 Factory"当成六个并列模块——Trainer 是消费者，其余五个才是生产者，CheckpointManager 既是生产（装配）也是消费（运行时存读）。
