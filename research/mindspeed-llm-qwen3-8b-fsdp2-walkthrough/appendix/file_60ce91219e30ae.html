class Qwen3RMSNorm(nn.Module):              # HF transformers 原版
    def forward(self, hidden_states):
        input_dtype = hidden_states.dtype
        hidden_states = hidden_states.to(torch.float32)
        variance = hidden_states.pow(2).mean(-1, keepdim=True)
        hidden_states = hidden_states * torch.rsqrt(variance + self.variance_epsilon)
        return self.weight * hidden_states.to(input_dtype)
差异讲解

左侧是 HF 上游 Qwen3RMSNorm.forward 在 NVIDIA GPU 上的标准实现：upcast 到 fp32 → pow(2) → mean(-1) → rsqrt → 乘权重 → downcast，五步、多个中间张量（variance、rsqrt 结果），每个中间张量都是一次显存访存。右侧 patch 后：npu_rms_norm(x, w, epsilon=eps) 把这五步融成单个 CANN 算子，一次 kernel launch、中间 fp32 不物化、权重乘法内置。差异本质：数学等价（数值上可对拍），省的是访存与 launch 开销；NPU 上 CANN 对这种 elementwise+reduce 融合有专门优化。对照 NVIDIA 侧若想要同等融合，得靠 torch.compile 或 TransformerEngine 的 fused RMSNorm，PyTorch eager 不会自动融。这也解释了为何 patch 受 use_fused_rmsnorm 开关保护——关掉即回左侧等价但更慢的路径，便于跨厂商数值对拍。

def rotate_half(x):                       # HF helpers
    x1 = x[..., : x.shape[-1] // 2]
    x2 = x[..., x.shape[-1] // 2 :]
    return torch.cat((-x2, x1), dim=-1)

def apply_rotary_pos_emb(q, k, cos, sin, position_ids=None, unsqueeze_dim=1):
    cos = cos.unsqueeze(unsqueeze_dim)
    sin = sin.unsqueeze(unsqueeze_dim)
    q_embed = (q * cos) + (rotate_half(q) * sin)
    k_embed = (k * cos) + (rotate_half(k) * sin)
    return q_embed, k_embed
差异讲解

左侧是 HF 上游 apply_rotary_pos_emb 在 GPU 上的标准实现：rotate_half（concat-flip 拆半取反）+ (q*cos) + (rotate_half(q)*sin) 的逐元素乘加，q 和 k 各跑一遍，多个中间张量（rotate_half 结果、两次乘法、一次加法）。右侧 patch 后：npu_rotary_mul(q, cos, sin) 把“rotate_half + cos 乘 + sin 乘 + 加”融成单个 CANN 算子，q/k 各一次 kernel launch。差异本质：数学等价（标准 RoPE），省的是 rotate_half 的 concat-flip 中间张量与多次逐元素访存；NPU CANN 对 RoPE 这种固定模式有专门融合内核。对照 NVIDIA 侧：HF 的 eager 实现就是左侧这样手写，要融合得靠 FlashAttention 内置的 RoPE 或 torch.compile；MindSpeed 这里用 patch 强制走融合内核，且对调用方透明（签名、返回值结构都不变，只是内部 mul 换了内核）。position_ids 哑参数两侧一致——HF 原版签名也带但实现里同样未用于算 cos/sin，MindSpeed 是照抄签名以兼容 patch 机制。

左侧是 HF 上游 Qwen3MLP.forward 在 GPU 上的标准实现：gate/up 两次独立 Linear（两次 matmul）+ F.silu + 逐元素 mul，四个 kernel、多个中间张量（gate、up、act、inter）。右侧 fused_mlp_forward 的思路是：先 cat 权重再单次 matmul（:21-22，把两次 Linear 合一次）、再用 npu_swiglu(.., dim=-1)（:24）把 silu+mul 融成单个 CANN 算子。数学上等价（cat 权重沿 dim=0 后单 matmul = 原两次 matmul 拼接），省的是一次 matmul launch + silu/mul 中间张量。但重点：右侧这段没被接入（全仓零调用，见讲解），它只是正确的写法样板；真正生效的 swiglu 融合在 MoE 专家里是直接内联调 torch_npu.npu_swiglu。NVIDIA 侧若想等价融合，得用 Megatron 的 ColumnParallelLinear + fused SiLU kernel，或 torch.compile；HF eager 就是左侧这样分开算。两侧对照凸显：本 wrapper 设计意图正确、但工程上未连线，别高估其运行时作用。

差异讲解

把三座桥在 NVIDIA DDP 与 Ascend FSDP2 两条路线上的落点并排看：第一座桥（HF backbone 复用）两边完全一致，都 transformers.Qwen3Model(config)——这是 MindSpeed 能跟 HF 上游升级的根基。

第二座桥差异最大：NVIDIA 侧融合算子靠 torch.compile / TransformerEngine / FlashAttention 自动融合或库内置，不 monkey-patch HF 源符号；Ascend 侧因 CANN 内核在 torch_npu、HF 上游不感知 NPU，只能用 MindSpeedPatchesManager.register_patch 按“符号全名”运行期替换 HF 的 Qwen3RMSNorm.forward / apply_rotary_pos_emb。两侧都数学等价，但替换机制不同：NVIDIA 是“库 + 编译器自动”，Ascend 是“运行期 monkey-patch + 开关保护”。use_fused_* 开关关掉即回 HF 原生，这是跨厂商数值对拍的逃生口。

第三座桥差异本质：NVIDIA DDP 是梯度后 all-reduce、参数不分片；Ascend FSDP2 是参数按 DTensor 分片、前向 unshard + 后向 reshard + 梯度 reduce。但融合算子在这两套里都不感知分布式——:9 的 self.weight 在 DDP 下是全量 Tensor、在 FSDP2 下是本地分片视图，npu_rms_norm 都只跑本地数据。所以融合算子层与 FSDP2 层正交可叠加，EP12 下“全卡 patch norm/rope + 专家走 grouped_matmul + 全卡 fully_shard”三件套同时生效，互不踩脚。
