行 1-1 · env_config.sh：Ascend 适配的真正藏身处

NPU / FSDP2（本项目 :1-:1）
source examples/fsdp2/env_config.sh
GPU / DDP（你熟悉）
# GPU DDP 脚本典型开头
source env.sh  # 注入 CUDA_HOME / NCCL_DEBUG / PATH
# 或：module load cuda/12.2 nccl/2.18
差异讲解

差异在“注入什么”与“由谁管”。GPU DDP 习惯自己 source env.sh 或 module load cuda nccl，注入的是 CUDA/NCCL；Ascend 这边注入的是 CANN/HCCL，且 env_config.sh 往往还得调 source set_env.sh 拉起 CANN 工具链。GPU 侧的 NCCL_DEBUG 由 env 控制，Ascend 侧的 HCCL 通信参数同样靠 env（如 HCCL_TIMEOUT）。另一差异：GPU 脚本常用 module load 这种已标准化的工具切换版本，Ascend CANN 的版本切换常落在 env_config.sh 内部 source CANN path，脚本本身更“裸”。

GPU / DDP（你熟悉）
# GPU DDP 等价
GPUS_PER_NODE=8
MASTER_ADDR=localhost
MASTER_PORT=6499
NNODES=1
NODE_RANK=0
WORLD_SIZE=$(($GPUS_PER_NODE*$NNODES))
差异讲解

差异只有一处但致命：变量名 NPUS_PER_NODE vs GPUS_PER_NODE。其余拓扑量（MASTER_ADDR/PORT、NNODES、NODE_RANK、WORLD_SIZE 算式）与 GPU DDP 逐字一致——因为 torchrun 的 rendezvous 协议是硬件无关的，进程组协商不区分后端。真正的硬件差异在通信库：GPU 走 NCCL（由 torchrun 默认 backend=gloo+nccl），Ascend 走 HCCL（由 env_config 注入 torch_npu 后端插件），但这一层在本段拓扑变量里看不见，被推迟到 :20 的 torchrun 实际拉起进程后由后端插件接管。

差异讲解

差异：几乎为零。torchrun 五元组是硬件无关协议，GPU/NPU 逐字相同。唯一要注意的是“看见 torchrun 不等于走 NCCL”：GPU 这边 torchrun 默认拉起后用 NCCL 做 allreduce；Ascend 这边同样 torchrun 拉起，但进程组 backend 由 env_config 把 torch_npu 注册的 hccl backend 接入，allreduce 实际走 HCCL。所以从命令行看不出硬件差异，差异全压在 env_config.sh 和后端插件里——这是最易误导 DDP 老手的地方：以为改成 torchrun_npu 才对，其实不用，torchrun + 注入即可。

行 18-19 · 注释：CLI 覆盖 YAML 的优先级语义

# Commonly used parameters are passed as CLI args here; see companion YAML for full config.
# CLI args take precedence over the YAML when both are set. All args can also be moved into the YAML if preferred.

:18-19 两行注释明确了两条优先级语义，是本仓“壳与芯分离”设计的总纲：

  • :18 “Commonly used parameters are passed as CLI args here; see companion YAML for full config.”——常用参数走 CLI，完整配置看配套 YAML（pretrain_qwen3_8b_4k_fsdp2_A2.yaml）。即本脚本只暴露“会常调的旋钮”（fsdp_size、batch_size 等），YAML 承载“不常动的全量”。
  • :19 “CLI args take precedence over the YAML when both are set. All args can also be moved into the YAML if preferred.”——CLI 与 YAML 同名时CLI 整体覆盖，不是合并；且所有 CLI 参数都可搬进 YAML。

对 DDP 老手的提醒：这不是 argparse 默认值那种“CLI 给了就用 CLI、没给用默认”的简单覆盖——YAML 里写了 per_device_train_batch_size: 4，CLI 给 --training.per_device_train_batch_size 1（:26），最终是 1 不是 4+1。点号路径 --training.xxx 对应 YAML 的嵌套 section（training: 下），是“扁平 CLI 映射嵌套 YAML”的约定。

python train.py config.json \

    --dataset_path /data/train.parquet
# 多进程对照（GPU torchrun + NCCL）

差异讲解

三处差异。(1) 入口二进制：GPU DDP 常直接 torchrun ... train.py 或 HF 的 accelerate launch，本仓用自造的 train_fsdp2.py（非 train.py），它内部封装了 FSDP2 的 mesh 构建、DTensor 分片、HCCL backend 注入——GPU 侧这些靠 accelerate 或 transformers Trainer 自动处理，本仓显式化。(2) 配置载体：GPU 侧常见 config.json 或 argparse，本仓用 YAML + 点号 CLI 覆盖（--training.xxx 映射嵌套 section），且明确声明 CLI>YAML。(3) 数据参数：GPU 侧常传裸路径 --dataset_path，本仓传 JSON 字符串 --data.dataset '{"file_name":...}'，预留多字段扩展（如可加 split/num_samples）。单进程 python train.py 在本仓不存在——FSDP2 必须 torchrun 多进程，因为 DTensor/mesh 依赖进程组。

行 23-25 · --parallel.fsdp_size 8 + ep_size/ep_fsdp_size 1：分片组宽度与关闭 EP

NPU / FSDP2（本项目 :23-:25）
    --parallel.fsdp_size 8 \
    --parallel.ep_size 1 \
    --parallel.ep_fsdp_size 1 \
GPU / DDP（你熟悉）
# GPU DDP 对照：无 fsdp_size 概念
# DDP：每卡持完整参数副本，allreduce 梯度
torchrun --nproc_per_node=8 train.py \
    --ddp_find_unused_parameters False
# GPU FSDP1（无 fsdp_size，默认全分片）
torchrun --nproc_per_node=8 train.py \
    --fsdp_transformer_layer_cls_to_wrap Qwen3DecoderLayer \
    --use_fsdp True
差异讲解

差异有三。(1) fsdp_size 概念在 DDP 不存在：DDP 每卡完整副本，没有“分片组宽度”这回事；到 FSDP1 才引入 sharding，但 FSDP1 没有 fsdp_size 这个旋钮（要么全分片要么不分片），只有 FSDP2/HSDP 才暴露 fsdp_size 让你做“组间复制+组内分片”的混合。本仓的 fsdp_size 正是 FSDP2 DTensor mesh 的分片维度。(2) 通信原语：DDP 走 allreduce 梯度（NCCL allreduce），FSDP 走 all-gather 参数 + reduce-scatter 梯度，通信量与激活峰值都不同——8B 模型 DDP 每卡 16GB 权重副本根本放不下，所以必须 FSDP。(3) EP 旋钮：GPU 侧 FSDP1 不带 ep_size/ep_fsdp_size，专家并行靠 Megatron-Deepspeed 或 vLLM 风格手动切；本仓把 EP 也收进同一套 --parallel.* 命名空间，dense 时全置 1 关掉，MoE 时再开，统一在 FSDP2 DTensor mesh 上表达。

差异很小但值得点出。(1) 命名前缀：本仓 --training.per_device_train_batch_size 带 training. 前缀映射 YAML 嵌套 section，HF Trainer 是裸 --per_device_train_batch_size 无前缀——这是“扁平 CLI 映射嵌套 YAML”约定 vs “argparse 扁平”约定的区别。(2) dtype：GPU 脚本通常显式 --bf16 True，本 A2 脚本里没有 dtype 旋钮——它落在 env_config 或 train_fsdp2.py 内部默认（Ascend 侧 bf16/fp16 由 NPU 配置决定），不像 HF Trainer 暴露成 CLI。(3) 全局 batch 推导相同：per_device × grad_accum × world_size，FSDP2 与 DDP 在这层语义一致，差异只在 fsdp_size 影响的是参数分片而非 batch。

NPU / FSDP2（本项目 :18-:28）

    --data.dataset '{"file_name": "your origin data path.example: /home/train-00000-of-a09b74b3ef9c3b56.parquet"}' \
    --parallel.fsdp_size 8 \
    --parallel.ep_size 1 \
    --parallel.ep_fsdp_size 1 \
    --training.per_device_train_batch_size 1 \
    --training.gradient_accumulation_steps 1 \
    --training.output_dir ./output
GPU / DDP（你熟悉）
# GPU 侧对照：recompute 与 fused 的等价旋钮
# A2 风格（显存紧）
torchrun --nproc_per_node=8 train.py \
    --gradient_checkpointing True       # = recompute True
# A3 风格（显存宽）
torchrun --nproc_per_node=16 train.py \
    --gradient_checkpointing False \
    --attn_implementation flash_attention_2 \   # = use_flash_attn True
    # fused rmsnorm/rotary 在 GPU 侧通常无独立旋钮，靠 kernel 内部融合
差异讲解

差异与对照。(1) recompute 命名：GPU/HF 侧叫 gradient_checkpointing，本仓叫 recompute + recompute_modules（按模块点名而非整模型开关），粒度更细——HF 只能整模型 on/off，本仓可只对 model.layers.{*} 重算而留 embed/lm_head 不重算。(2) fused 算子：GPU 侧 flash attention 有 --attn_implementation flash_attention_2 标准旋钮，但 fused rmsnorm/rotary 通常无独立 CLI，靠 kernel 实现（如 Apex FusedRMSNorm）内部融合或编译期决定；本仓把这三个都暴露成独立 --optimization.use_fused_* 开关，因为 Ascend NPU 的算子库融合需显式 opt-in，不像 CUDA kernel 默认就融。(3) HCCL 零拷贝：GPU 侧无等价旋钮——NCCL 本就是 device 直通无 host 拷贝；Ascend HCCL 有零拷贝开关是因为历史上有 host 中转路径，A3 开它对齐 NCCL 的直通语义。这正体现“Ascend 适配层把 GPU 默认就有的特性变成显式 env”的通则。
