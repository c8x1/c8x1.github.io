行 1-3 · model 段：信任边界与加载来源

model:
  trust_remote_code: False
  train_from_scratch: False

整个 YAML 的第一段只有两行有效配置，但它们划定了 MindSpeed-LLM 装载模型时的信任边界与权重来源，是 EP06「拿什么模型上车」这一场景的入口。

:2 trust_remote_code: False —— 不允许执行远端自定义代码。在 DDP/NVIDIA 语境里这相当于拒绝 transformers 的 AutoModel.from_pretrained(..., trust_remote_code=True) 那条会拉取并执行仓库里 modeling_*.py 的路径。这里关掉它，意味着 Qwen3 的权重走的是已注册到 transformers 库的标准实现（HuggingFace 已合入 Qwen3 模型代码），不再触发动态代码加载。Ascend 侧同理：MindSpeed-LLM 依赖 transformers 标准路径映射到自身的模型实现，关掉 trust_remote_code 减少了「同一段权重在不同卡上被两份不同 modeling 文件解释」的隐患。

:3 train_from_scratch: False —— 不从随机初始化训，而是加载预训练权重做继续预训练（与下方 stage: pt 呼应）。这决定了下面 data_manager_type: mg 用的是「预训练语料」而非「微调对话」，也决定了 lr: 1e-05 这种偏小、适合已收敛权重做二次续训的学习率。对 A2（两卡）这种小规模场景，续训比从零训现实得多——从零训 8B 在 2 张卡上根本走不动。

这一段看似无关并行，但它是后面 fsdp_modules 里 model.layers.{*} 这些参数路径名能成立的隐含前提：标准 transformers 实现里 Qwen3 的模块层级就叫 model.layers、model.embed_tokens、lm_head。若开了 trust_remote_code 走魔改 repo，路径名可能变，下面的 FSDP2 模块匹配就会失配。

行 5-11 · data 段：模板、长度与数据管理器类型

data:
  template: qwen3
  cutoff_len: 4096
  max_samples: 100000
  overwrite_cache: True
  preprocessing_num_workers: 1
  data_manager_type: mg

这一段决定「喂进网络的是什么形状的 token」。Qwen3 的对话/续训模板由 :6 template: qwen3 选定，MindSpeed-LLM 用它把原始文本包装成模型期望的 input_ids/labels 序列。

:7 cutoff_len: 4096 —— 单条样本截到 4096。注意文件名 pretrain_qwen3_8b_4k_fsdp2_A2 里的 4k 就是这里来的。4096 而非更长的 8192/32768，是出于A2 两卡的显存预算：FSDP2 虽然把参数分片摊薄到卡上，但激活值随 seq_len 线性甚至二次增长，4096 是「能跑得起来且不必开重计算也勉强」的甜点长度——不过本文件还是开了重计算（见 parallel 段 :23），说明 4096 + 重计算才稳。

:8 max_samples: 100000 / :9 overwrite_cache: True / :10 preprocessing_num_workers: 1 —— 三条都是数据预处理侧。100k 条样本对续训做个小规模实验足够；overwrite_cache: True 每次都重算预处理、不读旧缓存，在调试配置频繁变动时避免「吃到上一次跑的脏缓存」；preprocessing_num_workers: 1 则是有意保守——tokenize 用单进程，避免多进程抢占 CPU/内存，A2 这种小机子尤其要稳。

:11 data_manager_type: mg —— 这一行是数据段最关键的一行。mg 指 MindSpeed-LLM 自有的 megatron-style 数据管理器（对标 Megatron-LM 的预训练数据流：把文档打包成分桶、按 cutoff_len 切块、产出 (input_ids, labels) 的 binary stream），与 hf（HuggingFace datasets 管线，偏向指令微调/对话模板）相对应。此处用 mg 配 stage: pt，形成预训练语料 → mg 管线 → 4096 长度的 token 块的一致组合。对习惯 DDP + HF Trainer 的读者：HF Trainer 默认走的是 datasets + data_collator，而 mg 更接近 Megatron 的 data_builder，无 collator、直接吐已切好块的张量——这也是 EP11「数据怎么进网」这一场景里需要点出的「不是 HF 那套」。

# FSDP1 (NVIDIA / torch.distributed.fsdp)
import torch.distributed.fsdp as fsdp
from torch.distributed.fsdp import FullyShardedDataParallel as FSDP
from torch.distributed.fsdp.wrap import size_based_auto_wrap_policy
import functools

model = AutoModelForCausalLM.from_pretrained("Qwen/Qwen3-8B")
# 整层包进 FlatParameter 大桶；按 size 阈值自动包
auto_wrap = functools.size_based_auto_wrap_policy(
    min_num_params=int(1e6))
model = FSDP(model,
    auto_wrap_policy=auto_wrap,
    sharding_strategy=fsdp.ShardingStrategy.FULL_SHARD,
    use_orig_params=True,  # 想做 per-parameter 才开，但语义别扭
)
# 没有 "ep_modules / ep_fsdp_modules / ep_dispatcher" 一等概念，
# MoE 的 EP 要靠 megablocks/DeepSpeed 另接一套。
差异讲解

分片单元不同：FSDP1 默认按「参数数量阈值」(size_based_auto_wrap_policy) 把模块整体打平成单个 FlatParameter，参数在桶里丢了原始身份，要做 per-parameter 优化器状态分离得特意开 use_orig_params=True，语义是补丁。FSDP2 则天生 per-parameter：像这里 fsdp_modules 按模块名列表精确指定，每个参数各自分片、各自 all-gather，优化器状态天然按参数分离，这是 FSDP2 相对 FSDP1 最大的设计反转。

EP 一等公民 vs 外挂：右侧 FSDP1 脚本里没有 ep_modules/ep_fsdp_modules/ep_dispatcher 这种概念——NVIDIA 侧要把 EP 叠在 FSDP 上，得请 DeepSpeed-MoE 或 megablocks 另起一套。而 MindSpeed-LLM 的 FSDP2 把 EP 钩子做成和 FSDP 同级的配置项（:18-:22），并且支持「FSDP 套 EP」的复合（ep_fsdp_modules 即「对专家再做 FSDP 分片」）。稠密模型这里 ep_size=1 让 EP 退化、不触发跨卡专家通信，但配置形态已为 MoE 备好。

recompute 的耦合位置：NVIDIA 侧重计算通常靠 CheckpointFunction 装饰或在 activation_checkpoint_apply_policy 里按模块名包，和 FSDP 是两条独立链路。这里 recompute_modules 用同样的模块名通配语法（model.layers.{*}），与 fsdp_modules 共用一套模块匹配机制，写法一致、心智负担低。

最后一段是训练超参，与上面 stage: pt + train_from_scratch: False 构成完整的「续训」配方。

:28 stage: pt —— 明确是 pre-training 阶段（续训）。这会触发 MindSpeed-LLM 的预训练 loss 路径（整序列因果 LM loss，不套指令模板、不算 prompt masking），与 data 段的 data_manager_type: mg 配对。

:29 dataloader_num_workers: 4 / :31 dataloader_drop_last: True —— 4 个数据加载 worker 喂两卡（每卡 2），A2 小机子够用；drop_last: True 丢掉凑不齐 batch 的尾巴，保证每个 step batch 形状一致——FSDP2 下这点尤其重要，形状不一致会触发反复 reshard、性能塌。

:32 optimizer: adamw + :33-:37 lr/weight_decay/beta/eps —— AdamW，lr: 1e-05 很小（续训已收敛权重，不能大步踹坏），adam_beta2: 0.95（比常用的 0.999 小，对语言模型续训更稳、方差估计更新更快），weight_decay: 0.01、adam_epsilon: 1e-08 是标准值。:38 max_grad_norm: 1.0 梯度裁剪。对 FSDP2 有个细节：梯度是分片后的梯度，裁剪前的 reduce-scatter 已发生，裁剪在分片粒度上做，行为与 DDP 全量梯度裁剪等价但实现路径不同。

:39 lr_scheduler_type: cosine + :40 warmup_ratio: 0.0 + :41 min_lr: 1e-06 —— 余弦退火，warmup_ratio: 0.0 意味着不做 warmup（续训权重已稳，直接从 1e-05 起跑），退到 min_lr: 1e-06。这与从零训（需要长 warmup 爬坡）形成鲜明对比，再次印证 train_from_scratch: False。

:42 num_train_epochs: 3.0 / :43 max_steps: 2000 / :44 save_steps: 500 / :45 logging_steps: 1 —— max_steps: 2000 会先于 3 epochs 触发结束（A2 两卡 100k 样本、小 batch，3 个 epoch 步数远超 2000），所以实际跑 2000 步；每 500 步存一次 checkpoint（共存 4 次），每步都记 loss。这是 EP08「跑多少、存多少、看什么」这一场景的具体落地：A2 小规模快速验证续训是否收敛，靠每步 loss + 每 500 步的 ckpoint 判断。

把这段放回三座桥：它桥接了 EP06（模型上车：续训权重 + 小 lr）、EP08（训练过程：cosine 无 warmup、2000 步快验证），并为 EP11（数据→mg→4k 块进网）和 EP14（EP 钩子虽配好但稠密 ep_size=1 退化）提供了「在 A2 两卡上真能跑完」的超参保障。整份 YAML 因此是「FSDP2 per-parameter + EP 留钩 + recompute 压激活 + mg 预训练管线」的一套自洽最小配方。
