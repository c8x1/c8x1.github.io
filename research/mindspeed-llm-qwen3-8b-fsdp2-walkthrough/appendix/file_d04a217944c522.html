行 1-1 · 文件头注释与 fsdp2 后端声明

NPU / FSDP2（本项目 :1-:1）
# Common environment variables for the fsdp2 backend
GPU / DDP（你熟悉）
# NVIDIA/CUDA 侧通常写在 launch 脚本前导:
export CUDA_VISIBLE_DEVICES=0,1,2,3,4,5,6,7
export NCCL_DEBUG=INFO
export NCCL_IB_DISABLE=1
export NCCL_NET_GDR_LEVEL=2
export NCCL_P2P_DISABLE=0
export NCCL_SOCKET_IFNAME=eth0
差异讲解

差异：变量命名空间完全不同。NVIDIA 用 CUDA_*/NCCL_* 前缀；Ascend 用 HCCL_*/NPU_*/TASK_QUEUE/CPU_AFFINITY/MULTI_STREAM 等无统一前缀、按子系统命名的变量。NCCL_* 控集合通信库（对应 Ascend 的 HCCL_*）；CUDA_VISIBLE_DEVICES 选卡（对应 Ascend 的 ASCEND_RT_VISIBLE_DEVICES，本文件未出现，可能在上层脚本）。NCCL_DEBUG 调日志（对应 HCCL_CONNECT_TIMEOUT 之外的 HCCL_*_LOG_LEVEL）。关键认知：NPU 侧没有 NCCL_*，全部走 HCCL/CANN runtime，所以从 NVIDIA 迁移过来时不能简单替换前缀，得逐个对照功能。

行 2-2 · TRAINING_BACKEND=mindspeed_fsdp —— 选择 MindSpeed FSDP 后端

NPU / FSDP2（本项目 :2-:2）
export TRAINING_BACKEND=mindspeed_fsdp
GPU / DDP（你熟悉）
# NVIDIA 侧选后端的等价方式(无单一 env 变量,
# 通常通过 torchrun + 训练脚本参数):
# DDP:
torchrun --nproc_per_node=8 train.py --backend ddp
# FSDP2(PyTorch>=2.4 原生):
#   from torch.distributed.fsdp import FullyShardedDataParallel as FSDP
#   from torch.distributed.fsdp import ShardingStrategy
#   model=FSDP(model, sharding_strategy=ShardingStrategy.FULL_SHARD)
差异讲解

差异：选后端的机制不同。NVIDIA/PyTorch 原生没有 TRAINING_BACKEND 这种单变量开关，是在 Python 代码里直接 FSDP(model,...) 或传 --strategy 参数；MindSpeed 把它提升为 env 变量，框架层据此分发到不同 wrapper。Ascend 侧的 mindspeed_fsdp 不是裸 PyTorch FSDP，而是经过 Ascend NPU 适配的版本（HCCL 集合通信、NPU 内存语义、可能针对昇腾 NPU 调了 sharding 粒度）。等价关系：TRAINING_BACKEND=mindspeed_fsdp ≈ NVIDIA 侧 FSDP(model, FULL_SHARD)，只是前者走 HCCL 后端通信。

行 3-3 · HCCL_CONNECT_TIMEOUT=1800 —— HCCL 建连超时（NCCL 等价物）

NPU / FSDP2（本项目 :3-:3）
export HCCL_CONNECT_TIMEOUT=1800
GPU / DDP（你熟悉）
# NVIDIA/NCCL 等价物:
export NCCL_TIMEOUT=1800
# 或在 PyTorch 代码里(秒):
#   dist.init_process_group(
#     backend='nccl',
#     timeout=datetime.timedelta(seconds=1800))
差异讲解

差异：库名与超时单位需核对。NVIDIA 用 NCCL_TIMEOUT（或 dist.init_process_group(timeout=...)）控制 NCCL 操作超时；Ascend 用 HCCL_CONNECT_TIMEOUT，名称里多了 CONNECT 暗示主要管建连阶段（而非每次集合操作的完成超时）。单位需核实：NCCL NCCL_TIMEOUT 秒级；HCCL 文档对 HCCL_CONNECT_TIMEOUT 的单位（秒 vs 毫秒）在不同版本表述不一，以 CANN 文档为准。这里 1800 假定是秒即 30 分钟，是常见的「大集群建连留余量」设定。

行 4-4 · TASK_QUEUE_ENABLE=2 —— NPU 任务队列（Ascend 专属，无 NCCL 对应）

NPU / FSDP2（本项目 :4-:4）
export TASK_QUEUE_ENABLE=2
GPU / DDP（你熟悉）
# NVIDIA 侧无 env 变量对应。
# 多流重叠靠显式 CUDA stream 管理:
#   s_compute=torch.cuda.Stream()
#   s_comm=torch.cuda.Stream()
#   with torch.cuda.stream(s_compute):
#     forward()
#   with torch.cuda.stream(s_comm):
#     all_gather()
差异讲解

差异：抽象层级不同。NVIDIA 把多流管理交给应用代码（torch.cuda.Stream() + with torch.cuda.stream(...)，或 FSDP 内部自管 compute_stream/comm_stream）；Ascend 则在 CANN runtime 层提供全局 TASK_QUEUE_ENABLE 开关，开启后 runtime 自动做算子排队/重排。取值语义：=2 具体档位含义（是否启用某级多流/异步重排）需以 CANN 文档为准——不同 CANN 版本枚举值可能调整。这是 NVIDIA→Ascend 迁移时最容易踩坑的「无对应物」变量。

行 5-5 · CPU_AFFINITY_CONF=1 —— CPU 亲和性绑定

NPU / FSDP2（本项目 :5-:5）
export CPU_AFFINITY_CONF=1
GPU / DDP（你熟悉）
# NVIDIA 侧等价(numactl/taskset 显式绑核):
numactl --cpunodebind=0 --membind=0 torchrun ...
# 或进程内:
#   os.sched_setaffinity(0, {0,1,2,...,15})
# PyTorch DataLoader 也支持 setaffinity:
差异讲解

差异：绑核主体不同。NVIDIA 侧通常用 numactl --cpunodebind=/taskset -c 在进程启动前由用户显式绑核，或在代码里 os.sched_setaffinity；Ascend 把它纳入 env 变量 CPU_AFFINITY_CONF，由框架/runtime 据此自动绑。值语义：=1 是启用档，具体绑哪些核（按 rank 等分？按 NUMA？）需以 CANN/MindSpeed 文档为准。对 NVIDIA 用户这是个「可选手动优化」；在 NPU 侧 MindSpeed 把它默认开了，因为 NPU 的 H2D 路径更依赖 CPU 稳定供给。

行 6-6 · MULTI_STREAM_MEMORY_REUSE=2 —— 多流显存复用

NPU / FSDP2（本项目 :6-:6）
export MULTI_STREAM_MEMORY_REUSE=2
GPU / DDP（你熟悉）
# NVIDIA 侧等价:PyTorch CUDA caching allocator 自动做,
# 显式控制靠以下 env:
export PYTORCH_CUDA_ALLOC_CONF=expandable_segments:True
export PYTORCH_CUDA_ALLOC_CONF=max_split_size_mb:512
差异讲解

差异：allocator 抽象层级与作用对象不同。NVIDIA 用 PYTORCH_CUDA_ALLOC_CONF 控制 PyTorch 层的 caching allocator（段扩展、分块阈值等），allocator 在单 device 多流间自动协调显存；Ascend 的 MULTI_STREAM_MEMORY_REUSE 作用在 CANN runtime/NPU driver 层，是更底层的多流显存策略开关。二者互补而非替代——下一行 PYTORCH_NPU_ALLOC_CONF 才是 PyTorch-NPU 层的 allocator 配置，与 NVIDIA 的 PYTORCH_CUDA_ALLOC_CONF 对应。值 =2 的档位语义以 CANN 文档为准。

行 7-7 · PYTORCH_NPU_ALLOC_CONF=expandable_segments:True —— NPU allocator 段扩展

NPU / FSDP2（本项目 :7-:7）
export PYTORCH_NPU_ALLOC_CONF=expandable_segments:True
GPU / DDP（你熟悉）
# NVIDIA 等价(几乎逐字对应):
export PYTORCH_CUDA_ALLOC_CONF=expandable_segments:True
差异讲解

差异：仅命名空间替换，语义几乎一致。PYTORCH_NPU_ALLOC_CONF ↔ PYTORCH_CUDA_ALLOC_CONF，键 expandable_segments:True 完全相同。这是本文件里最「干净」的 NVIDIA↔NPU 对照——说明 MindSpeed-LLM 团队尽量保持 PyTorch 原生 allocator 配置语义不变，只把 device 后端从 CUDA 切到 NPU。一个注意点：NPU allocator 实现底层走 CANN 的显存 API（aclrtMalloc 等），「expandable」在昇腾上的具体行为（段扩展步长、是否真的能归还给 driver）与 NVIDIA caching allocator 不完全等价，以 PyTorch-NPU/CANN 文档为准，但配置键名一致这点让迁移成本很低。

行 8-8 · TORCH_COMPILE_DEBUG=0 —— torch.compile 调试日志关闭

NPU / FSDP2（本项目 :8-:8）
export TORCH_COMPILE_DEBUG=0
GPU / DDP（你熟悉）
# NVIDIA/NPU 通用,完全一致:
export TORCH_COMPILE_DEBUG=0
# 或 Python 代码内:
#   import torch._dynamo
#   torch._dynamo.config.debug=False
差异讲解

差异：无差异。TORCH_COMPILE_DEBUG 是 PyTorch 原生 env，device 无关，NVIDIA 和 Ascend 行为一致。值得标注的是：编译产物不同——NVIDIA 侧 inductor 生成 Triton/CUDA kernel 走 GPU；Ascend 侧 inductor 经 torch_npu 生成 NPU 算子（可能落到 CANN op 或 Triton-like NPU IR）。语义对齐但目标指令集不同，这是「软件栈桥」的体现：同一 PyTorch 编译栈，输出到不同硬件后端。迁移时此变量直接照搬。

行 9-9 · CUBLAS_WORKSPACE_CONFIG=:4096:8 —— cuBLAS 工作区（核实：NPU 侧是否仍读）

NPU / FSDP2（本项目 :9-:9）
export CUBLAS_WORKSPACE_CONFIG=:4096:8
GPU / DDP（你熟悉）
# NVIDIA 原生:控制 cuBLAS 工作区,常配确定性模式:
export CUBLAS_WORKSPACE_CONFIG=:4096:8
# 配合:
#   torch.use_deterministic_algorithms(True)
#   或 export CUBLAS_WORKSPACE_CONFIG=:4096:8 +
#      export TORCH_USE_DETERMINISTIC_ALGORITHMS=1
差异讲解

差异：命名与作用对象存疑。NVIDIA 上 CUBLAS_WORKSPACE_CONFIG 明确作用于 cuBLAS（NVIDIA 的线性代数库）；Ascend 上线性代数走 aclblas（CANN 的 BLAS），通常不读 CUBLAS_* 变量。核实要点：在 NPU 侧需确认 (a) torch_npu 是否拦截并翻译此 env；(b) MindSpeed-LLM 是否有 env 适配层把它映射到 ASCEND_*_BLAS 或 ACL 配置；(c) 若无映射则此行为 no-op（无副作用但也没用）。结论以 CANN 文档/torch_npu 源码为准。这是「NPU 配置文件里残留 GPU 命名变量」的典型案例，迁移排查时建议特别留意。

差异：无差异。PYTHONPATH 是 Python 解释器原生 env，device 与加速器均无关，NVIDIA/NPU 行为完全一致。唯一注意点：$(pwd) 在 source 时展开，依赖 source 此脚本的工作目录正确——若从其他目录 source examples/fsdp2/env_config.sh，pwd 会是调用者所在目录而非 examples/fsdp2/，可能把错误目录加进 PYTHONPATH。NPU 侧因大量依赖 monkey-patch 模块，此变量失效的后果更严重（patch import 不到 → 训练走未适配的原生 PyTorch 路径 → NPU 上报错），排查 NPU 启动问题时值得先确认 pwd 正确。
