DAWZY — 可解释 AI 助手

arXiv: 2512.03289 会议: AI for Music Workshop 机构: San Diego State University 2025 年 12 月
↗ 查看原始论文
Human-in-the-Loop REAPER MCP Tools ReaScript 可解释 AI 语音控制
背景

背景：DAW 界面复杂性与创作心流的断裂

现代音乐制作以数字音频工作站（DAW）为核心，DAW 虽然将专业质量的创作民主化，却同时给用户带来了"选项过载"（option overload）的问题，常常打断创作心流。论文指出了一个核心矛盾：高层创作意图（如"让人声更温暖"）与底层 DAW 操作（如调整 EQ 频率、添加压缩器侧链、修改混响预延迟时间）之间存在巨大的认知鸿沟。

同时，现有的 AI 音乐生成器（如 Suno）通常是一次性生成（one-shot）模式 —— 用户输入一段描述，AI 直接输出完整的音轨，无法进行精细调整。这种模式将 AI 放在了创作的中心而非辅助的位置，限制了人类创作者的参与度和迭代优化的空间。

DAWZY 正是在这一背景下提出的。它的核心定位是：让 DAW 始终作为创作中心，AI 只是辅助工具，而非替代品。DAWZY 通过自然语言（文本/语音/哼唱）接收用户的创作意图，将其转化为 REAPER 中可执行、可撤销的操作，同时保持透明的解释和完整的人类控制权。

原理

核心原理：可解释 AI 在 DAW 控制中的应用

DAWZY 的核心设计原则可以用三个关键词概括：

1. Context Grounding（上下文接地）

与"盲目执行"的 AI 不同，DAWZY 在执行任何操作之前，会先通过 MCP 工具刷新并读取 REAPER 的当前工程状态（轨道列表、Clip 内容、FX 参数、路由配置等）。这意味着 LLM 的每次代码生成都是在理解当前上下文的基础上进行的，大幅降低了"幻觉"（hallucination）的风险。

2. Explain-as-you-go（解释即行动）

DAWZY 的每次编辑操作都伴随自然语言解释。用户不仅知道"做了什么"，还能理解"为什么这样做"。这一设计有两个目的：一是增强可审计性（auditability），用户可以审查 AI 的决策；二是支持学习，新手用户可以通过 AI 的解释学习 DAW 操作的专业知识。

3. Atomic Scripts + Undo（原子脚本与可撤销性）

DAWZY 生成的每段 Lua ReaScript 都是原子化的（atomic），即一个独立的、可独立撤销的操作单元。这确保了用户可以逐步回退 AI 的每一个修改，保持了完整的创作控制权。这与 Ableton MCP 方案形成鲜明对比 —— 论文作者特别指出，在测试中 Ableton-MCP 在修改音符和音量时受到 API 限制，甚至出现错误声称操作成功的情况。

核心创新

DAWZY 使用 LLM 代码生成（而非预定义 tool）作为主要的 DAW 交互方式。LLM 不是调用固定的 API 函数，而是直接生成 Lua ReaScript 代码，由 REAPER 的 ReaPy 框架执行。这种方式的优势是极大的灵活性 —— LLM 可以生成任意复杂的操作序列，而不受预定义 tool 的限制。

DAWZY 的架构由三个层次组成：

用户交互层（User Interaction Layer）

由一个基于 Electron.js 的轻量 GUI 应用承载，提供三种输入方式：

  • 文本输入 —— 用户在聊天框中输入自然语言指令
  • 语音输入 —— 通过 OpenAI Whisper 进行语音转文字，实现免手操作
  • 哼唱输入 —— "录制哼唱" 按钮捕获旋律片段，通过 Spotify BasicPitch 模型将哼唱转换为 MIDI，自动导入为新的 REAPER 轨道

界面设计强调极简 GUI（minimal-GUI）和语音优先（voice-first）原则，仅提供"开始"、"停止"、"录制"、"撤销"等核心按钮，其余操作全部通过自然语言完成。

处理层（Processing Layer）

这是 DAWZY 的智能核心，包含以下组件：

  • LLM（GPT-5） —— 负责理解用户意图、调用 MCP 工具、生成 Lua ReaScript 代码。论文明确指出，开源模型（如 Qwen3-Coder-480B）在 ReaScript 代码生成中表现不佳，频繁产生无效索引和幻觉函数，因此选用了 GPT-5。
  • MCP Tool: State Query —— 枚举 REAPER 工程中的轨道、Clip、FX 和路由信息，为 LLM 提供当前工程状态上下文
  • MCP Tool: FX Parameterization (fxparam) —— 将人类可读的单位（dB、ms）转换为 ReaScript 的滑块范围（0-1、0-4 等），解决 LLM 无法可靠完成单位转换的问题
  • MCP Tool: Beat Generation —— 在本地运行 Meta 的 MusicGen-small (300M) 模型，根据文本描述生成音频波形

执行层（Execution Layer）

负责在 REAPER 中安全、可逆地执行操作：

  • ReaScript 执行 —— GPT-5 生成的 Lua 代码通过 ReaPy 框架在 REAPER 中执行
  • 工具脚本 —— 专用脚本处理 FX 参数更新、工程状态摘要、音频/MIDI 导入等操作
Human-in-the-Loop

Human-in-the-Loop 设计理念

DAWZY 的论文标题中明确标注了 "Human in the Loop"，这一设计理念贯穿了整个系统：

透明性

用户可以看到 AI 生成的每一段 Lua 代码，理解 AI 正在执行的具体操作。这比"黑盒式"的 AI 音乐生成器提供了更强的可审计性和信任感。

可逆性

每个操作都是原子的、可撤销的。用户可以逐步回退 AI 的修改，甚至在 AI 执行后手动微调结果。DAW 保持为创作的主导环境，AI 只是辅助工具。

可修改性

由于 AI 生成的是代码而非直接修改音频，用户可以在代码执行前审查和修改。论文作者认为这种透明度是 DAWZY 相对于其他方案的关键优势。

教育性

AI 的自然语言解释帮助用户学习 DAW 操作。例如，当用户询问"第二条轨道的压缩器 Attack 参数做什么？"时，AI 不仅解释 Attack 的概念，还会结合当前工程的具体参数值给出实际建议。

实验

实验结果与用户研究

客观评估

研究团队设计了四项可复现的测试任务：

  1. 多指令 FX 处理 —— "将第一条轨道音量翻倍、增加 decay、设置 attack 为 10ms"
  2. GUI 导航 —— "打开第一条轨道的 FX 浏览器"
  3. 工作流自动化 —— "复制第一条轨道、升高一个八度、以 20% 混合"
  4. 教育交互 —— "第二条轨道压缩器的 attack time 是做什么的？"

每个任务执行 3 次试验，对比了 4 个 LLM：GPT-5、QWEN-480B、GPT-OSS-120B、GPT-OSS-20B。结果显示，开源基线模型仅达到 25-50% 的成功率，主要失败原因是幻觉产生无效的 ReaScript 函数和错误的参数索引。不过所有模型在教育任务上都通过了全部 3 次试验。

与 Ableton-MCP 的对比

在三个定性任务（Wavy、Bouncy、Fade）中，DAWZY 和 Ableton-MCP 均使用 Claude Sonnet 4.5 进行测试。结果：

  • DAWZY 在 Fade 任务中遇到 3 次执行错误，其余均成功
  • Ableton-MCP 在 Wavy 和 Fade 任务上失败（API 限制导致无法修改音符和音量）
  • Ableton-MCP 在 Bouncy 任务上错误地声称成功，或创建了新的片段而非修改现有片段

论文分析 DAWZY 性能更优的原因：REAPER 的 ReaScript 文档公开且充分出现在 LLM 的训练数据中，而 Ableton Live 的内部 API 较新且在训练数据中覆盖不足。

主观评估（MOS 测试）

21 名参与者对 DAWZY 进行了 Mean Opinion Score (MOS) 评分（满分 5 分）：

评估维度 评分 评价
Enjoyment（趣味性） 4.48 最高分，用户认为使用体验有趣
Learning（学习效果） 4.38 解释功能有效帮助用户学习
Collaboration（协作感） 4.29 用户感受到与 AI 的良好协作
Usability（可用性） 4.14 整体可用性获得认可
Control（控制感） 3.81 最低分，但仍在中立线以上

所有维度均高于中立值（3.0），其中"趣味性"和"学习效果"得分最高。Control 维度得分相对较低，反映出用户对 AI 自动操作仍有一定的不确定感，这也是未来需要改进的方向。

核心差异总结

DAWZY 的核心创新在于使用 LLM 代码生成而非预定义工具作为主要的 DAW 交互方式。这意味着 LLM 不受预定义 API 的限制，可以生成任意复杂的操作序列。但代价是对 LLM 能力的依赖更高 —— 论文明确指出开源模型在 ReaScript 代码生成中表现不佳。相比之下，MCP 方案通过预定义 tools 限制了 LLM 的操作范围，虽然灵活性较低，但对 LLM 的要求也更低，更容易使用不同模型运行。

展望

未来展望与局限性

论文诚实地指出了当前的限制和未来方向：

  • DAW 特定脚本语言 —— 当前仅支持 REAPER 的 Lua ReaScript，需要为每个 DAW 开发特定的适配层
  • LLM 依赖 —— 目前依赖 GPT-5，论文计划微调开源 LLM 以提升 ReaScript 代码生成的可靠性
  • 功能范围 —— 当前原型聚焦于核心功能，尚未覆盖高级插件特性
  • 跨 DAW 支持 —— 计划支持更多可脚本化的 DAW（论文提及 Ableton Live 12 扩展的脚本能力）
  • 定性用户研究 —— 计划开展更深入的定性研究，理解用户在实际创作流程中的体验

论文最后向社区发出邀请："我们邀请社区尝试我们的 demo，并在这一自然语言创意软件控制的开放框架上展开合作。"
