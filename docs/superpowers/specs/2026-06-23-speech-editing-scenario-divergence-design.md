# 语音编创场景发散→收敛设计（S9/S10 增量）

> 日期：2026-06-23
> 背景：在现有 8 场景（S1–S8）报告基础上，用户要求"再发散语音编创领域还有哪些场景和用户诉求"。经多轮 arXiv API 直查 + 反向 ID 验证，发散后收敛。
> 方法：候选场景 → arXiv API 直查（`--noproxy '*'` 直连，429 时 6–8s 间距）→ Atom XML 解析 → 反向 `id_list` 验证 title 匹配 → 只保留专门做该场景的论文 → 诚实判断是否属广义编辑。
> 关键决策：用户认可"广义编辑"把整段变换式（匿名化/构音障碍→清晰重建）也纳入"修/改/修音"范畴。

## 1. 发散过程与正交性发现

### 1.1 原 8 场景本身不正交
审后发现 S1–S8 混了两类东西：
- **场景/目标类**（改什么·为什么）：S1 改文本 / S2 修复 / S3 跨语言 / S4 改音色 / S5 改情感 / S6 改口音
- **横切维度类**（怎么改·什么约束）：S7 指令式（自然语言接口维度）/ S8 长时段（作用域/一致性维度）

S7/S8 跟 S1–S6 不在同一坐标轴上。本轮发散把这个问题放大。

### 1.2 候选分档
- 🔴 升新顶层场景候选：C1 实时 / C2 隐私匿名化 / C3 病理重建 / C4 会议多人 / C5 法证 / C6 档案
- 🟡 并入子情形候选：C7 呼吸流畅 / C8 时长对齐 / C9 年龄属性 / C10 转写修正 / C11 lip-resync / C12 游戏对话
- ⚫ 出范围：唱歌/声乐修音（音乐，同事负责）、纯 TTS 生成、SFX（待定归属）

## 2. 论文落地核实（核心门槛）

**原则**：原报告 S1–S8 每场景都有 API 核实过的 paper→问题映射；新场景必须同样落地，否则就是分类学想象，违反报告底线。

### 2.1 核实结果

| 候选 | arXiv 直查 + ID 反验 | 是否专门做该场景 | 是否广义编辑 | 结论 |
|---|---|---|---|---|
| **C2 隐私/匿名化** | ~30 篇核实 | ✅ VPC 生态专门做 | ✅ 整段去身份变换（广义编辑） | **升 S9** |
| **C3 病理/障碍重建** | ~14 篇核实 | ✅ DSR 专门做 | ✅ 病理→清晰转换（广义编辑） | **升 S10** |
| C4 会议/多人编辑 | ≈0 | ❌ 全是 diarization/分离/摘要 | — | 不加 |
| C5 法证增强 | 0 | ❌ 全是 deepfake/篡改检测 | — | 不加 |
| C6 档案修复 | 1 篇（phonograph 2606.18615） | ⚠ 太薄 | — | 不加（归横切附录） |
| C1 实时/低延迟 | 7 篇（streaming VC） | ⚠ 5/7 是流式 VC，3 篇与 S9 重叠 | ✅ | **归横切维度**，不当独立场景 |

### 2.2 关键洞察
- **PHONOS(2603.27001) 横跨 S6 口音与 S9 匿名化**——"中性化"同源，S9 回链 S6。
- **C5 法证是"检测 vs 增强"的范式分叉**：法证文献全是检测，"增强+不可伪造"是真空——这不是检索失败而是真空白。
- **实时/低延迟**本质是横切约束（cuts across VC/anonymization/accent），7 篇流式 VC 中 3 篇就是 S9 匿名化的流式版本——独立成场景会与 S9 重复计数。

## 3. 最终设计

### 3.1 新增场景卡片（S9 / S10，六块同 S1–S8 格式）

**S9 · 隐私/语音匿名化**
- 痛点与用户诉求：医疗/法律/客服通话录音匿名化、证人保护、数据合规脱敏；诉求=身份不可逆变换+保内容可懂+抗重识别（与 S4 娱乐换声目标相反：去身份 vs 保身份）
- 现状解决情况：~30 篇专门方法，VPC 挑战赛驱动；几乎全是整段匿名化变换（解耦 content/speaker/prosody → 换 speaker embedding → 重合成）；唯一沾"局部编辑"的是可逆对抗扰动/选定词遮蔽
- paper→问题映射（代表 8–10 篇核实 ID）：Zero-VC(2606.20218)、DDPO-VC(2606.15313)、DiffAnon(2604.26281)、Anonymization-Not-Elimination(2604.17000)、PHONOS(2603.27001)、StreamVoiceAnon(2601.13948)、DarkStream(2509.04667)、Content-Anonymization-Long-form(2510.12780)、EASY(2505.15004)、Private-kNN-VC(2505.17584)
- 技术点：content/speaker/prosody 解耦·pseudo-speaker 映射·流式零前瞻·情感保持·抗重识别
- 成熟度与缺口：🟢 活跃成熟（VPC 生态）；缺口=外科式编辑既有语音（现有皆整段变换）、抗强攻击者重识别仍弱
- 落地情况：研究为主，产品少（合规驱动蓝海）；VPC 挑战赛是事实评测基准；PHONOS 同时属 S6（中性化同源）
- ⚠ scope 注：广义编辑纳入整段变换；标明属"变换式编辑"非"外科编辑"

**S10 · 病理/障碍语音重建**
- 痛点与用户诉求：ALS/帕金森/喉切除/构音障碍患者——把受损语音编辑/恢复成清晰语音，或预先 bank 本人音色；诉求=保本人身份+从劣化语音恢复清晰；社会价值高
- 现状解决情况：~14 篇专门方法，以 DSR（Dysarthric Speech Reconstruction）为核心；绝大多数是整段"病理→清晰"转换，非外科编辑；唯一沾编辑的是 AP-GRPO（保留锚词、只重建锚间损坏段）；voice banking 仅以个性化 TTS/声音克隆形态出现，无专门 ALS/喉切除 voice-banking 生成论文
- paper→问题映射（代表 6–8 篇核实 ID）：UNIT-DSR(2401.14664)、DiffDSR(2506.00350)、CFM-DSR(2506.16127)、ClaritySpeech(2507.09282)、ChiReSSD(2509.19231)、AP-GRPO(2606.15540)、EL-Speech-Enhancement(2606.01905)、Personalized-TTS-Dysarthric(2508.10412)
- 技术点：content-speaker 解耦·speech-unit normalization·flow-matching 重建·锚词保留局部重建·个性化 TTS 声音克隆
- 成熟度与缺口：🟡 DSR 转换较活跃；缺口=外科式编辑病理语音（AP-GRPO 是唯一沾边）、ALS/喉切除 voice-banking 生成空白
- 落地情况：ModelTalker/VocaliD/Acapela My-Own-Voice（voice banking 产品）；DSR 编辑侧薄；数据集 TORGO/UASpeech/SAP/Project Euphonia(2409.09190)
- ⚠ scope 注：广义编辑纳入整段转换；标明属"转换式编辑"非"外科编辑"
